面试官问“为什么这样设计、遇到什么边界问题”，本质上不是让你背源码，而是判断你能否从需求、约束、方案权衡、异常场景和验证方式完整思考系统。

建议每个技术点都按照这套结构回答：

```
业务问题 → 设计目标 → 可选方案 → 为什么选当前方案
→ 核心实现 → 边界问题 → 解决办法 → 如何验证
```

**一、通用考虑维度**

分析任何模块时，都可以从以下方面展开：

1. **业务需求**：这个功能解决什么问题？谁使用？不做会怎样？
2. **数据来源**：数据来自前端配置、后端接口，还是本地缓存？
3. **状态变化**：初始化、更新、刷新、退出登录时，状态如何变化？
4. **执行时序**：接口、路由、组件挂载之间谁先谁后？
5. **异常处理**：接口失败、数据为空、重复操作、权限不足怎么办？
6. **一致性**：后端权限、前端路由、菜单和按钮状态是否一致？
7. **性能**：是否存在重复请求、重复渲染、内存泄漏、大包加载？
8. **安全性**：哪些校验只能在后端完成？前端能做什么、不能做什么？
9. **可维护性**：新增模块是否需要修改很多文件？是否存在重复代码？
10. **验证方式**：正常流程、异常流程、刷新和并发场景是否都测试过？

## 动态路由与权限

### 为什么这样设计

如果路由和菜单全部写死在前端，每次新增菜单或调整角色权限都需要重新修改、构建和发布前端。因此系统让后端返回当前用户拥有的菜单和权限点，前端根据这些数据动态生成路由、侧边栏和按钮权限。

选择动态路由的目标是：

- 不同角色看到不同页面。
- 菜单调整不需要修改前端权限配置。
- 菜单、路由和权限来源保持一致。
- 支持管理员在后台配置角色权限。

### 边界问题

- 用户直接输入无权限页面地址。
- 页面刷新后 Pinia 状态丢失，动态路由尚未注册。
- 首次访问时路由匹配发生在动态路由注册之前。
- 后端返回的组件路径不存在。
- 菜单存在，但用户没有对应按钮权限。
- Token 存在但已过期。
- 重复执行权限初始化，导致重复添加路由。
- 退出登录后动态路由和权限状态没有清理。
- 路由组件加载失败或后端返回非法菜单数据。

### 解决策略

- 路由守卫先校验 Token，再初始化用户和权限。
- 使用初始化标识避免重复加载动态路由。
- 注册完成后通过 `replace` 重新进入目标地址。
- 使用组件映射表检查后端组件路径，找不到时进入异常页。
- 前端控制展示，后端接口继续执行最终权限校验。
- 退出登录时清理 Token、权限集合、标签页和动态路由状态。

面试回答可以这样说：

> 我们选择后端驱动的动态路由，是为了让角色权限可以在线配置，并保持菜单和路由数据来源一致。实现中的关键问题不是调用 `addRoute`，而是权限初始化时序。页面刷新时，路由第一次匹配可能早于权限接口返回，因此我在守卫中等待权限初始化，注册完成后重新 `replace` 当前地址。同时增加初始化标识、组件路径兜底和退出清理，避免重复注册及旧权限残留。

## Axios 请求封装

### 为什么这样设计

如果每个业务接口都单独处理 Token、错误提示和登录失效，代码会大量重复，而且不同模块的处理方式可能不一致。因此将公共逻辑集中在请求层。

### 边界问题

- 多个接口同时返回 `401`，反复提示和跳转登录页。
- 用户连续点击按钮，产生重复提交。
- GET、POST 和文件上传的参数结构不同。
- 请求取消后不应显示“网络错误”。
- 请求签名时对象字段顺序不一致，导致签名不同。
- Token 刷新或失效期间又发出新请求。
- 下载文件返回 JSON 错误，而不是正常文件流。
- 网络超时、业务错误和 HTTP 错误需要不同处理。
- 前端环境变量中的密钥可以被用户读取。

### 设计权衡

重复请求不能一律取消。例如查询接口通常可以取消旧请求，但保存、支付等写操作不能简单取消，否则可能出现前端认为失败、后端已经执行成功的情况。

HMAC 密钥放在前端也不能成为真正秘密，因为构建产物可以被分析。它只能增加请求伪造成本，真正的身份认证、权限和防重放仍需由后端负责。

## 主题系统

### 为什么这样设计

如果每个组件分别维护亮色和暗色样式，后期修改主题颜色需要改很多位置。通过 Design Token 和 CSS Variables，可以让组件库、业务组件和图表共享一套主题来源。

### 边界问题

- 页面加载后才读取主题，产生由白到黑的闪烁。
- “跟随系统”模式下，系统主题运行时发生变化。
- 用户手动选择主题后，不应继续被系统设置覆盖。
- Ant Design Vue 已切换主题，但业务 CSS 没有变化。
- ECharts 使用初始化时的颜色，主题切换后不会自动更新。
- 本地存储值非法或旧版本配置不兼容。
- 多个浏览器标签页之间主题不同步。

核心设计原则是区分：

```
用户配置：light / dark / auto
实际主题：light / dark
```

`auto` 不是实际颜色，它需要根据系统偏好计算出最终主题。这个区分是主题系统中很容易被忽略的边界。

## ECharts 组件封装

### 为什么这样设计

多个页面直接创建 ECharts 实例，会重复编写初始化、更新、监听窗口变化和销毁逻辑，因此将生命周期管理封装成统一组件。

### 边界问题

- 组件挂载时容器宽高为 `0`。
- 标签页隐藏后再显示，图表尺寸不正确。
- 侧边栏折叠改变容器宽度，但没有触发 `window.resize`。
- 配置频繁变化导致重复重绘。
- 组件卸载后未执行 `dispose()`，造成内存泄漏。
- 暗色主题切换后，旧实例仍使用原主题。
- 接口请求返回时组件已经卸载。
- 一个页面包含多个图表，频繁 resize 造成性能问题。

因此使用 `ResizeObserver` 比只监听 `window.resize` 更合理，因为图表容器变化不一定来自浏览器窗口变化。与此同时需要防抖，并在卸载时断开 Observer、销毁实例。

## 你准备项目时的方法

针对每个简历技术点，提前写清楚下面八个问题：

1. 原始需求是什么？
2. 最简单的实现方案是什么？
3. 最简单的方案有什么缺点？
4. 当前项目为什么采用这个方案？
5. 核心数据结构是什么？
6. 完整执行流程是什么？
7. 至少有哪些三个异常或边界场景？
8. 如何证明解决方案有效？

尤其要准备“刷新、重复、失败、过期、为空、并发、卸载、越权”这八类场景。后台系统的大部分边界问题都可以从这些关键词展开。

同时必须区分“源码中已经解决的问题”和“你阅读后发现但项目尚未解决的问题”。后者可以这样表达：

> 当前实现采用了某种方案，但我分析后发现它在某个场景存在风险。如果继续完善，我会通过某种方式处理。

这种回答比把项目包装成毫无缺陷更可信，也更能体现技术判断。

### 1. 技术栈

Vue 3、TypeScript、Vite、Pinia、Vue Router、Ant Design Vue、Axios、UnoCSS、Less、Vue I18n、ECharts、VueUse、ByteMD、SortableJS

### 2. 项目介绍

Admin X 是一套基于 Vue 3 与 TypeScript 开发的企业级后台管理系统，包含登录注册、用户管理、角色管理、菜单管理、字典管理、文章管理、系统日志、数据看板和个人中心等功能。系统采用后端驱动的 RBAC 权限模型，根据用户角色动态生成菜单和业务路由，并实现页面级、菜单级及按钮级权限控制；同时支持多标签页、响应式布局、主题切换、国际化、Markdown 内容管理和 ECharts 数据可视化。

### 3. 技术重点实现策略

- **动态路由与 RBAC 权限控制：** 登录后从服务端获取用户信息、菜单树和权限标识，将菜单数据转换为 Vue Router 路由配置；通过动态组件映射和 `router.addRoute()` 注册业务路由，同时生成侧边栏菜单。结合全局路由守卫完成 Token 校验、权限初始化、无权限拦截及刷新恢复，并通过自定义 `v-perm` 指令实现按钮级权限控制。
    
- **统一请求基础设施：** 基于 Axios 封装通用请求类，在请求拦截器中统一注入 Token、时间戳、随机数、设备指纹等信息，并通过 HMAC-SHA256 生成请求完整性签名；根据请求方法、地址及参数生成请求标识，实现重复请求检测与取消。响应阶段统一处理业务异常、网络错误和 `401/403` 鉴权失效，并封装文件上传等公共能力。
    
- **Design Token 驱动的主题系统：** 基于 Ant Design Vue 主题算法实现亮色、暗色和跟随系统三种模式，将组件库 Design Token 转换为全局 CSS Variables，使 Ant Design Vue 组件、业务 Less 样式和 ECharts 图表共享统一的颜色配置；结合 VueUse 完成主题偏好持久化、系统主题监听及页面初始化防闪烁。
    
- **通用 ECharts 组件封装：** 统一管理图表实例的初始化、配置更新、尺寸适配和销毁，通过 `ResizeObserver` 监听容器尺寸变化，并结合防抖降低高频重绘开销；对外暴露图表更新、缩放和实例访问方法，解决侧边栏折叠、标签页切换以及响应式布局下的图表变形问题。
    
- **全局状态与多标签页管理：** 使用 Pinia 按职责拆分用户、权限、标签页和应用配置状态，结合本地存储恢复登录状态、主题配置和访问记录；通过路由变化自动维护标签页，并使用 SortableJS 实现标签页拖拽排序、关闭当前、关闭其他和重新加载等操作。
    
- **国际化与内容编辑：** 使用 Vue I18n 管理中英文语言资源，并与 Ant Design Vue 组件语言同步；基于 ByteMD 封装 Markdown 编辑器和预览组件，集成 GFM、代码高亮及数学公式插件，用于文章编辑、详情展示和评论互动。
    
- **前端工程化建设：** 使用 Vite 配置开发环境代理、自动导入、按模块分包、Gzip 压缩和图片压缩；通过 ESLint、Prettier、Stylelint、Husky、Commitlint 和 `vue-tsc` 统一代码规范、提交规范及 TypeScript 类型检查。
