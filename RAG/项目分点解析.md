
这个项目可以拆成 **8 个核心业务组件**：

1. **文档上传组件**
2. **文档解析组件**
3. **文本分块组件**
4. **知识入库 / 向量化组件**
5. **知识库管理组件**
6. **检索召回组件**
7. **对话问答组件**
8. **模型参数配置组件**

# 1. 文件上传

## 1. **文档上传组件**

> `app/components/DocumentUpload.vue`

- 负责知识库管理弹窗的整体 UI
- 提供拖拽上传区域
- 提供点击选择文件上传
- 限制可选文件类型：`.pdf / .txt / .md / .docx`
- 上传中展示进度条、上传百分比、已上传大小 / 总大小
- 展示错误提示和通知提示
- 展示已上传文档列表
- 展示每个文档的文件类型、文件大小、向量化状态、文本块数量
- 支持删除文档，删除前会提示“删除文档将同时删除其向量数据”
- 不直接实现上传逻辑，而是通过父组件传入的 `docState.uploadDocument(file)` 触发上传流程

==作用==：源码里可以看到，这个组件主要是 **知识库文档上传和文档列表管理的 UI 层**。它本身不做哈希计算、分片上传、文本分块、向量化，而是负责接收用户选择的文件，然后调用 `props.docState.uploadDocument(file)`。上传进度、上传百分比、上传字节数、错误信息、通知信息、文档列表也都是从 `docState` 中读取。

### 1.1 **文档上传组件的入口位置**

> `app/pages/index.vue`

- 首页中维护 `showDocPanel`
- 点击顶部“知识库”按钮后展示 `DocumentUpload`
- 通过 `:doc-state="docState"` 把文档上传状态传给上传组件
- 使用 `useDocumentsAdvanced()` 创建文档管理状态
- 页面挂载时调用：
    - `docState.loadDocuments()`
    - `docState.initWorker()`
- 页面卸载时调用：
    - `docState.terminateWorker()`

==作用==：`index.vue` 是这个上传面板的挂载入口。源码里 `DocumentUpload` 并不是路由页面，而是首页聊天布局里的一个侧边知识库面板。`index.vue` 创建 `docState = useDocumentsAdvanced()`，然后把它传给 `DocumentUpload`，所以真正的上传逻辑入口在 composable 里。

调用关系是：

```
app/pages/index.vue
   ↓
useDocumentsAdvanced()
   ↓
<DocumentUpload :doc-state="docState" />
```

### 1.2 **文档上传核心逻辑**

> `app/composables/useDocuments.js`

这个文件里有两个 composable：

```
useDocuments()
useDocumentsAdvanced()
```

真正对应你简历里“秒传、断点续传、分片上传、上传进度计算”的是：

> `useDocumentsAdvanced()`

它主要做这些事：

- 维护文档列表 `documents`
- 维护上传状态 `isUploading`
- 维护上传文案 `uploadProgress`
- 维护上传百分比 `uploadPercent`
- 维护已上传大小 `uploadedBytes`
- 维护总文件大小 `totalBytes`
- 维护错误信息 `error`
- 维护提示信息 `notice`
- 计算文件哈希 `computeFileHash(file)`
- 按固定大小切分文件，源码里 `chunkSize = 2 * 1024 * 1024`
- 请求 `/api/upload/check` 判断是否秒传或断点续传
- 跳过服务端已经存在的分片
- 使用 `XMLHttpRequest` 上传分片到 `/api/upload/chunk`
- 上传完成后请求 `/api/upload/merge`
- 合并完成后拿到解析出的文本
- 调用 Web Worker 进行文本分块
- 调用 `/api/embed` 生成向量并入库
- 最后重新加载文档列表

==作用==：`DocumentUpload.vue` 是 UI 层，`useDocumentsAdvanced()` 才是 **文件上传业务编排层**。源码里可以看到，完整上传流程不是一次性 POST 文件，而是：先算 SHA-256 文件哈希，再检查服务端状态，再分片上传缺失部分，最后合并文件、解析文本、分块、向量化。

核心流程可以写成：

```
用户选择文件
   ↓
DocumentUpload.vue 调用 docState.uploadDocument(file)
   ↓
useDocumentsAdvanced.uploadDocument(file)
   ↓
computeFileHash(file)
   ↓
POST /api/upload/check
   ↓
判断是否秒传 / 是否存在已上传分片
   ↓
file.slice() 切分文件
   ↓
POST /api/upload/chunk 上传缺失分片
   ↓
POST /api/upload/merge 合并分片并解析文本
   ↓
Web Worker 文本分块
   ↓
POST /api/embed 生成向量并写入 ChromaDB
   ↓
loadDocuments() 刷新文档列表
```

### 1.3 **上传检查接口**

> `server/api/upload/check.post.js`

- 接收前端传来的文件元信息
- 校验上传元数据是否合法
- 根据 `fileHash` 查询是否已有完整文档
- 如果已有完整文档，返回 `exists: true`
- 如果没有完整文档，则创建或更新上传会话
- 查询服务端已经存在的分片索引
- 计算已上传字节数
- 返回 `uploadedChunks / uploadedBytes / totalChunks`

==作用==：这个接口支撑 **秒传和断点续传**。源码里会先 `findDocumentByHash(body.fileHash)`，如果找到已有文档，就直接返回 `exists: true`；否则通过 `upsertUploadSession` 创建上传会话，并返回已经上传过的分片列表。

也就是：

```
前端算出 fileHash
   ↓
请求 /api/upload/check
   ↓
服务端查是否有同 hash 文档
   ↓
有：秒传
   ↓
没有：返回已上传分片，用于断点续传
```

### 1.4 **分片上传接口**

> `server/api/upload/chunk.post.js`

- 接收 multipart 分片数据
- 读取分片文件字段 `chunk`
- 读取上传元信息：
    - `fileHash`
    - `fileName`
    - `fileSize`
    - `totalChunks`
    - `chunkSize`
    - `chunkIndex`
    - `ext`
- 校验分片索引是否合法
- 校验上传元数据是否合法
- 创建或更新上传会话
- 可选校验 `chunkHash`
- 确保分片目录存在
- 将当前分片写入本地文件系统

==作用==：这个接口是 **单个分片的接收层**。前端每次上传一个 chunk，服务端就按照 `fileHash + chunkIndex` 把它保存成对应的 `.part` 文件。源码里最终是通过 `writeFile(getUploadChunkPath(meta.fileHash, chunkIndex), filePart.data)` 把分片落盘。

### 1.5 **分片合并接口**

> `server/api/upload/merge.post.js`

- 校验上传元数据
- 读取上传会话
- 再次检查是否已有相同 hash 的文档
- 获取已经上传的分片索引
- 检查是否还有缺失分片
- 按 `0 ~ totalChunks - 1` 的顺序读取分片
- 使用 `createWriteStream` 合并成完整文件
- 调用 `parseDocument(fileBuffer, session.fileName)` 提取文本
- 生成文档元信息 `docMeta`
- 写入文档元数据文件
- 删除临时分片目录
- 删除上传会话文件
- 返回文档元信息和解析出的文本

==作用==：这个接口是 **上传阶段和 RAG 入库阶段的分界点**。在它之前，系统处理的是文件分片；在它之后，系统已经拿到了完整文件和解析文本，后续就可以进入文本分块和向量化流程。源码里合并完成后会调用 `parseDocument` 提取文本，并写入 `documents/${documentId}.meta.json`，其中初始状态是 `hasVectors: false`、`chunksCount: 0`。

### 1.6 **上传会话与校验工具**

> `server/utils/uploadSessions.js`

- 定义最大文件大小：`20MB`
- 定义允许的文件类型：`pdf / txt / md / docx`
- 生成上传会话文件路径
- 生成分片目录路径
- 生成单个分片路径
- 校验上传元数据
- 创建分片目录
- 读取已上传分片索引
- 计算已上传字节数
- 读取 / 写入上传会话
- 根据 `fileHash` 查找已完成文档
- 提供服务端 `sha256` 校验方法

==作用==：这个文件是上传模块的 **底层工具层**。`check.post.js`、`chunk.post.js`、`merge.post.js` 都依赖它。比如文件类型和大小不是前端写死判断，而是服务端通过 `validateUploadMeta` 再校验一遍，保证上传安全性和数据一致性。

### 1.7 **向量化接口**

> `server/api/embed.post.js`

- 接收 `documentId` 和 `chunks`
- 读取模型配置和 API Key
- 批量调用 embedding 方法生成向量
- 每批最多处理 25 条 chunk
- 读取文档元信息
- 获取或创建 Chroma Collection
- 将：
    - chunk id
    - embedding
    - metadata
    - 原始 chunk 文本  
        写入 ChromaDB
- 同步写入本地 `vectors/${documentId}.json`
- 更新文档元信息：
    - `hasVectors = true`
    - `chunksCount = chunks.length`
    - `updatedAt = 当前时间`

==作用==：这个接口不是文件上传本身，但它是上传流程的最后一步。`useDocumentsAdvanced()` 在文件合并、文本解析、Web Worker 分块之后，会调用 `/api/embed`，把文本块转成向量并存入 ChromaDB。源码里可以看到它使用 `getEmbeddings(batch, settings)` 批量生成向量，然后 `collection.add(...)` 写入 ChromaDB。

### 面试时回答

> 首页入口在 `app/pages/index.vue`，里面通过 `useDocumentsAdvanced()` 创建文档状态，并把它作为 `doc-state` 传给 `DocumentUpload`。真正的上传逻辑在 `app/composables/useDocuments.js`，里面会先计算文件 SHA-256，然后请求 `/api/upload/check` 判断是否秒传或断点续传。如果需要上传，就按 2MB 固定大小切片，用 XHR 上传到 `/api/upload/chunk`，并通过 `xhr.upload.onprogress` 计算进度。所有分片上传完成后，再请求 `/api/upload/merge`，服务端会检查分片完整性、按顺序合并文件、解析文本并返回。前端拿到文本后，再用 Web Worker 分块，最后调用 `/api/embed` 生成向量并写入 ChromaDB。

### 模块完整流程

你这个上传模块，前端页面组件其实主要有两个：

```

  ↓ 创建 docState，并控制知识库面板显示

server/api/upload/check.post.js
server/api/upload/chunk.post.js
server/api/upload/merge.post.js
server/api/embed.post.js

整体关系可以先这样记：

```

  ↓
app/pages/index.vue 打开 DocumentUpload 面板
  ↓

  ↓

  ↓
useDocumentsAdvanced.uploadDocument(file)
  ↓

  ↓
/api/upload/check 检查秒传 / 断点续传
  ↓
/api/upload/chunk 上传缺失分片
  ↓
/api/upload/merge 合并分片并解析文本
  ↓
Web Worker 分块
  ↓
/api/embed 生成向量并写入 ChromaDB
  ↓
loadDocuments() 刷新知识库列表
```

#### 上传模块挂载入口

>文件：`app/pages/index.vue`

它是整个 RAG 聊天页面的入口，同时也是知识库上传面板的挂载位置。这个页面本身不负责上传细节，而是负责：

- 创建文档状态对象 `docState`
- 控制知识库面板是否显示
- 把 `docState` 传给 `DocumentUpload`
- 页面挂载时加载文档列表
- 页面挂载时初始化 Web Worker
- 页面卸载时销毁 Web Worker

源码里可以看到，首页通过 `useDocumentsAdvanced()` 创建文档管理状态，然后把它传给 `<DocumentUpload :doc-state="docState" />`。

##### 1. template 结构

核心 template 可以抽象成这样：

```vue
<div class="chat-layout">
  <ChatSidebar
    :documents-count="docState.documents.value.length"

  />

  <div class="chat-main">

        </span>
      </button>

      <NuxtLink to="/settings">
        设置
      </NuxtLink>
    </header>

    <ChatWindow />
    <ChatInput />
  </div>

  <DocumentUpload
    v-if="showDocPanel"
    @close="showDocPanel = false"
    :doc-state="docState"
  />
</div>
```

它做了三件事：

1. 左侧 `ChatSidebar` 展示文档数量，并提供打开知识库面板的入口。
2. 顶部 header 里有“知识库”按钮，点击后切换 `showDocPanel`。
3. 当 `showDocPanel = true` 时，渲染 `DocumentUpload` 组件，并把 `docState` 传进去。

这里的关键点是：

```vue
<DocumentUpload
  v-if="showDocPanel"
  @close="showDocPanel = false"
  :doc-state="docState"
/>
```

也就是说，`DocumentUpload.vue` 自己不创建上传状态，而是完全接收父组件传入的 `docState`。

##### 2. 核心变量

###### `chatStore`

```
const chatStore = useChatStore()
```

这个是聊天模块状态，和上传模块不是强绑定，但页面里会同时管理聊天状态和知识库状态。

它主要负责：

- 当前会话标题
- 是否正在流式生成
- 来源片段展示
- 会话列表加载

上传模块里只需要知道：它和文档模块并列存在于首页。
###### `docState`

```js
const docState = useDocumentsAdvanced()
```

这是上传模块最重要的变量。

`docState` 不是普通对象，而是 `useDocumentsAdvanced()` 返回的一整套文档管理状态和方法。

里面包括：

```js
{
  documents,
  isUploading,
  uploadProgress,
  uploadPercent,
  uploadedBytes,
  totalBytes,
  error,
  notice,
  loadDocuments,
  uploadDocument,
  deleteDocument,
  initWorker,
  terminateWorker
}
```

所以 `index.vue` 的职责不是自己处理上传，而是创建这个状态对象，然后交给子组件使用。

###### `showDocPanel`

```js
const showDocPanel = ref(false)
```

它控制知识库面板是否显示。

```
false：不显示上传面板
true：显示上传面板
```

这个变量对应 template 里的：

```vue
<DocumentUpload v-if="showDocPanel" />
```

以及按钮上的：

```vue
@click="showDocPanel = !showDocPanel"
```

##### 3. 核心函数 / 生命周期

==`onMounted`==

```js
onMounted(async () => {
  await chatStore.loadConversations()
  await docState.loadDocuments()
  docState.initWorker()
})
```

它做了三件事：

1. 加载历史会话。
2. 加载知识库文档列表。
3. 初始化文本分块用的 Web Worker。

对上传模块来说，关键是后两步：

```js
await docState.loadDocuments()
docState.initWorker()
```

这表示首页一进入，就会先把已有文档列表拉出来，这样知识库按钮上的数量和上传面板里的文档列表都是最新的。

同时提前初始化 Worker，后续上传文档并解析文本后，就可以直接把大文本交给 Worker 分块。

==`onUnmounted`==

```js
onUnmounted(() => {  docState.terminateWorker()})
```

页面销毁时关闭 Worker。

这个设计是合理的，因为 Worker 是独立线程，如果页面离开后不销毁，可能会造成资源占用。

##### 4. 这个页面在上传链路中的作用

可以这样总结：

```
index.vue 不负责具体上传
index.vue 只负责创建 docState
index.vue 控制 DocumentUpload 是否显示
index.vue 把 docState 传给 DocumentUpload
index.vue 在生命周期里初始化和销毁 Worker
```

###### 1.3 上传区域

```vue
<div
  class="upload-zone"
  :class="{ dragging: isDragging, uploading: docState.isUploading.value }"
  @dragover.prevent="isDragging = true"
  @dragleave="isDragging = false"

这块是上传组件的核心 UI。

其中：

```vue

```

表示前端文件选择器只允许选择这些类型。

但注意，这只是浏览器层面的限制，真正安全校验还在服务端 `validateUploadMeta` 里做。

它展示四类信息：

4. 文件总大小：`totalBytes`

比如上传过程中会显示：

这部分对应 `useDocumentsAdvanced()` 里的 `updateProgressState()`。

###### 1.5 上传提示区域

==`formatSize`==

```js
function formatSize(bytes) {
  if (!bytes) return '0 B'
  const units = ['B', 'KB', 'MB', 'GB']
  let i = 0
  let size = bytes

  while (size >= 1024 && i < units.length - 1) {
    size /= 1024
    i++
  }

  return `${size.toFixed(1)} ${units[i]}`
}
```

它负责把字节数格式化成人类可读的单位。

比如：

```
1024      → 1.0 KB
1048576   → 1.0 MB
```

这个函数在两个地方用到：

```
上传进度：uploadedBytes / totalBytes
文档列表：doc.size
```

##### 4. 这个组件在上传链路中的作用

可以这样总结：

```
DocumentUpload.vue 是上传 UI 层
它不做哈希计算
它不做分片切割
它不直接请求上传接口
它只负责：
  1. 选择文件
  2. 拖拽文件
  3. 调用 uploadDocument(file)
  4. 展示上传状态
  5. 展示文档列表
  6. 调用 deleteDocument(id)
```

面试里可以这样说：

> `DocumentUpload.vue` 是知识库上传面板组件，它负责上传区域、拖拽上传、点击选择文件、上传进度展示、错误提示、通知提示和文档列表展示。它本身不处理分片上传，而是通过父组件传入的 `docState` 调用 `uploadDocument(file)`。这样做可以把 UI 和业务逻辑拆开，组件只关心用户交互和状态展示，具体上传流程交给 `useDocumentsAdvanced()` 统一编排。

==`uploadDocument`==

这是整个上传模块最核心的函数。

它的完整流程可以拆成 9 步。

**第一步：初始化上传状态**

```js
isUploading.value = true
uploadProgress.value = '正在准备上传...'
uploadPercent.value = 0
uploadedBytes.value = 0
totalBytes.value = file.size || 0
error.value = null
notice.value = ''
```

作用是重置上传状态。

用户一选择文件，就先把 UI 切到上传中。

**第二步：计算文件 hash**

```js
const fileHash = await computeFileHash(file)
const ext = getFileExt(file.name)
const totalChunks = Math.ceil(file.size / chunkSize)
```

这里做了三件事：

1. 用文件内容计算 SHA-256。
2. 从文件名里取后缀。
3. 根据文件大小和 2MB 分片大小计算分片总数。

比如一个 9MB 文件：

```js
chunkSize = 2MB
totalChunks = Math.ceil(9 / 2) = 5
```

`fileHash` 的作用是标识文件内容。

同一个文件只要内容一样，即使文件名不同，hash 也应该一样。

**第三步：检查秒传 / 断点续传**

```js
const checkResult = await $fetch('/api/upload/check', {
  method: 'POST',
  body: {
    fileName: file.name,
    fileSize: file.size,
    fileHash,
    ext,
    chunkSize,
    totalChunks
  }
})
```

这一步会把文件元数据发给服务端。

服务端会判断：

```
这个 hash 的完整文档是否已经存在？
这个 hash 是否有上传过一部分分片？
```

如果已经存在完整文档：

```js
if (checkResult.exists) {
  uploadedBytes.value = file.size
  totalBytes.value = file.size
  uploadPercent.value = 100
  uploadProgress.value = '秒传完成，已复用现有文件'
  notice.value = checkResult.message || '文件已存在，已跳过重复上传'
  await loadDocuments()
  return true
}
```

这就是秒传。

所谓秒传，不是真的重新上传文件，而是服务端发现这个文件已经入库过了，前端直接复用已有结果。

**第四步：处理已上传分片**

```js
const uploadedChunkSet = new Set(checkResult.uploadedChunks || [])
let confirmedBytes = checkResult.uploadedBytes || 0
```

服务端会返回已经上传过的分片索引。

比如：

```js
uploadedChunks = [0, 1, 2]
```

说明前 3 个分片之前已经上传成功。

前端后面循环上传时会跳过这些分片：

```js
if (uploadedChunkSet.has(index)) continue
```

这就是断点续传。

**第五步：循环切片并上传缺失分片**

```js
for (let index = 0; index < totalChunks; index++) {
  if (uploadedChunkSet.has(index)) continue

  const start = index * chunkSize
  const end = Math.min(start + chunkSize, file.size)
  const chunk = file.slice(start, end)

  await uploadChunkWithProgress({
    file,
    chunk,
    chunkIndex: index,
    totalChunks,
    fileHash,
    ext,
    chunkSize,
    onProgress: (loaded) => {
      updateProgressState(
        confirmedBytes + loaded,
        file.size,
        `正在上传分片 ${index + 1}/${totalChunks}...`
      )
    }
  })

  confirmedBytes += chunk.size
}
```

这段逻辑是分片上传的核心。

它做了几件事：

1. 遍历所有分片索引。
2. 如果该分片已经存在，就跳过。
3. 用 `file.slice(start, end)` 切出当前分片。
4. 调用 `uploadChunkWithProgress()` 上传分片。
5. 通过 `onProgress` 更新总进度。
6. 当前分片上传完后，把 `confirmedBytes` 增加。

这里的设计点是：

```
不是每个分片独立展示进度
而是把“已确认上传字节数 + 当前分片 loaded”合成整体进度
```

所以 UI 上展示的是整个文件的上传进度。

**第六步：请求服务端合并文件**

```js
const result = await $fetch('/api/upload/merge', {
  method: 'POST',
  body: {
    fileName: file.name,
    fileSize: file.size,
    fileHash,
    ext,
    chunkSize,
    totalChunks
  }
})
```

所有分片上传完成后，前端通知服务端合并。

服务端会：

```
检查分片是否齐全
按 chunkIndex 顺序读取分片
写入完整文件
解析文档文本
生成文档 meta
删除临时分片
返回 text
```

也就是说，`merge` 接口返回的不只是“合并成功”，它还会返回解析后的文本。

**第七步：校验解析文本**

```js
if (!result.text || !result.text.trim()) {
  throw new Error('文件中未提取到有效文本')
}
```

如果 PDF、DOCX、TXT、MD 里没有提取出文本，就终止流程。

这个判断很重要，因为后续的文本分块和向量化都依赖 `text`。

**第八步：Web Worker 文本分块**

```js
const settings = await $fetch('/api/settings')

const chunks = await chunkTextWithWorker(result.text, {
  chunkSize: settings.chunkSize || 500,
  chunkOverlap: settings.chunkOverlap || 50
})
```

这一步进入 RAG 的文本处理阶段。

前面是文件上传 chunk。

这里是知识库文本 chunk。

它会先请求模型配置里的：

```
chunkSize
chunkOverlap
```

然后交给 Worker 分块。

**第九步：调用向量化接口**

```js
await $fetch('/api/embed', {
  method: 'POST',
  body: { documentId: result.id, chunks }
})
```

这一步把文本块送到服务端生成 embedding。

服务端 `/api/embed` 会批量生成向量，并写入 ChromaDB，同时更新文档 meta 的 `hasVectors` 和 `chunksCount`。

上传完成后：

```js
uploadProgress.value = '上传完成'
await loadDocuments()
```

重新拉取文档列表，UI 中就能看到：

```
✓ n 个块
```

==`chunkTextWithWorker`==

```js
function chunkTextWithWorker(text, options) {
  return new Promise((resolve, reject) => {
    initWorker()

    if (!worker) {
      resolve(fallbackChunkAdvanced(text, options.chunkSize))
      return
    }

    worker.onmessage = ({ data }) => {
      if (data.type === 'chunks') {
        resolve(data.payload)
      } else if (data.type === 'error') {
        reject(new Error(data.payload))
      }
    }

    worker.postMessage({
      type: 'chunk',
      payload: {
        text,
        chunkSize: options.chunkSize,
        chunkOverlap: options.chunkOverlap
      }
    })
  })
}
```

它负责把文本分块任务交给 Worker。

流程是：

```
主线程调用 chunkTextWithWorker
  ↓
初始化 Worker
  ↓
向 Worker postMessage
  ↓
Worker 执行 recursiveChunk
  ↓
Worker 返回 chunks
  ↓
主线程 resolve(chunks)
```

它还做了降级处理：

```js
if (!worker) {
  resolve(fallbackChunkAdvanced(text, options.chunkSize))
}
```

也就是说，如果 Worker 不可用，就退回主线程简单分块。

==`fallbackChunkAdvanced`==

```js
function fallbackChunkAdvanced(text, size = 500) {
  const chunks = []

  for (let i = 0; i < text.length; i += size) {
    const chunk = text.slice(i, i + size).trim()
    if (chunk) chunks.push(chunk)
  }

  return chunks
}
```

这是 Worker 不可用时的降级方案。

它比较简单，就是固定长度切分。

缺点是语义不如 Worker 的递归分块完整。

==`deleteDocument`==

    documents.value = documents.value.filter(doc => doc.id !== id)

    error.value = err.data?.message || err.message || '删除失败'
  }
}
```

它负责删除文档。

流程是：

```
调用 DELETE /api/documents/:id
  ↓
服务端删除文档和向量数据
  ↓
前端从 documents 中过滤掉该文档
```

==`updateProgressState`==

```js
function updateProgressState(currentBytes, fileSize, message) {
  uploadProgress.value = message
  uploadedBytes.value = Math.min(currentBytes, fileSize)
  totalBytes.value = fileSize
  uploadPercent.value = fileSize
    ? Math.round((uploadedBytes.value / fileSize) * 100)
    : 0
}
```

它是上传进度的统一更新函数。

每个分片上传时，都会调用它。

它统一维护四个状态：

```js
uploadProgress
uploadedBytes
totalBytes
uploadPercent
```

这样组件里只需要展示状态，不需要关心进度怎么算。

##### 3. 这个模块在上传链路中的作用

可以总结成：

```
useDocumentsAdvanced 是上传业务编排层
DocumentUpload.vue 负责 UI
server/api 负责实际文件处理和向量化
useDocumentsAdvanced 负责把这些步骤串起来
```

面试里可以这样说：

> `useDocumentsAdvanced()` 是上传模块的核心逻辑。它先维护上传状态和文档列表，然后在 `uploadDocument(file)` 中完成完整的上传编排：计算文件 SHA-256，调用 `/api/upload/check` 判断秒传或断点续传，按 2MB 固定大小切片，用 XHR 上传缺失分片，上传完成后调用 `/api/upload/merge` 合并并解析文本，然后通过 Web Worker 按配置进行文本分块，最后调用 `/api/embed` 生成向量并刷新文档列表。

#### 文本分块 Worker

>文件：`public/workers/textProcessor.worker.js`

这个文件负责上传后的大文本分块。

它运行在 Worker 线程里，不阻塞页面主线程。源码里它通过 `self.onmessage` 接收主线程任务，任务类型是 `chunk` 时，会调用 `recursiveChunk` 进行分块，然后通过 `self.postMessage` 把 chunks 返回给主线程。

##### 1. 核心入口

```js
self.onmessage = function ({ data }) {
  const { type, payload } = data

  switch (type) {
    case 'chunk':
      const chunks = recursiveChunk(payload.text, {
        chunkSize: payload.chunkSize || 500,
        chunkOverlap: payload.chunkOverlap || 50
      })
      self.postMessage({ type: 'chunks', payload: chunks })
      break

    default:
      self.postMessage({ type: 'error', payload: '未知的任务类型' })
  }
}
```

它做了三件事：

1. 接收主线程传来的消息。
2. 如果任务类型是 `chunk`，就执行文本分块。
3. 分块完成后，把结果发回主线程。

##### 2. 核心函数：

==`recursiveChunk`==

```js
function recursiveChunk(text, options) {
  const { chunkSize, chunkOverlap } = options

  const cleanedText = text.replace(/\n{3,}/g, '\n\n').trim()

  const paragraphs = cleanedText.split(/\n\s*\n/)
  const rawChunks = []

  ...
}
```

它的分块策略是递归式的：

```
先按段落切
  ↓
段落太长就按句子切
  ↓
句子太长就按固定长度切
  ↓
最后加 overlap
```

这个比简单固定长度切分更适合 RAG。

因为 RAG 检索依赖语义完整性，如果直接每 500 字切一次，可能会把一个完整语义切断。

==`chunkBySentences`==

```js
function chunkBySentences(text, chunkSize) {
  const sentenceEnders = /(?<=[。！？.!?\n])\s*/g
  const sentences = text.split(sentenceEnders).filter(s => s.trim())

  ...
}
```

它负责把超长段落按句子切开。

支持中文和英文常见句末符号：

```
。 ！ ？ . ! ?
```

逻辑是：

```
遍历句子
  ↓
能放进当前 chunk 就追加
  ↓
放不下就新开一个 chunk
  ↓
如果单句仍然过长，再走固定长度切分
```

==`chunkByFixedSize`==

```js
function chunkByFixedSize(text, size, overlap) {
  const chunks = []
  const step = Math.max(size - overlap, 1)

  for (let i = 0; i < text.length; i += step) {
    chunks.push(text.slice(i, i + size))
  }

  return chunks
}
```

这是最终兜底方案。

它按固定长度切分，并支持 overlap。

比如：

```js
chunkSize = 500
chunkOverlap = 50
step = 450
```

那么切分窗口是：

```
0 - 500
450 - 950
900 - 1400
```

这样相邻 chunk 之间会保留一部分上下文，减少语义断裂。

```
/api/upload/merge 返回 text
  ↓
useDocumentsAdvanced.chunkTextWithWorker(text)
  ↓
postMessage 给 textProcessor.worker.js
  ↓
recursiveChunk 分块
  ↓
postMessage 返回 chunks
  ↓
/api/embed 生成向量
```

面试里可以这样说：

> 上传文件合并并解析出文本后，我没有直接在主线程做大文本分块，而是把文本和分块参数传给 `public/workers/textProcessor.worker.js`。Worker 内部先按段落切分，如果段落超过 chunkSize，再按句子切分，如果句子仍然过长，最后按固定长度兜底，并根据 chunkOverlap 增加重叠内容。这样可以保证分块尽量语义完整，同时避免大文档处理阻塞 UI。

### 文件上传原理

这个项目的文件上传不是普通的“一次性把文件丢给后端”，而是做成了一个比较完整的 **大文件上传 + RAG 入库流程**。

它的核心链路是：

```
选择文件
  ↓
计算文件 hash
  ↓
检查是否秒传 / 断点续传
  ↓
按 2MB 切分文件
  ↓
上传缺失分片
  ↓
服务端合并分片
  ↓
服务端解析文本
  ↓
前端 Web Worker 分块
  ↓
调用 /api/embed 向量化入库
  ↓
刷新知识库文档列表
```

当前实现中，前端核心逻辑在 `app/composables/useDocuments.js` 的 `useDocumentsAdvanced()`，上传面板在 `app/components/DocumentUpload.vue`，服务端接口主要是 `server/api/upload/check.post.js`、`server/api/upload/chunk.post.js`、`server/api/upload/merge.post.js`。

#### 1. 为什么要做分片上传？

普通上传是这样：

```
前端选择一个文件
  ↓
FormData 一次性上传整个文件
  ↓
服务端接收完整文件
```

这种方式简单，但是有几个问题：

```
文件稍大时，上传失败就要全部重传
网络中断后不能恢复
无法展示比较精确的上传进度
重复文件也会重复上传
```

所以这个项目采用的是 **文件分片上传**。

也就是把一个完整文件切成多个小块：

```
file.pdf
  ├─ chunk 0
  ├─ chunk 1
  ├─ chunk 2
  ├─ chunk 3
  └─ chunk n
```

当前项目里上传分片大小是：

```js
const chunkSize = 2 * 1024 * 1024
```

也就是每个分片 2MB。

如果一个文件是 9MB：

```js
totalChunks = Math.ceil(file.size / chunkSize)// Math.ceil(9 / 2) = 5
```

最终会切成 5 个分片。

#### 2. 文件 hash 的作用是什么？

上传前，项目会先计算文件的 SHA-256：

```js
const fileHash = await computeFileHash(file)
```

它的作用是给文件生成一个“内容指纹”。

```
同一个文件内容相同
  ↓
fileHash 相同

文件内容发生变化
  ↓
fileHash 不同
```

所以 `fileHash` 可以用来做两件事：

```
1. 秒传
2. 断点续传
```

#### 3. 秒传的原理

秒传并不是真的“瞬间上传成功”，而是：

```
前端先计算 fileHash
  ↓
请求 /api/upload/check
  ↓
服务端根据 fileHash 查找是否已有完整文档
  ↓
如果已有，直接返回 exists: true
  ↓
前端不再上传文件，直接提示秒传完成
```

当前项目服务端在 `server/api/upload/check.post.js` 里会调用 `findDocumentByHash(body.fileHash)` 判断是否已经存在相同文件。如果存在，就返回 `exists: true` 和对应文档信息。

可以理解成：

```
用户 A 上传过 a.pdf
  ↓
系统已经保存过它的 fileHash
  ↓
用户再次上传同一个 a.pdf
  ↓
服务端发现 hash 已存在
  ↓
直接复用已有文档
```

面试可以这样说：

> 秒传的核心不是提升网络速度，而是通过文件 hash 判断服务端是否已经存在相同内容的文件。如果存在，就直接复用已有文件和向量数据，不再重复上传。

#### 4. 断点续传的原理

断点续传解决的是：

```
文件上传到一半失败了
下次不想从头传
只想传剩下的部分
```

当前项目的流程是：

```
前端计算 fileHash
  ↓
请求 /api/upload/check
  ↓
服务端查 uploads/chunks/{fileHash} 目录
  ↓
返回已经上传过的 chunkIndex
  ↓
前端跳过这些分片
  ↓
只上传缺失分片
```

比如一个文件有 5 个分片：

```
0 1 2 3 4
```

第一次上传时，已经成功上传了：

```
0 1 2
```

网络断了。

第二次重新上传时，服务端返回：

```
uploadedChunks = [0, 1, 2]
```

前端循环上传时会判断：

```js
if (uploadedChunkSet.has(index)) continue
```

所以只会继续上传：

```
3 4
```

这就是断点续传。

服务端的 `server/utils/uploadSessions.js` 会维护上传会话、分片目录、已上传分片索引和已上传字节数，`check.post.js` 会把这些信息返回给前端。

#### 5. 分片上传是怎么做的？

前端用 `file.slice(start, end)` 切片。

核心逻辑可以简化理解成：

```js
for (let index = 0; index < totalChunks; index++) {
  if (uploadedChunkSet.has(index)) continue

  const start = index * chunkSize
  const end = Math.min(start + chunkSize, file.size)
  const chunk = file.slice(start, end)

  await uploadChunkWithProgress({
    chunk,
    chunkIndex: index,
    totalChunks,
    fileHash
  })
}
```

每个分片上传时，会携带这些元信息：

```
fileHashfileName
fileSize
chunkIndex
totalChunks
chunkSize
ext
```

服务端 `server/api/upload/chunk.post.js` 接收到分片后，会把它保存成：

```
uploads/chunks/{fileHash}/{chunkIndex}.part
```

比如：

```
uploads/chunks/abc123/0.part
uploads/chunks/abc123/1.part
uploads/chunks/abc123/2.part
```

当前项目服务端会校验分片索引、文件元信息，然后把分片写入对应路径。

#### 6. 上传进度是怎么算的？

项目里不是只看当前分片的进度，而是算整个文件的总进度。

核心思路是：

```
整体已上传字节数 = 已确认上传的分片字节数 + 当前分片实时上传字节数
```

比如：

```
文件总大小：10MB
已经完成分片：4MB
当前分片已上传：1MB
```

那么当前总进度就是：

```js
currentBytes = 4MB + 1MB
percent = currentBytes / 10MB * 100
```

项目里用 `updateProgressState()` 统一维护：

```
uploadProgress
uploadedBytes
totalBytes
uploadPercent
```

所以页面组件只负责展示，不关心进度计算细节。

#### 7. 分片合并的原理

当前端所有分片都上传完后，会请求：

```
POST /api/upload/merge
```

服务端做几件事：

```
1. 读取上传会话
2. 检查分片是否齐全
3. 按 chunkIndex 从 0 到 totalChunks - 1 顺序读取分片
4. 使用 writeStream 合并成完整文件
5. 调用 parseDocument 解析文本
6. 写入文档 meta 信息
7. 删除临时分片目录
8. 删除上传会话文件
9. 返回解析后的文本给前端
```

关键点是：**分片必须按顺序合并**。

```
0.part + 1.part + 2.part + ... + n.part  ↓完整文件
```

当前项目的 `merge.post.js` 会检查缺失分片，如果还有分片没上传完，就直接报错，不允许合并。

#### 8. 为什么文件上传后还要分块？

这里要区分两个 chunk：

```
文件分片 chunk
  ↓
用于上传、断点续传、秒传

文本分块 chunk
  ↓
用于 RAG 检索、embedding、向量化
```

文件上传完成后，服务端会解析出文本。

然后前端把文本交给 Web Worker 做语义分块：

```
完整文本
  ↓
按段落切
  ↓
段落太长按句子切
  ↓
句子太长按固定长度切
  ↓
加 overlap
  ↓
得到文本 chunks
```

最后这些文本 chunks 会发给：

```
POST /api/embed
```

服务端生成 embedding，并写入 ChromaDB。`server/api/embed.post.js` 会批量调用 embedding，每批最多 25 条，然后把 ids、embeddings、metadatas、documents 存入 Chroma Collection。

### 优化
#### 文件 hash 计算可以优化

当前项目使用：

```js
const buffer = await file.arrayBuffer()
const digest = await crypto.subtle.digest('SHA-256', buffer)
```

这种方式简单，但是它会一次性把整个文件读进内存。

当前项目限制最大 20MB，所以问题不大。服务端也在 `uploadSessions.js` 里限制了最大文件大小是 20MB。

但是如果以后支持更大的文件，比如 200MB、1GB，这种方式就不太合适。

**可以优化成增量 hash**

思路是：

```
按分片读取文件
  ↓
每读一片就更新 hash
  ↓
不用一次性把整个文件放进内存
```

可以把 hash 计算也放到 Web Worker 里：

```
主线程选择文件
  ↓
把文件传给 hash worker
  ↓
worker 分片读取并计算 hash
  ↓
计算完成后返回 fileHash
```

面试可以这样说：

> 当前项目因为限制 20MB，所以直接用 `file.arrayBuffer()` 计算 SHA-256 是可以接受的。但如果后续支持更大文件，我会把 hash 计算改成分片增量计算，并放到 Web Worker 中，避免大文件读取和哈希计算阻塞主线程。

#### 分片上传可以增加并发控制

当前上传流程是：

```
上传 chunk 0
  ↓
上传 chunk 1
  ↓
上传 chunk 2
  ↓
上传 chunk 3
```

也就是串行上传。

串行上传的优点是简单、稳定、容易控制进度。

缺点是速度可能不够快。

**可以改成有限并发上传**

比如同时上传 3 个分片：

```
chunk 0   chunk 1   chunk 2
   ↓         ↓         ↓
chunk 3   chunk 4   chunk 5
```

可以用一个并发池：

```js
const concurrency = 3
const queue = chunks.filter(chunk => !uploadedChunkSet.has(chunk.index))

async function uploadWithPool(queue) {
  const workers = new Array(concurrency).fill(0).map(async () => {
    while (queue.length) {
      const chunk = queue.shift()
      await uploadChunk(chunk)
    }
  })

  await Promise.all(workers)
}
```

优化后可以提升上传速度。

但要注意：

```
并发数不能太高
否则浏览器请求过多
弱网环境下失败率可能升高
服务端压力也会变大
```

面试可以这样说：

> 当前项目是串行上传分片，逻辑稳定但速度一般。后续可以加有限并发池，比如同时上传 3 到 5 个分片，在保证服务端压力可控的情况下提升上传速度。

#### 分片上传可以增加失败重试

当前如果某个分片上传失败，整个上传流程会进入 catch。

可以优化成：

```
单个分片失败
  ↓
重试 2~3 次
  ↓
仍然失败才终止整个上传
```

示例逻辑：

```js
async function uploadChunkWithRetry(chunk, retry = 3) {
  for (let i = 0; i < retry; i++) {
    try {
      return await uploadChunk(chunk)
    } catch (err) {
      if (i === retry - 1) throw err
    }
  }
}
```

还可以加指数退避：

```
第一次失败：等待 500ms
第二次失败：等待 1000ms
第三次失败：等待 2000ms
```

这样在弱网场景下体验会更好。

#### 可以支持暂停、继续、取消上传

当前项目有断点续传，但没有明确的“暂停 / 继续 / 取消”按钮。

可以增加：

```
暂停上传
继续上传
取消上传
```

核心可以用：

```js
AbortController
```

或者当前项目里使用的是 `XMLHttpRequest`，也可以保存当前 xhr 实例：

```js
let currentXhr = null

function cancelUpload() {
  currentXhr?.abort()
}
```

状态设计可以是：

```
idle
hashing
checking
uploading
paused
merging
chunking
embedding
success
failed
canceled
```

面试可以这样说：

> 当前项目支持断点续传，但还没有显式的暂停和取消控制。后续可以维护上传任务状态，并保存当前请求控制器，用户点击暂停时 abort 当前分片请求，点击继续时重新请求 `/api/upload/check` 获取已上传分片，再继续上传缺失部分。

#### 服务端可以校验最终文件 hash

当前服务端支持可选的 `chunkHash` 校验，`chunk.post.js` 里有 `chunkHash` 判断逻辑，但前端当前上传分片时没有明显传这个字段。

可以优化成两层校验：

```
1. 每个分片上传时校验 chunkHash
2. 所有分片合并后校验 fullFileHash
```

完整流程：

```
前端计算 fileHash
  ↓
上传分片时带 chunkHash
  ↓
服务端校验每个分片是否损坏
  ↓
合并完成后重新计算完整文件 hash
  ↓
和前端 fileHash 比较
  ↓
一致才进入解析和入库
```

这样可以避免：

```
分片传输过程中损坏
分片顺序错误
服务端合并文件异常
```

#### 文本分块和向量化可以后移到服务端

当前项目的流程是：

```
服务端合并并解析文本
  ↓
把 text 返回给前端
  ↓
前端 Web Worker 分块
  ↓
前端把 chunks 再发给 /api/embed
```

这个设计的好处是：

```
前端可以控制 chunkSize / chunkOverlap
可以展示更细的处理阶段
Web Worker 避免主线程卡顿
```

但缺点是：

```
大文本要从服务端返回到前端
前端分块后又要把 chunks 发回服务端
数据来回传输一遍
```

如果文档变大，可以优化成：

```
服务端合并文件
  ↓
服务端解析文本
  ↓
服务端直接分块
  ↓
服务端直接 embedding
  ↓
前端只轮询处理状态
```

也就是把 RAG 入库变成后端任务。

面试可以这样说：

> 当前项目把文本分块放在前端 Worker 里，能体现前端性能优化能力。但从工程角度看，如果文档更大或多用户并发更多，我会考虑把解析、分块、embedding 全部放到服务端异步任务里，前端只负责上传和展示任务进度，这样可以减少文本在前后端之间的重复传输。

#### 可以增加上传任务队列

当前上传面板每次只处理一个文件：

```js
const file = e.target.files?.[0]
```

如果要支持多文件上传，可以改成：

```js
const files = Array.from(e.target.files || [])
```

然后维护任务队列：

```js
uploadTasks = [
  { file, status: 'waiting', progress: 0 },
  { file, status: 'uploading', progress: 30 },
  { file, status: 'success', progress: 100 }
]
```

队列模式可以支持：

```
多文件排队上传
单个文件失败不影响其他文件
每个文件单独展示进度
每个文件单独取消 / 重试
```

面试可以这样说：

> 当前项目偏单文件上传，适合个人知识库场景。后续如果支持批量导入文档，可以把上传状态从单个 `isUploading` 扩展成任务队列，每个任务维护自己的 hash、进度、状态和错误信息。

### 问答
#### 如何保证分片合并后的文件是正确的？

可以答：

> 首先服务端会检查分片数量是否完整，只有所有 chunkIndex 都存在时才允许合并。其次合并时必须按照 chunkIndex 从小到大顺序写入。
> 
> 更严谨的优化是：前端上传每个分片时带上 chunkHash，服务端校验每个分片；最终合并后服务端再计算完整文件 hash，和前端上传前计算的 fileHash 比较，一致才认为文件有效。

#### 如果某个分片上传失败怎么办？

可以答：

> 当前项目如果某个分片失败，会进入异常处理，提示上传失败。后续可以优化为分片级重试，比如每个分片最多重试 3 次，并配合指数退避。这样弱网情况下不会因为一次网络抖动就导致整个上传任务失败。
> 
> 如果重试后仍然失败，可以保留已经上传成功的分片。用户下次继续上传时，通过 `/api/upload/check` 拿到已上传分片列表，再继续传缺失部分。

#### 上传进度为什么不能简单用当前请求进度？

可以答：

> 因为当前项目是多个分片组成一个完整文件。单个分片的进度只能表示当前 chunk 上传了多少，不能代表整个文件进度。
> 
> 所以我会用“已完成分片的字节数 + 当前分片已上传字节数”来计算整体进度，再除以文件总大小得到百分比。这样用户看到的是整个文件级别的上传进度。


## 2. 文档解析组件
### 2.1 文件解析入口组件

> `server/api/upload/merge.post.js`

这个文件是当前分片上传流程里的 **解析入口**。

也就是说，在高级上传流程中，文件不是上传完立刻解析，而是：

```
前端上传所有分片
  ↓
服务端合并成完整文件
  ↓
读取完整文件 Buffer
  ↓
调用 parseDocument(fileBuffer, session.fileName)
  ↓
提取纯文本
```

在 `merge.post.js` 里，合并文件后会读取完整文件：

```js
const fileBuffer = await readFile(filePath)
const text = await parseDocument(fileBuffer, session.fileName)
```

然后判断是否提取到了有效文本：

```js
if (!text?.trim()) {
  throw new Error('未能从文件中提取到文本内容')
}
```

==作用==：`merge.post.js` 是 **分片上传流程中的文档解析触发点**。它负责在文件分片合并完成后，把完整文件交给解析工具 `parseDocument`，拿到纯文本后再生成文档元信息。源码里它还会写入 `docMeta`，包括 `id / name / ext / size / textLength / createdAt / fileHash / hasVectors / chunksCount` 等信息。

### 2.2 兼容旧版直传的解析入口

> `server/api/upload.post.js`

这个文件是另一个上传入口，属于 **普通直传上传接口**。

它没有走分片上传，而是直接：

```
接收 multipart 文件
  ↓
校验文件类型
  ↓
校验文件大小
  ↓
保存原始文件
  ↓
调用 parseDocument 提取文本
  ↓
写入文档元信息
  ↓
返回 text
```

里面同样会调用：

```js
const text = await parseDocument(file.data, file.filename)
```

==作用==：`upload.post.js` 是普通上传模式下的解析入口。它适合小文件或早期版本逻辑：前端一次性上传完整文件，服务端保存文件后立即解析文本。当前首页使用的是 `useDocumentsAdvanced()`，主要走分片上传链路，但这个文件仍然是项目里的一个解析入口。

### 2.3 文档解析核心模块

> `server/utils/documentParser.js`

这是文件解析组件最核心的模块。

它导出了一个函数：

```js
export async function parseDocument(buffer, filename)
```

它接收两个参数：

```
buffer：文件二进制内容
filename：文件名，用来判断文件类型
```

返回值是：

```
提取出来的纯文本 string
```

源码里根据文件后缀做分发：

```
pdf   → pdf-parse
docx  → mammoth
txt   → Buffer.toString('utf-8')
md    → Buffer.toString('utf-8')
其他  → 抛出不支持文件类型错误
```

==作用==：`documentParser.js` 是 **文件类型分发器 + 具体解析器封装层**。业务接口不需要关心 PDF 怎么解析、DOCX 怎么解析，只需要把 `buffer` 和 `filename` 传给 `parseDocument`。具体解析逻辑都集中在这个工具模块里。

### 2.4 PDF 解析子模块

> `server/utils/documentParser.js` 中的 `case 'pdf'`

项目里 PDF 解析使用的是：

```js
import pdf from 'pdf-parse'
```

对应逻辑是：

```js
case 'pdf': {
  const data = await pdf(Buffer.from(buffer))
  return data.text || ''
}
```

它做的事情是：

```
接收 PDF Buffer
  ↓
调用 pdf-parse 解析 PDF
  ↓
从 data.text 中拿到文本
  ↓
返回纯文本
```

==作用==：PDF 解析子模块负责把 PDF 文件里的文字内容提取出来，供后续文本分块和向量化使用。它的输出不是结构化页面对象，而是纯文本字符串。

需要注意：这个方案主要适合 **文本型 PDF**。如果是扫描件 PDF，里面本质上是图片，没有真实文字层，`pdf-parse` 通常提取不到有效文本。

### 2.5 DOCX 解析子模块

> `server/utils/documentParser.js` 中的 `case 'docx'`

项目里 DOCX 解析使用的是：

```js
import mammoth from 'mammoth'
```

对应逻辑是：

```js
case 'docx': {
  const result = await mammoth.extractRawText({ buffer: Buffer.from(buffer) })
  return result.value || ''
}
```

它做的事情是：

```
接收 DOCX Buffer
  ↓
调用 mammoth.extractRawText
  ↓
提取 Word 文档里的原始文本
  ↓
返回 result.value
```

==作用==：DOCX 解析子模块负责把 Word 文档里的段落文本提取出来。这里用的是 `extractRawText`，所以它更偏向“拿纯文本”，而不是保留复杂样式、图片、表格结构。

### 2.6 TXT / Markdown 解析子模块

> `server/utils/documentParser.js` 中的 `case 'txt'` 和 `case 'md'`

对应逻辑是：

```js
case 'txt':
case 'md': {
  return Buffer.from(buffer).toString('utf-8')
}
```

它做的事情很简单：

```
接收文件 Buffer
  ↓
按 utf-8 转成字符串
  ↓
直接返回文本
```

==作用==：TXT 和 Markdown 本身就是文本格式，不需要额外解析库，所以直接把 Buffer 转成 UTF-8 字符串即可。

这里要注意一个点：

```
Markdown 当前只是作为普通文本读取
并没有解析 Markdown AST
也没有单独保留标题层级、代码块、引用等结构
```

所以后续如果优化 RAG 效果，Markdown 是一个很好的优化点。

### 2.7 文件类型校验组件

文件类型校验目前出现在两个位置。

#### 2.7.1 普通直传接口中的校验

> `server/api/upload.post.js`

里面定义：

```js
const allowedExts = ['pdf', 'txt', 'md', 'docx']
```

然后判断：

```js
if (!allowedExts.includes(ext)) {
  throw createError({
    statusCode: 400,
    message: `不支持的文件类型: .${ext}`
  })
}
```

==作用==：这个校验用于普通直传模式，防止不支持的文件进入解析流程。

#### 2.7.2 分片上传流程中的校验

> `server/utils/uploadSessions.js`

分片上传流程中，文件类型校验在上传会话工具里：

```js
const ALLOWED_EXTS = ['pdf', 'txt', 'md', 'docx']
```

以及：

```js
validateUploadMeta(...)
```

==作用==：这个校验用于分片上传流程。因为分片上传分为 `check / chunk / merge` 多个接口，所以文件类型校验被抽到了 `uploadSessions.js` 里复用。服务端会限制只允许 `pdf / txt / md / docx`，并限制文件大小不能超过 20MB。

### 2.8 文档元信息生成组件

文件解析完成后，项目会生成文档元信息。

在分片上传链路里，对应位置是：

> `server/api/upload/merge.post.js`

解析完成后生成：

```js
const docMeta = {
  id: session.documentId,
  name: session.fileName,
  ext,
  size: session.fileSize,
  textLength: text.length,
  createdAt: new Date().toISOString(),
  fileHash: body.fileHash,
  hasVectors: false,
  chunksCount: 0
}
```

==作用==：这个模块负责把“一个原始文件”转成系统里的“知识库文档记录”。其中 `textLength` 表示解析出的文本长度，`hasVectors` 初始为 `false`，表示还没有完成向量化；`chunksCount` 初始为 `0`，后面 `/api/embed` 向量化成功后会更新。

### 当前解析模块组成

```
文件解析组件
  ├─ 解析入口：server/api/upload/merge.post.js
  ├─ 旧版直传入口：server/api/upload.post.js
  ├─ 核心解析器：server/utils/documentParser.js
  │   ├─ PDF 解析：pdf-parse
  │   ├─ DOCX 解析：mammoth.extractRawText
  │   └─ TXT / MD 解析：Buffer.toString('utf-8')
  ├─ 文件类型校验：upload.post.js / uploadSessions.js
  └─ 文档元信息生成：merge.post.js
```

### 面试里可以这样说

> 文件解析组件主要在服务端实现，核心模块是 `server/utils/documentParser.js`。它暴露了一个 `parseDocument(buffer, filename)` 方法，根据文件后缀做解析分发：PDF 使用 `pdf-parse` 提取文本，DOCX 使用 `mammoth.extractRawText` 提取 Word 文档中的原始文本，TXT 和 MD 直接通过 `Buffer.toString('utf-8')` 转成字符串。
> 
> 在当前分片上传流程中，解析入口在 `server/api/upload/merge.post.js`。服务端先把所有分片按顺序合并成完整文件，然后读取文件 Buffer，调用 `parseDocument(fileBuffer, session.fileName)` 提取文本。如果没有提取到有效文本，就直接返回错误；如果成功，就生成文档元信息，包括文件名、类型、大小、文本长度、fileHash、向量化状态和 chunk 数量。
> 
> 项目里还保留了一个普通直传入口 `server/api/upload.post.js`，它也会在接收完整文件后调用 `parseDocument`，但当前高级上传流程主要走的是 `check / chunk / merge` 这套分片上传链路。


### 模块完整流程

先明确一点：**文件解析模块没有独立的前端页面组件**。

它不是一个单独的 Vue 页面，而是嵌在“文档上传链路”里的一个阶段。

当前项目的解析流程是：

```
DocumentUpload.vue
  ↓ 用户选择 / 拖拽文件

  ↓ 合并文件后调用 parseDocument
server/utils/documentParser.js

  ↓ 前端拿到 text 后交给 Web Worker 分块
/api/embed
  ↓ 向量化入库
```

2. 前端流程编排：useDocumentsAdvanced()
3. 服务端解析入口：upload/merge.post.js

##### 3. 核心函数

==`uploadDocument`==

这是文件解析流程在前端的主入口。

和解析相关的部分是：

```js
uploadProgress.value = '正在合并文件并提取文本...'

2. 调用 /api/upload/merge

```

注意：

### 文件解析原理

当前项目的文件解析核心在：

> `server/utils/documentParser.js`

它对外暴露：

```js
parseDocument(buffer, filename)
```

然后根据文件后缀分发不同解析策略：

```
pdf   → pdf-parse
docx  → mammoth.extractRawText
txt   → Buffer.toString('utf-8')
md    → Buffer.toString('utf-8')
```

在分片上传链路中，解析入口是：

> `server/api/upload/merge.post.js`

也就是所有分片上传完成后，服务端先合并完整文件，再读取完整文件 Buffer，然后调用 `parseDocument(fileBuffer, session.fileName)` 提取文本。

核心解析逻辑集中在 `documentParser.js` 中，PDF 用 `pdf-parse`，DOCX 用 `mammoth`，TXT 和 MD 直接转字符串。

#### 1. 为什么要做文件解析？

RAG 系统不能直接处理原始文件。

比如用户上传的是：

```
前端面试资料.pdf
Vue笔记.docx
项目总结.md
```

但是后续的 RAG 流程需要的是：

```
纯文本 text
```

因为后面要做：

```
文本分块
  ↓
生成 embedding
  ↓
存入向量库
  ↓
用户提问时语义检索
```

所以文件解析组件的作用就是：

```
原始文件 Buffer
  ↓
parseDocument
  ↓
纯文本 text
```

可以这样理解：

> 文件解析是 RAG 入库链路中的“格式转换层”，负责把不同格式的文件统一转换成可分块、可向量化的纯文本。

#### 2. PDF 解析原理

##### 当前项目怎么做？

当前项目里 PDF 解析是：

```js
const data = await pdf(Buffer.from(buffer))return data.text || ''
```

也就是使用 `pdf-parse` 从 PDF Buffer 中提取 `data.text`。

##### PDF 解析本质是什么？

PDF 本质上不是像 HTML 那样的结构化文档。

它更像是一个“排版结果文件”。

里面通常存的是：

```
文字内容
字体信息
坐标位置
页面信息
图片资源
绘制指令
```

`pdf-parse` 做的事情大概是：

```
读取 PDF Buffer
  ↓
解析 PDF 内部对象
  ↓
找到文本对象
  ↓
按照页面顺序提取文字
  ↓
拼成 data.text
```

所以它适合解析 **文本型 PDF**。

比如：

```
电子书 PDF
论文 PDF
导出的简历 PDF
文档转出的 PDF
```

这些 PDF 内部本身有文字层，所以可以提取出来。

##### PDF 解析的局限

如果用户上传的是扫描件 PDF：

```
扫描版合同
拍照生成的 PDF
图片版论文
```

这种 PDF 里面看起来有字，但本质上是图片。

这时 `pdf-parse` 很可能提取不到文本。

也就是说：

```
文本型 PDF  → 可以提取
扫描型 PDF  → 需要 OCR
```

面试里可以主动说：

> 当前项目的 PDF 解析依赖 PDF 本身存在文字层，所以对文本型 PDF 支持较好。如果是扫描件 PDF，后续需要接 OCR，把图片中的文字识别出来后再进入分块和向量化流程。

#### 3. DOCX 解析原理

##### 当前项目怎么做？

当前项目里 DOCX 解析是：

```js
const result = await mammoth.extractRawText({
  buffer: Buffer.from(buffer)
})
return result.value || ''
```

也就是用 `mammoth.extractRawText` 提取 Word 文档中的原始文本。

##### DOCX 本质是什么？

DOCX 不是一个普通文本文件。

它本质上是一个压缩包，里面有很多 XML 文件。

可以理解成：

```
example.docx
  ├─ word/document.xml
  ├─ word/styles.xml
  ├─ word/media/*
  ├─ docProps/*
  └─ ...
```

真正的正文内容通常在：

```
word/document.xml
```

`mammoth` 做的事情是：

```
读取 DOCX Buffer
  ↓
解压内部 XML
  ↓
读取正文段落
  ↓
提取文本内容
  ↓
返回 result.value
```

当前项目用的是 `extractRawText`，所以目标是：

```
尽快拿到纯文本
```

而不是完整还原 Word 样式。

##### DOCX 解析的局限

当前这种方式不会精细保留：

```
标题层级
加粗 / 斜体
表格结构
图片
页眉页脚
批注
复杂编号结构
```

所以如果 DOCX 里有表格：

```
| 参数 | 含义 |
| ---- | ---- |
| topK | 检索条数 |
```

直接提取纯文本后，可能会变成：

```
参数 含义 topK 检索条数
```

结构信息会丢失。

面试里可以说：

> 当前 DOCX 解析使用 `mammoth.extractRawText`，优点是简单稳定，能快速拿到正文文本。但它偏纯文本提取，表格、标题层级、图片等结构化信息会丢失。后续如果要提升 RAG 效果，可以考虑把 DOCX 解析成带层级结构的 Markdown，再进入分块流程。

#### 4. TXT / Markdown 解析原理

##### 当前项目怎么做？

当前项目里 TXT 和 MD 是：

```js
return Buffer.from(buffer).toString('utf-8')
```

也就是直接把 Buffer 按 UTF-8 转字符串。

##### 为什么 TXT / MD 可以直接转？

因为 TXT 和 Markdown 本身就是文本文件。

不像 PDF / DOCX 那样需要解析复杂格式。

所以处理链路是：

```
Buffer
  ↓
toString('utf-8')
  ↓
文本内容
```

##### Markdown 当前的局限

当前项目把 Markdown 当普通文本处理。

这能用，但没有充分利用 Markdown 的结构。

Markdown 里其实天然有层级：

```
# 一级标题

## 二级标题

- 列表项

```js
const a = 1
```

```

如果只是普通字符串读取，后续分块时可能不会特别照顾：

```text
标题和正文的关系
代码块完整性
列表完整性
引用块完整性
```

更好的方式是：

```
Markdown
  ↓
解析成 AST
  ↓
按标题层级组织内容
  ↓
转成带结构的文本块
```

比如一个 chunk 可以带上标题上下文：

```
标题路径：RAG 系统 / 文本分块 / overlap
内容：overlap 的作用是减少语义断裂...
```

这样检索效果通常更好。

### 优化

#### 不要只靠文件后缀判断类型

当前解析分发主要依赖：

```js
const ext = filename.split('.').pop().toLowerCase()
```

问题是文件后缀可以伪造。

比如用户把一个图片改名成：

```
test.pdf
```

后端会认为它是 PDF，然后尝试解析，最终可能报错。

更好的方式是增加 **文件魔数校验**。

比如：

```
PDF 文件通常以 %PDF 开头
DOCX 本质是 zip，通常以 PK 开头
```

优化后可以变成：

```
先看后缀
  ↓
再看 MIME / magic number
  ↓
两者一致才进入解析
```

面试可以这样说：

> 当前项目主要通过文件后缀判断类型，这对个人项目够用，但生产环境不够严谨。后续可以增加 magic number 校验，防止用户伪造后缀导致解析异常或安全风险。

#### 增加文本清洗 normalize

当前 `parseDocument` 返回的是原始提取文本。

但是 PDF / DOCX 提取出来的文本可能会有很多问题：

```
多余空行
页眉页脚
页码
断行
连续空格
不可见字符
乱码字符
```

可以在解析后统一做一层清洗：

```js
function normalizeText(text) {
  return text
    .replace(/\r\n/g, '\n')
    .replace(/\n{3,}/g, '\n\n')
    .replace(/[ \t]{2,}/g, ' ')
    .trim()
}
```

这样后续分块质量会更好。

面试可以说：

> 文件解析不只是把文本拿出来，还要尽量保证文本质量。后续我会在解析后增加 normalizeText，比如统一换行、去除多余空白、清理页码和页眉页脚，减少无效内容进入向量库。

#### PDF 按页解析并保留页码

当前 PDF 解析返回的是一个整体字符串。

问题是：

```
后续召回 chunk 时，无法知道它来自 PDF 第几页
```

更好的做法是保留 metadata：

```json
{
  text: 'Vue3 使用 Proxy 实现响应式...',
  metadata: {
    fileName: 'Vue面试.pdf',
    page: 12
  }
}
```

这样问答时可以展示来源：

```
参考来源：Vue面试.pdf 第 12 页
```

这对 RAG 项目很加分。

面试可以说：

> 当前项目解析后只返回纯文本，metadata 比较少。后续可以把 PDF 按页解析，并把 pageNumber 写入 chunk metadata。这样检索结果不仅能回答，还能展示来源页码，提升可解释性和可信度。

#### Markdown 结构化解析

当前 Markdown 是普通字符串读取。

可以优化成：

```
Markdown
  ↓
marked / markdown-it 解析
  ↓
识别标题、列表、代码块、引用
  ↓
按标题层级组织 chunk
```

比如：

```
# Vue
## 响应式
Vue3 使用 Proxy...
```

可以转换成：

```
{
  titlePath: ['Vue', '响应式'],
  content: 'Vue3 使用 Proxy...',
}
```

好处是：

```
chunk 语义更完整
标题上下文不容易丢
检索结果更准确
```

面试可以说：

> Markdown 本身有天然结构，后续可以不再简单 `toString`，而是解析 Markdown AST，按标题层级组织 chunk，并把标题路径作为 metadata 或拼入 chunk 内容，提高召回质量。

#### DOCX 表格结构处理

当前 DOCX 用 `extractRawText`，会丢失表格结构。

如果文档里有大量表格，比如：

```
接口字段说明
配置参数说明
实验结果对比
```

纯文本提取后效果可能不好。

可以优化成：

```
DOCX
  ↓
提取段落
  ↓
提取表格
  ↓
表格转 Markdown table
  ↓
再进入分块
```

例如把表格转成：

```
| 参数 | 含义 |
| --- | --- |
| topK | 检索召回条数 |
| temperature | 模型随机性 |
```

这样向量化后的语义更清楚。

#### 扫描版 PDF 接 OCR

当前 `pdf-parse` 只能处理文本层。

如果遇到扫描件，需要 OCR。

优化链路可以是：

```
pdf-parse 提取文本
  ↓
如果 text 为空或过短
  ↓
判断是否扫描件
  ↓
将页面转图片
  ↓
调用 OCR
  ↓
得到文本
```

可以用：

```
Tesseract
PaddleOCR
云 OCR 服务
```

面试可以说：

> 当前项目对文本型 PDF 支持较好，但扫描件 PDF 无法解析。后续可以增加 OCR fallback：如果 `pdf-parse` 提取文本为空，就将 PDF 页面转成图片，再通过 OCR 提取文字。

#### 解析任务异步化

当前 `/api/upload/merge` 做了：

```
合并文件
  ↓
解析文件
  ↓
返回 text
```

如果文件变大，解析时间会变长，请求可能阻塞。

可以优化成异步任务：

```
上传完成
  ↓
创建 parsing job
  ↓
立即返回 documentId
  ↓
服务端后台解析
  ↓
前端轮询 / SSE 监听状态
  ↓
解析完成后继续分块和向量化
```

文档状态可以变成：

```
uploaded
parsing
parsed
chunking
embedding
indexed
failed
```

面试可以说：

> 当前项目是同步解析，个人项目没问题。但如果文件更大或并发更多，我会把解析改成异步任务，前端只展示处理状态，服务端后台完成解析、分块和向量化，避免单个请求耗时过长。

#### 解析缓存

同一个 `fileHash` 的文件如果已经解析过，就没必要再次解析。

当前项目已经通过 `fileHash` 做秒传。

可以进一步把解析结果也缓存：

```
fileHash
  ↓
parsedText / parsedBlocks
```

下次同文件上传时：

```
命中 fileHash
  ↓
直接复用解析结果和向量数据
```

当前秒传已经有类似思路，但可以把缓存设计得更明确。

### 问答
#### 文件解析在 RAG 链路中处于什么位置？

可以答：

> 文件解析位于上传之后、文本分块之前。用户上传的是 PDF、DOCX、TXT、MD 等原始文件，但 RAG 后续需要处理的是纯文本。所以服务端会先把文件解析成 text，然后前端再把 text 交给 Web Worker 做分块，最后送到 embedding 接口生成向量。

---

#### 为什么要在服务端解析，而不是前端解析？

可以答：

> 服务端解析更稳定，也更安全。PDF、DOCX 这类格式解析依赖 Node 生态库，比如 `pdf-parse` 和 `mammoth`，放在服务端更方便处理 Buffer、文件存储和异常。
> 
> 另外，服务端解析可以统一控制文件大小、文件类型和安全校验，避免把复杂解析逻辑和大文件处理压力放到浏览器端。前端只负责上传和展示处理状态。

#### PDF 解析为什么有时候提取不到文字？

可以答：

> 因为有些 PDF 是扫描件，里面没有真实文字层，本质上只是图片。`pdf-parse` 只能提取 PDF 中已有的文本对象，不能识别图片里的文字。所以扫描版 PDF 需要接 OCR，把页面图片转成文本后再进入分块流程。

#### DOCX 解析为什么用 mammoth？

可以答：

> DOCX 本质是一个压缩包，正文在内部 XML 中。`mammoth` 可以读取 DOCX Buffer 并提取正文文本。当前项目用的是 `extractRawText`，目标是快速拿到纯文本，供后续分块和向量化使用。
> 
> 不过它不会完整保留样式、表格、图片等结构，所以如果文档里表格很多，后续可以把 DOCX 转成 Markdown 或结构化 blocks。

#### Markdown 为什么直接 `toString`，这样好吗？

可以答：

> 当前直接 `Buffer.toString('utf-8')` 是因为 Markdown 本身就是文本格式，这样实现简单，能快速进入分块流程。
> 
> 但从 RAG 效果看，Markdown 可以优化。因为 Markdown 有标题、列表、代码块等结构，后续可以解析 Markdown AST，按标题层级切分，并把 headingPath 作为 metadata，这样 chunk 的语义会更完整。

#### 解析后的文本为空怎么办？

可以答：

> 当前项目在 `merge.post.js` 中会校验 `text?.trim()`，如果为空就抛出“未能从文件中提取到文本内容”。这样可以避免空文本继续进入分块和向量化流程。
> 
> 后续可以把错误信息做得更细，比如区分“扫描件 PDF 需要 OCR”“文件损坏”“编码不支持”“文档为空”等，让用户知道具体原因。

#### 解析结果为什么不能直接拿去问大模型？

可以答：

> 因为文档可能很长，直接把全文塞给大模型会超过上下文长度，也会浪费 token。RAG 的做法是先把解析出来的文本切成多个 chunk，然后对 chunk 做 embedding 存入向量库。用户提问时，只召回相关 chunk，再把相关上下文交给大模型回答。

#### 解析和分块有什么区别？

可以答：

> 解析是把不同格式的文件统一转成文本，比如 PDF 转 text、DOCX 转 text。
> 
> 分块是把解析后的长文本切成多个小文本片段，比如按段落、句子、固定长度切分，并设置 overlap。
> 
> 所以解析解决的是“文件格式转换”问题，分块解决的是“长文本如何适配向量检索和模型上下文”问题。

#### 如果一个 PDF 有页眉页脚，会影响 RAG 吗？

可以答：

> 会影响。页眉、页脚、页码这类内容如果反复出现在每一页，会进入很多 chunk，导致向量库里出现大量噪声。用户检索时可能召回无效内容。
> 
> 所以后续可以在解析后增加文本清洗，比如识别重复页眉页脚、页码、连续空行和特殊字符，提升 chunk 质量。

#### 怎么提升解析后的检索效果？

可以答：

> 可以从三个方向优化。
> 
> 第一是保留结构，比如 PDF 保留页码，Markdown 保留标题路径，DOCX 表格转 Markdown table。
> 
> 第二是做文本清洗，比如去掉页眉页脚、空白字符、乱码和重复内容。
> 
> 第三是增强 metadata，比如每个 chunk 记录 documentId、fileName、pageNumber、headingPath、chunkIndex。这样检索结果不仅能召回内容，还能展示来源。

#### 文件类型校验只看后缀安全吗？

可以答：

> 不够安全。后缀可以伪造，比如把图片改成 `.pdf`。当前项目对个人项目来说够用，但生产环境应该增加 MIME 或 magic number 校验。比如 PDF 通常以 `%PDF` 开头，DOCX 本质是 zip，通常以 `PK` 开头。这样可以避免伪造后缀导致解析异常或安全问题。

#### 如果要支持更多格式怎么扩展？

可以答：

> 当前 `documentParser.js` 是根据 ext 做 switch 分发。要支持新格式，可以继续增加 case，比如 `html`、`csv`、`xlsx`、`pptx`。
> 
> 如果格式越来越多，可以把 switch 优化成策略映射，比如：
> 
> ```
> const parsers = {  pdf: parsePdf,  docx: parseDocx,  md: parseMarkdown,  txt: parseTxt}
> ```
> 
> 这样新增格式只需要注册新的 parser，结构会更清晰。


## 3. 文本分块组件

```
文本分块组件
  ├─ 调度层：app/composables/useDocuments.js
  │   ├─ chunkTextWithWorker
  │   └─ fallbackChunkAdvanced
  │
  ├─ 执行层：public/workers/textProcessor.worker.js
  │   ├─ self.onmessage
  │   ├─ recursiveChunk
  │   ├─ chunkBySentences
  │   └─ chunkByFixedSize
  │
  └─ 配置来源：/api/settings
      ├─ chunkSize
      └─ chunkOverlap
```

文本分块这一块没有独立页面组件，它属于 **文档上传后的文本处理模块**。当前主要由两个文件承担：

```
app/composables/useDocuments.js
public/workers/textProcessor.worker.js
```

前者负责调度，后者负责真正分块。

### 3.1 文本分块调度模块

>文件：`app/composables/useDocuments.js`

- 位于前端业务编排层
- 在文件解析完成后接收服务端返回的 `result.text`
- 请求 `/api/settings` 获取分块配置
- 调用 `chunkTextWithWorker`
- 把长文本传给 Web Worker
- 接收 Worker 返回的 `chunks`
- 把 `chunks` 继续传给 `/api/embed` 做向量化
- 在 Worker 不可用时，使用降级分块函数

==作用==：这个模块不是具体执行分块算法的地方，而是 **文本分块流程的调度层**。它负责决定什么时候开始分块、使用什么分块参数、分块完成后把结果交给谁。当前代码里是在上传合并并解析出 `text` 后，读取 `settings.chunkSize` 和 `settings.chunkOverlap`，然后调用 Worker 处理文本。

### 3.2 Web Worker 分块执行组件

>文件：`public/workers/textProcessor.worker.js`

- 位于浏览器 Worker 线程
- 接收主线程传来的文本和配置
- 根据 `type: 'chunk'` 判断任务类型
- 调用 `recursiveChunk` 执行递归分块
- 分块成功后返回 `{ type: 'chunks', payload: chunks }`
- 分块失败后返回 `{ type: 'error', payload: error.message }`
- 避免大文本处理阻塞页面主线程

==作用==：这是文本分块真正执行的地方。因为解析后的文档文本可能比较长，如果直接在主线程切分，会影响页面交互，所以项目把这部分逻辑放到了 Web Worker 中。Worker 接收文本后，在后台线程完成段落、句子、固定长度的递归切分，再把结果返回给主线程。

### 3.2 递归分块策略模块

>文件：`public/workers/textProcessor.worker.js`

- 负责文本分块的主算法
- 先清理连续空行
- 优先按段落切分
- 如果段落过长，再按句子切分
- 如果句子仍然过长，最后按固定长度切分
- 支持 `chunkOverlap`
- 返回最终的文本块数组

==作用==：这是项目里最核心的分块策略。它不是简单地每隔固定长度截断，而是优先保证语义完整：能按段落切就按段落切，段落太长再按句子，句子还太长才用固定长度兜底。这样做比纯固定长度切分更适合 RAG，因为 chunk 的语义边界更自然。

### 3.4 句子级分块模块

>文件：`public/workers/textProcessor.worker.js`

- 负责处理超长段落
- 根据中文和英文句末符号切句
- 支持 `。！？.!?` 等标点
- 尽量把多个句子合并到一个 chunk 中
- 如果单个句子仍然超过 `chunkSize`，继续交给固定长度分块兜底

==作用==：这个模块是递归分块里的第二层策略。它解决的是“某个段落太长，但直接固定长度切又容易切断语义”的问题。通过句子级切分，可以让每个 chunk 尽量保留完整句子。

### 3.5 固定长度兜底分块模块

文件：`public/workers/textProcessor.worker.js`

- 负责最终兜底切分
- 按固定 `size` 截取文本
- 支持 `overlap`
- 使用 `step = size - overlap` 控制滑动窗口
- 保证无论文本多长，都能被切成可处理的 chunk

==作用==：这是最后的保底策略。当前面的段落切分、句子切分都无法处理超长文本时，就用固定长度强制切开。它保证分块流程一定能产出结果，不会因为单段或单句过长导致后续 embedding 无法继续。

### 3.6 分块参数来源模块

文件：服务端设置接口，前端通过 `$fetch('/api/settings')` 获取

- 提供 `chunkSize`
- 提供 `chunkOverlap`
- 支持用户在设置页调整分块参数
- 让分块策略不是写死在 Worker 里
- 上传时读取最新配置后再传给 Worker

==作用==：它是文本分块的配置来源。当前分块大小和重叠长度不是完全写死，而是上传流程中动态读取设置，再传给 Worker。这样用户可以根据文档类型调整分块策略，比如长文档可以调大 `chunkSize`，需要更多上下文连续性时可以调大 `chunkOverlap`。

### 3.7 Worker 降级分块模块

文件：`app/composables/useDocuments.js`

- 当浏览器不支持 Worker 或 Worker 初始化失败时使用
- 在主线程按固定长度简单切分
- 不支持复杂的段落、句子递归策略
- 保证极端情况下流程仍然可以继续

==作用==：这是容错模块。正常情况下项目会优先使用 Worker 分块，但如果 Worker 不可用，就退回到主线程固定长度分块。它的效果没有 Worker 里的递归分块好，但可以保证上传入库流程不中断。

### 面试里回答

> 文本分块模块主要由 `useDocuments.js` 和 `textProcessor.worker.js` 组成。`useDocuments.js` 负责在文件解析出 text 后读取分块配置，并调用 `chunkTextWithWorker` 把文本交给 Worker；`textProcessor.worker.js` 负责真正执行分块算法。它采用递归策略，优先按段落切分，段落过长再按句子切分，句子仍然过长时按固定长度兜底，并通过 overlap 保留相邻 chunk 的上下文连续性。

### 模块完整流程
#### 文本分块调度模块

>文件：`app/composables/useDocuments.js`

这个模块不直接执行复杂分块算法，而是负责在合适的时机调用 Worker。

在当前高级上传流程里，文件完成分片上传后，会请求：

```
POST /api/upload/merge
```

服务端合并并解析文件后返回：

```
result.text
```

前端拿到 `result.text` 后，才进入文本分块阶段。

##### 核心变量

==result.text==

这是文件解析后得到的完整文本。

代码里会先判断它是否有效：

```js
if (!result.text || !result.text.trim()) {
  throw new Error('文件中未提取到有效文本')
}
```

只有解析出了有效文本，才会继续分块。

==uploadProgress==

用于更新上传面板中的处理状态。

进入文本分块阶段时，会设置为：

```js
uploadProgress.value = '正在分块处理...'
```

分块完成后进入向量化阶段，会设置为：

```js
uploadProgress.value = `正在生成向量 (${chunks.length} 个文本块)...`
```

所以用户看到的“正在分块处理”，本质上就是这个变量驱动的。

==settings==

通过：

```js
const settings = await $fetch('/api/settings')
```

读取系统配置。

这里主要用到两个参数：

```js
settings.chunkSize
settings.chunkOverlap
```

如果没有配置，则使用默认值：

```js
chunkSize: settings.chunkSize || 500
chunkOverlap: settings.chunkOverlap || 50
```

这说明分块参数不是写死在 Worker 里的，而是可以通过设置模块动态调整。

==chunks==

这是 Worker 返回的文本块数组。

```js
const chunks = await chunkTextWithWorker(result.text, {
  chunkSize: settings.chunkSize || 500,
  chunkOverlap: settings.chunkOverlap || 50
})
```

分块完成后会校验：

```js
if (!chunks.length) {
  throw new Error('文本分块结果为空')
}
```

只有 `chunks` 不为空，才会继续调用 `/api/embed`。

==worker==

这是 Web Worker 实例。

```js
let worker = null
```

它通过 `initWorker()` 创建，指向：

```
/workers/textProcessor.worker.js
```

作用是把大文本分块任务放到后台线程执行，避免阻塞主线程。

##### 核心函数

==uploadDocument(file)==

这是整个上传入库流程的主函数。

和文本分块相关的部分可以简化成：

```

  ↓

  ↓

  ↓

  ↓

  ↓

  ↓

```

也就是说，`uploadDocument` 不负责具体怎么切文本，它只是把“文本分块”这个阶段串进完整上传流程里。

==chunkTextWithWorker(text, options)==

这是文本分块调度的核心函数。

它主要做六件事：

```
1. 调用 initWorker 初始化 Worker
2. 判断 Worker 是否可用
3. Worker 不可用时走 fallbackChunkAdvanced
4. 设置 60 秒超时保护
5. 监听 worker.onmessage 接收结果
6. 通过 worker.postMessage 把任务发送给 Worker
```

主线程发送给 Worker 的数据结构是：

```js
worker.postMessage({
  type: 'chunk',
  payload: {
    text,
    chunkSize: options.chunkSize,
    chunkOverlap: options.chunkOverlap
  }
})
```

这里的重点是：

```
主线程不直接调用 recursiveChunk
而是通过 postMessage 把任务交给 Worker
```

==initWorker()==

用于创建 Worker。

```js
if (typeof Worker !== 'undefined' && !worker) {
  worker = new Worker('/workers/textProcessor.worker.js')
}
```

它有两个判断：

```
当前环境支持 Worker
当前还没有创建过 Worker
```

这样可以避免重复创建多个 Worker。

==terminateWorker()==

用于销毁 Worker。

```
worker.terminate()
worker = null
```

这个函数通常在页面卸载时调用，避免 Worker 一直常驻，占用浏览器资源。

==fallbackChunkAdvanced(text, size)==

这是 Worker 不可用时的降级方案。

它直接按固定长度切文本：

```js
for (let i = 0; i < text.length; i += size) {
  const chunk = text.slice(i, i + size).trim()
  if (chunk) chunks.push(chunk)
}
```

它没有段落优先、句子优先、overlap 这些策略，只是保证在 Worker 不可用时，文档仍然可以继续入库。

`useDocuments.js` 在文本分块链路里承担的是 **调度层** 角色。

#### Web Worker 分块执行模块

>文件：`public/workers/textProcessor.worker.js`

这是文本分块真正执行的地方。

它运行在浏览器的 Worker 线程中，和主线程分离。

主线程发送任务：

```js
worker.postMessage(...)
```

Worker 接收任务：

```js
self.onmessage
```

Worker 分块完成后返回：

```js
self.postMessage({ type: 'chunks', payload: chunks })
```

##### 核心变量

==type==

来自主线程发送的消息。

```js
const { type, payload } = data
```

当前主要支持一种任务类型：

```
chunk
```

如果不是 `chunk`，Worker 会返回错误：

```js
self.postMessage({
  type: 'error',
  payload: '未知的任务类型: ' + type
})
```

==payload==

主线程传给 Worker 的任务数据。

它包含：

```
text
chunkSize
chunkOverlap
```

其中：

```
text：完整文档文本
chunkSize：目标分块长度
chunkOverlap：相邻分块重叠长度
```

==chunks==

这是 Worker 执行分块后得到的结果。

```js
const chunks = recursiveChunk(payload.text, {
  chunkSize: payload.chunkSize || 500,
  chunkOverlap: payload.chunkOverlap || 50
})
```

最后返回给主线程：

```js
self.postMessage({ type: 'chunks', payload: chunks })
```

##### 核心函数

==self.onmessage==

这是 Worker 的入口函数。

它负责接收主线程任务，判断任务类型，然后执行对应逻辑。

核心流程是：

```
接收 data
  ↓
取出 type 和 payload
  ↓
如果 type === 'chunk'
  ↓
调用 recursiveChunk
  ↓
返回 chunks
```

如果分块过程中报错，会返回：

```js
self.postMessage({  type: 'error',  payload: error.message})
```

主线程收到 `type: 'error'` 后，会 reject 当前 Promise。

==recursiveChunk(text, options)==

这是文本分块的主算法。

它采用递归式策略：

```
先按段落切
  ↓
段落太长按句子切
  ↓
句子太长按固定长度切
  ↓
最后应用 overlap
```

它不是简单粗暴地每 500 个字符切一次，而是尽量保证每个 chunk 的语义完整。

==chunkBySentences(text, chunkSize)==

这是句子级分块函数。

当某个段落超过 `chunkSize` 时，会进入这个函数。

它会用中英文句末符号切分：

```js
const sentenceEnders = /(?<=[。！？.!?\n])\s*/g
```

支持的句子边界包括：

```
。 ！ ？ . ! ? 换行
```

它的作用是：在段落太长时，尽量按句子边界切，避免直接从句子中间截断。

==chunkByFixedSize(text, size, overlap)==

这是最终兜底函数。

当文本没有段落、没有句号，或者单句过长时，就按固定长度切。

核心逻辑是：

```js
const step = Math.max(size - overlap, 1)

for (let i = 0; i < text.length; i += step) {
  chunks.push(text.slice(i, i + size))
}
```

这里的 `step` 很关键。

比如：

```js
size = 500
overlap = 50step = 450
```

切分结果就是：

```
第 1 块：0 - 500
第 2 块：450 - 950
第 3 块：900 - 1400
```

相邻 chunk 会有 50 个字符重叠。

##### 在链路中的作用

`textProcessor.worker.js` 是 **文本分块执行层**。

它负责：

```
接收 text / chunkSize / chunkOverlap
  ↓
执行 recursiveChunk
  ↓
生成 chunks
  ↓
返回给主线程
```

面试里可以这样说：

> `textProcessor.worker.js` 是文本分块的执行层。主线程把解析后的长文本、分块大小和重叠长度通过 `postMessage` 发给 Worker，Worker 在后台线程里执行 `recursiveChunk`。分块完成后，再通过 `self.postMessage` 把 `chunks` 返回给主线程。这样可以避免大文本处理阻塞页面 UI。

#### 递归分块主算法模块

文件：`public/workers/textProcessor.worker.js`

这是整个文本分块模块中最核心的算法。

它决定 chunk 的语义质量。

##### 核心变量

==cleanedText==

清理后的文本。

```js
const cleanedText = text.replace(/\n{3,}/g, '\n\n').trim()
```

它会把连续 3 个以上换行压缩成 2 个换行。

这样后续按段落切分时，不会因为多余空行产生很多空段落。

==paragraphs==

按段落切出来的数组。

```js
const paragraphs = cleanedText.split(/\n\s*\n/)
```

这是第一层分块依据。

==rawChunks==

初步分块结果。

段落分块和句子分块产生的结果都会先放到这里。

==currentChunk==

当前正在累积的 chunk。

算法会不断尝试把新的段落拼进去：

```
拼接后不超过 chunkSize
  ↓
继续累积

拼接后超过 chunkSize
  ↓
保存当前 chunk
  ↓
重新开始下一个 chunk
```

##### 核心函数

==recursiveChunk(text, options)==

它的完整逻辑可以拆成五步。

第一步，空文本兜底：

```js
if (!text || text.trim().length === 0) return []
```

第二步，清理多余空行：

```js
const cleanedText = text.replace(/\n{3,}/g, '\n\n').trim()
```

第三步，按段落切分并累积：

```
遍历 paragraphs
  ↓
把当前段落和 currentChunk 拼接
  ↓
如果 combined.length <= chunkSize
      currentChunk = combined
  ↓
否则保存 currentChunk
```

第四步，处理超长段落：

```
如果单个段落 trimmedPara.length > chunkSize
  ↓
调用 chunkBySentences
```

第五步，应用 overlap：

```js
const prevChunk = rawChunks[i - 1]
const overlap = prevChunk.slice(-chunkOverlap)
overlapped.push(overlap + rawChunks[i])
```

这一步会把上一个 chunk 末尾的一段内容拼到当前 chunk 前面。
##### 在链路中的作用

`recursiveChunk` 是文本分块质量的核心。

它决定：

```
chunk 是否尽量语义完整
chunk 是否控制在合理长度内
chunk 之间是否保留上下文
后续 embedding 的文本质量
检索召回的准确性
```

面试里可以这样说：

> 文本分块算法不是简单按固定长度截断，而是递归处理。它先按段落切，因为段落通常是自然语义单位；如果段落太长，再按句子切；如果句子仍然太长，最后按固定长度兜底。这样可以尽量保证 chunk 的语义完整性，同时通过 overlap 保留上下文连续性。

#### 句子级分块模块

>文件：`public/workers/textProcessor.worker.js`
##### 核心变量

==sentenceEnders==

句子分隔正则。

```js
const sentenceEnders = /(?<=[。！？.!?\n])\s*/g
```

它用来识别中文和英文句子结束位置。

==sentences==

按句子拆出来的数组。

```js
const sentences = text.split(sentenceEnders).filter(s => s.trim())
```

==current==

当前正在累积的句子块。

多个句子会尽量合并到同一个 `current` 中，只要不超过 `chunkSize`。

==chunks==

句子级分块结果。

##### 核心函数

==chunkBySentences(text, chunkSize)==

它的逻辑是：

```
遍历 sentences
  ↓
尝试把当前句子拼到 current
  ↓
如果 combined.length <= chunkSize
      current = combined
  ↓
否则保存 current
      current = sent
```

如果某个句子本身就超过 `chunkSize`：

```js
if (current.length > chunkSize) {
  const forced = chunkByFixedSize(current, chunkSize, 0)
}
```

就会调用固定长度分块强制切开。

##### 在链路中的作用

`chunkBySentences` 是第二层分块策略。

它解决的是：

```
段落太长但直接按固定长度切会破坏句子完整性
```

所以它先尽量按句子边界切，让 chunk 更自然。

#### 固定长度兜底模块

>文件：`public/workers/textProcessor.worker.js`

##### 核心变量

==size==

固定切分长度。

通常等于 `chunkSize`。


==overlap==

重叠长度。

在 `recursiveChunk` 最后的 overlap 处理中，会使用用户配置的 `chunkOverlap`。

在某些强制分割场景下，代码会传入 `0`，避免过度重复。

==step==

滑动步长。

```js
const step = Math.max(size - overlap, 1)
```

如果：

```js
size = 500
overlap = 50
```

那么：

```js
step = 450
```

##### 核心函数

==chunkByFixedSize(text, size, overlap)==

它是最底层的兜底分块方式。

逻辑是：

```
从 i = 0 开始
  ↓
取 text.slice(i, i + size)
  ↓
i += step
  ↓
直到文本结束
```

它保证无论文本结构多差，最终都能切出 chunk。

##### 在链路中的作用

这是整个文本分块算法的保底机制。

比如遇到：

```
没有段落
没有标点
一整段超长代码
一整段连续文本
```

段落切分和句子切分都不适合时，就会由它强制切分。

逻辑是固定长度切分：

### 面试版总结

> 文本分块模块发生在文件解析之后、向量化之前。当前项目在 `useDocumentsAdvanced()` 中拿到 `/api/upload/merge` 返回的 `result.text` 后，会先读取 `/api/settings` 获取 `chunkSize` 和 `chunkOverlap`，然后调用 `chunkTextWithWorker`，通过 `worker.postMessage` 把文本和配置发送给 `public/workers/textProcessor.worker.js`。
> 
> Worker 接收到 `type: 'chunk'` 的任务后，会调用 `recursiveChunk` 执行递归分块。这个算法先清理多余空行，再优先按段落切；如果段落超过 `chunkSize`，就按句子切；如果句子仍然过长，最后按固定长度兜底。分块完成后，如果配置了 `chunkOverlap`，会把上一个 chunk 尾部的一段内容拼到当前 chunk 前面，减少上下文断裂。
> 
> 最后 Worker 通过 `postMessage` 把 `chunks` 返回主线程。主线程收到后校验 `chunks.length`，再把这些文本块传给 `/api/embed` 做向量化入库。这个设计的价值是：既保证 chunk 尽量语义完整，又把大文本处理放到 Worker 后台线程，避免阻塞页面主线程。

### 原理

#### 为什么 RAG 需要文本分块？

RAG 不能直接把整篇文档拿去做 embedding。

原因有几个：

```
1. 文档可能很长，超过 embedding 模型的输入限制
2. 整篇文档向量化后语义太粗，检索不精准
3. 用户提问通常只对应文档中的某一小段内容
4. 大模型最终回答时也不应该塞入整篇文档
```

所以要把：

```
一整篇文档 text
```

切成：

```
chunk 1
chunk 2
chunk 3
...
chunk n
```

后续每个 chunk 单独生成 embedding。

用户提问时，也是拿“问题向量”去和这些 chunk 的向量做相似度检索。

所以文本分块质量会直接影响：

```
召回是否准确
上下文是否完整
回答是否容易幻觉
向量库是否有大量噪声
```

#### 为什么不用简单固定长度切分？

最简单的分块方式是：

```
每 500 个字符切一块
```

比如：

```
0 - 500
500 - 1000
1000 - 1500
```

这种方式实现简单，但问题很明显：**容易从语义中间切断**。

例如：

```
Vue3 的响应式核心是通过 Proxy 拦截对象的 get 和 set 操作...
```

如果刚好切成：

```
chunk 1：Vue3 的响应式核心是通过 Proxy 拦截对象的
chunk 2：get 和 set 操作...
```

那两个 chunk 单独看都不完整。

所以你项目采用的是递归分块：

```
优先按段落切
  ↓
段落太长再按句子切
  ↓
句子还太长最后按固定长度切
```

也就是：

```
能保留大语义单位，就不要过早打碎
实在太长，再逐级降级
```

#### 按段落切分的原理

Worker 里首先会清理多余空行：

```js
const cleanedText = text.replace(/\n{3,}/g, '\n\n').trim()
```

然后按段落切：

```js
const paragraphs = cleanedText.split(/\n\s*\n/)
```

也就是说，两个换行附近会被认为是段落边界。

段落一般是自然语义单位，比如：

```
第一段：介绍 RAG 是什么
第二段：介绍 embedding 的作用
第三段：介绍向量检索流程
```

如果一个段落加到当前 chunk 后没有超过 `chunkSize`，就继续累积。

如果超过了，就把当前 chunk 保存，然后处理新的段落。

这里的核心变量是：

==currentChunk==：当前正在拼接的文本块。

==rawChunks==：初步切出来的文本块数组。

==chunkSize==：单个 chunk 的目标长度。

#### 按句子切分的原理

如果某个段落本身就超过 `chunkSize`，那就不能直接作为一个 chunk。

这时会调用：

==chunkBySentences(text, chunkSize)==

它用这个正则识别句子边界：

```js
const sentenceEnders = /(?<=[。！？.!?\n])\s*/g
```

也就是支持：

```
中文句号：。
中文感叹号：！
中文问号：？
英文句号：.
英文感叹号：!
英文问号：?
换行符
```

这样就可以把长段落拆成多个句子，再尽量把多个句子合并成一个不超过 `chunkSize` 的 chunk。

它解决的是：

```
段落太长
但不想直接从句子中间切断
```

#### overlap 的原理

`chunkOverlap` 是相邻 chunk 之间的重叠内容。

为什么需要 overlap？

因为即使按段落和句子切，也还是可能出现语义依赖跨 chunk 的情况。

例如：

```
上一段介绍问题背景
下一段介绍解决方案
```

如果两个 chunk 完全没有重叠，检索时可能只召回下一段，丢掉背景。

所以 overlap 会把上一个 chunk 末尾的一部分拼到当前 chunk 前面。

你项目里是这样做的：

```js
const prevChunk = rawChunks[i - 1]
const overlap = prevChunk.slice(-chunkOverlap)
overlapped.push(overlap + rawChunks[i])
```

这样可以让相邻文本块之间有一点上下文衔接。

面试里可以这样说：

> overlap 的作用是降低分块边界带来的语义断裂。因为用户问题可能刚好命中文档中跨段落的内容，如果两个 chunk 完全独立，召回时可能缺少前后文。通过让相邻 chunk 保留一部分重复内容，可以提升召回上下文的完整性。


### 优化

#### overlap 拼接方式可以更精细

当前实现：

```js
const overlap = prevChunk.slice(-chunkOverlap)
overlapped.push(overlap + rawChunks[i])
```

当前问题是：

```
1. overlap 和当前 chunk 之间没有分隔符
2. 可能把半句话、半个词直接拼到当前 chunk 前面
3. 拼接后 chunk 长度可能超过 chunkSize
4. 重复内容变多，可能增加 embedding 成本
```

比如上一个 chunk 结尾是：

```
...响应式系统的核心是依赖
```

当前 chunk 开头是：

```
收集和派发更新
```

拼起来可能变成：

```
核心是依赖收集和派发更新
```

这次刚好不错，但如果切在词中间，就可能很怪。

可以怎么改：

```
优先按句子或段落生成 overlap
而不是简单取最后 N 个字符
```

比如：

```
从上一个 chunk 中取最后 1~2 个完整句子作为 overlap
```

面试可以这样说：

> 当前 overlap 是按字符截取的，简单有效，但可能截断句子。后续可以优化成语义 overlap，比如取上一个 chunk 的最后一个完整句子或最后一个段落片段，这样既能保留上下文，又能减少无意义重复。

#### 分块结果可以携带 metadata

当前 Worker 返回的是：

```js
chunks = ['文本块1', '文本块2', ...]
```

也就是纯字符串数组。

问题是后续向量化时，chunk 本身缺少更丰富的信息。

可以优化成：

```js
[
  {
    content: '文本块内容',
    chunkIndex: 0,
    startOffset: 0,
    endOffset: 500,
    strategy: 'paragraph',
    overlapFromPrevious: false
  }
]
```

如果前面文件解析阶段能提供页码、标题路径，还可以带上：

```js
{
  page: 3,
  headingPath: ['RAG', '文本分块'],
  documentId: 'xxx'
}
```

好处是：

```
检索结果可以展示来源
删除文档时更好定位 chunk
调试召回结果更方便
可以分析每个 chunk 来自哪里
```

面试可以这样说：

> 当前分块结果只是字符串数组，比较轻量。后续可以把 chunk 升级成结构化对象，增加 chunkIndex、字符区间、页码、标题路径、切分策略等 metadata。这样向量库里的数据可解释性更强，检索结果也能展示更准确的来源。

#### Markdown 可以按标题层级切分

当前文本分块是通用策略：

```
段落句子固定长度
```

但 Markdown 本身有结构：

```
# 一级标题
## 二级标题
### 三级标题
```

当前如果把 Markdown 当普通文本切，会丢掉标题层级对 chunk 的指导意义。

可以优化为：

```
Markdown
  ↓
解析 AST
  ↓
按 heading 分区
  ↓
每个 chunk 带 headingPath
```

例如：

```js
{
  headingPath: ['Vue', '响应式原理'],
  content: 'Vue3 使用 Proxy...'
}
```

面试可以这样说：

> 对 Markdown 文档，我会考虑单独做结构化分块。因为 Markdown 天然有标题层级，按标题切分比普通段落切分更符合语义。后续可以把 headingPath 拼进 chunk 或作为 metadata，提升检索召回的准确性。

#### 代码块需要特殊保护

当前分块策略会把代码当普通文本处理。

如果文档里有代码：

```js
function reactive(obj) {
  return new Proxy(obj, {
    get() {},
    set() {}
  })
}
```

普通分块可能从代码中间切开，导致：

```
chunk 1：function reactive(obj) { return new Proxy
chunk 2：(obj, { get() {}, set() {} })
```

这样会影响代码语义。

可以优化：

```
识别 Markdown 代码块 ``` ```
代码块尽量整体保留
如果代码块过长，再按函数/行切
```

面试可以这样说：

> 当前分块策略更适合普通文本。如果文档里有大量代码，可以识别 Markdown 代码块，尽量保证一个代码块不要被截断。代码块过长时，可以按函数、空行或行数切分，而不是直接按字符切。

#### 可以增加 chunk 质量过滤

当前只判断：

```js
if (!chunks.length) throw new Error('文本分块结果为空')
```

但没有过滤低质量 chunk。

比如：

```
只有页码
只有空白
只有目录编号
只有几个符号
重复页眉页脚
```

这些内容如果进入向量库，会污染检索结果。

可以增加过滤规则：

```
长度太短的 chunk 丢弃
纯数字 / 页码丢弃
重复出现的页眉页脚丢弃
特殊字符占比过高的 chunk 丢弃
```

面试可以这样说：

> 当前项目只校验分块结果是否为空，后续可以增加 chunk 质量过滤，比如过滤过短文本、页码、重复页眉页脚和特殊字符占比过高的内容，减少无效 chunk 进入向量库，提高召回质量。

#### Worker 可以支持任务 id，避免并发任务串扰

当前 `chunkTextWithWorker` 每次调用都会设置：

```js
worker.onmessage = ...worker.onerror = ...
```

如果未来支持多文件同时上传，多个分块任务共用一个 Worker，就可能出现：

```
任务 A 还没返回
任务 B 又覆盖了 worker.onmessage
任务 A 的结果被 B 的 Promise 接走
```

当前项目单文件上传问题不大，但多文件并发时需要优化。

可以给任务加 `taskId`：

```js
worker.postMessage({
  type: 'chunk',
  taskId,
  payload: {...}
})
```

Worker 返回：

```js
self.postMessage({
  type: 'chunks',
  taskId,
  payload: chunks
})
```

主线程根据 `taskId` 找对应 Promise。

面试可以这样说：

> 当前项目是单文件上传，所以直接覆盖 `worker.onmessage` 没问题。后续如果支持多文件并发上传，就需要给 Worker 任务增加 taskId，并维护 pendingTasks 映射，避免多个分块任务之间结果串扰。

#### 大文本可以做流式分块或增量分块

当前是把完整 `text` 一次性传给 Worker。

流程是：

```
完整 text
  ↓
postMessage 给 Worker
  ↓
Worker 一次性返回 chunks
```

如果文档很大，会有两个问题：

```
一次性传输大字符串有内存开销
必须等所有 chunks 生成完才能继续
```

可以优化成增量分块：

```
按段落批量传给 Worker
Worker 分批返回 chunks
主线程边接收边展示进度
甚至可以边分块边 embedding
```

面试可以这样说：

> 当前分块是一次性把完整文本传给 Worker，个人知识库场景够用。如果后续支持更大文件，可以改成流式或增量分块，让 Worker 分批返回 chunks，前端可以展示更细的处理进度，服务端也可以边分块边向量化，减少等待时间和内存峰值。

#### 分块参数可以根据文档类型动态调整

当前分块参数来自：

```js
settings.chunkSize
settings.chunkOverlap
```

这意味着所有文档默认使用同一套配置。

但不同文档适合不同策略：

```
短 FAQ：chunkSize 可以小一些
长论文：chunkSize 可以大一些
代码文档：按标题/代码块切
Markdown：按 heading 切
PDF：按页码和段落切
```

可以优化为：

```
根据 ext / textLength / 文档结构选择 chunkSize
```

例如：

```js
if (ext === 'md') {
  useMarkdownChunker()
} else if (text.length > 50000) {
  chunkSize = 800
} else {
  chunkSize = 500
}
```

面试可以这样说：

> 当前项目的 chunkSize 和 overlap 是全局配置。后续可以根据文档类型和文本长度动态选择分块策略，比如 Markdown 按标题切，长文档增大 chunkSize，FAQ 类短文本减小 chunkSize，让分块更贴合内容结构。

### 面试追问

#### 为什么 RAG 里一定要做文本分块？

可以答：

> 因为原始文档通常很长，不能直接整体做 embedding，也不适合整体塞给大模型。分块后，每个 chunk 都能单独生成向量，用户提问时可以只召回最相关的 chunk。这样检索更精准，也能减少 token 成本。

#### 为什么你的分块策略不是直接按固定长度切？

可以答：

> 固定长度切分实现简单，但容易从语义中间切断。我的项目采用递归分块，优先按段落切，因为段落通常是自然语义单位；段落太长再按句子切；句子仍然太长时才按固定长度兜底。这样可以尽量保证 chunk 的语义完整性。

#### chunkSize 设置大一点好还是小一点好？

可以答：

> 不是越大越好，也不是越小越好。
> 
> chunkSize 太小，检索粒度更细，但上下文容易不完整，回答时可能缺少背景。
> 
> chunkSize 太大，上下文更完整，但每个 chunk 包含的信息更多，可能引入噪声，检索不够精准，也会增加 embedding 成本。
> 
> 所以它需要结合文档类型和业务场景调整。

#### chunkOverlap 是干什么的？

可以答：

> chunkOverlap 是相邻 chunk 的重叠内容，用来减少切分边界造成的上下文丢失。比如一个知识点刚好跨越两个 chunk，如果没有 overlap，检索时可能只召回后半段，导致上下文不完整。设置 overlap 后，相邻 chunk 会共享一部分内容，召回结果更稳定。

#### overlap 越大越好吗？

可以答：

> 不是。overlap 大可以增强上下文连续性，但也会带来重复内容，增加向量数量和 embedding 成本，还可能导致检索结果重复。一般要在上下文完整性和冗余成本之间取平衡。当前项目默认是 50，属于比较轻量的重叠。

#### 为什么要用 Web Worker？

可以答：

> 文本分块是 CPU 密集型的字符串处理。如果直接在主线程执行，大文本会导致页面卡顿，上传进度条、按钮点击和 Vue 渲染都会受影响。Web Worker 可以把分块任务放到后台线程执行，主线程只负责 UI 和状态更新，从而提升上传阶段的页面响应速度。

#### Worker 和主线程怎么通信？

可以答：

> 主线程通过 `worker.postMessage` 把任务发给 Worker，消息里包含 `type: 'chunk'`、完整文本、chunkSize 和 chunkOverlap。Worker 通过 `self.onmessage` 接收任务，调用 `recursiveChunk` 分块，完成后用 `self.postMessage({ type: 'chunks', payload: chunks })` 返回结果。如果出错，就返回 `type: 'error'`。

#### 如果 Worker 不可用怎么办？

可以答：

> 当前项目做了降级处理。如果 Worker 初始化失败，`chunkTextWithWorker` 会调用 `fallbackChunkAdvanced`，在主线程按固定长度简单切分。这个降级方案没有递归分块那么精细，但可以保证流程不中断，文档仍然能继续向量化入库。

#### 这个分块策略有什么不足？

可以答：

> 主要有几个不足。第一，overlap 当前是按字符截取，可能截断句子；第二，返回的是字符串数组，缺少 page、headingPath、chunkIndex 等 metadata；第三，Markdown 和代码块没有单独结构化处理；第四，当前 Worker 通信没有 taskId，如果未来多文件并发，需要防止任务串扰。

#### 如果文档是 Markdown，你会怎么优化？

可以答：

> Markdown 有天然标题层级，所以我不会只按普通段落切。可以先解析 Markdown AST，按 heading 分区，再把标题路径作为 metadata 或拼进 chunk 内容。这样 chunk 会带有上下文标题，比如“Vue / 响应式原理”，检索时语义会更明确。

#### 如果文档里有代码块怎么办？

可以答：

> 当前策略会把代码块当普通文本处理，可能从代码中间切开。后续可以识别 Markdown 代码块，尽量保持代码块完整。如果代码块太长，再按函数、空行或行数切，而不是简单按字符切，这样对技术文档检索更友好。

#### 为什么分块完成后还要校验 chunks.length？

可以答：

> 因为解析出来的文本可能异常，或者分块算法可能因为空文本、清洗后无内容等原因返回空数组。如果不校验，后面还会调用 embedding 接口，浪费资源甚至写入无效向量。所以当前项目在分块后判断 `chunks.length`，为空就直接抛出“文本分块结果为空”。

#### 分块质量会影响哪些后续模块？

可以答：

> 会影响向量化质量、检索召回质量和最终回答质量。chunk 如果太碎，召回时上下文不完整；chunk 如果太大，语义不聚焦，可能召回噪声；chunk 如果包含页眉页脚等垃圾内容，会污染向量库。所以文本分块是 RAG 效果里非常关键的一环。

#### 如果支持多文件批量上传，当前 Worker 逻辑有什么问题？

可以答：

> 当前 `chunkTextWithWorker` 每次调用会覆盖 `worker.onmessage`。单文件上传没问题，但多文件并发时，可能出现任务 A 还没返回，任务 B 覆盖了回调，导致结果串扰。优化方式是给每个任务加 taskId，并维护 pendingTasks 映射，Worker 返回时根据 taskId resolve 对应 Promise。

#### 文本分块是在前端做好，还是后端做好？

可以答：

> 当前项目放在前端 Worker，优点是可以体现前端性能优化能力，也能结合用户设置动态调整 chunkSize 和 overlap。但从生产 度看，如果文档很大或并发较多，解析、分块、embedding 放到服务端异步任务里会更合理，可以减少大文本在前后端之间来回传输，也便于统一做任务队列、重试和状态管理。


## 向量化与知识库入库

它的核心价值是：**把分块后的文本 chunks 转成向量，并写入向量数据库，让后续用户提问时可以做语义检索。**

### 一、组件部分

这个模块前端入口在 DocumentUpload.vue (line 1)，但它本身不直接生成向量，只负责展示状态和触发上传流程。

这个组件里你重点看三块：

1. 上传处理状态

```vue
<div v-if="docState.isUploading.value" class="upload-progress">
  <span class="spinner"></span>
  <p>{{ docState.uploadProgress.value }}</p>
</div>
```

当向量化开始时，uploadProgress 会变成类似：

```js
`正在生成向量 (xx 个文本块)...`
```
这个状态来自 useDocuments.js (line 328)。

2. 文档向量化状态

```vue
<span v-if="doc.hasVectors" class="doc-status done">
  {{ doc.chunksCount }} 个块
</span>
<span v-else class="doc-status pending">
  待向量化
</span>
```

这里的 hasVectors 和 chunksCount 不是前端算出来的，而是后端 /api/embed 向量化成功后更新文档元数据得到的。

3. 业务逻辑入口

js

`await props.docState.uploadDocument(file)`

DocumentUpload.vue 不关心上传、分块、向量化细节，它只调用 docState.uploadDocument(file)。这是一种比较典型的前端分层：**组件负责交互和展示，composable 负责业务流程，server API 负责数据处理。**

**二、模块完整流程**

这个模块从分块完成后开始。

前端流程在 useDocuments.js (line 217) 的 uploadDocument(file) 里。前面的上传、合并、解析、分块你已经学过了，向量化从这里开始：

js

``uploadProgress.value = `正在生成向量 (${chunks.length} 个文本块)...` await authFetch('/api/embed', { method: 'POST', body: { documentId: result.id, chunks } })``

这一步发送两个核心数据：

js

`{ documentId: result.id, chunks }`

其中：

- documentId：当前文档 ID，用来绑定文档和向量。
- chunks：文本分块数组，每个元素是一段文本。
- /api/embed：后端向量化接口。

后端入口是 embed.post.js (line 1)。

后端完整流程如下：

1. 读取请求参数

js

`const { documentId, chunks } = body`

如果没有 documentId 或 chunks，直接返回 400。

2. 读取当前登录用户

js

`const user = event.context.user`

这个用户信息来自后端鉴权中间件。后续所有向量都会带上 userId，这是多用户隔离的关键。

3. 读取模型配置

js

`const settings = getSettings(user.id)`

配置里包含：

- apiKey
- baseUrl
- embeddingModel
- chunkSize
- chunkOverlap

如果没有 API Key，直接报错。这说明向量化依赖外部 embedding 模型服务。

4. 批量生成 embedding

源码里设置了：

js

`const BATCH_SIZE = 25`

然后按 25 个 chunk 一批调用：

js

`const embeddings = await getEmbeddings(batch, settings)`

getEmbeddings 在 embeddings.js (line 11)。

这里为什么要批量？因为一个文档可能有很多 chunks，如果每个 chunk 单独请求一次，网络开销大、速度慢；如果一次全传，又可能超过模型 API 限制。所以用固定批大小折中。

5. 调用 embedding API

核心代码在 embeddings.js (line 20)：

js

``const response = await fetch(`${settings.baseUrl}/embeddings`, { method: 'POST', headers: { 'Authorization': `Bearer ${settings.apiKey}`, 'Content-Type': 'application/json' }, body: JSON.stringify(requestBody) })``

如果模型是 text-embedding-v3，项目额外传了：

js

`requestBody.dimensions = 1024`

这表示把每个文本块转成 1024 维向量。

6. 准备写入 ChromaDB 的数据

在 embed.post.js (line 39)：

js

``const ids = chunks.map((_, i) => `${user.id}_${documentId}_chunk_${i}`)``

每个 chunk 都有唯一 ID。这个 ID 由三部分组成：

- 用户 ID
- 文档 ID
- chunk 序号

这样可以避免不同用户、不同文档之间冲突。

metadata 结构：

js

`{ documentId, userId: String(user.id), name: docMeta?.name || 'unknown', chunkIndex: i }`

metadata 非常重要。它不参与向量计算，但参与过滤、溯源和展示。

7. 写入 ChromaDB

js

`await collection.add({ ids, embeddings: allEmbeddings, metadatas, documents: chunks })`

这一步把四类信息一起写入向量数据库：

- ids：每个 chunk 的唯一标识。
- embeddings：每个 chunk 的向量。
- metadatas：文档名、用户 ID、chunk 序号等。
- documents：原始 chunk 文本。

ChromaDB collection 来自 chroma.js (line 8)：

js

`getOrCreateCollection({ name: "rag_docs", metadata: { "hnsw:space": "cosine" } })`

也就是说，所有知识库 chunks 都存在 rag_docs 这个 collection 里，并且使用 cosine 余弦距离做相似度计算。

8. 更新文档状态

向量入库成功后，后端更新文档元信息：

js

`docMeta.hasVectors = true docMeta.chunksCount = chunks.length docMeta.updatedAt = new Date().toISOString()`

所以前端文档列表才能显示“已完成向量化”和 chunk 数量。

这个模块在链路中的作用可以总结成一句话：

**它把“可阅读文本”变成“可语义检索的数据”。**

没有这个模块，后面用户提问时只能普通问答，没法从知识库里召回相关文档内容。

**三、这个模块涉及的原理**

第一个原理是 **Embedding 向量化**。

文本本身是字符串，计算机无法直接判断“语义相似”。Embedding 模型会把文本转成一组数字，比如：

js

`"如何配置 API Key" -> [0.12, -0.33, 0.08, ...]`

这些数字组成一个高维向量。语义越接近的文本，向量在空间中的距离越近。

举个例子：

- “如何配置 API Key”
- “在哪里填写密钥”
- “模型接口鉴权怎么设置”

这三句话字面不同，但语义接近。关键词搜索可能不一定都能命中，但 embedding 检索更容易认为它们相关。

第二个原理是 **向量数据库**。

普通数据库擅长精确查询，比如 id = 1。向量数据库擅长相似度查询，比如：

js

`找出和这个问题向量最接近的 5 个文档块`

项目里用的是 ChromaDB。它保存每个 chunk 的向量，后续用户提问时，也会先把问题转成向量，再去 ChromaDB 里查相似 chunks。

第三个原理是 **余弦相似度**。

项目里 Chroma collection 设置了：

js

`"hnsw:space": "cosine"`

cosine 关注两个向量的方向是否相似，而不是绝对长度。文本语义检索通常更适合用余弦相似度。

你面试时可以这样说：

> 文本 embedding 的重点是语义方向，余弦相似度可以衡量两个向量方向是否接近，因此适合判断两段文本语义是否相关。

第四个原理是 **metadata 过滤**。

向量数据库不是只存向量。这个项目还给每个向量存了：

js

`userId documentId name chunkIndex`

后续检索时可以用 userId 限制只查当前用户的数据，避免 A 用户召回 B 用户的文档。这是 RAG 项目里非常重要的安全点。

**四、有什么可以优化的地方**

1. 向量入库可以改成 upsert

当前使用：

js

`collection.add()`

如果同一个文档重复向量化，可能出现 ID 冲突。更稳的做法是：

- 先按 documentId + userId 删除旧向量；
- 再重新 add；
- 或者直接使用 upsert 能力。

面试时可以说：

> 当前实现适合首次入库，但重复入库时需要考虑幂等性。生产环境我会把向量写入设计成 upsert 或“先删后写”，避免重复 ID 或脏数据。

2. 向量化进度可以更细

现在前端只知道“正在生成向量”，不知道完成了多少批。可以把 /api/embed 改成 SSE，后端每处理完一批返回：

js

`{ currentBatch: 3, totalBatch: 10 }`

这样用户体验会更好。

3. 批量请求可以做有限并发

现在是串行批处理：

js

`for (...) { await getEmbeddings(batch) }`

稳定但慢。可以做并发池，比如同时处理 2-3 批，既提升速度，又不至于打爆 API。

4. 增加失败重试

Embedding API 可能因为网络、限流、超时失败。现在失败后整个流程结束。可以加：

- 单批重试；
- 指数退避；
- 失败批次记录；
- 断点续向量化。

5. 增加文本清洗和去重

如果 chunk 里有页眉、页脚、空行、重复目录，会污染向量库。可以在向量化前做：

- 去空白；
- 去重复 chunk；
- 过滤过短文本；
- 清理页码、目录、版权声明等噪声。

6. 删除文档时同步删除向量

这个点面试很容易被问。文档删了以后，对应 ChromaDB 里的向量也应该删掉，否则后续还能召回已删除文档内容。

你可以回答：

> 文档删除不仅要删元数据和原文件，还要按 userId + documentId 删除 ChromaDB 中对应 chunks，否则会造成知识库脏召回。

**五、面试追问回答**

面试官问：为什么 RAG 项目需要向量化？

可以回答：

> 因为用户的问题和文档里的原文表达不一定完全一致。向量化可以把文本转成语义空间里的向量，通过相似度找到语义相关的内容，而不是只靠关键词匹配。这样用户问“密钥怎么设置”，也可能召回包含“API Key 配置”的文档片段。

面试官问：为什么不把整篇文档直接向量化？

可以回答：

> 整篇文档太长，可能超过 embedding 模型输入限制；而且整篇向量粒度太粗，召回后也不方便放进 prompt。分块后向量化可以提升检索精度，让系统只召回和问题最相关的局部内容，同时控制后续 prompt token 数量。

面试官问：这个模块怎么保证用户数据隔离？

可以回答：

> 入库时每个 chunk 的 metadata 都带了 userId，chunk ID 也拼接了 user.id。后续检索时会用 where: { userId: String(user.id) } 过滤，所以每个用户只能检索自己的文档向量。

面试官问：metadata 有什么作用？

可以回答：

> metadata 主要用于过滤和溯源。比如 userId 用于权限隔离，documentId 用于删除或更新某个文档的所有 chunks，name 用于前端展示来源文档，chunkIndex 可以用于后续扩展相邻 chunk 补全上下文。

面试官问：为什么用 ChromaDB？

可以回答：

> ChromaDB 是轻量级向量数据库，适合本地 RAG 项目快速落地。它支持 collection、embedding 存储、metadata 过滤和相似度查询。对于学习型或中小型知识库项目，接入成本比 Milvus、Pinecone 这类系统更低。

面试官问：这里的 BATCH_SIZE = 25 是什么考虑？

可以回答：

> 主要是平衡性能和 API 限制。单条请求效率低，一次传太多又容易超过接口限制或导致超时。按固定批次发送可以减少网络请求，同时控制每次请求体大小。

面试官问：如果向量化成功一半失败了怎么办？

可以回答：

> 当前实现还比较简单，失败后会返回错误，但可能存在部分向量已写入的情况。生产环境应该把这个流程设计成可恢复的：记录 batch 状态，失败批次可重试；或者先写临时状态，全部成功后再把文档标记为 hasVectors = true。同时重复执行时要保证幂等，避免重复写入。

面试官问：这个模块和后面的检索模块是什么关系？

可以回答：

> 向量化入库是离线或准离线阶段，负责把知识库内容预处理成向量索引。检索模块是在线阶段，用户每次提问时把问题也转成向量，再从这个索引里查相似 chunks。前者决定知识库数据质量，后者决定召回效果。
