
这个项目可以拆成 **8 个核心业务组件**：

1. **文档上传组件**
2. **文档解析组件**
3. **文本分块组件**
4. **知识入库 / 向量化组件**
5. **知识库管理组件**
6. **检索召回组件**
7. **对话问答组件**
8. **模型参数配置组件**

## 1. 文件上传

### 1. **文档上传组件 / 文件上传组件**

> `src/pages/knowledge/index.vue`  
> 或  
> `src/views/knowledge/components/upload/index.vue`

- 负责知识库文档上传入口
- 支持用户选择 PDF、DOCX、TXT、Markdown 等文件
- 对文件类型、文件大小进行前置校验
- 维护当前上传文件列表和上传状态
- 调用分片上传逻辑，展示整体上传进度
- 上传完成后触发后续的文档解析、文本分块、向量化入库流程
- 上传状态变化后通知知识库列表组件刷新数据

==作用==：这个组件是整个 RAG 知识入库流程的入口。源码里可以把它理解成“上传控制层”，它不直接处理所有复杂逻辑，而是负责接收用户文件、展示上传状态，并调用下层的上传 composable 或 API 方法完成文件上传。你的简历里提到项目支持 PDF、DOCX、TXT、Markdown 等文档上传，并实现了文件哈希校验、分片索引上传、断点续传、秒传和整体进度展示，所以这个组件就是这些能力的业务入口。

### FileSelector：文件选择组件

负责让用户选择本地文件。

例如：

- 点击上传按钮
- 拖拽上传
- 多文件选择
- 获取文件对象 File

它拿到的是浏览器原生的 `File` 对象。

```
{  
	name: "前端面试八股.pdf",
	size: 1024 * 1024 * 8,
	type: "application/pdf"
}
```

### FileValidator：文件校验组件

负责判断这个文件能不能进入知识库。

常见校验包括：

```
文件格式是否合法
文件大小是否超限
文件名是否为空
是否重复上传
```

比如只允许：

```
.pdf
.docx
.txt
.md
.markdown
```

### ChunkUploader：分片上传组件

这是上传组件中比较有技术含量的部分。

它负责把大文件切成多个固定大小的片段，例如每片 2MB 或 5MB。

```
file
 ├─ chunk-0
 ├─ chunk-1
 ├─ chunk-2
 └─ chunk-n
```

每个分片上传时会携带：

```json
{
	fileHash: "xxx",
	chunkIndex: 0,
	chunkSize: 2 * 1024 * 1024,
	totalChunks: 10
}
```

它的作用是：

- 大文件不用一次性上传
- 上传失败后只重传失败分片
- 支持断点续传
- 支持整体进度计算

### UploadProgress：上传进度组件

负责展示上传阶段的状态。

例如：

```
等待上传
计算哈希中
上传中 35%
解析中
入库中
完成
失败
```

这个组件本质上是一个状态展示组件，但它和业务流程强相关，因为 RAG 的文档处理不是上传完就结束，而是后面还有解析、分块、向量化、入库。

所以它不应该只展示“文件上传进度”，而应该展示“知识入库进度”。

## 2. 文件解析

文档解析组件负责把不同格式的文件转成纯文本内容。

因为 RAG 后续的分块、向量化、检索都不是直接处理 PDF 或 DOCX，而是处理解析后的文本。

它的输入是：

```
PDF / DOCX / TXT / Markdown 文件
```

输出是：

```
纯文本内容 text
```

## 不同文件的解析方式

可以这么讲：

```
PDF       → 提取页面文本DOCX      → 提取段落文本TXT       → 直接读取文本Markdown  → 保留标题、列表、代码块等结构
```

你简历中明确提到项目支持 PDF、MD 等多格式文档解析。

## 可以抽象出的业务组件

```
DocumentParser ├─ PdfParser ├─ DocxParser ├─ TxtParser └─ MarkdownParser
```

### PdfParser

负责解析 PDF。

PDF 解析的难点是：

- PDF 不是天然的结构化文本
- 有些 PDF 是扫描件，普通解析无法拿到文字
- 页眉页脚、换行、分页会影响文本质量

面试时可以说：

> PDF 解析后我会把每一页文本合并，并对多余换行、空格、页码等内容做基础清洗，尽量保证后续 chunk 语义连续。

### MarkdownParser

负责解析 Markdown。

Markdown 对 RAG 其实比较友好，因为它天然带结构：

```
# 标题## 小节- 列表```js代码块
```

```
处理 Markdown 时，可以保留标题层级，这样后面分块时可以让 chunk 带上上下文标题。例如：```text标题：Vue 响应式原理内容：Vue 3 使用 Proxy 实现响应式...
```

这样召回效果通常会更好。

### TxtParser

TXT 最简单，基本就是直接读取文本内容。

但也需要做：

- 编码处理
- 空行清理
- 特殊字符清理

### DocxParser

DOCX 一般按段落提取。

如果文档里面有标题、表格、编号列表，也可以在解析阶段尽量转成文本结构。

# 3. 文本分块组件

## 组件职责

文本分块组件负责把一篇长文档切成多个小的知识片段，也就是 RAG 里的 chunk。

为什么要分块？

因为大模型和向量检索都不适合直接处理超长文档。

如果一篇文档有 10 万字，不能直接全部塞给模型，需要先拆成多个语义片段。

你的简历中提到：引入 **Web Worker** 处理大文本分块任务，并采用“按段落、按句子、按固定长度”的递归分块策略，避免大文档处理阻塞主线程。

## 可以抽象出的业务组件

```
TextSplitter ├─ ParagraphSplitter ├─ SentenceSplitter ├─ FixedLengthSplitter ├─ OverlapProcessor └─ WorkerBridge
```

### ParagraphSplitter：按段落切分

优先按段落切。

例如：

```
第一段：RAG 是检索增强生成...第二段：它通常包括文档解析、向量化、检索...第三段：用户提问时会先召回相关片段...
```

优点是语义完整。

### SentenceSplitter：按句子切分

如果某个段落太长，就继续按句子切。

例如：

```
RAG 是检索增强生成。它可以减少大模型幻觉。它会先从知识库中召回相关内容。
```

优点是比固定长度更自然。

### FixedLengthSplitter：按固定长度切分

如果句子仍然过长，最后再按固定字符长度切。

例如每个 chunk 控制在：

```
500 - 1000 字
```

这就是你简历中提到的递归分块策略。

面试可以这样讲：

> 我的分块策略不是简单粗暴地按固定长度截断，而是优先保证语义完整。系统会先按段落拆，如果段落超长，再按句子拆；如果句子仍然超过阈值，最后才按固定长度兜底。

### OverlapProcessor：重叠窗口处理

RAG 分块一般会设置 overlap，也就是相邻 chunk 之间保留一部分重复内容。

例如：

```
chunkSize = 800overlap = 100
```

这样可以减少语义被切断的问题。

例如：

```
chunk1: A B C D Echunk2: E F G H I
```

`E` 就是重叠部分。

面试可以说：

> 我支持配置分块大小和重叠长度，避免因为切分边界导致上下文丢失。

### WorkerBridge：Web Worker 通信组件

这是前端性能优化点。

大文档分块可能非常耗时，如果直接在主线程执行，会导致页面卡顿。

所以用 Web Worker 把分块任务放到子线程。

调用关系大概是：

```
主线程上传文件   ↓解析出文本   ↓把文本传给 Web Worker   ↓Worker 执行分块   ↓Worker 返回 chunks   ↓主线程更新入库状态
```

# 4. 知识入库 / 向量化组件

## 组件职责

知识入库组件负责把文本 chunk 转成向量，并保存到知识库中。

它是 RAG 系统中非常核心的一步。

输入：

```
[  {    content: "Vue 3 使用 Proxy 实现响应式...",    metadata: {      documentId: "doc_001",      chunkIndex: 0,      fileName: "vue响应式原理.md"    }  }]
```

输出：

```
[  {    chunkId: "chunk_001",    content: "Vue 3 使用 Proxy 实现响应式...",    embedding: [0.123, 0.456, ...],    metadata: {...}  }]
```

## 可以抽象出的业务组件

```
KnowledgeIndexer ├─ EmbeddingService ├─ VectorStoreAdapter ├─ MetadataBuilder └─ IndexStatusManager
```

### EmbeddingService：向量生成服务

负责调用 embedding 模型，把文本转成向量。

例如：

```
"Vue 3 响应式原理是什么？"        ↓[0.12, -0.08, 0.33, ...]
```

向量的作用是表示文本语义。

意思相近的文本，在向量空间中的距离更近。

### VectorStoreAdapter：向量存储适配器

负责把向量和文本片段存起来。

真实项目中可以是：

```
本地存储数据库向量数据库服务端接口
```

你简历中写的是“向量化存储”，没有明确写具体向量库，所以面试时不要强行说用了 Milvus、FAISS 或 Chroma，除非你确实用了。

可以说：

> 项目中我把向量化和存储封装成独立服务，前端只关心入库状态和文档管理，具体向量生成和存储由服务接口完成，便于后续替换不同的 embedding 模型或向量库。

### MetadataBuilder：元信息构建器

每个 chunk 不能只存文本，还要存来源信息。

例如：

```
{  documentId: "doc_001",  fileName: "前端面试.md",  chunkIndex: 3,  chunkSize: 800,  createdAt: "2026-05-11"}
```

这些 metadata 的作用是：

- 检索结果可以展示来源
- 删除文档时可以删除对应 chunks
- 回答时可以追溯引用
- 后续支持按文档、按知识库过滤检索范围

### IndexStatusManager：入库状态管理

负责维护文档入库状态。

例如：

```
uploadedparsingchunkingembeddingindexedfailed
```

这是知识库管理页面展示状态的基础。

# 5. 知识库管理组件

## 组件职责

知识库管理组件负责管理已经上传并入库的文档。

它面向用户，属于业务管理页面。

你简历里提到做了文档数据管理、删除同步清理、状态展示等功能。

## 可以抽象出的业务组件

```
KnowledgeBaseManager ├─ DocumentList ├─ DocumentStatusTag ├─ DocumentDetail ├─ DeleteDocumentAction └─ ReindexAction
```

### DocumentList：文档列表

展示所有上传过的文档。

通常包括：

```
文件名文件类型文件大小上传时间入库状态chunk 数量操作按钮
```

### DocumentStatusTag：文档状态

展示文档当前处理状态。

例如：

```
解析中分块中向量化中已入库入库失败
```

这对用户很重要，因为 RAG 入库不是瞬时完成的。

### DeleteDocumentAction：删除文档

删除文档时，不能只删除文件记录。

还需要同步删除：

```
文档元信息原始文件文本 chunks向量数据历史引用关系
```

面试可以强调：

> 删除知识库文档时，我做了同步清理，保证文档记录和向量数据一致，避免用户已经删除了文档，但检索时仍然召回旧内容。

这是一个很好的业务完整性点。

# 6. 检索召回组件

## 组件职责

检索召回组件负责在用户提问时，从知识库中找到和问题最相关的内容。

这是 RAG 里的 Retrieval。

输入：

```
用户问题：Vue 3 响应式原理是什么？
```

处理流程：

```
问题向量化   ↓向量相似度搜索   ↓召回 TopK 个相关 chunk   ↓组装成上下文 context
```

输出：

```
[  {    content: "Vue 3 使用 Proxy 实现响应式...",    score: 0.89,    source: "vue响应式原理.md"  },  {    content: "依赖收集发生在 getter 中...",    score: 0.82,    source: "前端面试.md"  }]
```

## 可以抽象出的业务组件

```
Retriever ├─ QueryEmbeddingService ├─ SimilaritySearchService ├─ TopKSelector ├─ ContextBuilder └─ SourceFormatter
```

### QueryEmbeddingService：问题向量化

把用户的问题也转成向量。

这样才能和知识库里的 chunk 向量做相似度比较。

### SimilaritySearchService：相似度搜索

根据向量相似度找最相关的 chunk。

常见算法是：

```
余弦相似度欧氏距离点积相似度
```

前端面试不用展开太深，可以说：

> 检索阶段会把用户问题转成向量，然后和知识库中 chunk 的向量做相似度计算，选出最相关的 TopK 片段作为上下文。

### TopKSelector：检索条数控制

你简历里提到模型参数配置支持“检索条数”动态调整。

也就是：

```
topK = 3 / 5 / 10
```

TopK 太小，可能召回不完整。

TopK 太大，可能上下文冗余，甚至影响回答质量。

### ContextBuilder：上下文构建器

负责把召回的 chunks 拼接成 prompt 上下文。

例如：

```
以下是知识库中检索到的内容：[片段1]Vue 3 使用 Proxy 实现响应式...[片段2]effect 用于收集依赖...请基于以上内容回答用户问题：Vue 3 响应式原理是什么？
```

这个组件决定了最终问答质量。

# 7. 对话问答组件

## 组件职责

对话问答组件负责用户和 AI 的交互。

你简历中提到实现了基于 **SSE 的流式对话能力**，并配合 Pinia 管理会话状态、消息列表、流式输出内容和中断控制。

## 可以抽象出的业务组件

```
ChatModule ├─ ChatInput ├─ MessageList ├─ MessageBubble ├─ StreamRenderer ├─ StopGenerateButton ├─ SessionStore └─ AutoScrollController
```

### ChatInput：输入组件

负责用户输入问题。

常见功能：

```
输入问题回车发送按钮发送禁用空消息发送中禁止重复提交
```

### MessageList：消息列表

负责展示历史消息。

消息通常分为：

```
user messageassistant messagesystem message
```

结构可能是：

```
{  id: "msg_001",  role: "user",  content: "RAG 是什么？",  createdAt: 1710000000000}
```

### MessageBubble：消息气泡组件

负责展示每一条消息。

用户消息和 AI 消息样式不同。

AI 消息还需要支持 Markdown 渲染。

### StreamRenderer：流式输出组件

这个是核心组件。

它负责接收 SSE 返回的一段段 token，并逐字展示在页面上。

流程是：

```
用户发送问题   ↓创建 assistant 空消息   ↓发起 SSE 请求   ↓服务端持续返回 token   ↓前端不断追加 content   ↓页面实时渲染
```

### StopGenerateButton：中断生成组件

负责终止当前流式请求。

通常会结合：

```
AbortController
```

当用户点击停止时：

```
abort 当前请求更新消息状态保留已生成内容
```

### SessionStore：会话状态管理

你简历中提到用 Pinia 统一管理会话状态。

它可以管理：

```
当前会话 ID历史会话列表当前消息列表当前是否生成中当前流式内容中断控制器
```

### AutoScrollController：自动滚动组件

你简历中提到用 ResizeObserver 监听消息气泡 DOM 高度变化，实现长对话场景下自动滚动吸底。

它解决的问题是：

```
AI 回复是不断变长的消息气泡高度持续变化页面需要自动滚动到底部但不能疯狂 scroll 导致抖动
```

# 8. 模型参数配置组件

## 组件职责

模型参数配置组件负责让用户调整 RAG 和大模型相关参数。

你简历中提到支持配置：

```
API Key模型名称温度分块大小重叠长度检索条数
```

## 可以抽象出的业务组件

```
ModelConfigPanel ├─ ApiKeyConfig ├─ ModelSelector ├─ TemperatureSlider ├─ ChunkSizeInput ├─ OverlapInput ├─ TopKInput └─ ConfigStore
```

### ApiKeyConfig

负责配置 DashScope API Key。

注意：面试时可以补一句安全意识：

> API Key 不应该直接硬编码在前端代码里，正式环境应该通过服务端代理请求模型服务，前端只保存必要配置或通过后端读取。

### ModelSelector

负责选择模型。

例如：

```
qwen-plusqwen-turboqwen-max
```

不用说死具体模型，除非你项目里真的用了。

### TemperatureSlider

控制模型回答的随机性。

```
temperature 低 → 回答更稳定temperature 高 → 回答更多样
```

知识库问答一般更适合较低 temperature，因为要求准确。

### ChunkSizeInput

控制每个 chunk 的大小。

例如：

```
chunkSize = 500 / 800 / 1000
```

### OverlapInput

控制相邻 chunk 的重叠长度。

例如：

```
overlap = 50 / 100 / 150
```

### TopKInput

控制每次检索召回多少个片段。

例如：

```
topK = 3 / 5 / 8
```
