1. **文档上传组件**
2. **文档解析组件**
3. **文本分块组件**
4. **知识入库 / 向量化组件**
5. **知识库管理组件**
6. **检索召回组件**
7. **对话问答组件**
8. **模型参数配置组件**

# 1. 文件上传

## 1. **文档上传组件**

> `app/components/DocumentUpload.vue`

- 负责知识库管理弹窗的整体 UI
- 提供拖拽上传区域
- 提供点击选择文件上传
- 限制可选文件类型：`.pdf / .txt / .md / .docx`
- 上传中展示进度条、上传百分比、已上传大小 / 总大小
- 展示错误提示和通知提示
- 展示已上传文档列表
- 展示每个文档的文件类型、文件大小、向量化状态、文本块数量
- 支持删除文档，删除前会提示“删除文档将同时删除其向量数据”
- 不直接实现上传逻辑，而是通过父组件传入的 `docState.uploadDocument(file)` 触发上传流程

==作用==：源码里可以看到，这个组件主要是 **知识库文档上传和文档列表管理的 UI 层**。它本身不做哈希计算、分片上传、文本分块、向量化，而是负责接收用户选择的文件，然后调用 `props.docState.uploadDocument(file)`。上传进度、上传百分比、上传字节数、错误信息、通知信息、文档列表也都是从 `docState` 中读取。

### 1.1 **文档上传组件的入口位置**

> `app/pages/index.vue`

- 首页中维护 `showDocPanel`
- 点击顶部“知识库”按钮后展示 `DocumentUpload`
- 通过 `:doc-state="docState"` 把文档上传状态传给上传组件
- 使用 `useDocumentsAdvanced()` 创建文档管理状态
- 页面挂载时调用：
    - `docState.loadDocuments()`
    - `docState.initWorker()`
- 页面卸载时调用：
    - `docState.terminateWorker()`

==作用==：`index.vue` 是这个上传面板的挂载入口。源码里 `DocumentUpload` 并不是路由页面，而是首页聊天布局里的一个侧边知识库面板。`index.vue` 创建 `docState = useDocumentsAdvanced()`，然后把它传给 `DocumentUpload`，所以真正的上传逻辑入口在 composable 里。

调用关系是：

```
app/pages/index.vue
   ↓
useDocumentsAdvanced()
   ↓
<DocumentUpload :doc-state="docState" />
```

### 1.2 **文档上传核心逻辑**

> `app/composables/useDocuments.js`

这个文件里有两个 composable：

```
useDocuments()
useDocumentsAdvanced()
```

真正对应你简历里“秒传、断点续传、分片上传、上传进度计算”的是：

> `useDocumentsAdvanced()`

它主要做这些事：

- 维护文档列表 `documents`
- 维护上传状态 `isUploading`
- 维护上传文案 `uploadProgress`
- 维护上传百分比 `uploadPercent`
- 维护已上传大小 `uploadedBytes`
- 维护总文件大小 `totalBytes`
- 维护错误信息 `error`
- 维护提示信息 `notice`
- 计算文件哈希 `computeFileHash(file)`
- 按固定大小切分文件，源码里 `chunkSize = 2 * 1024 * 1024`
- 请求 `/api/upload/check` 判断是否秒传或断点续传
- 跳过服务端已经存在的分片
- 使用 `XMLHttpRequest` 上传分片到 `/api/upload/chunk`
- 上传完成后请求 `/api/upload/merge`
- 合并完成后拿到解析出的文本
- 调用 Web Worker 进行文本分块
- 调用 `/api/embed` 生成向量并入库
- 最后重新加载文档列表

==作用==：`DocumentUpload.vue` 是 UI 层，`useDocumentsAdvanced()` 才是 **文件上传业务编排层**。源码里可以看到，完整上传流程不是一次性 POST 文件，而是：先算 SHA-256 文件哈希，再检查服务端状态，再分片上传缺失部分，最后合并文件、解析文本、分块、向量化。

核心流程可以写成：

```
用户选择文件
   ↓
DocumentUpload.vue 调用 docState.uploadDocument(file)
   ↓
useDocumentsAdvanced.uploadDocument(file)
   ↓
computeFileHash(file)
   ↓
POST /api/upload/check
   ↓
判断是否秒传 / 是否存在已上传分片
   ↓
file.slice() 切分文件
   ↓
POST /api/upload/chunk 上传缺失分片
   ↓
POST /api/upload/merge 合并分片并解析文本
   ↓
Web Worker 文本分块
   ↓
POST /api/embed 生成向量并写入 ChromaDB
   ↓
loadDocuments() 刷新文档列表
```

### 1.3 **上传检查接口**

> `server/api/upload/check.post.js`

- 接收前端传来的文件元信息
- 校验上传元数据是否合法
- 根据 `fileHash` 查询是否已有完整文档
- 如果已有完整文档，返回 `exists: true`
- 如果没有完整文档，则创建或更新上传会话
- 查询服务端已经存在的分片索引
- 计算已上传字节数
- 返回 `uploadedChunks / uploadedBytes / totalChunks`

==作用==：这个接口支撑 **秒传和断点续传**。源码里会先 `findDocumentByHash(body.fileHash)`，如果找到已有文档，就直接返回 `exists: true`；否则通过 `upsertUploadSession` 创建上传会话，并返回已经上传过的分片列表。

也就是：

```
前端算出 fileHash
   ↓
请求 /api/upload/check
   ↓
服务端查是否有同 hash 文档
   ↓
有：秒传
   ↓
没有：返回已上传分片，用于断点续传
```

### 1.4 **分片上传接口**

> `server/api/upload/chunk.post.js`

- 接收 multipart 分片数据
- 读取分片文件字段 `chunk`
- 读取上传元信息：
    - `fileHash`
    - `fileName`
    - `fileSize`
    - `totalChunks`
    - `chunkSize`
    - `chunkIndex`
    - `ext`
- 校验分片索引是否合法
- 校验上传元数据是否合法
- 创建或更新上传会话
- 可选校验 `chunkHash`
- 确保分片目录存在
- 将当前分片写入本地文件系统

==作用==：这个接口是 **单个分片的接收层**。前端每次上传一个 chunk，服务端就按照 `fileHash + chunkIndex` 把它保存成对应的 `.part` 文件。源码里最终是通过 `writeFile(getUploadChunkPath(meta.fileHash, chunkIndex), filePart.data)` 把分片落盘。

### 1.5 **分片合并接口**

> `server/api/upload/merge.post.js`

- 校验上传元数据
- 读取上传会话
- 再次检查是否已有相同 hash 的文档
- 获取已经上传的分片索引
- 检查是否还有缺失分片
- 按 `0 ~ totalChunks - 1` 的顺序读取分片
- 使用 `createWriteStream` 合并成完整文件
- 调用 `parseDocument(fileBuffer, session.fileName)` 提取文本
- 生成文档元信息 `docMeta`
- 写入文档元数据文件
- 删除临时分片目录
- 删除上传会话文件
- 返回文档元信息和解析出的文本

==作用==：这个接口是 **上传阶段和 RAG 入库阶段的分界点**。在它之前，系统处理的是文件分片；在它之后，系统已经拿到了完整文件和解析文本，后续就可以进入文本分块和向量化流程。源码里合并完成后会调用 `parseDocument` 提取文本，并写入 `documents/${documentId}.meta.json`，其中初始状态是 `hasVectors: false`、`chunksCount: 0`。

### 1.6 **上传会话与校验工具**

> `server/utils/uploadSessions.js`

- 定义最大文件大小：`20MB`
- 定义允许的文件类型：`pdf / txt / md / docx`
- 生成上传会话文件路径
- 生成分片目录路径
- 生成单个分片路径
- 校验上传元数据
- 创建分片目录
- 读取已上传分片索引
- 计算已上传字节数
- 读取 / 写入上传会话
- 根据 `fileHash` 查找已完成文档
- 提供服务端 `sha256` 校验方法

==作用==：这个文件是上传模块的 **底层工具层**。`check.post.js`、`chunk.post.js`、`merge.post.js` 都依赖它。比如文件类型和大小不是前端写死判断，而是服务端通过 `validateUploadMeta` 再校验一遍，保证上传安全性和数据一致性。

### 1.7 **向量化接口**

> `server/api/embed.post.js`

- 接收 `documentId` 和 `chunks`
- 读取模型配置和 API Key
- 批量调用 embedding 方法生成向量
- 每批最多处理 25 条 chunk
- 读取文档元信息
- 获取或创建 Chroma Collection
- 将：
    - chunk id
    - embedding
    - metadata
    - 原始 chunk 文本  
        写入 ChromaDB
- 同步写入本地 `vectors/${documentId}.json`
- 更新文档元信息：
    - `hasVectors = true`
    - `chunksCount = chunks.length`
    - `updatedAt = 当前时间`

==作用==：这个接口不是文件上传本身，但它是上传流程的最后一步。`useDocumentsAdvanced()` 在文件合并、文本解析、Web Worker 分块之后，会调用 `/api/embed`，把文本块转成向量并存入 ChromaDB。源码里可以看到它使用 `getEmbeddings(batch, settings)` 批量生成向量，然后 `collection.add(...)` 写入 ChromaDB。

### 面试时回答

> 首页入口在 `app/pages/index.vue`，里面通过 `useDocumentsAdvanced()` 创建文档状态，并把它作为 `doc-state` 传给 `DocumentUpload`。真正的上传逻辑在 `app/composables/useDocuments.js`，里面会先计算文件 SHA-256，然后请求 `/api/upload/check` 判断是否秒传或断点续传。如果需要上传，就按 2MB 固定大小切片，用 XHR 上传到 `/api/upload/chunk`，并通过 `xhr.upload.onprogress` 计算进度。所有分片上传完成后，再请求 `/api/upload/merge`，服务端会检查分片完整性、按顺序合并文件、解析文本并返回。前端拿到文本后，再用 Web Worker 分块，最后调用 `/api/embed` 生成向量并写入 ChromaDB。

### 模块完整流程

你这个上传模块，前端页面组件其实主要有两个：

```

server/api/upload/check.post.js
server/api/upload/chunk.post.js
server/api/upload/merge.post.js
server/api/embed.post.js
```

整体关系可以先这样记：

```
用户点击“知识库”
  ↓
app/pages/index.vue 打开 DocumentUpload 面板
  ↓
DocumentUpload.vue 选择文件 / 拖拽文件
  ↓
调用 docState.uploadDocument(file)
  ↓
useDocumentsAdvanced.uploadDocument(file)
  ↓
计算文件 hash
  ↓
/api/upload/check 检查秒传 / 断点续传
  ↓
/api/upload/chunk 上传缺失分片
  ↓
/api/upload/merge 合并分片并解析文本
  ↓
Web Worker 分块
  ↓
/api/embed 生成向量并写入 ChromaDB
  ↓
loadDocuments() 刷新知识库列表
```

#### 上传模块挂载入口

>文件：`app/pages/index.vue`

它是整个 RAG 聊天页面的入口，同时也是知识库上传面板的挂载位置。这个页面本身不负责上传细节，而是负责：

- 创建文档状态对象 `docState`
- 控制知识库面板是否显示
- 把 `docState` 传给 `DocumentUpload`
- 页面挂载时加载文档列表
- 页面挂载时初始化 Web Worker
- 页面卸载时销毁 Web Worker

源码里可以看到，首页通过 `useDocumentsAdvanced()` 创建文档管理状态，然后把它传给 `<DocumentUpload :doc-state="docState" />`。

##### 1. template 结构

核心 template 可以抽象成这样：

```vue
<div class="chat-layout">
  <ChatSidebar
    :documents-count="docState.documents.value.length"
    @open-docs="showDocPanel = true"
  />

  <div class="chat-main">
    <header class="chat-header">
      <button
        class="btn-header"
        :class="{ active: showDocPanel }"
        @click="showDocPanel = !showDocPanel"
      >
        知识库
        <span v-if="docState.documents.value.length">
          {{ docState.documents.value.length }}
        </span>
      </button>

      <NuxtLink to="/settings">
        设置
      </NuxtLink>
    </header>

    <ChatWindow />
    <ChatInput />
  </div>

  <DocumentUpload
    v-if="showDocPanel"
    @close="showDocPanel = false"
    :doc-state="docState"
  />
</div>
```

它做了三件事：

1. 左侧 `ChatSidebar` 展示文档数量，并提供打开知识库面板的入口。
2. 顶部 header 里有“知识库”按钮，点击后切换 `showDocPanel`。
3. 当 `showDocPanel = true` 时，渲染 `DocumentUpload` 组件，并把 `docState` 传进去。

这里的关键点是：

```vue
<DocumentUpload
  v-if="showDocPanel"
  @close="showDocPanel = false"
  :doc-state="docState"
/>
```

也就是说，`DocumentUpload.vue` 自己不创建上传状态，而是完全接收父组件传入的 `docState`。

##### 2. 核心变量

###### `chatStore`

```
const chatStore = useChatStore()
```

这个是聊天模块状态，和上传模块不是强绑定，但页面里会同时管理聊天状态和知识库状态。

它主要负责：

- 当前会话标题
- 是否正在流式生成
- 来源片段展示
- 会话列表加载

上传模块里只需要知道：它和文档模块并列存在于首页。
###### `docState`

```js
const docState = useDocumentsAdvanced()
```

这是上传模块最重要的变量。

`docState` 不是普通对象，而是 `useDocumentsAdvanced()` 返回的一整套文档管理状态和方法。

里面包括：

```js
{
  documents,
  isUploading,
  uploadProgress,
  uploadPercent,
  uploadedBytes,
  totalBytes,
  error,
  notice,
  loadDocuments,
  uploadDocument,
  deleteDocument,
  initWorker,
  terminateWorker
}
```

所以 `index.vue` 的职责不是自己处理上传，而是创建这个状态对象，然后交给子组件使用。

###### `showDocPanel`

```js
const showDocPanel = ref(false)
```

它控制知识库面板是否显示。

```
false：不显示上传面板
true：显示上传面板
```

这个变量对应 template 里的：

```vue
<DocumentUpload v-if="showDocPanel" />
```

以及按钮上的：

```vue
@click="showDocPanel = !showDocPanel"
```

#### 3. 核心函数 / 生命周期

###### `onMounted`

```js
onMounted(async () => {
  await chatStore.loadConversations()
  await docState.loadDocuments()
  docState.initWorker()
})
```

它做了三件事：

1. 加载历史会话。
2. 加载知识库文档列表。
3. 初始化文本分块用的 Web Worker。

对上传模块来说，关键是后两步：

```js
await docState.loadDocuments()
docState.initWorker()
```

这表示首页一进入，就会先把已有文档列表拉出来，这样知识库按钮上的数量和上传面板里的文档列表都是最新的。

同时提前初始化 Worker，后续上传文档并解析文本后，就可以直接把大文本交给 Worker 分块。

###### `onUnmounted`

```js
onUnmounted(() => {  docState.terminateWorker()})
```

页面销毁时关闭 Worker。

这个设计是合理的，因为 Worker 是独立线程，如果页面离开后不销毁，可能会造成资源占用。

#### 4. 这个页面在上传链路中的作用

可以这样总结：

```
index.vue 不负责具体上传
index.vue 只负责创建 docState
index.vue 控制 DocumentUpload 是否显示
index.vue 把 docState 传给 DocumentUpload
index.vue 在生命周期里初始化和销毁 Worker
```

### 知识库上传面板组件

>文件：`app/components/DocumentUpload.vue`

它负责知识库上传面板的 UI，包括：

#### 1.3 上传区域

```
<div  class="upload-zone"  :class="{ dragging: isDragging, uploading: docState.isUploading.value }"  @dragover.prevent="isDragging = true"  @dragleave="isDragging = false"  @drop.prevent="handleDrop"  @click="triggerFileInput">  <input    ref="fileInputRef"    type="file"    accept=".pdf,.txt,.md,.docx"    class="file-input"    @change="handleFileSelect"  />  <div v-if="docState.isUploading.value" class="upload-progress">    上传进度  </div>  <div v-else class="upload-prompt">    上传提示  </div></div>
```

这块是上传组件的核心 UI。

其中：

```

```

表示前端文件选择器只允许选择这些类型。

但注意，这只是浏览器层面的限制，真正安全校验还在服务端 `validateUploadMeta` 里做。

#### 1.4 上传进度区域

```
<div v-if="docState.isUploading.value" class="upload-progress">  <span class="spinner"></span>  <p>{{ docState.uploadProgress.value }}</p>  <div class="progress-track">    <div      class="progress-fill"      :style="{ width: `${docState.uploadPercent.value || 0}%` }"    ></div>  </div>  <p class="upload-stats">    {{ formatSize(docState.uploadedBytes.value) }}    /    {{ formatSize(docState.totalBytes.value) }}    ({{ docState.uploadPercent.value || 0 }}%)  </p></div>
```

它展示四类信息：

4. 文件总大小：`totalBytes`

比如上传过程中会显示：

这部分对应 `useDocumentsAdvanced()` 里的 `updateProgressState()`。

当没有上传任务时，显示上传提示。

它告诉用户：

```
支持 PDF、DOCX、TXT、MD最大 20MB
```

这里和服务端保持一致，服务端 `uploadSessions.js` 里也限制了最大 20MB 和允许类型。

#### 1.6 错误和通知提示

```
<div v-if="docState.error.value" class="alert alert-error">  {{ docState.error.value }}</div><div v-if="docState.notice.value" class="alert alert-info">  {{ docState.notice.value }}</div>
```

这两个区域分别展示：

```
error：上传失败、解析失败、分块失败、向量化失败notice：秒传成功、跳过重复上传等提示
```

比如秒传时，后端返回文件已存在，前端会设置：

```
notice.value = checkResult.message || '文件已存在，已跳过重复上传'
```

#### 1.7 文档列表

```
<div class="doc-list">  <h3 class="list-title">    已上传文档    <span class="count">      {{ docState.documents.value.length }}    </span>  </h3>  <div v-if="!docState.documents.value.length" class="doc-empty">    <p>暂无文档，上传后即可开始问答</p>  </div>  <TransitionGroup name="list" tag="div">    <div      v-for="doc in docState.documents.value"      :key="doc.id"      class="doc-item"    >      ...    </div>  </TransitionGroup></div>
```

这其实就把后端入库状态反馈到了 UI 上。

### 2. 核心变量

#### `props.docState`

```
const props = defineProps({  docState: { type: Object, required: true }})
```

这是整个组件最核心的数据来源。

`DocumentUpload.vue` 不自己维护上传状态，而是全部从 `docState` 里读。

常用字段包括：

```
docState.documentsdocState.isUploadingdocState.uploadProgressdocState.uploadPercentdocState.uploadedBytesdocState.totalBytesdocState.errordocState.noticedocState.uploadDocumentdocState.deleteDocument
```

这种写法的好处是：

```
UI 组件只负责展示和交互上传逻辑全部收敛到 composable父组件负责创建状态子组件负责消费状态
```

---

#### `isDragging`

```
const isDragging = ref(false)
```

它表示当前是否有文件拖拽到上传区域上方。

对应 template 里的动态 class：

```
:class="{ dragging: isDragging, uploading: docState.isUploading.value }"
```

当用户拖拽文件进入区域时：

```
isDragging = true
```

离开或者放下时：

```
isDragging = false
```

它主要用于控制上传区域的视觉反馈。

---

#### `fileInputRef`

```
const fileInputRef = ref(null)
```

它指向隐藏的文件选择 input。

template 里：

```
<input  ref="fileInputRef"  type="file"  class="file-input"/>
```

因为 input 被隐藏了，所以点击上传区域时，要通过 JS 主动触发：

```
fileInputRef.value?.click()
```

这样用户体验上就是“点击整个上传区域都可以选择文件”。

### 3. 核心函数

#### `triggerFileInput`

```
function triggerFileInput() {  if (!props.docState.isUploading.value) {    fileInputRef.value?.click()  }}
```

它的作用是点击上传区域时，打开文件选择窗口。

这里有一个细节：

```
if (!props.docState.isUploading.value)
```

如果当前正在上传，就不允许再次打开文件选择器。

这样可以避免同一时间重复触发多个上传任务。

---

#### `handleFileSelect`

```
async function handleFileSelect(e) {  const file = e.target.files?.[0]  if (file) {    await props.docState.uploadDocument(file)    e.target.value = ''  }}
```

它处理点击选择文件后的逻辑。

流程是：

```
用户点击上传区域  ↓触发隐藏 input  ↓用户选择文件  ↓handleFileSelect 拿到 file  ↓调用 docState.uploadDocument(file)  ↓清空 input value
```

这里清空：

```
e.target.value = ''
```

是一个很重要的小细节。

如果不清空，用户连续选择同一个文件时，`change` 事件可能不会再次触发。

---

#### `handleDrop`

```
async function handleDrop(e) {  isDragging.value = false  const file = e.dataTransfer?.files?.[0]  if (file) {    await props.docState.uploadDocument(file)  }}
```

它处理拖拽上传。

流程是：

```
用户拖拽文件到上传区域  ↓drop 事件触发  ↓从 e.dataTransfer.files[0] 取出文件  ↓调用 docState.uploadDocument(file)
```

它和 `handleFileSelect` 最终都会走同一个上传入口：

```
props.docState.uploadDocument(file)
```

也就是说：

```
点击上传拖拽上传   ↓统一进入 uploadDocument(file)
```

这保证了两种交互方式复用同一套上传逻辑。

---

#### `confirmDeleteDoc`

```
function confirmDeleteDoc(id) {  if (window.confirm('删除文档将同时删除其向量数据，确定吗？')) {    props.docState.deleteDocument(id)  }}
```

它负责删除文档前的确认。

这里的提示文案很关键：

```
删除文档将同时删除其向量数据
```

因为 RAG 系统里删除文档不是只删列表记录，还要删除对应向量数据，否则用户删除文档后，聊天检索时仍然可能召回旧内容。

---

#### `formatSize`

```
function formatSize(bytes) {  if (!bytes) return '0 B'  const units = ['B', 'KB', 'MB', 'GB']  let i = 0  let size = bytes  while (size >= 1024 && i < units.length - 1) {    size /= 1024    i++  }  return `${size.toFixed(1)} ${units[i]}`}
```

它负责把字节数格式化成人类可读的单位。

比如：

```
1024      → 1.0 KB1048576   → 1.0 MB
```

这个函数在两个地方用到：

```
上传进度：uploadedBytes / totalBytes文档列表：doc.size
```

### 4. 这个组件在上传链路中的作用

可以这样总结：

```
DocumentUpload.vue 是上传 UI 层它不做哈希计算它不做分片切割它不直接请求上传接口它只负责：  1. 选择文件  2. 拖拽文件  3. 调用 uploadDocument(file)  4. 展示上传状态  5. 展示文档列表  6. 调用 deleteDocument(id)
```

面试里可以这样说：

> `DocumentUpload.vue` 是知识库上传面板组件，它负责上传区域、拖拽上传、点击选择文件、上传进度展示、错误提示、通知提示和文档列表展示。它本身不处理分片上传，而是通过父组件传入的 `docState` 调用 `uploadDocument(file)`。这样做可以把 UI 和业务逻辑拆开，组件只关心用户交互和状态展示，具体上传流程交给 `useDocumentsAdvanced()` 统一编排。
