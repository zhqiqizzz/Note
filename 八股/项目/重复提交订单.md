在电商场景下，防止用户因网络卡顿或手速过快而重复提交订单（导致重复扣款、重复下单），是前端必须守住的**第一道防线**。

单纯依赖后端去重（幂等性）虽然能兜底，但会增加服务器压力且用户体验不好（用户会看到多次“提交中”或报错）。

以下是 **5 种层层递进的防御策略**，建议**组合使用**：

### 1. 按钮状态锁 (UI Lock)

**原理**：点击后立即禁用按钮，直到请求结束。  
**适用**：绝大多数普通表单提交。

```vue
<template>
  <button 
    @click="submitOrder" 
    :disabled="isSubmitting"
    class="submit-btn"
  >
    {{ isSubmitting ? '提交中...' : '立即支付' }}
  </button>
</template>

<script setup>
import { ref } from 'vue';
import { createOrderApi } from '@/api/order';

const isSubmitting = ref(false);

const submitOrder = async () => {
  // 1. 如果正在提交，直接返回（防抖核心）
  if (isSubmitting.value) return;

  try {
    isSubmitting.value = true; // 🔒 上锁
    
    await createOrderApi({ /* 订单数据 */ });
    
    // 成功跳转
    router.push('/success');
  } catch (error) {
    // 失败提示，但通常**不要**立即解锁，让用户手动重试或等待超时
    // 或者根据错误类型决定是否解锁
    console.error(error);
  } finally {
    // ⚠️ 注意：如果是网络完全断开导致的超时，这里解锁后用户可能会再次点击
    // 最佳实践：成功后跳转，失败后保留锁定状态几秒或显示“重试”按钮
    setTimeout(() => {
      isSubmitting.value = false; 
    }, 2000); 
  }
};
</script>
```

**缺点**：如果用户通过浏览器的“后退/前进”或快速刷新，状态可能重置。

### 2. 请求级防重 (Request Deduplication)

**原理**：在 Axios 拦截器层面，对**相同参数**的正在进行的请求进行拦截。如果上一个请求还没回来，新的相同请求直接取消。  
**适用**：用户疯狂连点，或者组件重复渲染导致多次调用。

**实现逻辑**（结合之前封装的 Axios）：  
维护一个 `Map`，Key 是 `方法+URL+参数` 的哈希值，Value 是 `AbortController`。

`abort()` 是 **`AbortController`** 对象的一个方法，它的核心作用是：**立即强制终止（取消）一个正在进行的网络请求（或任何关联了该 signal 的异步操作）**。

在 Axios 和前端开发中，它主要用于解决 **“请求竞态”**、**“节省资源”** 和 **“防止重复提交”** 问题。

#### 1. `abort()` 到底做了什么？

当你调用 `controller.abort()` 时，会发生以下连锁反应：

1. **切断连接**：
    - 浏览器会立即关闭与该请求对应的 TCP 连接。
    - 如果服务器正在发送大数据（如下载文件、大列表），数据传输会**立刻停止**，节省用户流量和带宽。
2. **触发异常**：
    - 该请求对应的 `Promise` 会立即变为 **`rejected`** 状态。
    - 抛出的错误对象通常是一个 `DOMException`，其 `name` 为 `"AbortError"`，`message` 为 "The operation was aborted"。
3. **清理资源**：
    - 浏览器会释放与该请求相关的内存和网络句柄。

#### 2. 如何使用

##### 步骤 A：创建控制器并绑定信号


```javascript
// 1. 实例化控制器
const controller = new AbortController();

// 2. 将 controller.signal 传递给 axios
axios.get('/api/large-data', {
  signal: controller.signal // 👈 关键：建立关联
})
.then(res => console.log('成功', res))
.catch(err => {
  // 3. 捕获错误
  if (err.name === 'AbortError') {
    console.log('⚠️ 请求被用户手动取消了！');
  } else {
    console.error('其他错误', err);
  }
});
```

##### 步骤 B：在需要时调用 abort()

```javascript
// 比如用户点击了“取消”按钮，或者组件卸载了
button.addEventListener('click', () => {
  controller.abort(); // 👈 调用这里，上面的请求立刻中断
});
```

#### 3. 代码演示

```typescript
// utils/request.ts (增强版)
const pendingRequests = new Map<string, AbortController>();

const generateReqKey = (config: InternalAxiosRequestConfig) => {
  const { method, url, params, data } = config;
  return [method, url, JSON.stringify(params), JSON.stringify(data)].join('&');
};

service.interceptors.request.use((config) => {
  const reqKey = generateReqKey(config);
  
  // 如果存在相同的未完成请求，取消它
  if (pendingRequests.has(reqKey)) {
    const controller = pendingRequests.get(reqKey);
    controller?.abort(); // 取消旧请求
    console.warn('⚠️ 检测到重复请求，已自动取消:', reqKey);
    // 注意：这里可以选择直接 return config (允许新请求覆盖旧请求) 
    // 或者 return Promise.reject() (直接丢弃新请求，保留旧请求) -> **推荐后者用于提交操作**
    return Promise.reject(new Error('Duplicate Request'));
  }

  // 创建新的控制器
  const controller = new AbortController();
  config.signal = controller.signal;
  pendingRequests.set(reqKey, controller);

  return config;
});

service.interceptors.response.use((response) => {
  const reqKey = generateReqKey(response.config);
  pendingRequests.delete(reqKey); // 请求完成，移除记录
  return response;
}, (error) => {
  const reqKey = generateReqKey(error.config);
  pendingRequests.delete(reqKey);
  
  // 如果是人为取消的重复请求，静默处理，不弹窗报错
  if (axios.isCancel(error) && error.message === 'Duplicate Request') {
    return Promise.reject(error); 
  }
  return Promise.reject(error);
});
```

**效果**：用户点击 10 次，只有第 1 次会发出去，后面 9 次在发出前就被拦截器掐断了。

### 3. 防抖 (Debounce) vs 节流 (Throttle)

- **防抖 (Debounce)**：n 秒内只执行最后一次。
    - ❌ **不适用于提交订单**。因为用户点完后，你可能想要立即发送，而不是等他停手。如果用户点了一下就卡住了，防抖会导致请求一直不发。
- **节流 (Throttle)**：n 秒内只执行第一次。
    - ✅ **适用于提交**。例如设置 `delay: 1000`，1 秒内的所有点击只触发一次请求。

```javascript
// 简单的节流函数
function throttle(fn, delay) {
  let lastTime = 0;
  return function(...args) {
    const now = Date.now();
    if (now - lastTime >= delay) {
      lastTime = now;
      fn.apply(this, args);
    }
  };
}

// 使用
const throttledSubmit = throttle(submitOrder, 1000);
// <button @click="throttledSubmit">
```

**建议**：通常**按钮锁 (方案 1)** 比 节流函数更好用，因为按钮锁有明确的视觉反馈（Loading 态），而节流用户可能不知道自己的点击被忽略了。

### 总结

对于**电商提交订单**这种高危场景，请严格执行以下组合：

1. **UI 层**：点击后立即 **禁用按钮** + **显示 Loading 转圈** + **文字变为“提交中”**。（防止肉眼可见的连点）
2. **逻辑层**：使用 **请求拦截器去重**（基于 URL+Params 的 Map），确保代码层面不会发出第二个同名请求。（防止代码逻辑漏洞或组件重复渲染）
3. **协议层**：生成 **唯一幂等 Token (UUID)** 传给后端。（防止网络波动导致的重试、多标签页操作、以及前端防御失效后的最后防线）
4. **后端层**：后端必须实现 **幂等性校验**（基于 Token 查 Redis）。（绝对兜底）
