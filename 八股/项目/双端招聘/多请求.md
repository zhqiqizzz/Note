这是前端开发中**搜索、筛选、分页、Tab 切换**场景下的高频经典问题，结合你的招聘系统场景（比如求职者快速切换「前端 / 后端 / 产品」职位筛选、HR 快速翻页看简历列表），我给你 3 套从「简单保底」到「体验最优」的落地方案，全部配 Vue3 + Axios 代码示例。

## 一、先搞懂问题是怎么发生的（场景还原）

### 招聘系统职位列表场景

1. 用户点击「前端」筛选 → 发起请求 A（`/api/job?type=frontend`）
2. 请求 A 还没回来，用户又快速点击「后端」 → 发起请求 B（`/api/job?type=backend`）
3. **网络波动**：请求 B 先回来，列表显示「后端职位」
4. 紧接着请求 A 也回来了，列表被覆盖成「前端职位」
5. **结果**：用户选的是「后端」，但列表显示的是「前端」，逻辑完全错乱。

### 核心原因

- 请求是**异步并发**的，返回顺序不可控；
- 旧请求后回来，直接覆盖了新请求的结果。

## 二、方案一：请求标识 + 忽略旧请求（最简单保底方案）

### 核心思路

给每个请求加一个**唯一标识（ID / 时间戳）**，只保留「最新请求的标识」，请求回来时判断：如果不是最新的标识，直接忽略结果，不更新列表。

### 完整代码示例（Vue3 + Axios）

```vue
<template>
  <div class="job-list">
    <!-- 筛选Tab -->
    <div class="tabs">
      <div
        v-for="tab in tabs"
        :key="tab.value"
        :class="{ active: currentTab === tab.value }"
        @click="changeTab(tab.value)"
      >
        {{ tab.label }}
      </div>
    </div>

    <!-- 职位列表 -->
    <div v-loading="loading" class="list">
      <div v-for="job in jobList" :key="job.id" class="job-item">
        {{ job.title }}
      </div>
    </div>
  </div>
</template>

<script setup>
import { ref } from 'vue'
import request from '@/utils/request'

const tabs = [
  { label: '全部', value: 'all' },
  { label: '前端', value: 'frontend' },
  { label: '后端', value: 'backend' },
  { label: '产品', value: 'pm' }
]
const currentTab = ref('all')
const jobList = ref([])
const loading = ref(false)

// ==================== 核心：记录最新请求的标识 ====================
let latestRequestId = 0

const getJobList = async (type) => {
  loading.value = true
  // 1. 生成当前请求的唯一标识（时间戳即可）
  const currentRequestId = Date.now()
  // 2. 更新「最新请求标识」
  latestRequestId = currentRequestId

  try {
    const res = await request.get('/api/job', { params: { type } })

    // ==================== 核心：判断是不是最新请求的结果 ====================
    if (currentRequestId !== latestRequestId) {
      // 不是最新的，直接忽略，不更新列表
      console.log('旧请求结果，已忽略')
      return
    }

    // 是最新的，才更新列表
    jobList.value = res.data.list
  } catch (e) {
    console.error('请求失败', e)
  } finally {
    // 只有最新请求结束才关闭loading，避免闪烁
    if (currentRequestId === latestRequestId) {
      loading.value = false
    }
  }
}

const changeTab = (type) => {
  currentTab.value = type
  getJobList(type)
}

// 初始化
getJobList('all')
</script>
```

### 方案优缺点

|优点|缺点|
|---|---|
|代码简单，无需修改请求库，兼容性最好|旧请求依然会发出去，浪费服务器资源和流量|
|逻辑清晰，容易理解|无法真正取消请求，只是前端忽略结果|
## 三、方案二：AbortController 取消旧请求（体验最优，推荐方案）

### 核心思路

利用浏览器原生的 **`AbortController`** API，发起新请求前，**主动取消上一个未完成的请求**，从根源上避免旧请求回来覆盖。

> 这是目前行业内的标准方案，Axios、Fetch 原生支持。

### 完整代码示例（Vue3 + Axios）

```vue
<template>
  <div class="job-list">
    <div class="tabs">
      <div
        v-for="tab in tabs"
        :key="tab.value"
        :class="{ active: currentTab === tab.value }"
        @click="changeTab(tab.value)"
      >
        {{ tab.label }}
      </div>
    </div>
    <div v-loading="loading" class="list">
      <div v-for="job in jobList" :key="job.id" class="job-item">
        {{ job.title }}
      </div>
    </div>
  </div>
</template>

<script setup>
import { ref, onUnmounted } from 'vue'
import request from '@/utils/request'

const tabs = [
  { label: '全部', value: 'all' },
  { label: '前端', value: 'frontend' },
  { label: '后端', value: 'backend' }
]
const currentTab = ref('all')
const jobList = ref([])
const loading = ref(false)

// ==================== 核心：保存上一个请求的 AbortController ====================
let abortController = null

const getJobList = async (type) => {
  // 1. 发起新请求前，先取消上一个未完成的请求
  if (abortController) {
    abortController.abort()
  }

  // 2. 为当前请求创建新的 AbortController
  abortController = new AbortController()
  const signal = abortController.signal

  loading.value = true
  try {
    // 3. 把 signal 传给 Axios
    const res = await request.get('/api/job', {
      params: { type },
      signal // 核心：传入 signal
    })

    // 只有成功回来的请求才更新列表（被取消的请求会走 catch，不会到这里）
    jobList.value = res.data.list
  } catch (e) {
    // 核心：判断是不是「主动取消」的错误，如果是，不报错
    if (e.name === 'CanceledError' || e.message === 'canceled') {
      console.log('上一个请求已被取消')
      return
    }
    console.error('请求失败', e)
  } finally {
    loading.value = false
  }
}

const changeTab = (type) => {
  currentTab.value = type
  getJobList(type)
}

// ==================== 组件卸载时，取消未完成的请求 ====================
onUnmounted(() => {
  if (abortController) {
    abortController.abort()
  }
})

getJobList('all')
</script>
```

### 方案优缺点

|优点|缺点|
|---|---|
|真正取消请求，节省服务器资源和流量|需要浏览器支持 AbortController（现代浏览器都支持，IE 不行）|
|从根源上避免旧请求覆盖，体验最好|需要稍微修改请求逻辑|
|组件卸载时可以取消请求，避免内存泄漏||

## 四、方案三：防抖（Debounce）+ 方案二（从根源减少并发）

### 核心思路

如果用户快速点击筛选（比如 1 秒内点了 5 次），前 4 次请求其实都是无效的，直接用**防抖**延迟发起请求，等用户停止操作后（比如 300ms）再发起请求，配合方案二的取消请求，体验拉满。

### 完整代码示例（Vue3 + Lodash Debounce）

```bash
# 安装 lodash.debounce（也可以自己手写防抖）
npm i lodash.debounce
```

```vue
<template>
  <div class="job-list">
    <!-- 搜索框 + 筛选Tab -->
    <div class="search-bar">
      <input v-model="searchKeyword" placeholder="搜索职位" />
    </div>
    <div class="tabs">
      <div
        v-for="tab in tabs"
        :key="tab.value"
        :class="{ active: currentTab === tab.value }"
        @click="changeTab(tab.value)"
      >
        {{ tab.label }}
      </div>
    </div>
    <div v-loading="loading" class="list">
      <div v-for="job in jobList" :key="job.id" class="job-item">
        {{ job.title }}
      </div>
    </div>
  </div>
</template>

<script setup>
import { ref, watch, onUnmounted } from 'vue'
import debounce from 'lodash.debounce'
import request from '@/utils/request'

const tabs = [
  { label: '全部', value: 'all' },
  { label: '前端', value: 'frontend' },
  { label: '后端', value: 'backend' }
]
const currentTab = ref('all')
const searchKeyword = ref('')
const jobList = ref([])
const loading = ref(false)
let abortController = null

// ==================== 核心：防抖 + 取消请求 ====================
const getJobList = debounce(async (keyword, type) => {
  // 取消上一个请求
  if (abortController) abortController.abort()
  
  abortController = new AbortController()
  loading.value = true

  try {
    const res = await request.get('/api/job', {
      params: { keyword, type },
      signal: abortController.signal
    })
    jobList.value = res.data.list
  } catch (e) {
    if (e.name === 'CanceledError') return
    console.error('请求失败', e)
  } finally {
    loading.value = false
  }
}, 300) // 防抖延迟300ms

// 监听搜索框变化，防抖请求
watch(searchKeyword, (newVal) => {
  getJobList(newVal, currentTab.value)
})

const changeTab = (type) => {
  currentTab.value = type
  getJobList(searchKeyword.value, type)
}

// 组件卸载时取消请求
onUnmounted(() => {
  if (abortController) abortController.abort()
  // 取消防抖定时器
  getJobList.cancel()
})

getJobList('', 'all')
</script>
```

### 方案优缺点

|优点|缺点|
|---|---|
|从根源上减少请求数量，大幅降低服务器压力|有 300ms 延迟，用户体验稍微有一点点影响（但 300ms 用户感知不强）|
|配合取消请求，双重保险，绝对不会出现覆盖||

## 五、三种方案对比与选择建议

| 方案                        | 适用场景                | 推荐指数       |
| ------------------------- | ------------------- | ---------- |
| 方案一：请求标识 + 忽略旧请求          | 简单场景、不想改请求库、兼容老浏览器  | ⭐⭐⭐        |
| 方案二：AbortController 取消旧请求 | 绝大多数场景、体验要求高        | ⭐⭐⭐⭐⭐      |
| 方案三：防抖 + 取消请求             | 搜索框、快速切换 Tab、高频操作场景 | ⭐⭐⭐⭐⭐（最推荐） |

## 六、Axios 封装优化

如果你不想在每个页面都写 `AbortController`，可以在封装 Axios 时全局处理：

```javascript
// utils/request.js
import axios from 'axios'

// 存储每个接口的 abortController
const pendingRequests = new Map()

// 生成请求的唯一key
const generateRequestKey = (config) => {
  return `${config.method}-${config.url}-${JSON.stringify(config.params || config.data)}`
}

const request = axios.create({ baseURL: '/api' })

// 请求拦截器
request.interceptors.request.use(config => {
  // 1. 生成当前请求的key
  const key = generateRequestKey(config)
  // 2. 如果之前有相同的请求，取消它
  if (pendingRequests.has(key)) {
    pendingRequests.get(key).abort()
  }
  // 3. 创建新的 abortController，存起来
  const abortController = new AbortController()
  config.signal = abortController.signal
  pendingRequests.set(key, abortController)
  
  // 携带token
  config.headers.Authorization = 'Bearer ' + localStorage.getItem('token')
  return config
})

// 响应拦截器
request.interceptors.response.use(
  res => {
    // 请求成功，从 pendingRequests 中删除
    const key = generateRequestKey(res.config)
    pendingRequests.delete(key)
    return res.data
  },
  error => {
    // 主动取消的错误，不处理
    if (error.name === 'CanceledError') {
      return Promise.reject(error)
    }
    // 其他错误，从 pendingRequests 中删除
    if (error.config) {
      const key = generateRequestKey(error.config)
      pendingRequests.delete(key)
    }
    return Promise.reject(error)
  }
)

export default request
```

封装后，你在页面里直接正常调用接口即可，**不需要写任何 AbortController 代码**，全局自动处理旧请求取消，绝对不会出现覆盖问题。
