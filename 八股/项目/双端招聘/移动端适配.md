_**基于 [PostCSS](../../vue/Vue/PostCSS.md) 实现以下功能**_
### 一、移动端适配：px 转 rem / vw

在移动端开发中，设计师通常给的是 **750px** (iPhone 6/7/8 Plus 逻辑像素) 或 **375px** 的设计稿。但手机屏幕千差万别（从 320px 到 400+ px 宽度不等）。如果直接写 `px`，在不同屏幕上要么太大要么太小。

**解决方案核心**：使用相对单位（`rem` 或 `vw`），配合 PostCSS 插件，**在构建时自动将设计稿的 `px` 转换为相对单位**。

#### 1. 方案对比：Rem vs VW

| 特性             | **Rem (`root em`)**                                                            | **VW (`Viewport Width`)**     |
| -------------- | ------------------------------------------------------------------------------ | ----------------------------- |
| **定义**         | 相对于根元素 `<html>` 的 `font-size`。                                                 | 相对于视口宽度的百分比 (1vw = 1% 视口宽度)。  |
| **原理**         | 1. JS 动态设置 `<html>` 的 `font-size`。  <br>2. CSS 写 `rem`，浏览器根据 html 字体大小计算实际 px。 | 直接使用 `vw`，浏览器根据当前屏幕宽度自动计算。    |
| **优点**         | 兼容性好，可控性强（可通过修改 html font-size 整体缩放）。                                          | 无需 JS 参与，纯 CSS 方案，性能略好，代码更直观。 |
| **缺点**         | 需要引入少量 JS (如 `lib-flexible`) 来动态计算 font-size。                                  | 在某些极端场景（如字体大小限制、输入框）可能需要额外处理。 |
| **PostCSS 插件** | `postcss-pxtorem`                                                              | `postcss-px-to-viewport` (推荐) |

> **目前趋势**：**`vw` 方案越来越流行**，因为现代浏览器支持极好，且不需要依赖 JS 去动态设置 font-size，更纯粹。但 `rem` 方案在一些老项目或需要精细控制缩放比例的场景依然常用。

#### 2. 实战配置 (以 Vite + Vue 为例)

假设设计稿宽度为 **750px**。

##### 方案 A：使用 `postcss-px-to-viewport` (转 vw，推荐)

这是目前最主流的方案，无需 JS 辅助，纯 CSS 解决。

1. **安装插件**：
    
```bash
    npm install -D postcss-px-to-viewport-8-plugin
    # 注意：原版 postcss-px-to-viewport 已停止维护，推荐使用社区维护的 -8-plugin 版本以兼容 PostCSS 8+
```
    
2. **创建配置文件**：  
    在项目根目录创建 `postcss.config.js` (或 `.cjs`)：
    
```javascript
    // postcss.config.js
    export default {
      plugins: {
        'postcss-px-to-viewport-8-plugin': {
          viewportWidth: 750, // 设计稿宽度，一般是 750 或 375
          viewportUnit: 'vw', // 转换成的单位，默认就是 vw
          unitPrecision: 5,   // 小数点后保留位数
          propList: ['*'],    // 需要转换的属性，'*' 表示所有属性
          selectorBlackList: [], // 黑名单，这些选择器里的 px 不转换 (如 '.no-convert')
          minPixelValue: 1,   // 小于等于 1px 的不转换 (避免 1px 边框问题)
          mediaQuery: false,  // 是否在 @media 查询中也转换
        },
      },
    };
```
    
3. **编写代码**：  
    现在你可以**直接按照设计稿写 `px`**！
    
```css
    /* 设计稿是 750px 宽，这里写 375px */
    .box {
      width: 375px; 
      height: 100px;
      font-size: 32px;
    }
```
    
**构建后自动变成**：
当你运行 `npm run dev` (开发环境) 或 `npm run build` (生产环境) 时，Vite (或 Webpack) 会启动构建流程：

1. 读取你的 `.vue` 或 `.css` 文件。
2. 交给 **PostCSS** 处理。
3. PostCSS 加载你配置的 `postcss-px-to-viewport` 插件。
4. 插件遍历代码，发现 `750px`，根据配置的 `viewportWidth: 750` 进行计算：750÷750=100%=100vw。
5. **输出结果**：浏览器最终接收到的 CSS 已经是转换后的单位了。
	
```css
.box { 
	width: 50vw; /* 375 / 750 = 0.5 = 50% */ 
	height: 13.33333vw; 
	font-size: 4.26667vw; 
}
```

- **结论**：你全程只需要写 `px`，转换工作由构建工具在**编译阶段**自动完成。

##### 方案 B：使用 `postcss-pxtorem` (转 rem)

如果你需要兼容非常老的设备，或者团队习惯用 rem。

1. **安装依赖**：
    
```bash
    npm install -D postcss-pxtorem
    npm install -D amfe-flexible # 或者 lib-flexible，用于动态设置 html font-size
```
    
2. **引入 JS 适配脚本** (在 `main.js` 入口文件)：
    
```javascript
    import 'amfe-flexible'; 
    // 这个脚本会自动根据屏幕宽度设置 <html> 的 font-size
    // 例如：屏幕 375px -> font-size 设为 37.5px (假设基准是 10)
```
    
3. **配置 PostCSS** (`postcss.config.js`)：
    
```javascript
    export default {
      plugins: {
        'postcss-pxtorem': {
          rootValue: 75, // 关键！设计稿750px，通常设为 75 (即 750/10) 或 100
          // 如果 rootValue=75，那么 75px = 1rem。
          // 设计稿 375px -> 375/75 = 5rem
          propList: ['*'],
          selectorBlackList: [],
        },
      },
    };
```
    
4. **编写代码**：  
    同样直接写 `px`：
	
```css
    .box {
      width: 375px; /* 构建后变为 5rem (假设 rootValue=75) */
    }
```

#### ⚠️ 注意事项

- **不要转换所有文件**：如果是 PC 端页面或后台管理系统，通常不需要转换。可以通过配置 `include/exclude` 只处理特定目录（如 `/src/mobile`）。
- **1px 边框问题**：在高分屏（Retina）上，`1px` 可能看起来太粗。通常配合 `postcss-px-to-viewport` 的 `minPixelValue` 设置，或者使用 `border-image` / `transform: scale(0.5)` 来解决物理 1px 问题。
- **第三方库**：如果引入了 UI 库（如 Vant），它们通常已经内置了适配方案，配置时要注意避免重复转换或冲突（使用 `selectorBlackList` 排除 `.van-` 开头的类名）。

### 二、使用 Tailwind CSS

Tailwind CSS 是一个**实用优先 (Utility-First)** 的 CSS 框架。它不提供现成的组件（如 `.btn`），而是提供大量的原子类（如 `flex`, `pt-4`, `text-center`），让你直接在 HTML 中组合出样式。

**PostCSS 在其中的角色**：Tailwind CSS **本身就是作为一个 PostCSS 插件运行的**。它扫描你的 HTML/JS/Vue 文件，提取出你用到的类名，然后生成对应的 CSS。

#### 1. 为什么需要 PostCSS？

Tailwind 的核心功能（按需生成 CSS、前缀处理、压缩）完全依赖 PostCSS 引擎来执行。没有 PostCSS，Tailwind 无法工作。

#### 2. 实战配置 (Vite + Vue 3 + Tailwind CSS v3/v4)

> **注**：Tailwind CSS v4 正在逐步普及，它简化了配置，甚至不再强制需要 `tailwind.config.js`，但目前主流稳定版仍是 v3。以下以 **v3** 为主，兼顾 v4 趋势。

##### 步骤 1：安装依赖

```bash
npm install -D tailwindcss postcss autoprefixer
```

##### 步骤 2：初始化配置

运行以下命令生成配置文件：

```bash
npx tailwindcss init -p
```

这会生成两个文件：

- `tailwind.config.js`：Tailwind 的配置文件（定义主题、颜色、扫描路径等）。
- `postcss.config.js`：PostCSS 的配置文件（自动注册了 tailwindcss 和 autoprefixer 插件）。

生成的 `postcss.config.js` 长这样：

```javascript
// postcss.config.js
export default {
  plugins: {
    tailwindcss: {},
    autoprefixer: {},
  },
}
```

**看！这就是 PostCSS 的典型用法：加载 `tailwindcss` 插件。**

##### 步骤 3：配置扫描路径 (`tailwind.config.js`)

告诉 Tailwind 去哪里查找你写的类名，以便生成对应的 CSS。

```javascript
/** @type {import('tailwindcss').Config} */
export default {
  content: [
    "./index.html",
    "./src/**/*.{vue,js,ts,jsx,tsx}", // 扫描 src 下所有 vue/js 文件
  ],
  theme: {
    extend: {
      // 可以在这里自定义主题颜色、字体等
      colors: {
        primary: '#3498db',
      }
    },
  },
  plugins: [],
}
```

##### 步骤 4：在 CSS 中引入 Tailwind

在你的全局样式文件（如 `src/style.css` 或 `src/main.css`）顶部添加：

```css
@tailwind base;
@tailwind components;
@tailwind utilities;
```

这三行指令会被 PostCSS 处理，替换为 Tailwind 生成的重置样式、组件类和工具类。

##### 步骤 5：开始使用

现在你可以在 Vue 模板中直接写原子类：

```vue
<template>
  <div class="flex items-center justify-center h-screen bg-gray-100">
    <button class="px-4 py-2 text-white bg-primary rounded hover:bg-blue-600">
      点击我
    </button>
  </div>
</template>
```

**构建时**：PostCSS 调用 Tailwind 插件 -> 扫描文件发现 `bg-primary` -> 查找配置中的 `primary` 颜色 -> 生成 `.bg-primary { background-color: #3498db; }` -> 输出最终 CSS。

#### Tailwind v4 的新变化 (前瞻)

Tailwind v4 (目前处于 Beta/RC 阶段) 进行了重大重构：

- **零配置**：不再强制需要 `tailwind.config.js`，可以直接在 CSS 中使用 `@theme` 块定义变量。
- **更快的引擎**：基于 Rust 重写，速度极快。
- **PostCSS 依然是核心**：虽然配置简化了，但它依然是作为 PostCSS 插件运行在构建流中。

配置示例 (v4 风格)：

```css
/* style.css */
@import "tailwindcss";

@theme {
  --color-primary: #3498db;
}
```

PostCSS 配置依然简单：

```javascript
export default {
  plugins: {
    "@tailwindcss/postcss": {}, // v4 的插件名可能变化
  },
}
```
### 总结对比

|特性|**移动端适配 (px 转 vw/rem)**|**Tailwind CSS**|
|---|---|---|
|**核心目的**|**响应式布局**：让一套代码适配不同屏幕尺寸。|**开发效率**：通过原子类快速构建 UI，无需想类名。|
|**PostCSS 插件**|`postcss-px-to-viewport` 或 `postcss-pxtorem`|`tailwindcss` (本身就是一个插件)|
|**工作方式**|**转换单位**：把写死的 `px` 变成相对的 `vw/rem`。|**按需生成**：扫描代码，只用到的样式才打包进去。|
|**配置文件**|需配置 `viewportWidth` (设计稿宽度) 等参数。|需配置 `content` (扫描路径) 和 `theme` (主题)。|
|**代码写法**|依然写传统 CSS，只是单位写 `px` (由插件转)。|直接在 HTML class 中写 `w-40`, `text-red-500` 等。|
|**典型场景**|H5 活动页、移动端 Web 应用。|后台管理系统、SaaS 平台、快速原型开发。|

**两者可以共存吗？**  
**完全可以！**  
很多项目会同时使用：

1. 用 **Tailwind** 快速构建布局和组件。
2. 用 **postcss-px-to-viewport** 处理某些必须写 `px` 的特殊场景（比如引用了第三方不带 rem/vw 的 CSS 库，或者设计师强制要求某些元素固定物理像素）。

只需在 `postcss.config.js` 中同时配置两个插件即可：

```javascript
export default {
  plugins: {
    tailwindcss: {},
    'postcss-px-to-viewport-8-plugin': {
      viewportWidth: 750,
      // 注意：通常要把 Tailwind 生成的类名加入黑名单，避免重复转换或冲突
      selectorBlackList: ['.tl-'], 
    },
    autoprefixer: {},
  },
}
```

## 三、vw 和 rem对比
### 1. 彻底摆脱 JavaScript 依赖（最核心原因）

- **Rem 方案的痛点**：  
    在 `rem` 方案中，CSS 单位是相对的（相对于 `<html>` 的 `font-size`）。但屏幕宽度是动态的，浏览器不会自动根据屏幕宽度去修改 `font-size`。  
    因此，你必须引入一段 **JavaScript 代码**（如 `lib-flexible`），在页面加载时计算屏幕宽度，并动态设置 `<html style="font-size: ...">`。
    
    - **风险**：如果 JS 加载慢了、报错了、或者执行晚了，页面会出现**“闪屏”**（先显示默认大小，突然跳动变正常）或**布局错乱**。
    - **依赖**：没有 JS，适配就失效了。
- **VW 方案的优势**：  
    `vw` (Viewport Width) 是 **CSS 原生单位**。
    
    - **定义**：`1vw` = 视口宽度的 1%。
    - **机制**：浏览器**原生支持**，不需要任何 JS 计算。当屏幕宽度变化（如旋转屏幕、调整窗口大小时），浏览器会自动重新计算所有 `vw` 单位的值。
    - **结果**：**纯 CSS 解决方案**。即使 JS 挂了、没加载，布局依然是完美适配的。这极大地提高了稳定性和首屏渲染速度。

### 2. 性能更优（减少重排与 JS 执行）

- **Rem 方案**：  
    每次窗口大小改变（resize），JS 监听器需要触发 -> 计算新的 font-size -> 修改 DOM 样式 -> 浏览器触发重排（Reflow）。虽然现代优化做得很好，但这依然是一个“JS 驱动”的过程。
- **VW 方案**：  
    这是浏览器内核层面的能力。浏览器在渲染引擎（Rendering Engine）内部直接处理 `vw` 的计算，效率极高，且不需要经过 JS 线程。对于动画和复杂布局，`vw` 通常表现得更流畅。

### 3. 逻辑更直观，换算更简单

- **Rem 方案**：  
    你需要维护一个“基准值”概念。
    - 设计稿 750px，基准设为 75px (10倍) 还是 100px？
    - 写代码时心里要默算：`375px` / `75` = `5rem`。
    - 如果不小心改了 `rootValue` 配置，整个项目的尺寸比例都要重新评估。
- **VW 方案**：  
    逻辑是线性的百分比，非常直观。
    - 设计稿 750px 代表 100% 屏幕宽度。
    - 元素宽 375px -> 就是 50% 屏幕宽度 -> 直接写 `50vw` (或者让插件把 `375px` 转成 `50vw`)。
    - **PostCSS 插件的作用**：正是因为 `vw` 的逻辑如此简单，PostCSS 插件 (`postcss-px-to-viewport`) 才能完美地让你**继续写 `px`**，它在后台帮你做简单的除法 ( px/设计稿宽度∗100px/设计稿宽度∗100 )，开发者完全无感。

### 4. 更好的响应式体验（横竖屏切换）

- **场景**：用户把手机从竖屏转到横屏。
- **Rem**：需要 JS 监听到 `resize` 事件 -> 重新计算 -> 更新 `font-size`。在这个极短的时间差内，可能会出现布局抖动。
- **VW**：浏览器渲染引擎实时响应视口变化，丝滑过渡，没有任何延迟或抖动。
