#### 1. 你在项目里是怎么引入 Element Plus 的？为什么要用按需引入代替全局引入？

- **【原理揭秘】**：
    
    - **全局引入**：把整个 Element Plus 的代码（包括你根本没用到的日历组件、穿梭框组件等）全部打包进了最终的 `vendor.js` 文件里。这会导致打包体积暴增，用户打开网页的首屏白屏时间极长。
        
    - **按需引入**：利用 Vite 插件在**编译阶段**扫描你的 `.vue` 模板文件，发现你写了 `<el-button>`，就在底层自动帮你 `import { ElButton } from 'element-plus'` 并引入对应的 CSS。做到了“用多少，打包多少”（也就是常说的 Tree-shaking 摇树优化）。
        
- **【满分回答话术】**：
    
    > “考虑到电商项目对首屏加载速度要求极高，我没有采用常规的全局引入，而是配置了**自动按需引入**。
    > 
    > 具体来说，我在 `vite.config.js` 中引入了 `unplugin-vue-components` 和 `unplugin-auto-import` 两个插件，并配置了 `ElementPlusResolver`。 这样配置后，我在写业务代码时，完全不需要手动去 import 某个具体的 Element 组件，直接在模板里写标签即可。Vite 在打包编译时，会自动帮我分析依赖，并仅打包我实际用到的组件代码和样式。这极大减小了产物体积，提升了站点的加载性能。”
    

#### 2. 电商项目通常有自己的品牌色（比如兔鲜的绿色），你是如何把 Element Plus 默认的蓝色主题改成你们项目的品牌色的？

- **【原理揭秘】**： Element Plus 底层的样式是使用 SCSS 编写的，它的默认主题色（蓝色）是通过一个 SCSS 变量（如 `$el-color-primary`）控制的。只要我们在它编译之前，用我们自己的变量覆盖掉它的默认变量，就能实现全局主题定制。
    
- **【满分回答话术】**：
    
    > “我是通过**覆盖 Element Plus 底层 SCSS 变量**来实现主题定制的。
    > 
    > 1. 首先，我在 `ElementPlusResolver` 插件配置中，开启了 `importStyle: "sass"`，让它按需导入 SCSS 源文件而不是编译后的 CSS 文件。
    >     
    > 2. 然后，我新建了一个全局的变量文件（比如 `var.scss`），在里面重新定义了 Element 的主色调变量，改成我们电商的品牌色。
    >     
    > 3. 最关键的一步，我在 `vite.config.js` 的 `css.preprocessorOptions.scss` 配置中，利用 `additionalData` 属性，把我的 `var.scss` 和 Element 的基础变量文件自动注入到每一个样式块中。 这样，在整个项目编译时，我们的品牌色就会自动替换掉官方的默认颜色。这种做法不仅改得彻底，而且维护成本极低，不需要去写恶心的 `::v-deep` 强制覆盖。”
    >     
    

#### 3. 看你配置了 `@` 别名（Alias），为什么要配置这个？它的底层原理是什么？

- **【原理揭秘】**： 随着项目文件嵌套越来越深，如果用相对路径导入文件，会出现像 `import user from '../../../stores/user.js'` 这样的“寻址地狱”。配置别名可以将 `@` 映射到系统绝对路径下的 `src` 目录。
    
- **【满分回答话术】**：
    
    > “配置 `@` 别名主要是为了解决复杂目录结构下，文件相互引用的**路径地狱**问题，让代码更易读、重构更方便。
    > 
    > 在 `vite.config.js` 中，我通过 `resolve.alias` 将 `@` 映射到了绝对路径下的 `./src` 目录。其底层原理是：当 Vite 或 Webpack 解析模块的 import 语句时，只要遇到以 `@` 开头的路径，就会直接替换为我们配置好的根绝对路径去寻找文件。 另外，为了让 VSCode 编辑器在写代码时能识别 `@` 并给出路径提示，我还会在根目录配合配置 `jsconfig.json` 或 `tsconfig.json` 的 `paths` 属性。”
    

#### 4. 在前端工程化中，Vite 相对于 Webpack，它的热更新（HMR）为什么这么快？（🌟 高阶原理题）

- **【原理揭秘】**：
    
    - **Webpack 热更新**：你改了一个文件，它可能需要把这个文件所在的整个模块/依赖链条重新走一遍编译流程，项目越大，更新越慢（有时按个保存要等好几秒才刷新浏览器）。
        
    - **Vite 热更新**：Vite 在开发环境下是利用浏览器原生的 ES Module 机制。当一个文件被修改时，Vite 只需要精确地让这一个特定文件在浏览器端的缓存失效，并重新请求这一个文件。因此它的热更新时间几乎是恒定的（毫秒级），不会随着项目体积变大而变慢。
        
- **【满分回答话术】**：
    
    > “Vite 的热更新（HMR）机制和 Webpack 有着本质的区别。
    > 
    > Webpack 在检测到文件修改时，需要重新构建并打包相关的模块，然后通过 WebSocket 将打好的新模块推送给浏览器。项目体积一旦庞大，这个重新构建的过程就会非常耗时。
    > 
    > 而 Vite 充分利用了现代浏览器原生支持 ESM（ES Module）的特性。它构建的是一条由原文件直接映射到浏览器的请求链。当我修改了某个 Vue 组件保存时，Vite 服务器只会精确地把这个被修改的模块标记为‘失效’。浏览器发起请求时，只会重新加载这一个组件的代码包。所以无论我们的电商项目变得多庞大，Vite 的热更新始终能保持在毫秒级别的极速响应，极大地提升了我的开发效率。”

#### 16. 打包优化的最深水区——`package.json` 的 `sideEffects`

我们在前面聊过 Vite 的按需引入（Tree-Shaking），但很多人不知道，Tree-Shaking 是有条件的。

- **【面试官的挑战】**：你说你用了 Vite 打包，那你知道为什么有时候你写了一个工具函数根本没用到，它还是被打包进产物里了吗？怎么彻底剔除它们？
    
- **【满分降维打击】**：
    
    > “因为打包工具（如 Rollup/Webpack）在遇到模块时，出于安全考虑，默认认为所有模块都是有‘副作用 (Side Effects)’的。比如一个 JS 文件里不仅导出了函数，还在顶层写了一句 `window.xxx = 1`，如果打包工具把它删了，整个程序可能就崩了。
    > 
    > 为了实现真正的极致 Tree-Shaking，如果我封装的是纯函数工具库（比如通用的格式化时间工具、数学计算工具），我会在项目的 `package.json` 中手动加上一行 **`"sideEffects": false`**。
    > 
    > 这句话等同于向上帝发誓：‘我这个项目里的所有模块都是纯粹的，没有任何副作用！’ 这样 Vite 在打包时，只要发现哪个 `export` 没被其他文件 `import` 使用，就会毫不留情地把它当做死代码（Dead Code）彻底抹除，将打包体积压缩到绝对的极限。”
    

#### 17. Vite 真的“完全不打包”吗？（依赖预构建 esbuild）

你前面回答了 Vite 是基于 ESM 模块按需加载的（No-Bundle），所以冷启动快。

- **【面试官的终极试探】**：你说 Vite 启动时不打包，那为什么第一次启动 Vite 的时候，控制台会提示 `Optimizing dependencies`（优化依赖）？Vite 底层到底有没有用到打包工具？
    
- **【满分绝杀话术】**：
    
    > “这是一个非常经典的误区。Vite 所谓的‘不打包’，指的是对我们自己写的**业务代码**（如 `.vue` 文件）不打包。但是对于 `node_modules` 里的第三方依赖，Vite 绝对是会进行**依赖预构建（Pre-bundling）**的。
    > 
    > 这里 Vite 用到了 **esbuild**（Go 语言编写，速度极快）。预构建解决两个致命问题：
    > 
    > 1. **CommonJS 兼容**：像 `axios` 等很多老包还是 CommonJS 格式，浏览器原生是不认识 `require()` 的。esbuild 会把它们统一转化为 ESM 格式。
    >     
    > 2. **解决网络瀑布流**：比如以前用到 `lodash-es`，如果你引入一个函数，它内部可能又 import 了 100 个微型文件。如果不打包，浏览器会瞬间发起 100 个 HTTP 请求导致卡死。esbuild 会把这 100 个文件预先打包成单个模块。 所以，Vite 的快，是**‘业务代码按需编译’**和**‘底层依赖 esbuild 极速预构建’**共同配合的结果。”
    >
