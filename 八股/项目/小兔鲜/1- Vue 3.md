### 1. Vue 3 的响应式原理（Proxy vs Object.defineProperty）

这是 Vue 3 面试必问的 Top 1 问题，必须能流利地说出两者差异。

- **【原理揭秘】**：
    
    - **Vue 2 (`Object.defineProperty`)**：只能劫持对象的“属性”。Vue 初始化时需要深度遍历对象的所有属性，给它们加上 `getter` 和 `setter`。
        
        - _致命缺点_：无法监听到对象属性的**新增**和**删除**（所以 Vue 2 需要 `$set` 和 `$delete`）；无法完美监听数组的索引和长度变化；初始化时深度遍历大对象非常消耗性能。
            
    - **Vue 3 (`Proxy`)**：劫持的是“整个对象”。ES6 的 Proxy 可以理解为在目标对象之前架设一层“拦截”，外界对该对象的任何操作（包括新增属性、删除属性、通过索引改变数组）都会经过这层拦截。
        
        - _优势_：原生支持监听对象属性的新增/删除和数组变化；采用“懒处理”模式，只有当访问到深层属性时，才会去把深层属性也变成响应式，极大提升了首屏渲染性能。
            
- **【满分回答话术】**：
    
    > “Vue 3 最大的底层升级就是放弃了 `Object.defineProperty`，全面拥抱了 ES6 的 `Proxy`。
    > 
    > 以前在写 Vue 2 时，最头疼的就是动态给对象新增属性，视图不会更新，必须用 `$set`。这是因为 `defineProperty` 只能劫持初始化时就存在的属性。而且如果数据层级很深，Vue 2 在初始化时就会递归遍历所有属性，非常消耗性能。
    > 
    > Vue 3 的 `Proxy` 是在对象外层包了一层代理，不管我是新增属性、删除属性，还是直接修改数组的索引，`Proxy` 都能直接拦截到。并且 Vue 3 的响应式是‘按需’的，只有在 UI 或逻辑中真正访问到了对象的深层嵌套属性，它才会被转化为响应式对象，这让大列表、复杂表单的初始化性能有了质的飞跃。”
    

### 2. `ref` 和 `reactive` 的底层差异与选型思考

在实际写代码时，天天都在用这两个 API，面试官会通过这个问题考察你对 API 边界的理解。

- **【原理揭秘】**：
    
    - **`reactive`**：它的底层直接调用了 `Proxy`，所以它**只能接收对象或数组**（复杂数据类型）。如果你直接解构它（如 `const { name } = reactive({name: '兔子'})`），解构出来的值会丢失响应式。
        
    - **`ref`**：它是为了解决基本数据类型（字符串、数字、布尔值）无法使用 Proxy 的问题而诞生的。底层封装了一个 `RefImpl` 类，通过类的 `get value()` 和 `set value()` 来实现响应式拦截。如果给 `ref` 传入一个复杂对象，它内部其实会自动调用 `reactive`。
        
- **【满分回答话术】**：
    
    > “在电商项目中，我是这样规范使用它们的：
    > 
    > 对于基本数据类型，比如控制弹窗显隐的布尔值、分页的页码、倒计时的秒数，我毫无疑问会使用 `ref`。
    > 
    > 对于具备明确结构的对象，比如复杂的表单数据模型（登录表单、收货地址）或者接口返回的复杂配置对象，我会使用 `reactive`。这样在模板和逻辑层引用时，直接点属性名即可，不需要满屏的 `.value`，代码可读性更好。
    > 
    > 从底层原理来说，`reactive` 依赖 ES6 的 Proxy，仅支持对象；而 `ref` 包装了一个对象，通过 `.value` 的属性访问器拦截读写。如果必须要对 `reactive` 对象进行解构，我会配合 `toRefs` 辅助函数，它会把对象里的每个属性都包裹成一个单独的 `ref`，从而防止响应式丢失。”
    

### 3. Composition API (组合式) 与 自定义 Hook

你的项目中存在 `src/composables/useCountDown.js` 这种文件，这就是典型的自定义 Hook，也是 Vue 3 解决逻辑复用的终极方案。

- **【原理揭秘】**：
    
    - **Vue 2 (Options API + Mixins)**：数据写在 `data`，方法写在 `methods`，业务逻辑被强行按“组件选项”拆分。如果要复用逻辑，只能用 Mixin。Mixin 的缺点是：变量来源不清晰（不知道是从哪个 Mixin 混入的）、命名容易冲突（两个 Mixin 有同名变量会相互覆盖）。
        
    - **Vue 3 (Composition API + Hooks)**：可以把实现同一个业务功能的所有代码（状态、计算属性、侦听器、生命周期）写在一起，甚至抽离成一个独立的 JS 文件（函数）。
        
- **【满分回答话术】**：
    
    > “组合式 API 最大的价值在于**逻辑复用和代码组织**。
    > 
    > 比如在我的项目中，电商支付页面和验证码发送都会用到‘倒计时’功能。我利用 Composition API 封装了一个 `useCountDown` 的 Hook。我把倒计时的响应式状态（剩余时间）、定时器的开启与清除逻辑（在 `onUnmounted` 中销毁防止内存泄漏）全部封装在这个函数里。
    > 
    > 相比于 Vue 2 的 Mixins，Hook 有三个压倒性优势： 第一，**数据来源极其清晰**。组件通过解构引入 Hook 的返回值（如 `const { time, start } = useCountDown()`），一眼就能看出变量是从哪来的。 第二，**彻底解决了命名冲突**。如果有重名变量，可以在解构时直接重命名。 第三，**TypeScript 支持更完美**。Hook 本质上就是普通函数，TS 的参数和返回值推导非常自然，而 Mixins 的类型推导一直是痛点。”
    

### 4. 编译时优化：`<script setup>` 语法糖到底做了什么？

现代 Vue 3 项目基本全在使用这个语法糖，但很多人不知道它为什么能省去 `return`。

- **【原理揭秘】**： `<script setup>` 是在**编译阶段**发力的。普通的 `setup()` 函数需要手动把变量和方法 return 出去，模板才能用到。而 `<script setup>` 经过 Vite 和 `@vitejs/plugin-vue` 编译后，会将里面声明的所有顶层变量、函数和 import 导入的内容，自动闭包并暴露给模板，省去了大量的样板代码。 此外，它还可以和构建工具结合，在编译时做更多的静态分析（比如提升静态节点、缓存事件处理函数等）。
    
- **【满分回答话术】**：
    
    > “`<script setup>` 不仅仅是少写了 `return` 那么简单，它是 Vue 3 极其重要的编译时优化手段。
    > 
    > 在底层，当构建工具（如 Vite）处理 `.vue` 文件时，`<script setup>` 里的代码会被编译成组件 `setup()` 函数的函数体。它内部声明的顶层变量和 import 导入的组件，会自动暴露给模板使用。
    > 
    > 另外，因为 `<script setup>` 提供了更明确的上下文，Vue 的编译器能够更容易地进行静态分析，执行比如‘静态节点提升’和‘补丁标记 (PatchFlag)’等优化机制，这让 Vue 3 的运行时 Diff 算法速度远超 Vue 2。”
    

### 5. Vue 3 的组件通信机制

在电商项目中，父子组件嵌套非常深（比如 `Home` -> `HomePanel` -> `GoodsItem`）。

- **【面试官可能会问】**：在 Vue 3 的 `<script setup>` 中，你是怎么做组件通信的？如何调用子组件内部的方法？
    
- **【原理揭秘】**：`<script setup>` 默认是**封闭**的。父组件即使拿到了子组件的 `ref`，也无法直接访问子组件里的变量和方法，除非子组件主动暴露。
    
- **【满分回答话术】**：
    
    > “在 Vue 3 中，针对不同的场景，我有不同的通信方案：
    > 
    > 1. **常规父子通信**：我使用 `defineProps` 接收父组件传来的数据，使用 `defineEmits` 声明并触发自定义事件来通知父组件。这两个都是编译宏，不需要额外 import 就能直接用。
    >     
    > 2. **调用子组件方法（非常高频）**：如果我要在父组件中重置子组件的表单，我会给子组件绑定一个 `ref`。但因为 `<script setup>` 默认闭合，我必须在子组件内部使用 **`defineExpose({ resetForm })`** 把方法主动暴露出来，父组件才能通过 `childRef.value.resetForm()` 调用。
    >     
    > 3. **跨层级通信**：如果是像换肤、多语言这种全局级别的，我会用 Pinia；如果是单纯的几层父子嵌套（比如深层级的筛选组件），我会用 `provide` 和 `inject`，并且为了防止子孙组件乱改数据，我会把传下去的数据包一层 `readonly`。”
    >     
    

### 6. 样式穿透与 Scoped 原理

在使用 Element Plus 时，我们经常需要修改它组件内部的默认样式。

- **【面试官可能会问】**：Vue 组件中 `<style scoped>` 的原理是什么？在 Vue 3 中如果想修改 Element Plus 的内部样式，该怎么做？
    
- **【原理揭秘】**：`scoped` 是通过给 DOM 节点加上随机的 `data-v-hash` 属性来实现样式隔离的。
    
- **【满分回答话术】**：
    
    > “`scoped` 的底层原理是：Vue 的编译器会给当前组件的所有 DOM 元素加上一个唯一的自定义属性（比如 `data-v-12345`），同时在 CSS 选择器末尾加上 `[data-v-12345]`，从而保证样式只对当前组件生效。
    > 
    > 但这也导致我们无法直接修改嵌套在里面的 Element Plus 子组件的样式。在 Vue 2 中，我们通常用 `::v-deep` 或 `/deep/` 来实现样式穿透。
    > 
    > **但在 Vue 3 中，官方废弃了旧的穿透写法，推出了全新的伪类 `:deep()`。** 如果我要修改 `<el-input>` 内部的输入框背景色，我会写成 `:deep(.el-input__inner) { background: red; }`。它的原理是将 `data-v-hash` 属性提升到穿透选择器的前面，从而成功匹配到子组件的 DOM。”
    

### 7. 极限性能优化：海量数据下的 `shallowRef`

在电商后台或者秒杀活动页，有时候接口会一次性返回成千上万条商品数据。

- **【面试官可能问】**：如果接口返回了 10000 条只用于列表展示的商品数据，你直接用 `ref` 存，页面会卡死，为什么？怎么解决？
    
- **【满分回答】**：
    
    > “因为 Vue 3 的 `ref`（对对象而言）和 `reactive` 默认是**深度响应式**的。它会递归遍历这 10000 条数据的所有深层属性，全部用 Proxy 包装一遍。这在纯展示场景下是极大的性能浪费。
    > 
    > **我的解决方案是使用 `shallowRef`（浅层响应式）**。 `shallowRef` 只有对 `.value` 的修改（也就是直接替换整个大数组）才会触发视图更新，而不会去深层劫持数组内部的每个对象。配合 `Object.freeze()` 冻结那些不需要修改的对象，可以把长列表的初始化渲染时间从几秒钟直接缩短到几十毫秒。此外，如果列表超长，我还会结合**虚拟列表（Virtual List）**技术，只渲染可视区域的 DOM 节点。”


### 8. 倒计时 Hook 的“休眠失真”问题

在你的 `src/composables/useCountDown.js` 中，你封装了验证码/订单支付的倒计时逻辑。

- **【面试官的死亡追问】**：“你封装的倒计时是用 `setInterval` 每次减 1 秒实现的吗？如果用户把浏览器切换到后台，或者手机锁屏了，过了 5 分钟再切回来，你的倒计时还会准吗？”
    
- **【原理揭秘】**：现代浏览器为了省电，当标签页处于非活跃状态（后台运行）时，会**强制限制或暂停 `setInterval` 的执行**。如果只靠 `time--`，切回前台时，时间会发生严重的滞后（失真）。
    
- **【满分绝杀话术】**：
    
    > “我最初封装 `useCountDown` 时确实考虑过这个问题。因为浏览器对后台标签页的定时器有节流机制（休眠），如果单纯用 `setInterval` 每次减一秒，切回页面时时间必定是不准的。
    > 
    > **为了解决这个问题，我采用了‘时间戳差值法’。** 在开启倒计时的时候，我不仅声明了定时器，还会计算出一个绝对的结束时间戳（`endTime = Date.now() + 剩余秒数 * 1000`）。 在定时器的每一次回调里，我不做简单的减法，而是**用 `endTime` 减去当前的 `Date.now()`** 来重新计算剩余时间。这样一来，无论浏览器后台怎么休眠、怎么节流，只要用户一切回页面，执行的依然是基于当前真实物理时间的减法计算，倒计时永远是精准的。”
    

### 9. 自定义指令的内存泄漏与重复触发（考察 `lazyPlugin`）

在 `src/main.js` 里，有一行代码是 `app.use(lazyPlugin)`，这是你自己封装的图片懒加载插件。

- **【面试官的死亡追问】**：“你用 IntersectionObserver 实现了图片懒加载，那当图片滚动到视野内加载成功后，用户再把页面滚上去又滚下来，它会重复触发加载吗？如果页面销毁了，你的观察器清除了吗？”
    
- **【原理揭秘】**：交叉观察器会持续监听，如果不手动停止，不仅会重复触发逻辑，还会造成严重的内存泄漏。
    
- **【满分绝杀话术】**：
    
    > “在封装 `lazyPlugin` 时，性能和内存溢出是我重点关注的两个点。
    > 
    > 第一，**防止重复加载**。在使用 `@vueuse/core` 的 `useIntersectionObserver`（或者原生 API）监听图片进入可视区 (`isIntersecting === true`) 时，我会立刻将真实的 URL 赋给 `img.src`。紧接着，我会立刻调用监听器返回的 **`stop()` 方法（停止观察）**。这样无论用户怎么上下滚动，这张图片都只会被处理一次。
    > 
    > 第二，**防止内存泄漏**。我将这个逻辑封装成了 Vue 3 的自定义指令（比如 `v-lazy`）。我会利用指令的生命周期，在 `mounted` 钩子里初始化观察器；同时，一定会在指令的 **`unmounted` 钩子中执行销毁逻辑**，确保当商品卡片被卸载时（比如切换路由），绑定的观察器也会被浏览器垃圾回收机制彻底清理，绝不留下内存泄漏的隐患。”

### 11. 吸顶导航的性能优化（对应 `LayoutFixed.vue`）

- **【面试官可能会问】**：你这个吸顶导航是怎么实现的？如果是监听滚动事件，如何保证性能不卡顿？

### 13. Vue 3 到底快在哪里？（编译时优化：PatchFlag 与静态提升）

在电商首页（如 `LayoutFooter` 或 `HomeBanner`），有大量的静态文本（比如“关于我们”、“联系客服”等）和少量动态数据。

- **【满分绝杀话术】**：
    
    > “Vue 3 渲染快，核心在于它在 **编译阶段（Compile-Time）** 做了大量优化，主要体现在两个机制：
    > 
    > 第一是**静态提升（Static Hoisting）**。在 Vue 2 中，无论元素是否是静态的，每次组件更新重渲染时，都会重新创建这些 VNode 节点。而在 Vue 3 中，像电商 Footer 这种纯静态的 DOM，在编译时会被提升到渲染函数之外，只创建一次，后续更新直接复用。
    > 
    > 第二是**补丁标记（PatchFlag）和 Block 树**。Vue 2 的 Diff 是全量遍历对比新旧虚拟 DOM 树，非常慢。Vue 3 在编译模板时，如果发现某个 `div` 里绑定了动态变量（比如购物车的 `{{ count }}`），就会给它打上一个 `PatchFlag`（比如标记为 1 代表 Text 变化）。在运行时更新时，Vue 3 会跳过所有静态节点，直接根据 PatchFlag 去靶向更新这个动态节点。**这就把 Diff 算法的复杂度从 O(整个模板大小) 降到了 O(动态节点数量)！**”
    

### 14. 弹窗组件的 CSS 坑与 Vue 3 `<Teleport>` 传送门

电商项目里充满了各种弹窗（登录弹窗、确认收货弹窗、地址选择弹窗）。

- **【面试官的终极试探】**：如果在商品详情页（深层嵌套的组件中）需要弹出一个全局的全屏对话框，你经常会遇到什么 CSS 问题？在 Vue 3 中怎么优雅地解决？
    
- **【满分绝杀话术】**：
    
    > “在深层组件中写全局弹窗，最容易踩的坑就是 **CSS 上下文污染**。比如父级组件设置了 `overflow: hidden` 或者 `z-index` 层级较低，那么深层的弹窗即使设置了 `position: fixed`，也会被父容器裁切掉，或者被其他组件遮挡。
    > 
    > 在 Vue 2 时，我们通常要把弹窗组件挂载到 `document.body` 上，逻辑非常繁琐。
    > 
    > **但在 Vue 3 中，我引入了官方的新特性 `<Teleport>`（传送门）。** 哪怕我的弹窗代码写在深达 5 层的子组件内部（方便复用组件的状态和方法），我只需要用 `<Teleport to="body">` 把弹窗的 DOM 包裹起来。Vue 3 在渲染时，会神奇地把这部分 DOM 结构直接“瞬移”挂载到页面的 `<body>` 标签下，脱离原有的 DOM 树。这样既保持了组件内部逻辑的内聚，又完美避开了所有 CSS 的层级和溢出问题。”
