#### 1. 项目为什么从 Vuex 换成了 Pinia？它们俩到底有什么区别？（🌟必考题）

- **【原理揭秘】**： Vuex 4 虽然也能用在 Vue 3 中，但它依然带着很多 Vue 2 Options API 的历史包袱（比如繁琐的 mutations，复杂的命名空间机制）。Pinia 是专门为 Vue 3 的 Composition API 重新设计的，可以说它就是“去掉历史包袱的 Vuex 5”。
    
- **【满分回答话术】**：
    
    > “在我的项目中选择 Pinia 主要有三个核心原因：
    > 
    > 第一，**舍弃了繁琐的 Mutations**。在 Vuex 中，哪怕只是改一个布尔值，也得写 Action 提交给 Mutation，非常啰嗦。Pinia 移除了 Mutation，所有的同步和异步业务逻辑都可以直接写在 Action 里，直接修改 State，代码量锐减。
    > 
    > 第二，**扁平化的架构设计**。Vuex 是一个单一的全局大 Store，如果模块多，还得套娃一样写 `modules` 和命名空间（namespaced）。而 Pinia 中每个 Store（比如 userStore, cartStore）都是平级的、独立的。用哪个就 import 哪个，这非常利于 Vite 进行代码分割和 Tree-shaking 优化。
    > 
    > 第三，**完美的 TypeScript 支持**。因为 Pinia 没有嵌套模块和一堆魔法字符串（像 `commit('moduleA/xxx')`），它的类型推断非常自然，不需要像 Vuex 那样写冗长的泛型包装。”
    

#### 2. 你从 Pinia 中解构数据时，遇到过响应式丢失的问题吗？怎么解决的？（💣高频陷阱题）

- **【原理揭秘】**： Pinia 的 `state` 和 `getters` 内部是用 Vue 3 的 `reactive` 包装的。如果在组件里直接使用 ES6 的方式解构（比如 `const { userInfo } = useUserStore()`），拿到的只是一个失去响应式的普通对象。但 Action 是普通的函数，可以直接解构。
    
- **【满分回答话术】**：
    
    > “遇到过。因为 Pinia 的 state 本质上是 reactive 对象，如果直接进行 ES6 解构，会导致响应式链接断开，数据变了但视图不会更新。
    > 
    > **我的解决方案是**： 如果我只想要提取里面的 Action（方法），我会直接解构：`const { clearUserInfo } = useUserStore()`。 但如果我要解构 State 里的数据或者 Getters（比如用户的 token，或者购物车的商品总数），我会使用 Pinia 提供的辅助函数 **`storeToRefs`**。 比如 `const { userInfo } = storeToRefs(useUserStore())`，它的底层原理类似于 Vue 的 `toRefs`，会把解构出来的数据包一层 `ref`，从而完美保留响应式。”
    

#### 3. 电商项目中，用户刷新页面会导致 Pinia 的状态丢失，你是怎么处理的？

- **【原理揭秘】**： Vuex 和 Pinia 的数据都是存在浏览器内存里的，一按 F5 刷新，内存清空，数据（比如登录的 Token、没结算的购物车商品）就全丢了。所以必须配合本地存储（localStorage / sessionStorage）使用。
    
- **【满分回答话术】**：
    
    > “这是一个典型的痛点。比如用户好不容易把商品加入购物车，一刷新页面，购物车清空了，体验非常差。
    > 
    > 传统的做法是：每次修改 Pinia 数据时，手动写代码把它 `localStorage.setItem` 存一份；初始化时，再 `getItem` 读出来赋给 Pinia。但这种做法代码侵入性太强，且极难维护。
    > 
    > **我在项目中的优化方案是**：引入了 **`pinia-plugin-persistedstate`** 持久化插件。 只需要在定义 Store 时，开启 `persist: true` 配置。这个插件会在底层自动监听 Store 的变化，并将其序列化保存到 `localStorage` 中。当页面刷新 Pinia 重新实例化时，它又会自动从本地存储中把数据注水（Hydrate）回 Store 中。不仅实现了状态的无缝持久化，还解耦了业务代码。”
    

#### 4. 聊聊你在项目中是怎么用 Pinia 管理购物车（Cart）数据的？为什么不用组件传值？（💼业务实战题）

- **【原理揭秘】**： 面试官想看你是否真正理解“什么样的数据该放进全局状态库”。局部数据放全局是滥用，全局共享数据通过 Props 传也是灾难。
    
- **【满分回答话术】**：
    
    > “在电商场景下，**购物车数据是典型的跨多层级共享状态**。 顶部导航栏（Header）需要实时展示购物车商品种类数；侧边栏/抽屉需要展示简略列表；单独的购物车页面（CartList）需要展示明细并做增删改查；最后的结算页（Checkout）还要拉取勾选的商品算钱。如果用 Props 一层层传或者用 EventBus，代码会变成难以维护的“意大利面条”。
    > 
    > **我的设计思路是**：
    > 
    > 1. 在 `cart.js` 的 Store 里，定义一个数组 `cartList` 作为核心 State。
    >     
    > 2. 将计算总价和总件数的逻辑全部写在 **Getters** 里。比如利用数组的 `reduce` 方法累加选中商品的价格。因为 Getters 具有缓存特性，只要 `cartList` 里的内容不改，总价就不会重复计算，性能很好。
    >     
    > 3. 在 **Actions** 里封装统一的业务行为：比如 `addCart`、`delCart`、`checkAll`。当未登录时，这些 Action 只操作本地的 `cartList` 数组；当用户登录后，Action 内部会去发起网络请求，同步后端的购物车接口数据。这样就将复杂的业务逻辑从 Vue 视图组件中彻底抽离出来了。”

#### 5. 购物车的“离线与在线”合并机制

你在 `src/stores/cart.js` 中管理了购物车，但在真实的电商业务中，面试官必问一个场景：**“如果用户没有登录，挑了几件商品放进购物车，然后他点击结算并进行了登录，这时候他之前挑的商品去哪了？”**

- **【满分逻辑】**：
    
    > “在项目中，我处理了严格的**购物车合并逻辑**。 当用户处于未登录状态时，添加购物车的操作只会修改 Pinia 中的本地 `cartList` 数组，并通过持久化插件存在 LocalStorage 中。 当用户在登录页面（`Login`）登录成功拿到 Token 后，我会在登录成功的异步回调中，立即派发一个合并购物车的 Action。我会把本地存储的商品列表（如 skuId 和 数量）组装成一个数组发送给后端的‘合并购物车’接口。合并成功后，再调用一次获取最新购物车列表的接口，用云端的数据覆盖掉本地的数据。这样既保证了用户体验（未登录也能加购），又保证了多端数据的一致性。”
