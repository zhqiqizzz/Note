#### 1. 电商项目中有大量的页面（如个人中心、去支付、我的订单），你是怎么做路由鉴权（登录拦截）的？

- **【原理揭秘】**： 前端路由鉴权的核心机制是**全局前置守卫 (`router.beforeEach`)**。配合路由配置中的 `meta`（路由元信息）字段，我们可以给需要保护的路由打上“标签”。当路由发生跳转时，守卫会拦截并检查目标路由的标签和本地用户的 Token 状态。
    
- **【满分回答话术】**：
    
    > “在电商项目中，像购物车列表、结算页 (`/checkout`)、个人中心 (`/member`) 都是强依赖用户身份的页面。我主要通过 Vue Router 的全局前置守卫结合 Pinia 来实现鉴权拦截。
    > 
    > **具体流程是这样的**：
    > 
    > 1. 在 `router/index.js` 中，我会给需要登录才能访问的路由添加自定义的元信息，比如 `meta: { requiresAuth: true }`。
    >     
    > 2. 注册 `router.beforeEach` 守卫。每次路由跳转前，先判断目标路由的 `to.meta.requiresAuth` 是否为 true。
    >     
    > 3. 如果为 true，我会去 Pinia 的 UserStore 里获取用户的 Token。如果 Token 存在且有效，直接 `next()` 放行；
    >     
    > 4. 如果没有 Token，我会强制拦截，通过 `next('/login')` 跳转到登录页。为了提升用户体验，我还会把用户本来想访问的路径作为 query 参数带在 URL 上（如 `/login?redirect=/checkout`），这样用户登录成功后，我会读取这个参数，直接把他送回刚才想去的结算页，实现无缝衔接。”
    >     
    

#### 2. 看你的代码 `import Home from "@/views/Home/index.vue"`，这样直接导入路由有什么性能隐患吗？你会怎么优化？（🌟高频挑刺题）

- **【原理揭秘】**： 你当前在 `router/index.js` 中使用的是**同步导入**。这意味着在使用 Vite 打包时，所有的页面组件（Home、Login、Detail 等）都会被打包进同一个巨大的 `index.[hash].js` 文件中。用户第一次打开首页时，必须把所有页面的代码都下载完才能看到内容，导致**首屏白屏时间极长**。 **解决方案是：路由懒加载（动态导入）**。
    
- **【满分回答话术】**：
    
    > “如果采用直接 import 的同步引入方式，会导致所有路由组件被打包成一个巨大的 JS 文件，严重拖慢首屏加载速度。
    > 
    > **为了优化首屏性能，我会使用路由懒加载**。具体做法是把组件的引入方式改为箭头函数配合动态 `import()`，例如：`const Home = () => import('@/views/Home/index.vue')`。
    > 
    > **它的底层原理是**：结合 Vite (或 Webpack) 的代码分割（Code Splitting）机制，构建工具遇到动态 `import()` 时，会自动将这个路由组件单独打包成一个独立的 JS 文件（chunk）。只有当用户在浏览器里真正点击跳转到该路由时，浏览器才会按需发送网络请求去下载这个对应的 JS 文件。这就极大地减小了首屏渲染时的代码体积。”
    

#### 3. Vue 3 (Vue Router 4) 和 Vue 2 (Vue Router 3) 在组件内使用路由有什么区别？

- **【原理揭秘】**： Vue 2 使用 Options API，组件实例上有 `this`，所以我们习惯用 `this.$router.push` 和 `this.$route.query`。 Vue 3 使用 Composition API (`<script setup>`)，组件内部**没有 `this`**，因此 Vue Router 4 提供了组合式 API 函数。
    
- **【满分回答话术】**：
    
    > “最大的区别在于脱离了 `this` 上下文。在 Vue 3 的 `<script setup>` 中，我们无法再使用 `this.$router` 和 `this.$route`。
    > 
    > Vue Router 4 为此提供了两个专门的 Hook：
    > 
    > 1. 我会按需引入 `useRouter`，调用 `const router = useRouter()` 拿到**路由器实例**，用它来执行编程式导航，比如 `router.push('/login')` 或 `router.back()`。
    >     
    > 2. 引入 `useRoute`，调用 `const route = useRoute()` 拿到**当前路由信息对象**。当我在商品详情页 (`/detail/:id`) 需要获取 URL 上的商品 ID 时，我会通过 `route.params.id` 来动态获取，或者通过 `route.query` 拿到问号后面的参数。”
    >     
    

#### 4. 你在项目中大量使用了嵌套路由（基于 Layout 的子路由），为什么这么设计？

- **【原理揭秘】**： 电商项目有明显的**区域复用**特征。比如顶部导航栏（搜索框、购物车入口）、底部的网站信息（Footer），在首页、分类页、详情页都是一模一样的。嵌套路由配合 `<router-view>` 就是为了解决这种布局级别的复用问题。
    
- **【满分回答话术】**：
    
    > “这种设计主要是为了**布局抽象和组件复用**。
    > 
    > 在我的电商项目中，像顶部导航栏 (`LayoutHeader`) 和底部 (`LayoutFooter`) 是大多数页面公用的核心骨架。如果每个页面组件都去 import 一次头部和底部，代码不仅冗余，而且在页面切换时，头部和底部组件会被反复销毁和重新创建，导致页面闪烁。
    > 
    > 我把 `Layout` 作为一级路由，里面固定放置 Header 和 Footer 组件，并在中间留出一个 `<router-view />` 占位。然后把首页 (`Home`)、分类 (`Category`)、详情页 (`Detail`) 等作为 `Layout` 的 `children`（二级路由）。
    > 
    > 这样，当用户从首页跳转到分类页时，外层的 `Layout` 及其内部的导航栏完全不会被卸载，只有中间 `<router-view>` 区域的内容会被替换。这不仅精简了代码，更保证了极其流畅的页面切换体验。”
    

#### 5. 单页路由的极致体验—— `<KeepAlive>` 与状态缓存控制

电商项目有一个非常经典的交互痛点：用户在列表页（Category）滑到了第 50 个商品，点进去看详情页（Detail），看完后点浏览器的“返回”，结果列表页刷新了，又回到了顶部！用户体验瞬间爆炸。

- **【面试官的挑战】**：Vue Router 切换页面时组件会销毁，你怎么解决从详情页返回列表页时，保持之前的滚动位置和数据状态不丢失？
    
- **【满分降维打击】**：
    
    > “我会利用 Vue 3 的内置组件 **`<KeepAlive>`** 结合 Vue Router 的 `meta` 属性来实现精准的页面缓存。
    > 
    > 1. 在 `router/index.js` 中，我给需要缓存的页面（比如 `Home`、`Category`）配置 `meta: { keepAlive: true }`。
    >     
    > 2. 在外层的 `App.vue` 或 `Layout` 的 `<router-view>` 中，使用 `v-slot` 拿到 Component，并用 `<KeepAlive>` 包裹起来，通过 `include` 属性或直接判断 `route.meta.keepAlive` 来决定是否缓存当前组件。
    >     
    > 3. **核心细节**：被 `<KeepAlive>` 缓存的组件，不会再触发 `mounted` 和 `unmounted` 生命周期。所以，如果有部分数据确实需要每次进入都刷新（比如用户的实时余额），我会在它专属的 **`onActivated`** 钩子中去发起针对性的局部数据请求，既保证了滚动位置和核心列表不丢失，又保证了敏感数据的实时性。”
    >
