#### 1. SKU 规格选择算法（🌟 顶级高频、拉开差距的绝杀题）

**【业务场景】**：在商品详情页（Detail），一件衣服有“颜色（红/黑）”和“尺码（L/XL/XXL）”。有的组合（如：红-XXL）可能没库存。当用户点击“红色”时，“XXL”按钮必须立刻变成灰色不可点状态。

- **【原理揭秘】**： 小白的做法：用户每次点击，前端都去拿着当前选中的值（比如“红色”），去遍历后端返回的几百条 SKU 数据，看看有没有“红色+XXL”且库存大于 0 的商品。这种做法时间复杂度极高，在规格多的时候页面会卡顿。 **大厂的高阶做法（也就是你项目里的 `power-map.js`）**：使用 **“路径字典（幂集算法）”**。空间换时间。
    
    1. 页面初始化时，拿到有库存的 SKU 数组（如 `['红', 'XL']` 和 `['黑', 'L']`）。
        
    2. 根据有库存的数组，生成它们所有的子集组合（幂集），并存入一个字典对象中。比如：`{ '红': true, 'XL': true, '红-XL': true, '黑': true, 'L': true, '黑-L': true }`。
        
    3. 用户点击时，直接去这个字典里查（比如查 `dict['红-XXL']`）。如果返回 `undefined`，说明没货，直接禁用该按钮。查找复杂度瞬间降为 O(1)。
        
- **【满分回答话术】**：
    
    > “在我的项目中，商品详情页的 SKU 规格联动是一个技术难点。因为多规格组合很容易出现某几个特定组合没有库存的情况，没库存的按钮必须置灰。
    > 
    > 我没有采用低效的‘每次点击都去循环遍历所有 SKU 列表’的做法，而是引入了**无向图/路径字典的算法思想（幂集算法）**。
    > 
    > **我的具体实现是这样的**：
    > 
    > 1. 组件初始化时，我首先过滤出后端返回的所有 `库存 > 0` 的 SKU 组合。
    >     
    > 2. 然后，我封装了一个 `power-map.js` 工具函数，利用算法算出这些有效组合的所有子集，并将它们拼接成字符串作为 Key，存入一个路径字典（Map/Object）中。
    >     
    > 3. 在 UI 渲染阶段，当用户选中了某个规格（比如红色），我就拿着‘红色’和其余未选中的规格（比如 XL、XXL）去拼接。
    >     
    > 4. 然后直接去路径字典里读取。如果在字典里找不到这个 Key，说明这个组合必定无货，我马上把对应尺码的按钮状态设为 disabled。
    >     
    > 
    > **这种做法最大的好处是‘以空间换时间’。不管商品的规格有多少层级，用户点击时的判断逻辑永远是 O(1) 的属性读取，保证了商品详情页极致的交互流畅度。**”
    

#### 2. 订单结算与第三方支付闭环（💼 业务闭环题）

**【业务场景】**：对应你项目中的 `CartList` -> `Checkout` -> `Pay` -> `PayBack`。面试官想知道你是否清楚前端在这个过程中的职责边界，尤其是如何处理支付状态。

- **【原理揭秘】**： 前端不负责扣钱，只负责“串流程”和“展示状态”。
    
    1. **结算页 (Checkout)**：根据购物车勾选的商品，生成预览订单（收集地址、商品明细）。
        
    2. **提交订单**：将地址 ID、商品参数发给后端，后端生成真实的 `orderId` 返回给前端。前端拿到 `orderId` 跳转到支付页 (`/pay`)。
        
    3. **发起支付**：前端把 `orderId` 和回调地址给后端，后端返回一段支付宝/微信的 URL。前端直接 `window.location.href` 跳过去。
        
    4. **支付回调 (PayBack)**：用户在支付宝付完钱，支付宝会把页面跳回你预先配置的 `/paycallback` 页面，并在 URL 后缀带上支付结果参数。**注意：前端绝不能直接相信 URL 上的成功参数，必须拿着 URL 里的 `orderId` 再去向我们自己的后端发请求，核实是否真的到账了**。
        
- **【满分回答话术】**：
    
    > “在我的项目中，我完整负责了从购物车结算到第三方支付的闭环链路。整个流程分为四个核心步骤：
    > 
    > 1. **订单确认 (`Checkout`)**：在购物车中点击结算，进入确认订单页。这里我会拉取用户的默认收货地址，并结合 Pinia 中的购物车数据，计算出最终的运费和总价。
    >     
    > 2. **提交订单与防重发**：用户点击‘提交订单’时，我会立即锁定按钮状态（loading）。接口请求成功后，后端会返回一个唯一的 `orderId`，我随即将页面路由跳转至支付页 (`/pay?id=xxx`)，并在提交成功后清空购物车中已结算的商品。
    >     
    > 3. **跳转第三方支付**：在支付页，用户选择支付宝/微信。点击支付时，前端不处理金钱逻辑，而是调用后端的支付接口，后端会返回一串带有第三方支付平台签名的跳转 URL。我使用 `window.location.href` 重定向到该地址，让用户在第三方平台完成付款。
    >     
    > 4. **回调与安全性核验 (`PayBack`)**：付款完成后，第三方平台会将页面重定向回我的支付回调页 (`/paycallback`)。此时 URL 上会携带一些参数表明支付状态。**但是出于安全考虑，我绝不会直接依赖前端的 URL 参数来判定支付成功。** 我会在回调页的 `onMounted` 阶段，提取路由参数中的 `orderId`，再次调用我们自己后端的‘查询订单状态’接口。只有后端接口明确返回支付成功，我才会向用户展示支付成功的绿色打勾页面，否则提示异常并引导去‘我的订单’页查看。”
    >

#### 3. 如果用户在支付页面，不付款直接按浏览器的“返回键”或者关闭页面，你怎么处理？

- **【满分回答话术】**：
    
    > “这在电商中是非常常见的‘取消支付 / 待支付’场景。 如果用户在跳转第三方支付前或支付一半时后退，我会利用 Vue Router 的生命周期或路由守卫进行处理。 其实在用户点击‘提交订单’并拿到 `orderId` 的那一刻，这个订单在后端的数据库里就已经生成了，状态为‘待支付’。 所以，如果用户退出了支付流程，没关系，他的购物车依然是被清空的状态（因为订单已生成）。我会引导用户去个人中心的‘我的订单’(`MemberOrder` 页面) 中寻找这笔‘待支付’的订单。在那里，他可以重新点击‘去支付’按钮，我会拿着这个旧的 `orderId` 再次走支付跳转流程。同时，后端会对待支付订单设置一个 30 分钟的倒计时（结合我封装的 `useCountDown` Hook），超时未支付则自动取消订单并释放商品库存。”
    

#### 4. 电商经典组件库封装（商品放大镜组件）

在你的项目目录中有一个 `src/components/ImageView/index.vue`。这是电商项目必考的交互组件。

- **【面试官可能会问】**：商品详情页的图片放大镜效果是怎么实现的？如果是你自己封装，需要考虑哪些点？
    
- **【原理揭秘】**：核心是**两张图片（一大一小）+ 鼠标坐标计算**。当鼠标在小图上移动时，计算出遮罩层（滑块）的 X/Y 坐标，同时利用比例关系，反向移动大图（通常使用 `background-position` 或 `transform`），从而形成局部放大的视觉效果。
    
- **【满分回答话术】**：
    
    > “在实现商品放大镜组件时，难点在于**精准的坐标计算和边界判断**。 我没有手动去监听 `mousemove` 写繁琐的坐标获取，而是结合了 VueUse 库中的 `useMouseInElement` Hook。它可以极其方便地帮我获取当前鼠标在图片容器内的相对坐标（elementX, elementY）以及是否在容器外部（isOutside）。 拿到坐标后，我要做两步处理：
    > 
    > 1. **限制遮罩层边界**：通过计算，保证半透明的遮罩层永远不会滑出小图的边界。
    >     
    > 2. **大图反向移动**：按照小图遮罩层移动的距离，乘以大图的放大比例（通常是 2 倍），使用 `background-position` 反向移动大图。利用 Vue 3 的响应式数据绑定这些坐标值，整个放大镜的交互如丝般顺滑。”
    

#### 5. 支付状态的实时性——从 HTTP 轮询到 WebSocket 实时推送

在前面的支付闭环中，我们提到在 `PayBack` 页面通过重发 HTTP 请求去后端查询支付状态。

- **【面试官的挑战】**：如果第三方支付（比如微信/支付宝）的回调延迟了，你前端怎么保证用户能第一时间看到支付成功的状态？总不能让前端写个 `setInterval` 疯狂轮询后端接口吧？那样服务器压力太大了。
    
- **【满分降维打击】**：
    
    > “常规做法确实是前端搞个定时器，每隔 3 秒去查一次订单状态，但这极其消耗服务器资源。
    > 
    > **为了追求极致的实时性和降低服务器压力，我会在这里引入 WebSocket**。
    > 
    > 当用户跳转到第三方支付的同时，前端在后台建立一个带有该 `orderId` 的 WebSocket 长连接。一旦用户在手机上指纹支付成功，微信服务器通知了我们的后端，我们的后端立刻通过 WebSocket 主动向前端推送一条 `{"status": "success", "orderId": "xxx"}` 的消息。 前端接收到消息后，直接由等待状态切换为支付成功页面。这样不仅真正做到了毫秒级的状态同步，还彻底干掉了低效的 HTTP 轮询。”
