#### 1. 你在项目中是如何封装 Axios 的？为什么要进行二次封装？

- **【原理揭秘】**： 原生 Axios 虽然好用，但在企业级项目中，如果每个组件都单独引入并配置接口地址、单独处理错误提示，代码会极其冗余。二次封装的核心目的是：**统一配置（BaseURL/超时时间）、统一鉴权（携带 Token）、统一错误处理（全局 Toast 提示/状态码分发）**。
    
- **【满分回答话术】**：
    
    > “在我的项目中，我没有在组件里直接使用 axios 发请求，而是单独创建了一个 `http.js` 进行二次封装。 首先，我通过 `axios.create` 配置了基础的 `baseURL` 和 10000ms 的超时时间。 其次，利用**请求拦截器**，我从 Pinia 的用户 Store 中取出 Token，按照后端的规范拼接成 `Bearer ${token}` 注入到请求头的 `Authorization` 字段中，实现了所有接口自动鉴权。 最后，利用**响应拦截器**做两件事：一是剥离掉 Axios 默认包装的那层数据结构，直接 `return res.data` 给业务层；二是对非 2xx 的业务报错，统一调用 Element Plus 的 `ElMessage` 组件弹出后端返回的错误信息，避免了在每个页面写 try-catch 的麻烦。”
    

#### 2. 遇到 401 状态码你是怎么处理的？（进阶追问：如何实现 Token 无感刷新？）

- **【原理揭秘】**： 401 (Unauthorized) 代表未授权，通常是因为用户没登录，或者 Token 过期了。 在这个兔鲜电商项目中，目前的处理方式比较“硬”：发现 401 就清空本地数据踢回登录页。但在复杂业务中，面试官往往期望听到“双 Token 机制（无感刷新）”。
    
- **【满分回答话术（结合项目与进阶思考）】**：
    
    > “在我目前的电商项目中，我是在 Axios 的响应拦截器里捕获 401 状态码的。一旦触发，我会调用 Pinia 中的 `clearUserInfo` 方法清空本地用户信息，并通过 Vue Router 强制让用户跳转回 `/login` 页面重新登录。 **（主动抛出进阶方案）** 但如果是更偏向管理后台或者体验要求更高的项目，我会采用**双 Token 无感刷新方案**。 登录时后端会返回 `AccessToken`（短效）和 `RefreshToken`（长效）。当请求接口返回 401 时，响应拦截器会先拦截这个错误，并挂起当前请求。接着，我会在底层悄悄携带 `RefreshToken` 去请求刷新接口获取新的 `AccessToken`。拿到新 Token 后，把它更新到本地，并把刚才挂起的那个失败请求带上新 Token 重新发出去。整个过程对用户是完全透明的，他们只会觉得刚才那次请求稍微慢了一点点，而不会被强制踢下线。”
    

#### 3. 电商场景下，如果用户在网络卡顿的情况下疯狂点击“提交订单”，如何防止前端发送重复请求？

- **【原理揭秘】**： 这是非常高频的业务场景题。防抖（Debounce）和节流（Throttle）只能在短时间内拦截，如果网络很慢，接口 3 秒才返回，防抖也防不住用户在第 4 秒再次点击。最根本的解决方案是**状态锁定**或**取消重复的 Axios 请求**。
    
- **【满分回答话术】**：
    
    > “应对这种电商高频场景，我通常会从三个维度来做防护：
    > 
    > **第一层是 UI 交互层（最简单有效）**：点击提交按钮的瞬间，将按钮的 `loading` 属性设为 true，或者将按钮置灰（disabled）。只有当 `await` 等待接口有了明确的成功或失败回调后，才恢复按钮状态。
    > 
    > **第二层是逻辑层（防抖/节流）**：如果是搜索框联想提示这种高频触发的操作，我会配合 `@vueuse/core` 中的 `useDebounceFn` 做一个 500ms 的防抖，减少无效的网络开销。
    > 
    > **第三层是底层网络层（Axios 请求取消机制）**：这是最严谨的做法。我会在 Axios 请求拦截器中，利用 `AbortController` API（旧版是用 `CancelToken`）建立一个请求池。把当前请求的 `method` 和 `url` 拼接成一个独一无二的 Key 存入 Map 中。如果在上一个请求还没回来时，又进来了完全一样的请求，我就直接调用 `abort()` 方法把前面的请求或者当前的重复请求给杀掉。这样可以从根源上杜绝重复订单的产生。”
    

#### 4. 为什么有时候请求接口时，浏览器会自动发一个 OPTIONS 请求？

- **【原理揭秘】**： 这就是著名的 **CORS 跨域预检请求 (Preflight Request)**。浏览器处于安全考虑（同源策略），在发送“复杂请求”之前，会先发一个类型为 `OPTIONS` 的探测请求，问问服务器“我跨域发这个带自定义请求头的请求，你允许吗？”。
    
- **【满分回答话术】**：
    
    > “这是因为触发了跨域资源共享 (CORS) 的预检机制。 浏览器把请求分为‘简单请求’和‘复杂请求’。像我们项目中，因为在请求头里自定义添加了 `Authorization: Bearer token`，这就让它变成了一个‘复杂请求’。 浏览器在发送真正的业务请求前，会自动先发出一个 `OPTIONS` 方法的预检请求去询问服务器。如果服务器响应的 `Access-Control-Allow-Headers` 中允许了我们携带 `Authorization`，并且允许了跨域来源，浏览器紧接着才会发送真实的 GET/POST 请求。”
    

#### 5. 电商项目的前端安全（XSS 与 CSRF）

电商项目涉及真实的交易和金钱，安全考点是高级前端绕不开的话题。

- **【面试官可能会问】**：你们的电商系统在前端层面有做哪些安全防护？比如怎么防范 XSS 和 CSRF 攻击？
    
- **【满分回答话术】**：
    
    > “安全防护通常是前后端配合的，在前端层面我主要做了两点：
    > 
    > **第一，防范 XSS（跨站脚本攻击）**： XSS 主要是恶意用户注入了 Script 脚本。得益于 Vue 框架本身的机制，我们在模板中使用的双大括号 `{{ }}` 进行数据绑定时，Vue 内部会自动使用 `textContent` 而不是 `innerHTML` 来更新 DOM，这天然就防御了绝大多数的 XSS。此外，我会在项目中严格限制 `v-html` 的使用，如果必须用（比如渲染富文本商品详情），我会先引入 `DOMPurify` 库对后端返回的 HTML 字符串进行一波转义和过滤。
    > 
    > **第二，防范 CSRF（跨站请求伪造）**： CSRF 的核心是利用了浏览器会自动携带 Cookie 的机制来伪造用户请求。因为我们的项目使用的是 **Token 鉴权机制**，而不是传统的 Cookie-Session。我们将 Token 存在 LocalStorage 中，每次请求手动塞进 Header 的 `Authorization` 里。因为 CSRF 攻击无法读取本地 LocalStorage，也无法自动在请求头里加东西，所以这种架构天然就免疫了 CSRF 攻击。”
