浏览器渲染机制（也叫页面渲染流水线），是浏览器渲染引擎（如 Chrome 的 Blink、Safari 的 WebKit）将 HTML、CSS、JavaScript 等资源，解析、转换、绘制为用户可见可交互页面的核心流程。整个过程是有序的分阶段流水线，核心分为**6 个主步骤**，同时包含资源加载、阻塞处理、性能优化等配套逻辑。

## 核心渲染流程（按执行顺序）

### 1. 解析 HTML，构建 DOM 树（Document Object Model）

这是渲染流程的起点，核心是将 HTML 文本转换为浏览器可理解的结构化对象。

- **字节解码**：浏览器从网络 / 本地获取 HTML 二进制字节流，按文档指定的编码（如 UTF-8）解码为字符文本；
- **词法分析（Token 化）**：将 HTML 字符串拆解为一个个标准化 Token，区分标签开始 / 结束、属性名 / 值、文本内容、注释等类型，比如`<div class="box">`会被拆分为`div`标签 Token、`class`属性 Token；
- **语法分析与树构建**：按照 HTML 语法规范和嵌套规则，将 Token 转换为一个个 DOM 节点，最终生成一棵完整的**DOM 树**。DOM 树的每个节点对应 HTML 中的标签 / 文本，完整描述了文档的结构、内容和节点关系，是后续所有操作的基础。

**关键特性**：HTML 解析是**增量式**的，浏览器边接收字节流边解析，无需等待全量 HTML 加载完成；遇到外链资源（图片、视频、CSS）会发起异步请求，不会阻塞 HTML 解析（同步`<script>`标签除外）。

### 2. 解析 CSS，构建 CSSOM 树（CSS Object Model）

与 DOM 树构建并行执行，核心是将 CSS 样式转换为浏览器可理解的样式对象模型。

- **样式来源**：包括 HTML 内联`<style>`标签、外链`.css`文件、元素行内`style`属性、浏览器默认样式（用户代理样式表）；
- **解析过程**：将 CSS 字节流解码、Token 化，解析出选择器、样式规则、属性值，同时处理 CSS 的核心规则：层叠优先级、继承关系、单位转换（如`rem`转`px`）、颜色标准化等；
- **最终生成 CSSOM 树**：树形结构，每个节点对应 DOM 节点可应用的完整样式规则，精准描述了每个元素的视觉表现。

**关键特性**：**CSSOM 构建会阻塞渲染**，因为没有完整的 CSSOM，浏览器无法生成渲染树，不会渲染无样式的内容（避免 FOUC 无样式内容闪烁）；同时 CSSOM 构建会阻塞后续 JavaScript 的执行，因为 JS 可以读取、修改 CSSOM。

### 3. 合并 DOM+CSSOM，生成渲染树（Render Tree）

当 DOM 树和 CSSOM 树都构建完成后，渲染引擎会将两者合并，生成**渲染树（也叫渲染对象树）**，这是后续布局、绘制的直接依据。

- **核心作用**：渲染树只包含**需要渲染到页面上的可见节点**，过滤掉所有不可见内容；
- **构建规则**：
    
    1. 从 DOM 树的根节点（`<html>`）开始遍历，筛选可见节点；
    2. 不可见节点会被完全排除：比如`<head>`、`<script>`等非视觉标签，以及`display: none`的节点（注意：`visibility: hidden`、`opacity: 0`的节点会被保留，因为它们仍占据布局空间，只是视觉不可见）；
    3. 为每个可见节点匹配 CSSOM 树中的对应样式规则，将样式附加到节点上；
    
- **最终结果**：生成一棵只包含可见元素、且每个元素都带有完整样式信息的渲染树，每个节点称为「渲染对象（Render Object）」。

### 4. 布局（Layout / 回流 / 重排）

渲染树生成后进入布局阶段，业内也叫**回流（Reflow）**，核心是**计算渲染树中每个节点的精确几何位置和尺寸**。

- **核心原因**：渲染树只知道节点的存在和样式，但不知道节点具体放在页面的哪个位置、宽高是多少，布局阶段就是解决这个问题；
- **计算过程**：
    
    1. 从根节点（对应浏览器视口）开始，递归遍历渲染树；
    2. 按照盒模型、流式布局、Flex/Grid、定位等 CSS 规则，计算每个节点的坐标、宽高、边距、内边距、换行规则等所有几何属性；
    3. 布局是相对计算：父节点的布局会决定子节点的初始位置，子节点的内容尺寸也可能反向影响父节点的尺寸。
    

**关键特性**：布局是 CPU 密集型的递归计算，性能开销极大；**当修改元素的布局属性（宽高、位置、边距、`display`、字体大小等）时，会触发重新布局，也就是重排**，重排会重新计算整个或部分渲染树的几何信息。

### 5. 绘制（Paint / 重绘）

布局完成后进入绘制阶段，业内也叫**重绘（Repaint）**，核心是**将渲染树的每个节点，按照布局信息转换为屏幕上的像素点**。

- **绘制内容**：包括元素的背景色、文字、边框、阴影、渐变、图片、圆角等所有视觉效果；
- **绘制过程**：渲染引擎会将绘制任务拆分为多个**绘制层（Paint Layer）**，按照「从下到上」的堆叠顺序绘制（先绘制背景，再绘制内容，再绘制边框，最后绘制浮动 / 定位元素），确保元素的 z-index、层叠上下文规则正确执行。

**关键特性**：绘制是 CPU 密集型操作，性能开销小于重排；**修改元素的非布局样式（颜色、背景色、阴影、文字颜色等）时，不会触发布局，只会触发重绘**，因为不需要重新计算几何位置。

> 补充： 使用 [虚拟 DOM](../vue/页面渲染/虚拟%20DOM.md) 是为了减少重排和重绘

### 6. 合成（Composite）

这是渲染流程的最后一步，核心是**将拆分的多个绘制图层，按照正确的顺序合并，最终输出到屏幕上**。

- **核心原理**：现代浏览器为了极致优化渲染性能，会将页面拆分为多个独立的**合成层（Graphics Layer）**，每个图层独立绘制；绘制完成后，由浏览器的**合成线程（Compositor Thread）** 处理图层合并，最终提交给 GPU 渲染到屏幕。
- **分层的核心优势**：
    
    1. 某个图层的内容变化时，只需要重新绘制该图层，不需要重绘整个页面，大幅降低渲染开销；
    2. 可以利用 GPU 加速图层的变换，`transform`、`opacity` 等属性的修改，不会触发重排和重绘，仅需要在合成阶段调整图层的位置 / 透明度，性能开销极小，是动画优化的首选方案。
    
- **常见的分层触发条件**：CSS 3D 变换、`will-change`、`<video>`/`<canvas>`元素、z-index 较高的定位元素、CSS 滤镜等。

**最终结果**：合成线程将所有图层合并完成后，提交给 GPU，最终渲染到屏幕上，用户即可看到完整的页面。

## 配套核心知识点

### 1. 渲染阻塞规则

- **同步`<script>`标签**：会阻塞 HTML 解析和后续渲染，因为 JS 可以修改 DOM 树和 CSSOM，浏览器必须等待 JS 下载并执行完成，才能继续解析后续 HTML；
- **CSS 资源**：会阻塞渲染和后续 JS 执行，因为没有完整的 CSSOM 无法生成渲染树；
- **优化方案**：CSS 放在`<head>`中尽早加载，JS 使用`async`/`defer`属性避免阻塞 HTML 解析。

### 2. 重排、重绘、合成的性能对比

|操作|触发场景|性能开销|
|---|---|---|
|重排（回流）|修改元素布局属性（宽高、位置、边距、`display`等）|极大，需要重新计算布局 + 重绘 + 合成|
|重绘|修改非布局视觉属性（颜色、背景、阴影等）|中等，仅需重新绘制 + 合成|
|合成|修改`transform`、`opacity`等 GPU 加速属性|极小，仅需合并图层，GPU 加速执行|

### 3. 关键渲染路径

指从浏览器接收 HTML，到首次渲染出页面内容的完整链路。核心优化目标是缩短首次内容绘制（FCP）、最大内容绘制（LCP）等核心性能指标，核心手段是：减少关键资源数量、缩小关键资源体积、优化关键资源的加载顺序。
