垃圾回收（Garbage Collection，简称 GC）是 JavaScript 引擎自动管理内存的核心机制，它的作用是**自动识别并回收不再被使用的内存空间**，避免内存泄漏，保证程序的稳定运行。

与 C/C++ 等需要手动管理内存的语言不同，JavaScript 的垃圾回收是**自动执行**的，但理解其原理能帮助我们写出更高效、更少内存泄漏的代码。

## 一、前置基础：内存生命周期

JavaScript 中，内存的使用分为三个阶段：

1. **分配内存**：声明变量、创建对象 / 函数时，引擎自动分配内存；
2. **使用内存**：读写变量、调用函数，使用分配的内存；
3. **释放内存**：当内存不再被需要时，垃圾回收器自动回收这部分内存。

垃圾回收的核心任务就是**判断 “内存是否不再被需要”**，并执行释放。

## 二、核心垃圾回收算法

JavaScript 引擎的垃圾回收算法经历了迭代，目前主流是**标记清除（Mark-and-Sweep）** 及其优化版本，早期的**引用计数（Reference Counting）** 因存在缺陷已被淘汰。

### 1. 引用计数（早期算法，已淘汰）

#### 核心原理

给每个内存对象（如对象、数组、函数）维护一个 “引用计数”：

- 每有一个变量 / 属性引用该对象，计数 +1；
- 每有一个引用失效（变量被销毁、赋值为 `null`），计数 -1；
- 当计数为 0 时，说明该对象不再被使用，垃圾回收器立即回收它。
#### 示例

```javascript
let obj1 = { name: 'Alice' }; // 对象 A 被 obj1 引用，计数 = 1
let obj2 = obj1;                // 对象 A 又被 obj2 引用，计数 = 2
obj1 = null;                    // obj1 引用失效，计数 = 1
obj2 = null;                    // obj2 引用失效，计数 = 0 → 对象 A 被回收
```

#### 致命缺陷：循环引用

当两个对象互相引用时，它们的引用计数永远不会为 0，导致内存永远无法回收，这是引用计数被淘汰的核心原因。

**循环引用示例**：

```javascript
function createLeak() {
  const objA = {};
  const objB = {};
  objA.ref = objB; // objA 引用 objB
  objB.ref = objA; // objB 引用 objA → 两者计数都是 1
}
createLeak();
// 函数执行完，objA 和 objB 互相引用，计数永远不为 0，无法回收
```

### 2. 标记清除（Mark-and-Sweep，现代主流基础算法）

#### 核心原理

从 “根对象（Root）” 开始，遍历所有能访问到的对象（标记为 “存活”），然后回收所有未被标记的对象（“垃圾”）。

这个算法解决了引用计数的循环引用问题，是目前所有现代 JavaScript 引擎（V8、SpiderMonkey、JavaScriptCore）的基础算法。

#### 关键概念：根对象（Root）

根对象是垃圾回收的起点，包括：

- 全局对象（浏览器中的 `window`、Node.js 中的 `global`）；
- 执行栈中的局部变量和参数（当前正在执行的函数的变量）；
- 闭包中的变量。

#### 执行过程

分为两个阶段：

1. **标记阶段（Mark）**：
    
    - 从根对象开始，递归遍历所有能通过引用链访问到的对象；
    - 每访问到一个对象，就给它打上 “存活” 的标记。
    
2. **清除阶段（Sweep）**：
    
    - 遍历整个堆内存，回收所有**没有被标记**的对象；
    - 同时清除所有对象的标记，为下一次垃圾回收做准备。
#### 为什么能解决循环引用？

回到之前的循环引用示例：

```javascript
function createLeak() {
  const objA = {};
  const objB = {};
  objA.ref = objB;
  objB.ref = objA;
}
createLeak();
```

函数执行完后，`objA` 和 `objB` 离开了执行栈，从根对象（`window`）出发，无法访问到它们，因此不会被标记，会被垃圾回收器回收 —— 完美解决了循环引用问题。

#### 缺点：内存碎片

清除阶段后，存活的对象在堆内存中可能不连续，产生 “内存碎片”，导致后续无法分配大的连续内存块。

### 3. 标记整理（Mark-Compact，标记清除的优化）

为了解决标记清除的内存碎片问题，现代引擎在标记清除的基础上增加了 “整理” 阶段：

- 标记阶段：和标记清除一致；
- 整理阶段：将所有存活的对象向堆内存的一端移动，排列成连续的内存块；
- 清除阶段：直接清理掉连续内存块之外的所有空间。

这样就消除了内存碎片，后续可以更高效地分配大内存。

### 4. 分代回收（Generational Collection，V8 等引擎的核心优化）

现代引擎（如 Chrome 的 V8、Node.js）基于一个观察：**对象的生命周期差异很大，大多数对象 “朝生夕死”，少数对象会长期存活**。

因此，引擎将堆内存分为两个 “代”，对不同代采用不同的回收策略，大幅提升回收效率：

- **新生代（Young Generation）**：存放新创建的对象，生命周期短，回收频繁；
- **老生代（Old Generation）**：存放经过多次新生代回收后仍存活的对象，生命周期长，回收频率低。

#### 新生代回收（Scavenge 算法）

新生代空间较小（通常几十 MB），采用 **Scavenge 算法**（基于复制的算法）：

1. 将新生代分为两个相等的空间：`From` 空间和 `To` 空间；
2. 新对象分配在 `From` 空间；
3. 垃圾回收时，将 `From` 空间中**存活的对象**复制到 `To` 空间，排列成连续内存；
4. 清空 `From` 空间，然后交换 `From` 和 `To` 的角色，等待下一次回收。

**晋升机制**：如果一个对象在新生代中经过多次（通常 2 次）回收后仍存活，就会被 “晋升” 到老生代。

#### 老生代回收（Mark-Sweep/Mark-Compact）

老生代空间大，对象存活时间长，不适合用 Scavenge 算法（复制大对象成本高），因此采用 **标记清除 + 标记整理**：

- 当老生代内存不足时，先执行标记清除；
- 如果内存碎片严重，再执行标记整理。

## 三、垃圾回收的触发时机

垃圾回收是**自动触发**的，引擎会根据内存使用情况决定何时执行，常见触发时机：

- 新生代空间满了，触发新生代 GC；
- 老生代空间满了，触发老生代 GC；
- 程序空闲时（浏览器会尝试在空闲时间执行 GC，避免阻塞主线程）。

⚠️ 注意：我们无法手动触发垃圾回收（虽然部分环境有 `gc()` 函数，但不推荐在生产环境使用），只能通过合理的代码编写，配合 GC 工作。

## 四、前端开发中如何配合 GC？

虽然 GC 是自动的，但我们可以通过良好的编码习惯，减少 GC 的压力，避免内存泄漏：

1. **及时释放不必要的引用**：
    
    - 不再使用的全局变量、闭包变量，及时赋值为 `null`：
        
```javascript
    let bigData = new Array(1000000).fill('数据');
    // 使用完后
    bigData = null; // 释放引用，让 GC 可以回收
```
    
2. **避免意外的全局变量**：
    
    - 使用严格模式 `'use strict'`，防止未声明的变量变成全局变量。
    
3. **及时清理 “副作用”**：
    
    - 组件销毁时，清理定时器、事件监听器、第三方库的实例（参考之前的 “内存泄漏” 章节）。
    
4. **优先使用弱引用数据结构**：
    
    - `WeakMap`、`WeakSet` 的引用是 “弱引用”，不会阻止 GC 回收对象，适合临时存储对象引用：
        
```javascript
    const weakMap = new WeakMap();
    const obj = {};
    weakMap.set(obj, '数据');
    obj = null; // obj 被回收，weakMap 中的对应项也会自动消失
```
## 总结

- **核心目标**：自动回收不再使用的内存，避免内存泄漏；
- **主流算法**：现代引擎以**分代回收**为核心，新生代用 Scavenge，老生代用 Mark-Sweep/Mark-Compact；
- **关键优势**：标记清除解决了循环引用问题，分代回收提升了效率；
- **开发者注意**：无需手动 GC，但要及时释放不必要的引用，避免内存泄漏。
