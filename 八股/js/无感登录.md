无感登录也叫**无感刷新登录态 / 静默登录**，是前后端分离项目中，兼顾用户体验与账号安全的身份认证方案。核心是：**在用户登录态过期时，无需中断当前操作、无需手动输入账号密码，自动完成登录态续期；甚至用户关闭页面后再次打开，也能自动恢复登录态，全程无感知**。

它解决了传统登录方案的两大核心痛点：

1. 短时效登录态过期后，强制用户跳登录页重登，打断操作流程，体验极差；
2. 长时效登录态虽不用频繁登录，但一旦泄露，攻击者可长期冒用用户身份，安全风险极高。

## 一、主流实现方案：双 Token 无感刷新机制（前后端分离首选）

这是目前前端开发中最通用、最标准的无感登录方案，核心是**职责分离的双 Token 设计**，用短时效令牌降低安全风险，用长时效令牌实现无感续期。

### 1. 双 Token 核心定义

| 令牌类型                       | 核心作用                        | 时效建议                   | 使用规则                                           |
| -------------------------- | --------------------------- | ---------------------- | ---------------------------------------------- |
| **Access Token（AT，访问令牌）**  | 业务接口的身份凭证，所有需要鉴权的接口都必须携带    | 短时效，1~2 小时             | 放在请求头 `Authorization: Bearer <AT>` 中，仅用于业务接口请求 |
| **Refresh Token（RT，刷新令牌）** | 仅用于刷新 Access Token，不能用于业务接口 | 长时效，7~30 天（根据业务安全等级调整） | 仅能调用后端的「刷新令牌接口」，禁止用于其他任何接口                     |

核心设计逻辑：AT 泄露后，因为时效极短，风险可控；RT 仅用于刷新接口，后端会做严格校验，泄露风险极低，同时实现了无需用户操作的登录态续期。

### 2. 完整执行流程（前端视角）

#### 阶段 1：首次登录，获取双 Token

1. 用户输入账号密码，前端发送登录请求；
2. 后端验证账号密码通过，生成并返回 AT + RT；
3. 前端存储双 Token（存储方案见安全规范），完成登录。

#### 阶段 2：正常业务请求

前端所有业务请求，在请求拦截器中自动携带 AT，后端校验 AT 有效性，校验通过则正常返回业务数据。

#### 阶段 3：AT 过期，无感刷新

1. AT 过期后，后端返回固定的 `401 Unauthorized` 状态码；
2. 前端响应拦截器捕获到 401，**中断后续业务请求**，用本地存储的 RT 调用后端的「刷新令牌接口」；
3. 后端校验 RT 有效性，校验通过后，返回**新的 AT**（进阶方案会同时返回新的 RT，即「RT 轮转」）；
4. 前端更新本地存储的 AT（和 RT），自动重试之前因 401 失败的业务请求；
5. 整个过程用户完全无感知，不会看到登录页，操作不会中断。

#### 阶段 4：RT 过期，兜底处理

如果 RT 也过期 / 无效，后端刷新接口会返回固定错误码，前端清除本地所有登录态，跳转到登录页，仅此时需要用户手动登录。

### 3. 前端核心实现（Axios 完整示例，带刷新锁）

前端实现的核心难点是**解决并发请求重复刷新的问题**：如果多个接口同时返回 401，会触发多次刷新请求，导致 RT 失效。因此必须加「刷新锁」，确保同一时间只有一个刷新请求在执行，其他请求等待刷新完成后自动重试。

```javascript
import axios from 'axios';

const request = axios.create({
  baseURL: '/api',
  timeout: 10000
});

// 核心：刷新锁，标记是否正在刷新Token
let isRefreshing = false;
// 等待重试的请求队列
let retryQueue = [];

// 1. 请求拦截器：自动携带 Access Token
request.interceptors.request.use(
  (config) => {
    const accessToken = localStorage.getItem('access_token');
    if (accessToken) {
      config.headers.Authorization = `Bearer ${accessToken}`;
    }
    return config;
  },
  (error) => Promise.reject(error)
);

// 2. 响应拦截器：处理Token过期，实现无感刷新
request.interceptors.response.use(
  // 响应成功直接返回
  (response) => response.data,
  // 响应错误处理
  (error) => {
    const originalConfig = error.config;
    // 非401错误直接抛出
    if (error.response?.status !== 401) {
      return Promise.reject(error);
    }

    // 核心：处理401 Token过期
    if (!isRefreshing) {
      isRefreshing = true;
      const refreshToken = localStorage.getItem('refresh_token');

      // 调用刷新Token接口
      return axios.post('/api/refresh-token', { refreshToken })
        .then((res) => {
          // 刷新成功，更新本地Token
          const { accessToken, newRefreshToken } = res.data;
          localStorage.setItem('access_token', accessToken);
          localStorage.setItem('refresh_token', newRefreshToken);

          // 重试队列中所有等待的请求
          retryQueue.forEach((cb) => cb(accessToken));
          retryQueue = [];

          // 重试当前失败的请求
          originalConfig.headers.Authorization = `Bearer ${accessToken}`;
          return request(originalConfig);
        })
        .catch(() => {
          // 刷新失败：RT过期，清除登录态，跳登录页
          localStorage.removeItem('access_token');
          localStorage.removeItem('refresh_token');
          window.location.href = '/login';
          return Promise.reject(error);
        })
        .finally(() => {
          // 释放刷新锁
          isRefreshing = false;
        });
    } else {
      // 正在刷新中，将请求加入等待队列，返回Promise
      return new Promise((resolve) => {
        retryQueue.push((newToken) => {
          originalConfig.headers.Authorization = `Bearer ${newToken}`;
          resolve(request(originalConfig));
        });
      });
    }
  }
);

export default request;
```

### 4. 进阶扩展：冷启动无感登录

无感登录不仅包含页面运行中的续期，还包含用户关闭页面后再次打开的**自动无感登录**：

1. 用户首次登录后，RT 持久化存储在本地；
2. 用户下次打开页面时，前端先判断本地是否有有效 RT；
3. 如果有，直接调用刷新接口换取新的 AT，完成自动登录，直接进入业务页面，无需跳登录页；
4. 如果没有 RT 或刷新失败，再跳转到登录页。

## 二、其他场景的无感登录方案

### 1. Session-Cookie 滑动过期方案（同域传统项目）

适合传统同域单体项目、服务端渲染（SSR）项目，无需前端做复杂逻辑，完全由后端实现。

- **核心原理**：后端给 Session-Cookie 设置**滑动有效期**（也叫滚动过期）。用户每次发起有效请求，后端自动刷新 Session 的过期时间；只要用户持续操作，登录态就会一直续期，不会过期；只有用户长期无操作，Session 才会过期。
- **前端操作**：无需任何额外代码，浏览器会自动携带 Cookie，全程用户无感。
- **优缺点**：实现简单，无需前端处理；但仅适用于同域项目，跨域场景下 Cookie 携带受限。

### 2. OAuth2 静默授权方案（第三方 / 企业内应用）

适合微信网页、企业微信 / 钉钉 / 飞书内应用、第三方联合登录场景。

- **核心原理**：基于 OAuth2 协议的静默授权模式（如微信的 `snsapi_base` 模式），无需用户手动点击「授权」按钮，前端通过页面重定向，直接从平台获取授权码，后端用授权码换取用户登录态，全程用户无感知。
- **适用场景**：企业内部应用、公众号 H5、第三方平台登录，无需用户输入账号密码。

### 3. 同域单点登录（SSO）无感同步

适合同一企业 / 品牌下的多个子域名系统，实现「一次登录，全平台通行」。

- **核心原理**：基于统一认证中心的根域名 Cookie 共享。用户在 A 系统登录后，认证中心将登录态 Cookie 种在根域名下；用户访问同根域名的 B 系统时，B 系统自动读取根域名下的登录态，完成自动登录，无需用户重复输入账号密码。

## 三、安全最佳实践（前端必知）

无感登录的核心是平衡体验与安全，以下是前端必须遵守的安全规范，避免出现账号安全漏洞：

1. **Refresh Token 必须用 HttpOnly Cookie 存储**
    
    绝对禁止将 RT 存在 `localStorage`/`sessionStorage` 中，这两个存储极易被 XSS 攻击窃取。必须让后端将 RT 种在 `HttpOnly; Secure; SameSite=Lax` 的 Cookie 中，前端 JS 无法读取，从根源上防止 XSS 窃取。
    
    > 上述代码示例用 localStorage 仅为演示逻辑，生产环境必须用 HttpOnly Cookie。
    
2. **严格的职责隔离**
    
    RT 只能用于「刷新令牌接口」，后端必须禁止 RT 调用任何业务接口，即使 RT 泄露，攻击者也无法调用业务接口，降低风险。
    
3. **Refresh Token 轮转机制**
    
    每次刷新 AT 时，后端必须同时生成新的 RT，旧的 RT 立即作废。避免一个 RT 长期有效，即使泄露，也会很快失效。
    
4. **刷新接口严格限流**
    
    后端必须对刷新接口做严格的限流，防止攻击者用泄露的 RT 暴力破解。
    
5. **黑名单机制**
    
    用户主动登出、修改密码、账号异地登录时，后端必须将对应的 RT 加入黑名单，立即作废，防止被盗用的 RT 继续使用。
    
6. **时效控制**
    
    AT 越短越安全，建议不超过 2 小时；RT 时长根据业务安全等级调整，金融 / 支付类建议不超过 7 天，普通工具类不超过 30 天。

## 四、前端落地常见坑与解决方案

1. **刷新接口死循环**
    
    刷新接口本身也会返回 401，必须在拦截器中排除刷新接口的 401 重试，否则会出现「刷新失败→继续重试刷新」的死循环。
    
2. **无痕模式兼容问题**
    
    Safari、Chrome 无痕模式会拦截第三方 Cookie，如果前后端跨域，RT 无法正常存储，需做降级处理：提示用户开启 Cookie 权限，或切换为同域代理方案。
    
3. **路由守卫提前校验**
    
    不要等接口返回 401 才刷新，可在路由守卫中提前校验 AT 的过期时间，快过期时提前刷新，避免页面加载后批量请求失败。
    
4. **网络异常兜底**
    
    刷新接口因网络问题失败时，不要直接跳登录页，给用户提供「重试」按钮，避免因临时网络波动导致用户被迫重登。
