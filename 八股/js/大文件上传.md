大文件上传是针对 GB 级超大文件（高清视频、系统镜像、数据集、大型压缩包等）的专项上传解决方案，核心解决传统单文件直传的四大核心痛点：网络波动 / 页面刷新导致的全量重传、服务器 / 反向代理对单请求体的大小限制、大文件全量加载导致的内存溢出、长时上传不可控的极差用户体验。

目前行业内的底层核心方案是**分片上传（分块上传）**，并基于此扩展断点续传、秒传、并发控制等能力，形成完整的大文件上传体系。

---

## 一、核心底层原理：分片上传

分片上传的核心逻辑是：将一个完整的大文件，按照固定大小切分为多个连续、独立的小数据块（业内称分片 / Chunk，通用大小为 2MB~10MB），客户端将这些分片独立上传至服务端；待全部分片上传完成后，服务端按分片编号顺序，将所有分片合并为完整的原文件，完成整个上传流程。

该方案的核心优势，精准对应传统直传的痛点：

1. 容错性极强：单分片上传失败，仅需重传该分片，无需全量重传；
2. 突破大小限制：彻底规避 Web 服务器、反向代理、CDN 对单 HTTP 请求体的大小上限；
3. 内存占用极低：客户端与服务端均只需处理单个分片，无需一次性加载全量文件，避免内存溢出；
4. 传输可控性强：支持暂停 / 恢复、并发调速、进度可视化，大幅提升长时上传的用户体验。

## 二、大文件上传完整标准流程

一套完整的大文件上传链路，需要客户端与服务端全流程协作，分为 4 个核心阶段，步骤如下：

### 1. 文件预处理与唯一标识生成

客户端获取待上传文件后，先完成基础信息处理，为后续流程做准备：

- 读取文件元信息：包括文件名、总大小、MIME 类型、修改时间等；
- 生成文件唯一标识：通过密码学哈希算法（MD5、SHA-1、SHA-256）计算文件的内容哈希，作为该文件的唯一 ID。针对超大文件，采用**流式增量哈希**计算，分片读取文件内容、增量更新哈希值，避免全量文件读入内存；同时可通过子线程 / Worker 执行计算，避免阻塞主线程；
- 分片规划：根据文件总大小、网络环境，确定单分片大小、分片总数，为每个分片分配从 0 开始的连续编号，确保服务端合并时顺序正确。

### 2. 前置校验（秒传 + 断点续传预检）

客户端将文件哈希、文件总大小、分片总数发送至服务端，执行两项核心校验，最大化减少无效传输：

- **秒传校验**：服务端检查存储系统中是否已存在该哈希对应的完整文件。若存在，直接返回上传成功，无需传输任何分片；同时可通过文件引用计数实现存储去重，节省服务器存储资源；
- **断点续传预检**：若不存在完整文件，服务端查询该哈希对应的已成功上传的分片列表，返回给客户端。客户端据此过滤已完成的分片，仅需上传剩余未完成的分片。

### 3. 分片传输

客户端根据预检结果，传输剩余分片，主流分为两种传输模式：

- **串行传输**：一个分片上传成功后，再传输下一个，稳定性高，适合弱网、高延迟环境；
- **并发传输**：同时发起多个分片的上传请求，提升传输效率，适合优质网络环境，通常限制并发数为 3~6 个，规避客户端同源请求限制、服务端限流规则。

每个分片的上传请求，必须携带核心标识信息：文件唯一哈希、上传任务唯一 ID（uploadId）、当前分片编号、分片总数、分片二进制数据、分片内容哈希（用于服务端完整性校验）。

服务端收到分片后，先校验分片哈希是否匹配，校验通过后，将分片暂存至临时目录（通常以文件哈希为目录名，分片编号为文件名），并返回该分片上传成功的响应。

### 4. 合并通知与最终校验

所有分片全部上传成功后，客户端向服务端发起**合并请求**，携带文件哈希、文件名、分片总数、文件总大小、原文件完整哈希。

服务端收到合并请求后，执行全流程收尾操作：

1. 全量校验：检查所有分片是否完整上传，无缺失、无损坏；
2. 流式合并：按分片编号的顺序，流式读取所有分片，合并为完整的原文件，避免全量读入内存导致的服务端内存溢出；
3. 最终完整性校验：计算合并后完整文件的哈希，与客户端上传的原文件哈希比对，确保文件无损坏、无丢包、顺序正确；
4. 收尾处理：校验通过后，将完整文件移入正式存储目录，生成文件访问地址，清理临时分片文件，返回上传成功的最终响应。

## 三、核心扩展能力

基于分片上传的基础能力，可扩展出大文件上传必备的核心增值能力，也是方案成熟度的关键：

### 1. 断点续传

断点续传的核心是**上传状态的持久化**，解决网络中断、页面刷新、客户端重启导致的上传进度丢失问题：

- 服务端：以文件哈希为维度，用 Redis 缓存已上传的分片状态，用本地磁盘 / 分布式存储暂存分片文件；同时设置过期清理规则（如 7 天未完成的分片自动清理），释放存储资源；
- 客户端：支持手动暂停上传（终止未完成的分片请求，停止发起新请求）、恢复上传（重新发起预检请求，获取已上传分片列表，继续传输剩余分片），即使程序重启，也能通过文件哈希恢复上传进度。

### 2. 秒传

秒传的本质是**基于内容哈希的文件去重**，核心逻辑是：相同内容的文件，生成的哈希值完全一致。服务端维护已上传文件的哈希索引，收到客户端的预检请求后，若索引中已存在该哈希的完整文件，直接复用已有文件，无需重复传输和存储，可极大节省带宽和存储成本，尤其适合多用户上传相同文件的场景（如企业内部通用安装包、热门视频素材）。

### 3. 异常处理与可靠性保障

大文件上传链路长、耗时久，完善的异常处理是方案稳定的核心：

- **失败重试**：单个分片上传失败（超时、网络波动、服务端临时错误）时，采用指数退避策略重试（重试间隔 1s/2s/4s 递增），达到最大重试次数（通常 3~5 次）仍失败时，终止上传并提示用户，避免无效请求占用资源；
- **分片级校验**：每个分片上传后都做哈希校验，确保单个分片传输无损坏，避免合并后全文件出错；
- **超时控制**：为每个分片请求设置独立的超时时间，避免长时挂起的无效请求。

### 4. 进度可视化与管控

客户端基于已上传成功的分片总大小，实时计算整体上传进度（已上传大小 / 文件总大小），对外暴露进度回调，用于进度条、上传速度预估、剩余时间展示，大幅提升长时上传的用户体验；同时支持手动暂停、恢复、取消上传的操作，给用户完全的控制权。

## 四、主流落地实现方案

### 1. 自研分片上传方案

前后端基于分片上传的核心原理，自主实现完整的预处理、预检、分片传输、合并、续传全流程逻辑。优势是灵活度极高，可根据业务需求深度定制（如自定义鉴权规则、文件审核流程、限速策略、多租户隔离），是中大型企业级业务的主流选择。

### 2. 基于标准化开源协议实现

最主流的是**tus 协议**：这是一套基于 HTTP 的断点续传开放标准协议，定义了标准化的上传、续传、暂停、恢复、扩展规范，提供了多语言的客户端、服务端 SDK，开箱即用，无需从零开发核心逻辑，适合中小型团队快速落地成熟的大文件上传能力。

### 3. 云存储厂商官方方案

主流云存储服务商（阿里云 OSS、腾讯云 COS、AWS S3、MinIO）均提供了成熟的分片上传能力，封装了完整的分片、续传、合并、校验逻辑，客户端可直接通过官方 SDK 调用，无需自行开发服务端逻辑，开发成本极低，适合基于云存储构建业务的场景。其典型流程为：初始化分片上传任务获取 uploadId → 分片上传 → 完成合并 / 取消上传。

## 五、关键优化与安全保障

### 性能优化

1. **哈希计算优化**：对安全性要求不高的场景，可采用抽样哈希（取文件开头、中间、结尾的分片数据 + 文件总大小计算哈希），大幅缩短哈希计算时间；高安全性场景采用流式全量哈希 + 异步线程计算，避免阻塞主线程。
2. **传输优化**：基于 HTTP/2 多路复用提升并发传输性能，规避 HTTP/1.1 的同源并发限制；根据实时网络质量动态调整分片大小和并发数，弱网环境缩小分片、切换串行传输，优质网络放大分片、提升并发数。
3. **服务端优化**：采用流式合并分片，避免全量文件加载到内存；用 Redis 缓存分片上传状态，提升预检查询效率；采用分布式存储系统，支撑大文件的高并发读写与海量存储。

### 安全防护

1. **上传鉴权**：所有上传请求必须绑定用户身份与鉴权凭证（Token、API 签名），限制单用户的上传总容量、单文件大小、允许的文件类型，防止未授权上传与恶意刷量。
2. **防篡改与完整性保障**：分片级哈希校验 + 全文件最终哈希校验，双重保障文件传输过程中无篡改、无损坏；上传任务与用户身份绑定，禁止越权操作他人的上传任务。
3. **内容安全管控**：上传完成后对文件进行类型校验、病毒查杀、违规内容检测，禁止恶意可执行文件、违规内容上传，规避业务安全风险。
4. **资源隔离与清理**：临时分片文件与正式存储隔离，设置自动过期清理规则，避免无效分片占用存储资源。
