事件委托是前端开发中基于**事件冒泡机制**的一种高效事件处理模式，核心思想是：**将子元素的事件监听器统一绑定在父元素（或祖先元素）上，利用事件冒泡机制，通过父元素的监听器来处理所有子元素的同类事件**。

它是优化事件绑定性能、处理动态元素的核心方案，能大幅减少内存占用、简化代码逻辑。

## 一、前置基础：事件冒泡机制

要理解事件委托，必须先明确事件流的三个阶段（以点击事件为例）：

1. **捕获阶段**：事件从 `window` 开始，自上而下向目标元素传递；
2. **目标阶段**：事件到达实际触发的目标元素；
3. **冒泡阶段**：事件从目标元素开始，自下而上向 `window` 传递。

事件委托的核心就是利用**冒泡阶段**：子元素触发事件后，会沿着 DOM 树向上冒泡，父元素会接收到这个事件，从而可以在父元素的监听器中统一处理。

## 二、事件委托的核心优势

### 1. 大幅减少内存占用

如果有大量同类子元素需要绑定事件（比如一个有 1000 个 `<li>` 的列表），传统方式需要给每个 `<li>` 都绑定一个监听器，会占用大量内存；而事件委托只需要给父 `<ul>` 绑定 1 个监听器，内存占用极低。

### 2. 自动处理动态元素

对于后续动态添加的子元素（比如通过 AJAX 加载新列表项、点击 “添加” 按钮新增元素），传统方式需要重新给新元素绑定监听器；而事件委托无需额外操作，动态添加的子元素触发事件时，仍会通过冒泡被父元素的监听器捕获处理。

## 三、事件委托的实现方式

### 核心步骤

1. **选择合适的父元素**：将监听器绑定在离目标子元素最近的、稳定的父元素上（避免绑定在 `document`/`window` 上，减少冒泡路径）；
2. **判断事件目标**：在父元素的监听器中，通过 `event.target` 获取实际触发事件的元素，判断是否是我们要处理的目标子元素；
3. **执行对应逻辑**：如果是目标元素，执行事件处理代码。

### 代码示例

#### 基础示例：列表项点击

```html
<ul id="list">
  <li class="item">列表项 1</li>
  <li class="item">列表项 2</li>
  <li class="item">列表项 3</li>
  <!-- 后续可能动态添加更多 <li> -->
</ul>
```

```javascript
// 传统方式：给每个 li 绑定监听器（不推荐，li 多了性能差）
// const items = document.querySelectorAll('.item');
// items.forEach(item => {
//   item.addEventListener('click', () => {
//     console.log('点击了:', item.textContent);
//   });
// });

// 事件委托方式：给父 ul 绑定监听器（推荐）
const list = document.getElementById('list');
list.addEventListener('click', (event) => {
  // 1. 获取实际触发事件的元素
  const target = event.target;
  // 2. 判断是否是目标元素（比如是否有 .item 类）
  if (target.classList.contains('item')) {
    // 3. 执行处理逻辑
    console.log('点击了:', target.textContent);
  }
});

// 测试：动态添加新 li，无需重新绑定监听器
const newItem = document.createElement('li');
newItem.className = 'item';
newItem.textContent = '动态添加的列表项';
list.appendChild(newItem);
// 点击新 li 也会触发事件，无需额外操作
```

#### 进阶示例：处理子元素嵌套的情况

如果目标元素内部还有子元素（比如 `<li>` 里有 `<span>`），`event.target` 可能是内部的 `<span>`，这时候需要用 `closest()` 方法向上查找最近的目标元素：

```html
<ul id="list">
  <li class="item">
    <span>列表项 1</span>
  </li>
</ul>
```

```javascript
const list = document.getElementById('list');
list.addEventListener('click', (event) => {
  // 用 closest() 向上查找最近的 .item 元素（包括 target 自身）
  const targetItem = event.target.closest('.item');
  if (targetItem) {
    console.log('点击了列表项:', targetItem.textContent);
  }
});
```

## 四、事件委托的注意事项

### 1. 选择合适的事件类型

只有**支持冒泡的事件**才能用事件委托，常见的支持冒泡的事件有：`click`、`mousedown`、`mouseup`、`keydown`、`keyup`、`input`、`change` 等。

**不支持冒泡的事件**（如 `focus`、`blur`、`mouseenter`、`mouseleave`）不能直接用事件委托，需要用它们的冒泡替代事件：

- `focus` → `focusin`
- `blur` → `focusout`
- `mouseenter` → `mouseover`
- `mouseleave` → `mouseout`

### 2. 选择合适的父元素

不要将监听器绑定在 `document` 或 `window` 上，原因：

- 冒泡路径太长，事件传递到父元素需要更多时间，影响性能；
- 多个事件委托都绑在 `document` 上，容易导致事件冲突。

**最佳实践**：将监听器绑定在离目标子元素最近的、稳定的父元素上（父元素不会被动态移除）。

### 3. 正确判断目标元素

- 优先用 `closest()` 方法，避免因目标元素内部嵌套子元素导致的判断失效；
- 可以通过 `tagName`、`classList`、`id`、`dataset` 等属性判断目标元素，比如：
    
```javascript
    // 通过 tagName 判断（注意 tagName 是大写）
    if (target.tagName === 'LI') { ... }
    // 通过 dataset 判断（自定义属性）
    if (target.dataset.type === 'button') { ... }
```
### 4. 避免过度委托

不要把所有事件都委托给同一个父元素，不同类型的事件可以委托给不同的父元素，保持代码逻辑清晰，减少判断复杂度。

## 五、总结

- **核心原理**：利用事件冒泡机制，将子元素事件统一绑定在父元素上；
- **核心优势**：减少内存占用、自动处理动态元素；
- **实现关键**：选择合适的父元素，用 `event.target`/`closest()` 判断目标元素；
- **注意事项**：仅支持冒泡事件，选择最近的稳定父元素，正确判断目标。

事件委托是处理大量同类元素、动态元素事件的首选方案，能显著提升性能和代码可维护性。
