防抖和节流是前端开发中**控制高频函数执行频率**的核心性能优化方案，专门解决高频触发的 DOM 事件（如输入框 input、窗口 resize、页面滚动 scroll、鼠标移动 mousemove 等）导致的函数频繁执行问题，避免页面卡顿、浏览器性能过载、接口请求泛滥等情况。

二者的核心差异可以一句话概括：**防抖是「触发停止后再执行」，节流是「固定时间内最多执行一次」**。

---

## 一、防抖（Debounce）

### 1. 核心原理

当事件被触发后，不会立即执行目标函数，而是等待一个指定的延迟时间；如果在这个延迟时间内，事件再次被触发，就会**重新计时**，直到延迟时间内没有新的触发，才会真正执行一次目标函数。

简单来说：**频繁触发的事件，只会在最后一次触发结束后，执行一次函数**。

### 2. 生活化类比

电梯关门逻辑：有人进入电梯后，电梯会等待 5 秒再关门；如果这 5 秒内又有人进入，就会重新等待 5 秒，直到连续 5 秒没人进入，电梯才会执行关门动作。

### 3. 代码实现

#### 基础版（非立即执行，最常用）

触发事件后不立即执行，等待延迟结束后执行，适合输入联想、窗口 resize 等场景：

```javascript
/**
 * 防抖函数
 * @param {Function} fn 要执行的目标函数
 * @param {Number} delay 延迟时间（毫秒）
 * @returns {Function} 防抖处理后的函数
 */
function debounce(fn, delay) {
  let timer = null; // 闭包保存定时器，避免全局变量污染
  return function(...args) {
    // 每次触发事件，清除之前的定时器，重新计时
    if (timer) clearTimeout(timer);
    // 重新设置定时器，延迟执行目标函数
    timer = setTimeout(() => {
      fn.apply(this, args); // 修正this指向，传递事件等参数
      timer = null;
    }, delay);
  };
}
```

#### 立即执行版（前缘防抖）

触发事件时立即执行函数，之后的延迟时间内再次触发不会执行，适合按钮防重复提交场景：

```javascript
function debounceImmediate(fn, delay) {
  let timer = null;
  return function(...args) {
    const canRun = !timer; // 定时器为空时，允许执行
    // 每次触发，重置定时器
    if (timer) clearTimeout(timer);
    timer = setTimeout(() => {
      timer = null; // 延迟结束后重置状态，允许下次执行
    }, delay);
    // 立即执行目标函数
    if (canRun) fn.apply(this, args);
  };
}
```

### 4. 典型应用场景

1. **搜索框输入联想**：用户输入停止后再发送搜索请求，避免每输入一个字符就触发一次接口，大幅减少请求次数；
2. **窗口 resize 事件**：窗口大小调整结束后，再重新计算布局、渲染元素，避免调整过程中频繁执行导致的卡顿；
3. **按钮防重复提交**：表单提交、秒杀按钮点击后立即执行，短时间内再次点击不生效，防止重复提交；
4. **文本编辑器实时保存**：用户输入停止后，再执行内容保存操作。

## 二、节流（Throttle）

### 1. 核心原理

规定一个固定的时间窗口，在这个时间窗口内，无论事件触发多少次，目标函数**最多只会执行一次**。如果在时间窗口内事件再次触发，不会执行函数，也不会重置计时，直到当前时间窗口结束，下一个时间窗口才会响应新的触发。

简单来说：**频繁触发的事件，会按固定间隔匀速执行函数，稀释执行频率**。

### 2. 生活化类比

游戏技能冷却：释放一次技能后，会有固定的冷却时间，冷却时间内无论按多少次技能按键，都不会触发技能；只有冷却结束后，才能再次释放技能。

### 3. 代码实现

#### 时间戳版（前缘执行）

触发事件时立即执行，固定间隔内只执行一次，适合滚动加载、高频点击等场景：

```javascript
/**
 * 节流函数
 * @param {Function} fn 要执行的目标函数
 * @param {Number} interval 执行间隔（毫秒）
 * @returns {Function} 节流处理后的函数
 */
function throttle(fn, interval) {
  let lastTime = 0; // 记录上一次执行的时间戳
  return function(...args) {
    const now = Date.now();
    // 达到间隔时间，才执行目标函数
    if (now - lastTime >= interval) {
      fn.apply(this, args); // 修正this指向，传递参数
      lastTime = now; // 更新上一次执行时间
    }
  };
}
```

#### 定时器版（后缘执行）

时间窗口结束后执行函数，能保留最后一次触发的状态，适合鼠标跟随、拖拽等场景：

```javascript
function throttleTimer(fn, interval) {
  let timer = null;
  return function(...args) {
    if (!timer) {
      timer = setTimeout(() => {
        fn.apply(this, args);
        timer = null;
      }, interval);
    }
  };
}
```

### 4. 典型应用场景

1. **页面滚动 scroll 事件**：滚动加载更多、滚动显示 / 隐藏回到顶部按钮，每隔固定时间执行一次，避免每滚动 1px 就触发函数；
2. **鼠标移动 mousemove 事件**：拖拽元素、鼠标跟随特效、画布绘制，固定时间内更新位置，避免频繁触发导致的性能问题；
3. **高频点击事件**：秒杀、抢票、点赞按钮，固定时间内只能触发一次，防止恶意高频点击；
4. **视频播放进度更新、实时数据上报**等需要持续反馈，但无需高频执行的场景。

## 三、防抖与节流的核心区别

|对比维度|防抖（Debounce）|节流（Throttle）|
|---|---|---|
|核心逻辑|频繁触发时，**只在最后一次触发结束后执行一次**，每次触发都会重置计时|频繁触发时，**固定时间窗口内最多执行一次**，触发不会重置计时，匀速执行|
|执行时机|延迟时间结束后执行（或触发时立即执行，之后延迟内禁止）|时间窗口开始时执行（前缘），或时间窗口结束时执行（后缘）|
|高频触发表现|只要持续触发，函数就一直不执行，直到触发完全停止|只要持续触发，函数会按固定间隔匀速执行，不会一直阻塞|
|核心适用场景|只关心「最终状态」的场景，如输入联想、窗口 resize、停止输入后保存|关心「过程中状态」的场景，如滚动加载、鼠标跟随、拖拽、高频点击|

## 四、实际开发使用指南

### 1. 成熟库的使用

实际开发中，无需手动手写实现，优先使用成熟的工具库（如 Lodash）提供的防抖节流函数，兼容性更强、功能更完善：

```javascript
// 安装：npm i lodash
import { debounce, throttle } from 'lodash';

// 防抖：输入联想，延迟500ms执行
const handleSearch = debounce((keyword) => {
  console.log('发送搜索请求：', keyword);
}, 500);

// 节流：滚动处理，间隔200ms执行
const handleScroll = throttle(() => {
  console.log('滚动位置：', window.scrollY);
}, 200);
```

Lodash 还支持通过配置项自定义执行时机：

```javascript
// 防抖：前缘执行，按钮防重复提交
const handleSubmit = debounce(submitForm, 300, {
  leading: true,  // 触发时立即执行
  trailing: false // 延迟结束后不执行
});

// 节流：前后缘都执行，滚动时立即执行，滚动停止后再执行一次最终状态
const handleScroll = throttle(scrollHandler, 200, {
  leading: true,
  trailing: true
});
```

### 2. 关键注意事项

1. **时间参数选择**：输入联想推荐 300-500ms，滚动 / 鼠标事件推荐 100-200ms，按钮防重复推荐 200-300ms，根据业务体验调整；
2. **函数缓存**：防抖 / 节流后的函数要提前定义并缓存，不要在事件绑定、渲染函数中每次生成新实例，否则会导致功能失效；
3. **内存泄漏处理**：组件卸载时，要移除事件监听，并调用 Lodash 提供的`cancel()`方法取消待执行的定时器，避免内存泄漏；
4. **场景选择**：只关心最终结果用防抖，需要过程中持续反馈用节流，不要混淆使用。
