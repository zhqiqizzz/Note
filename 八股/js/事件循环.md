事件循环是 JavaScript 实现**异步编程**的核心机制，它的作用是**协调同步任务、异步任务的执行顺序**，确保单线程的 JavaScript 既能高效处理同步代码，又能不阻塞地响应异步操作（如接口请求、定时器、DOM 事件）。

理解事件循环，是掌握 Promise、async/await 等异步语法的底层基础，也是解决异步执行顺序问题的关键。

## 一、前置基础：JavaScript 是单线程的

JavaScript 的核心设计是**单线程**：同一时间只能执行一个任务，所有任务都要排队执行。

为什么设计成单线程？

- 避免多线程操作 DOM 带来的冲突（比如一个线程修改 DOM，另一个删除 DOM，会导致不可预期的结果）；
- 保持语言简单，降低开发复杂度。

但单线程会导致一个问题：如果遇到耗时的异步任务（如等待 3 秒的定时器、等待接口返回），主线程会被阻塞，页面会 “卡死”。**事件循环就是为了解决这个问题而生的**。

## 二、事件循环的核心组成

事件循环由三个核心部分组成：**调用栈（Call Stack）**、**任务队列（Task Queue）**、**事件循环线程**。

### 1. 调用栈（Call Stack）

调用栈是一个 “后进先出” 的栈结构，用于存储**当前正在执行的同步任务**：

- 执行函数时，将函数压入栈；
- 函数执行完后，将函数弹出栈；
- 栈空时，说明当前同步任务执行完毕。

**示例**：

```javascript
function foo() {
  console.log('foo');
}
function bar() {
  foo();
  console.log('bar');
}
bar();
```

调用栈执行顺序：

1. 压入 `bar()`；
2. 压入 `foo()`；
3. 执行 `foo()`，输出 `foo`，弹出 `foo()`；
4. 执行 `bar()` 剩余代码，输出 `bar`，弹出 `bar()`；
5. 栈空。

### 2. 任务队列（Task Queue）

任务队列是一个 “先进先出” 的队列结构，用于存储**异步任务的回调函数**。当异步任务完成后（如定时器到时间、接口返回数据），它的回调函数会被放入任务队列，等待调用栈清空后执行。

任务队列分为两类，**执行优先级不同**：

#### （1）宏任务（Macro Task）

也叫 “任务”，是浏览器 / Node.js 底层发起的异步任务，常见的有：

- `setTimeout`/`setInterval`；
- DOM 事件（`click`、`input`、`scroll` 等）；
- `requestAnimationFrame`；
- `fetch`/`XMLHttpRequest` 的回调；
- Node.js 中的 `setImmediate`、I/O 操作。

#### （2）微任务（Micro Task）

也叫 “作业”，是 JavaScript 语言层面发起的异步任务，优先级**高于宏任务**，常见的有：

- `Promise.then()`/`catch()`/`finally()`；
- `async/await`（底层是 Promise）；
- `queueMicrotask()`；
- Node.js 中的 `process.nextTick`（优先级比普通微任务更高）。

### 3. 事件循环线程

事件循环线程是独立于主线程的后台线程，它的工作是**不断循环检查调用栈和任务队列**：

- 当调用栈为空时，先执行**所有微任务队列**中的任务；
- 微任务清空后，再从**宏任务队列**中取出一个任务执行；
- 执行完这个宏任务后，再次检查微任务队列，执行所有微任务；
- 如此循环往复。

## 三、事件循环的完整执行流程（核心）

事件循环的执行顺序是固定的，严格按照以下步骤循环：

1. **执行同步任务**：优先执行调用栈中的所有同步任务，直到调用栈清空；
2. **执行所有微任务**：调用栈清空后，立即执行**微任务队列中的所有任务**（一个不剩），直到微任务队列为空；
3. **执行一个宏任务**：微任务清空后，从**宏任务队列中取出第一个任务**执行（只执行一个）；
4. **回到步骤 2**：执行完这个宏任务后，再次检查微任务队列，执行所有新产生的微任务；
5. **循环往复**。

⚠️ **关键规则**：

- 微任务是 “当前事件循环的收尾”，必须在进入下一个宏任务前全部执行完；
- 执行宏任务的过程中，如果产生了新的微任务，会把微任务放入微任务队列，等当前宏任务执行完后，立即执行这些新微任务。

## 四、经典示例：理解执行顺序

用一个最常见的面试题示例，直观展示事件循环的执行顺序：

```javascript
console.log('1'); // 同步任务

setTimeout(() => { // 宏任务
  console.log('2');
}, 0);

Promise.resolve()
  .then(() => { // 微任务
    console.log('3');
  })
  .then(() => { // 微任务
    console.log('4');
  });

console.log('5'); // 同步任务
```

### 执行步骤解析

1. **执行同步任务**：
    
    - 压入 `console.log('1')`，输出 `1`，弹出；
    - 遇到 `setTimeout`，交给浏览器后台计时（0 毫秒后，回调放入宏任务队列）；
    - 遇到 `Promise.resolve().then()`，将第一个 `.then()` 回调放入微任务队列；
    - 压入 `console.log('5')`，输出 `5`，弹出；
    - **此时调用栈清空，同步任务执行完毕**。
    
2. **执行所有微任务**：
    
    - 取出微任务队列中的第一个回调（`console.log('3')`），执行，输出 `3`；
    - 执行过程中产生了第二个 `.then()` 回调，放入微任务队列；
    - 取出下一个微任务（`console.log('4')`），执行，输出 `4`；
    - **微任务队列清空**。
    
3. **执行一个宏任务**：
    
    - 取出宏任务队列中的 `setTimeout` 回调，执行，输出 `2`；
    - **宏任务执行完毕**。
    
4. **再次检查微任务队列**：无新微任务，进入下一个循环。
### 最终输出

```plaintext
1
5
3
4
2
```

## 五、async/await 的事件循环表现

`async/await` 是 Promise 的语法糖，它的执行顺序也遵循事件循环规则：

- `await` 之前的代码是**同步任务**，直接执行；
- `await` 后面的代码（包括 `await` 表达式的返回值处理）相当于 `Promise.then()` 的回调，属于**微任务**。

**示例**：

```javascript
async function asyncFn() {
  console.log('1'); // 同步任务
  const res = await Promise.resolve('2'); // await 等待 Promise，后面的代码是微任务
  console.log(res);
}

console.log('3');
asyncFn();
console.log('4');
```

### 执行步骤

1. 执行 `console.log('3')`，输出 `3`；
2. 调用 `asyncFn()`，执行 `console.log('1')`，输出 `1`；
3. 遇到 `await Promise.resolve('2')`，将 `console.log(res)` 放入微任务队列；
4. 执行 `console.log('4')`，输出 `4`；
5. 调用栈清空，执行微任务：输出 `2`。

### 最终输出

```plaintext
3
1
4
2
```

## 六、浏览器 vs Node.js 事件循环的差异

简单了解即可，前端开发重点掌握浏览器的事件循环：

- **浏览器**：只有一个宏任务队列和一个微任务队列，执行顺序是 “同步 → 所有微任务 → 一个宏任务 → 所有微任务”；
- **Node.js**：宏任务分为多个阶段（Timers、I/O、Poll、Check、Close），微任务在每个阶段切换时执行，且 `process.nextTick` 优先级高于普通微任务。

## 总结

- **核心作用**：协调单线程 JavaScript 的同步 / 异步任务执行顺序；
- **关键组成**：调用栈（同步）、微任务队列（高优先级）、宏任务队列（低优先级）；
- **执行顺序**：同步任务 → 所有微任务 → 一个宏任务 → 所有微任务 → 循环；
- **async/await**：`await` 后代码是微任务，遵循微任务优先规则。
