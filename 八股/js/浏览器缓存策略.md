浏览器缓存是浏览器对之前请求过的资源（HTML、CSS、JS、图片、接口数据、字体等），在本地内存 / 磁盘进行存储的机制。当再次请求相同资源时，浏览器会优先使用本地副本，无需重新从服务器下载，核心价值是**大幅提升页面加载速度、节省用户带宽、降低服务器压力**。

浏览器缓存策略整体分为**存储位置**、**核心缓存规则**、**完整执行流程**、**最佳实践**四大模块，其中核心规则分为**强缓存**和**协商缓存**两大类，是整个缓存机制的核心。

## 一、缓存的存储位置

浏览器会按优先级从高到低，依次查找以下缓存，命中则直接返回，全部未命中才会发起网络请求：

| 缓存类型                 | 存储位置 | 核心特点                             | 生命周期                          |
| -------------------- | ---- | -------------------------------- | ----------------------------- |
| Service Worker Cache | 磁盘   | 开发者完全可控的自定义缓存，支持离线缓存，是 PWA 的核心能力 | 永久存储，除非手动调用 API 删除或用户清除浏览器缓存  |
| Memory Cache         | 内存   | 读取速度最快，容量小，优先级最高                 | 会话级，与当前标签页绑定，标签页关闭后内存释放，缓存失效  |
| Disk Cache           | 硬盘   | 容量大、持久化存储，是浏览器最主要的缓存载体           | 长期存储，关闭浏览器 / 电脑后依然保留，过期后才会被清理 |
| Push Cache           | 内存   | HTTP/2 推送的专属缓存，兼容性有限             | 会话级，仅在当前会话生效，标签页关闭后失效，优先级最低   |

日常开发中最常接触的是 **Memory Cache** 和 **Disk Cache**：

- 页面当前正在使用的 JS、CSS、图片会优先存入 Memory Cache，刷新页面时优先从内存读取；
- 已关闭页面的资源、大体积资源会存入 Disk Cache，长期保留。

## 二、核心缓存策略

浏览器缓存分为两大核心体系：**强缓存（本地缓存）** 和 **协商缓存（对比缓存）**。

- 强缓存：命中后**完全不向服务器发送 HTTP 请求**，直接从本地读取资源，返回 `200 OK (from memory/disk cache)`；
- 协商缓存：强缓存失效后触发，浏览器会向服务器发送请求验证资源是否更新，未更新则返回 `304 Not Modified`，浏览器继续使用本地缓存；已更新则返回 `200 OK` 和新资源。

### 1. 强缓存（本地缓存）

强缓存是性能最优的缓存方案，核心逻辑是：浏览器请求资源时，先检查本地缓存是否存在，且缓存未过期，符合条件则直接读取本地资源，全程不与服务器通信。

强缓存通过两组 HTTP 响应头控制，**`Cache-Control`（HTTP/1.1）优先级高于 `Expires`（HTTP/1.0）**。

#### 1.1 Expires（HTTP/1.0 旧规范）

`Expires` 是 HTTP/1.0 的缓存字段，值为**服务器返回的绝对时间**，示例：

```http
Expires: Wed, 21 Oct 2026 07:28:00 GMT
```

浏览器会对比客户端本地时间和`Expires`的时间，本地时间早于过期时间则缓存有效。

**核心缺点**：依赖客户端本地时间，如果用户修改了电脑时间、客户端与服务器时间不一致，会导致缓存判断失效，目前已被`Cache-Control`替代。

#### 1.2 Cache-Control（HTTP/1.1 主流规范）

`Cache-Control` 是 HTTP/1.1 推出的缓存标准，解决了`Expires`的时间同步问题，支持多指令组合，是目前强缓存的核心控制字段。

常用核心指令：

| 指令             | 作用                                               | 适用场景                                 |
| -------------- | ------------------------------------------------ | ------------------------------------ |
| `max-age=xxx`  | 缓存有效时间，单位为秒，是**相对时间**（从资源请求成功的时刻开始计算），与客户端本地时间无关 | 所有静态资源，示例：`max-age=31536000`（缓存 1 年） |
| `public`       | 资源可被浏览器、CDN、代理服务器等所有中间节点缓存                       | 无用户信息的公共静态资源（图片、通用 JS/CSS）           |
| `private`      | 资源只能被浏览器（客户端）缓存，禁止 CDN / 代理服务器缓存                 | 含用户信息的私有资源（用户专属页面、个性化接口）             |
| `no-cache`     | 跳过强缓存，**强制走协商缓存**，每次请求都必须到服务器验证资源是否更新（注意：不是不缓存）  | HTML 入口文件、需要实时更新的资源                  |
| `no-store`     | 完全禁用所有缓存，每次请求都必须从服务器拉取完整资源                       | 敏感数据（支付接口、用户隐私数据）、实时性要求极高的资源         |
| `immutable`    | 缓存有效期内，即使用户强制刷新页面，也不会触发协商缓存，彻底禁止重新验证             | 带内容哈希的静态资源，避免无效的 304 请求              |
| `s-maxage=xxx` | 给 CDN / 代理服务器设置的缓存时间，优先级高于`max-age`，仅对共享缓存生效     | 配合 CDN 使用的静态资源                       |

示例：

```http
# 公共静态资源：缓存1年，所有节点可缓存
Cache-Control: public, max-age=31536000, immutable

# HTML入口：强制走协商缓存，不使用强缓存
Cache-Control: no-cache

# 敏感接口：完全不缓存
Cache-Control: no-store
```

#### 1. 核心区分：强缓存 vs 协商缓存

| 类型   | 是否发 HTTP 请求  | 状态码              | 核心逻辑                                       |
| :--- | :----------- | :--------------- | :----------------------------------------- |
| 强缓存  | ❌ 完全不发请求     | 200 (from cache) | 浏览器本地判断，全程不涉及服务器                           |
| 协商缓存 | ✅ 一定会发请求到服务器 | 304 Not Modified | 浏览器发请求带缓存标识，服务器判断缓存是否可用，可用则返回 304，浏览器本地读缓存 |

你看到的「发了请求，还是读了本地缓存」，其实是**协商缓存的 304 场景**，和强缓存完全是两回事。

#### 2. 哪些操作会强制跳过强缓存？

这些场景下，浏览器会直接忽略强缓存规则，强制发请求，所以你看不到 `from cache`：

1. **F5 刷新页面**：浏览器会强制跳过强缓存，直接发起带协商缓存标识的请求，走 304 校验；
2. **Ctrl+F5 / Cmd+Shift+R 强制刷新**：浏览器完全忽略所有缓存（强缓存 + 协商缓存），强制向服务器拉取全新的资源，请求头会带 `Cache-Control: no-cache` 和 `Pragma: no-cache`；
3. **开发者工具勾选了「Disable cache」**：Network 面板里的这个选项，会强制关闭所有缓存，每次都发请求；
4. **资源的强缓存规则配置错误**：比如 `Cache-Control: no-cache` / `no-store`，会直接禁用强缓存。

> 唯一能稳定复现强缓存的操作：**在地址栏输入 URL 回车，或通过 a 标签、页面跳转访问资源**，而不是刷新页面。

### 2. 协商缓存（对比缓存）

当强缓存失效（`max-age`过期、设置了`no-cache`）时，浏览器会触发协商缓存：向服务器发送带**缓存标识**的 HTTP 请求，服务器通过标识判断资源是否有更新，决定是返回 304 还是新资源。

协商缓存通过两组 HTTP 头控制，**`ETag/If-None-Match`（HTTP/1.1）优先级高于 `Last-Modified/If-Modified-Since`（HTTP/1.0）**。

#### 2.1 Last-Modified / If-Modified-Since（HTTP/1.0）

基于资源的**最后修改时间**实现，流程如下：

1. 浏览器第一次请求资源，服务器返回资源的同时，在响应头添加`Last-Modified`，值为资源在服务器上的最后修改时间，示例：
    
```http
    Last-Modified: Wed, 21 Oct 2025 07:28:00 GMT
```
    
2. 浏览器缓存资源和该时间，下次请求同一资源时，在请求头添加`If-Modified-Since`，值为缓存的`Last-Modified`；
3. 服务器收到请求后，对比资源当前的最后修改时间和`If-Modified-Since`的值：
    
    - 时间一致：资源无更新，返回`304 Not Modified`，不返回资源体，浏览器直接使用本地缓存；
    - 时间不一致：资源已更新，返回`200 OK`、新资源、新的`Last-Modified`，浏览器更新本地缓存。
**核心缺点**：

1. 时间精度仅到秒级，资源 1 秒内多次修改无法识别，会导致缓存失效；
2. 资源内容未变化，仅重新保存文件，修改时间会更新，导致不必要的缓存失效；
3. 分布式服务器的文件修改时间不一致，会导致缓存判断异常。

#### 2.2 ETag / If-None-Match（HTTP/1.1）

基于资源**内容唯一标识**实现，解决了`Last-Modified`的所有缺陷，是目前协商缓存的主流方案。

ETag 是服务器根据资源内容生成的唯一字符串（通常是内容哈希值，如 MD5），**只要资源内容发生变化，ETag 就会改变**，与修改时间无关。流程如下：

1. 浏览器第一次请求资源，服务器返回资源的同时，在响应头添加`ETag`，示例：
    
```http
    ETag: "abc123xyz789"
```
    
2. 浏览器缓存资源和该 ETag，下次请求同一资源时，在请求头添加`If-None-Match`，值为缓存的 ETag；
3. 服务器收到请求后，对比资源当前的 ETag 和`If-None-Match`的值：
    
    - 一致：资源无更新，返回`304 Not Modified`，浏览器使用本地缓存；
    - 不一致：资源已更新，返回`200 OK`、新资源、新的 ETag，浏览器更新本地缓存。
    
**核心优势**：

- 精准匹配资源内容，解决了秒级修改、内容不变修改时间变化的问题；
- 不受分布式服务器时间差异的影响，判断更可靠。

## 三、浏览器缓存完整执行流程

浏览器每次请求资源时，会严格按照以下流程执行：

1. **查找内存缓存**：优先检查 Memory Cache，命中且有效则直接返回`200 OK (from memory cache)`，流程结束；
2. **查找磁盘缓存**：内存缓存未命中，检查 Disk Cache，判断强缓存是否有效：
    
    - 强缓存有效（`max-age`未过期）：直接返回`200 OK (from disk cache)`，流程结束；
    - 强缓存无效（过期 / 设置了`no-cache`）：进入协商缓存流程；
    
3. **发起协商缓存请求**：浏览器向服务器发送请求，携带缓存标识`If-None-Match`/`If-Modified-Since`；
4. **服务器校验**：
    
    - 资源无更新：返回`304 Not Modified`，浏览器更新缓存新鲜度，读取本地缓存，流程结束；
    - 资源已更新：返回`200 OK`、新资源、新的缓存头，浏览器将新资源存入缓存，流程结束；
    
5. **无缓存可用**：本地无任何缓存，直接向服务器发起完整请求，获取资源后，根据响应头的缓存规则存入缓存。

## 四、前端开发最佳实践

### 1. 静态资源（JS、CSS、图片、字体、视频）

- 核心方案：**强缓存 + 内容哈希命名**
- 配置：`Cache-Control: public, max-age=31536000, immutable`
- 说明：给静态资源文件名添加内容哈希（如`app.8a2b3c.js`），只要资源内容变化，文件名就会改变，自动触发新请求；内容不变则文件名不变，一直命中强缓存，是性能最优的方案。

### 2. HTML 入口文件

- 核心方案：**协商缓存，禁用强缓存**
- 配置：`Cache-Control: no-cache`
- 说明：HTML 是页面的入口文件，不能设置长强缓存，否则页面更新后用户无法获取最新的 HTML，强制走协商缓存，保证每次更新都能生效。

### 3. 接口数据

- 公共无差别接口（如首页商品列表、公共配置）：设置短时长强缓存，`Cache-Control: public, max-age=60`（缓存 1 分钟），减少重复请求；
- 用户私有接口（如个人信息、订单列表）：`Cache-Control: private, no-cache`，走协商缓存，禁止代理服务器缓存；
- 敏感 / 实时接口（如支付、实时行情）：`Cache-Control: no-store`，完全禁用缓存，保证数据安全和实时性。

### 4. CDN 缓存优化

- 使用`s-maxage`单独设置 CDN 缓存时间，优先级高于`max-age`，实现 CDN 和浏览器缓存的分离控制；
- 静态资源 CDN 缓存时间可设置更长，配合缓存刷新功能，更新时主动刷新 CDN 缓存。

## 五、常见误区避坑

1. **`no-cache` 不是不缓存**：`no-cache`只是跳过强缓存，强制走协商缓存，资源依然会被浏览器缓存；真正完全不缓存的是`no-store`。
2. **强缓存的优先级**：`Cache-Control` 优先级高于 `Expires`，同时设置时只有`Cache-Control`生效。
3. **协商缓存的优先级**：`ETag/If-None-Match` 优先级高于 `Last-Modified/If-Modified-Since`，同时设置时服务器优先校验 ETag。
4. **304 请求的特点**：304 响应只有响应头，没有响应体，传输体积极小，仅用于通知浏览器使用本地缓存。
5. **用户强制刷新的影响**：用户按下`Ctrl+F5`强制刷新时，浏览器会跳过所有强缓存，直接发起协商缓存请求；按下`Shift+F5`超强制刷新时，会跳过所有缓存，直接拉取全新资源。
