同源策略是浏览器最核心、最基础的**安全机制**，它的作用是**限制不同源的页面 / 脚本之间的交互**，防止恶意网站窃取用户的敏感数据（如 Cookie、LocalStorage、DOM 内容），是保护用户隐私和 Web 安全的基石。

## 一、什么是 “同源”？

同源的判断标准非常严格，**协议、域名、端口**三者必须完全一致，只要有一个不同，就属于 “跨源（跨域）”。

以 `http://www.example.com:80`（默认端口 80 可省略）为例，对比以下 URL：

|URL|是否同源|原因|
|---|---|---|
|`http://www.example.com/path`|✅ 是|协议、域名、端口一致|
|`https://www.example.com`|❌ 否|协议不同（http vs https）|
|`http://api.example.com`|❌ 否|域名不同（www vs api）|
|`http://example.com`|❌ 否|域名不同（二级域名不同）|
|`http://www.example.com:8080`|❌ 否|端口不同（80 vs 8080）|

## 二、同源策略限制了什么？

同源策略主要限制三个方面的跨源交互：

### 1. 限制存储数据的读取

不同源的页面无法互相读取对方的 `Cookie`、`LocalStorage`、`SessionStorage`、IndexedDB 等存储数据。

- 例如：`http://evil.com` 无法读取 `http://bank.com` 的登录态 Cookie，防止恶意网站冒充用户登录。

### 2. 限制 DOM 的访问和操作

不同源的页面无法通过 `iframe`、`window.open()` 等方式访问或操作对方的 DOM。

- 例如：`http://evil.com` 嵌入 `http://bank.com` 的 iframe 后，无法读取 iframe 内的输入框内容，防止窃取密码。

### 3. 限制 AJAX/Fetch 请求

不同源的 AJAX、Fetch 请求会被浏览器拦截（注意：是浏览器拦截，请求可能已经发到服务器，但响应被浏览器挡住了）。

- 例如：前端在 `http://localhost:3000` 直接请求 `http://api.example.com` 的接口，会报跨域错误。

## 三、同源策略的 “例外”

同源策略并非完全禁止所有跨源交互，有一些常见的 “例外” 场景：

- **嵌入资源**：`<script src="...">`、`<img src="...">`、`<link href="...">`、`<video>`、`<iframe>` 等标签可以跨源加载资源（这也是 JSONP 的原理基础）；
- **表单提交**：HTML 表单可以跨源提交（但响应无法被读取）；
- **WebSocket**：WebSocket 不受同源策略限制，只要服务器支持即可连接。

## 四、常见跨域解决方案（前端开发必备）

同源策略是为了安全，但前后端分离、多域名协作的场景下，我们需要合法地跨源交互，以下是最常用的解决方案：
### 1. CORS（Cross-Origin Resource Sharing，跨域资源共享）

**最主流、最推荐的方案**，是 W3C 标准，通过后端配置 HTTP 响应头，告诉浏览器允许跨源请求。

#### 核心原理

后端在响应头中添加 CORS 相关字段，浏览器识别后，允许跨源请求的响应被读取。

#### 关键响应头

|响应头|作用|示例|
|---|---|---|
|`Access-Control-Allow-Origin`|允许的源（`*` 表示所有源，或具体域名）|`Access-Control-Allow-Origin: http://localhost:3000`|
|`Access-Control-Allow-Methods`|允许的请求方法|`Access-Control-Allow-Methods: GET, POST, PUT, DELETE`|
|`Access-Control-Allow-Headers`|允许的请求头|`Access-Control-Allow-Headers: Content-Type, Authorization`|
|`Access-Control-Allow-Credentials`|是否允许携带 Cookie（设为 `true` 时，`Origin` 不能是 `*`）|`Access-Control-Allow-Credentials: true`|
#### 简单请求 vs 预检请求

- **简单请求**：满足 `GET`/`POST`/`HEAD` 方法，且请求头仅包含简单字段（如 `Content-Type: text/plain`/`application/x-www-form-urlencoded`），浏览器直接发送请求，后端返回 CORS 头即可；
- **预检请求**：不满足简单请求条件的（如 `PUT`/`DELETE`、自定义请求头、`Content-Type: application/json`），浏览器会先发送一个 `OPTIONS` 预检请求，后端确认允许后，才会发送真实请求。
> 补充： [预检请求](../vue/网络/预检请求.md) 

#### 前端操作

前端几乎不需要额外配置，正常发送请求即可；如果需要携带 Cookie，需设置 `credentials: 'include'`（Fetch）或 `withCredentials: true`（Axios）。

### 2. 开发环境代理（Webpack/Vite Proxy）

**开发阶段最常用的方案**，通过本地开发服务器将请求代理到后端，避免跨域。

#### 核心原理

前端请求本地开发服务器（同源），开发服务器转发请求到后端（服务器之间没有同源策略），再将响应返回给前端。

#### 配置示例

- **Vite**（`vite.config.js`）：
    
```javascript
    export default {
      server: {
        proxy: {
          '/api': {
            target: 'http://api.example.com', // 后端地址
            changeOrigin: true,
            rewrite: (path) => path.replace(/^\/api/, '')
          }
        }
      }
    }
```
    
- **Webpack**（`vue.config.js`/`webpack.config.js`）：
    
```javascript
    module.exports = {
      devServer: {
        proxy: {
          '/api': {
            target: 'http://api.example.com',
            changeOrigin: true,
            pathRewrite: { '^/api': '' }
          }
        }
      }
    }
```

### 3. Nginx 反向代理（生产环境常用）

**生产环境的主流方案**，通过 Nginx 作为反向代理服务器，让前端和后端看起来是同源的。

#### 核心原理

用户访问 Nginx 服务器，Nginx 将静态资源请求转发到前端服务，将接口请求转发到后端服务，浏览器只和 Nginx 交互，不存在跨域。

#### 配置示例

```nginx
server {
  listen 80;
  server_name www.example.com;

  # 前端静态资源
  location / {
    root /usr/share/nginx/html;
    index index.html;
  }

  # 接口代理
  location /api {
    proxy_pass http://api.example.com; # 后端地址
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
  }
}
```

### 5. postMessage（跨窗口 / 跨 iframe 通信）

**专门用于不同源页面之间的通信**，比如父页面和嵌入的 iframe 之间、`window.open()` 打开的子页面之间。

#### 核心语法

- **发送消息**：`targetWindow.postMessage(data, targetOrigin)`
    
    - `targetWindow`：目标窗口对象（如 `iframe.contentWindow`、`window.opener`）；
    - `data`：要发送的数据；
    - `targetOrigin`：允许的目标源（`*` 或具体域名，建议设具体域名）。
    
- **接收消息**：监听 `window` 的 `message` 事件
    
```javascript
    window.addEventListener('message', (event) => {
      if (event.origin !== 'http://allowed.com') return; // 验证源，安全第一
      console.log('收到消息:', event.data);
    });
```

## 五、总结

- **同源策略核心**：协议、域名、端口一致，限制存储、DOM、AJAX 的跨源交互，目的是保护安全；
- **跨域解决方案**：
    - 开发环境：用 Webpack/Vite 代理；
    - 生产环境：优先用 CORS 或 Nginx 反向代理；
    - 跨窗口通信：用 postMessage；
    - 老系统：JSONP（不推荐）。
