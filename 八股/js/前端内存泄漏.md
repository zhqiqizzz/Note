前端内存泄漏是指**页面中已不再需要的内存空间，没有被浏览器的垃圾回收（GC）机制正确释放**，导致内存占用持续上升的现象。长期内存泄漏会造成页面卡顿、响应变慢，甚至浏览器崩溃，是前端性能优化中需要重点关注的问题。

## 一、什么是内存泄漏？

要理解内存泄漏，先明确两个核心概念：

- **内存占用**：页面运行时需要占用一定内存存储变量、DOM、事件等，这是正常的；
- **内存泄漏**：当某个对象 / 内存块 “不再被需要”，但仍被其他对象引用，导致垃圾回收器无法回收它，内存占用持续累积。

简单判断：反复进入 / 离开某个页面 / 组件，内存曲线持续上升且不回落，大概率存在内存泄漏。

## 二、前端常见内存泄漏场景（附示例）

### 1. 意外的全局变量

JavaScript 中，未声明直接使用的变量会自动成为全局变量（非严格模式下），全局变量会一直存在于内存中，直到页面关闭。

**示例**：

```javascript
function createUser() {
  // 未声明，自动变成全局变量
  userName = 'Alice'; 
}
createUser();
// 即使函数执行完，userName 仍在全局作用域，无法回收
```

**另一种情况**：`this` 指向全局对象

```javascript
function setName() {
  // 非严格模式下，函数独立调用时 this 指向 window
  this.name = 'Bob';
}
setName();
// name 变成全局变量
```

**解决**：使用严格模式 `'use strict'`，强制变量先声明再使用；避免不必要的全局变量。

### 2. 未清理的定时器 / 回调

`setInterval`、`setTimeout` 是最常见的泄漏源：如果定时器回调中引用了外部变量，且定时器未被及时清除，这些变量会一直被引用，无法回收。

**示例**：

```javascript
// 组件初始化时设置定时器
const timer = setInterval(() => {
  const data = fetchData();
  console.log(data);
}, 1000);

// 组件销毁时，忘记清除定时器
// clearInterval(timer); // 这行没写，定时器会一直运行，data 等变量无法回收
```

**解决**：在组件 / 页面销毁时（如 React 的 `componentWillUnmount`、Vue 的 `beforeUnmount`），及时调用 `clearInterval` 或 `clearTimeout`。

### 3. 未解绑的事件监听器

使用 `addEventListener` 绑定事件后，如果组件销毁时未调用 `removeEventListener`，事件回调会一直保留在内存中，尤其是回调中引用了 DOM 元素时，会导致 DOM 和回调都无法回收。

**示例**：

```javascript
const button = document.getElementById('btn');
// 绑定事件
button.addEventListener('click', () => {
  console.log('按钮被点击');
});

// 组件销毁时，按钮从页面移除，但事件监听器未解绑
// button.removeEventListener('click', handler); // 这行没写，回调和 button 都无法回收
```

**解决**：

- 组件销毁时，用相同的函数引用解绑事件（注意：匿名函数无法解绑，需保存函数引用）；
- 优先使用事件委托（将事件绑定到父元素），减少监听器数量。

### 4. 闭包引用导致的泄漏

闭包会保留外部函数的作用域链，如果闭包被长期引用（如赋值给全局变量、定时器回调），外部函数的变量即使不再需要，也会被闭包 “锁住”，无法回收。

**示例**：

```javascript
let globalHandler;

function init() {
  const largeData = new Array(1000000).fill('数据'); // 大数组
  // 闭包引用了 largeData
  globalHandler = function() {
    console.log(largeData.length);
  };
}
init();
// 即使 init 执行完，largeData 仍被 globalHandler 引用，无法回收
```

**解决**：

- 避免不必要的闭包；
- 闭包不再需要时，将引用它的变量设为 `null`：`globalHandler = null;`。

### 5. DOM 元素引用未释放

如果用变量长期保存 DOM 元素的引用，即使该 DOM 已从页面中移除，变量仍会 “抓住” 它，导致 DOM 无法被垃圾回收。

**示例**：

```javascript
// 保存 DOM 引用
const domRef = document.getElementById('box');

// 后来 DOM 从页面移除
document.body.removeChild(document.getElementById('box'));

// 但 domRef 还在，DOM 元素仍在内存中，无法回收
```

**解决**：DOM 移除后，将引用变量设为 `null`：`domRef = null;`。

### 6. 未清理的第三方库 / 插件

很多第三方库（如图表库 ECharts、地图库、富文本编辑器）内部会创建大量内存对象，如果组件销毁时未调用库提供的 `destroy`/`cleanup` 方法，库内部的内存会泄漏。

**示例**：

```javascript
// 初始化 ECharts 图表
const chart = echarts.init(document.getElementById('chart'));
chart.setOption({...});

// 组件销毁时，忘记调用销毁方法
// chart.dispose(); // 这行没写，图表内部的内存无法释放
```

**解决**：仔细阅读第三方库文档，在组件销毁时调用对应的清理方法。

### 7. 控制台日志的 “隐性” 泄漏

`console.log`、`console.error` 等控制台方法会保留对输出对象的引用（浏览器为了让你在控制台能查看对象详情），如果频繁输出大对象，且生产环境未移除日志，会导致内存泄漏。

**解决**：生产环境通过构建工具（如 Webpack、Vite）自动移除 `console` 语句，或仅输出必要的简单类型数据。

## 三、如何检测内存泄漏？

Chrome DevTools 的 **Memory 面板**是检测内存泄漏的核心工具，常用以下三种方式：

### 1. 堆快照（Heap Snapshot）

- **操作步骤**：
    
    1. 打开 DevTools → Memory → 选择 “Heap snapshot”；
    2. 点击 “Take snapshot” 拍第一次快照（基准）；
    3. 执行可能导致泄漏的操作（如反复进入 / 离开组件）；
    4. 拍第二次快照，选择 “Comparison” 对比两次快照；
    
- **判断依据**：对比 “Retained Size”（对象占用的总内存），如果某个对象的数量和内存持续增长，大概率存在泄漏。

### 2. 分配时间线（Allocation Timeline）

- **操作步骤**：
    
    1. 选择 “Allocation timeline”；
    2. 点击 “Record” 开始记录，执行可疑操作；
    3. 点击 “Stop” 停止；
    
- **判断依据**：观察内存分配的柱状图，如果有持续的蓝色（新分配内存）且不回落，说明存在泄漏。

### 3. Performance 面板辅助

- 打开 DevTools → Performance，勾选 “Memory”；
- 点击 “Record”，反复执行操作后停止；
- 观察内存曲线：如果曲线持续上升且无明显回落，提示可能存在泄漏。

## 四、如何预防内存泄漏？

1. **养成良好的编码习惯**：
    
    - 使用严格模式 `'use strict'`，避免意外全局变量；
    - 变量 “即用即丢”，不再需要时设为 `null`；
    - 避免不必要的闭包和全局变量。
    
2. **及时清理 “副作用”**：
    
    - 组件销毁时，清理定时器、事件监听器；
    - 调用第三方库的销毁方法；
    - 释放 DOM 引用。
    
3. **使用弱引用数据结构**：
    
    - 优先用 `WeakMap`、`WeakSet` 存储对象引用：它们的引用是 “弱引用”，不会阻止垃圾回收，适合临时存储对象。
    
4. **定期检测**：
    
    - 开发阶段定期用 DevTools Memory 面板检查内存；
    - 关注线上监控的内存指标（如通过 Performance API 上报）。
    
## 总结

前端内存泄漏的核心原因是 **“不再需要的对象仍被引用”**，重点关注定时器、事件监听器、闭包、第三方库这几个高频场景，养成 “创建 - 清理” 成对的编码习惯，配合 DevTools 定期检测，就能有效避免内存泄漏，提升页面稳定性。
