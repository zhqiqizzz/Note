### 1. 核心概念：为什么图片是性能优化的“头号敌人”？

在网页资源中，图片通常占据 **60% ~ 80%** 的流量体积。  
如果不做懒加载，当你打开一个包含 100 张图的长页面时：

1. **带宽拥堵**：浏览器会并发发起 100 个 HTTP 请求。根据 HTTP/2 或 HTTP/3 的机制，虽然多路复用解决了队头阻塞，但服务器的处理能力和客户端的解码能力是有限的。
2. **主线程阻塞**：图片下载完成后，浏览器需要进行**解码（Decode）**和**绘制（Paint）**。大量图片同时解码会占用大量 CPU 和内存，导致主线程卡顿，用户滑动页面时会出现掉帧（Jank）。
3. **无效流量**：统计数据显示，用户在长页面中平均只浏览 **20% ~ 30%** 的内容。剩下的 70% 图片流量被白白浪费。

**图片懒加载的目标**：  
将“一次性全部加载”改为 **“按需分批加载”**。只有当图片**即将进入**可视区域时，才发起网络请求。

### 2. 两种主流实现方案深度对比

目前主要有两种方案：**原生 HTML 方案** 和 **JavaScript IntersectionObserver 方案**。

#### 方案 A：原生 HTML `loading="lazy"`

这是 HTML5 标准的一部分，现代浏览器（Chrome 51+, Firefox 55+, Safari 14+, Edge 79+）已全面支持。

**代码写法：**

```html
<img 
  src="real-image.jpg" 
  loading="lazy" 
  width="800" 
  height="600" 
  alt="描述"
>
```

**优点：**

- **零 JavaScript 开销**：不需要写任何 JS 代码，没有事件监听，没有回调函数，性能损耗最低。
- **浏览器底层优化**：浏览器内核可以直接调度网络请求优先级，甚至在图片进入视口前就预加载（Preload），比 JS 判断更精准。
- **自动降级**：在不支持的老浏览器中，它会忽略该属性，直接加载图片（表现为无懒加载，但功能正常，不会报错）。

**局限性与坑点：**

1. **阈值不可控**：浏览器内部决定何时加载（通常是距离视口 1200px 左右），开发者无法自定义“提前 200px 加载”。在网络极慢的环境下，用户可能会看到图片加载过程中的空白闪烁。
2. **不支持背景图**：`loading="lazy"` 仅对 `<img>` 和 `<iframe>` 标签有效。对于 CSS `background-image` 无效。
3. **首屏误伤风险**：如果不小心给**首屏可见**的图片加了 `loading="lazy"`，浏览器可能会错误地延迟加载它们，导致 LCP（最大内容绘制）指标变差，影响 SEO 和用户体验。
    - _修正策略_：首屏图片（LCP 元素）**不要**加 `loading="lazy"`，或者使用 `loading="eager"`。

**最佳适用场景：**

- 常规的图文列表、文章详情页。
- 不需要复杂加载动画（如骨架屏、渐进式模糊）的场景。
- 追求极致性能和开发效率的项目。

#### 方案 B：JavaScript `IntersectionObserver`

这是目前手动实现懒加载的**行业标准 API**。它取代了老旧的 `scroll` 事件监听方案。

**核心原理：**  
创建一个“观察器”，告诉浏览器：“帮我盯着这些图片，一旦它们和视口（Viewport）的交叉面积超过阈值，就通知我。”

**完整代码实现：**
- **`options = {}`**：允许你在创建时传入自定义设置。比如你想让图片提前 500px 加载，就可以传 `{ rootMargin: '500px' }`。如果不传，就用默认值。
- **`this.rootMargin = '200px 0px'`**：
    - **比喻**：就像在屏幕上下各扩展了 200px 的“缓冲区”。
    - **作用**：图片还没完全进入屏幕，只要离屏幕还有 200px 远，就开始加载。这样用户滑到时，图片已经好了，不会看到白块。
- **`new IntersectionObserver(...)`**：
	- **这是谁？**：这是浏览器自带的一个**超级保安**。
	- **它的工作**：你告诉它：“帮我盯着页面上所有的图片。一旦有哪张图片进入了我的视线（或者接近视线），你就立刻喊我（触发回调函数）。”
- `threshold: 0.01`：表示只要图片露出 **1%** (0.01) 的面积，就认为它“出现了”，触发加载。
- `root: null`：表示相对于**浏览器窗口**（视口）来判断。

```javascript
class ImageLazyLoader {
  constructor(options = {}) {
    // 配置项
    this.rootMargin = options.rootMargin || '200px 0px'; // 提前 200px 加载
    this.threshold = options.threshold || 0.01;          // 露出 1% 即触发
    this.placeholderClass = options.placeholderClass || 'lazy-placeholder';
    
    // 创建观察器
    this.observer = new IntersectionObserver(this.handleIntersect.bind(this), {
      root: null, // 相对于视口
      rootMargin: this.rootMargin,
      threshold: this.threshold
    });
    
    this.init();
  }

  init() {
    // 选中所有带有 data-src 的图片
    const images = document.querySelectorAll('img[data-src]');
    images.forEach(img => this.observer.observe(img));
  }

  handleIntersect(entries, observer) {
    entries.forEach(entry => {
    // entry.isIntersecting 意思是：这张图片是否进入了视野（或缓冲区）？
      if (entry.isIntersecting) {
        const img = entry.target; // 拿到那张具体的图片元素
        
        // 1. 开始加载真实图片
        this.loadImage(img);
        
        // 2. 停止观察该图片（性能关键：避免重复触发）
        observer.unobserve(img);
      }
    });
  }

  loadImage(img) {
    const realSrc = img.dataset.src;
    const realSrcSet = img.dataset.srcset;

    // 如果已经在加载或加载过，跳过
    if (!realSrc || img.src === realSrc) return;

    // 创建临时 Image 对象预加载，避免直接在 DOM 上替换导致闪烁
    const tempImg = new Image();
    
    tempImg.onload = () => {
      // 加载成功，替换 src
      img.src = realSrc;
      // 如果有高清多套图 (srcset)，也一并赋值
      if (realSrcSet) img.srcset = realSrcSet;
      
      // 加个 class，方便 CSS 做淡入动画
      img.classList.add('loaded');
      // 把 data-src 删掉，标记为已处理
      img.removeAttribute('data-src');
      img.removeAttribute('data-srcset');
    };
    
	// 3. 定义“加载失败”后做什么
    tempImg.onerror = () => {
      // 加载失败，设置错误占位图
      img.src = '/images/error-placeholder.png';
      img.classList.add('error');
    };

    // 触发网络请求
    tempImg.src = realSrc;
    if (realSrcSet) tempImg.srcset = realSrcSet;
  }
  
  // 销毁方法（组件卸载时调用）
  destroy() {
    this.observer.disconnect();
  }
}

// 初始化
new ImageLazyLoader({ rootMargin: '300px 0px' });
```

- **`document.querySelectorAll('img[data-src]')`**：
    - **动作**：在页面上搜索所有 `<img>` 标签，且这些标签必须写着 `data-src="..."`。
    - **为什么是 `data-src`？**：因为真正的图片地址我们先藏在这里，`src` 里放的是占位图。如果直接写在 `src` 里，浏览器一看到就会立刻下载，懒加载就失效了。

- **为什么要用 `tempImg = new Image()`？**
    - **技巧**：这叫**预加载**。
    - **原因**：如果你直接把 `img.src = realSrc`，页面上的图片会瞬间变成空白（因为旧图没了，新图还没下好），用户体验很差。
    - **做法**：我们在内存里先悄悄下载 (`tempImg`)。等内存里的图**完全下好**了 (`onload`)，再“啪”地一下替换到页面上。这样用户看到的就是从“占位图”平滑过渡到“清晰图”，没有闪烁。
- **`img.dataset.src`**：
    - 在 JS 里，HTML 标签上的 `data-xxx` 属性，都会自动变成 `dataset.xxx` 来访问。
    - 例如：`<img data-src="a.jpg">` -> JS 里就是 `img.dataset.src` (值为 "a.jpg")。
    - 如果 `src` 为空，图片标签的高度可能会坍塌（变成0），导致页面布局错乱，文字突然跳动（CLS 问题）。
	- 所以最佳实践是：`src` 放一个**极小的占位图**（比如 1x1 像素的透明图，或者一个 Base64 的灰色块）。这样既骗过了浏览器（只下载极小文件），又撑住了布局位置。

**对应的 HTML 结构：**

```html
<!-- 
  1. src 放占位图（或透明 GIF）
  2. 真实地址放在 data-src
  3. 必须指定 width/height 防止布局抖动 (CLS)
-->
<img 
  src="data:image/svg+xml,%3Csvg xmlns='http://www.w3.org/2000/svg' viewBox='0 0 800 600'%3E%3Crect fill='%23f0f0f0' width='800' height='600'/%3E%3C/svg%3E"
  data-src="https://example.com/heavy-image.jpg"
  width="800" 
  height="600"
  alt="商品图"
  class="lazy-image"
/>
```

1. **`src="..."`**：这里放的是一个极小的、Base64 编码的灰色方块（或者一张模糊的小图）。
    - **目的**：占住位置，防止页面抖动，且几乎不耗流量。
2. **`data-src="..."`**：这里藏着**真正的大图地址**。
    - **目的**：骗过浏览器。浏览器看到 `src` 里有东西，就只加载那个小方块；它看不懂 `data-src`，所以不会主动请求大图。

**优点：**

- **完全可控**：可以自定义 `rootMargin`（提前加载距离），适应不同网速环境。
- **支持任意元素**：不仅可以处理 `<img>`，还可以处理 `div` 的背景图、视频 Poster、甚至动态组件的加载。
- **丰富的交互**：可以轻松实现**加载前占位**、**加载中骨架屏**、**加载失败重试**、**淡入动画**等高级效果。
- **性能优异**：异步回调，不阻塞主线程，且 `unobserve` 机制确保内存泄漏风险极低。

**缺点：**

- 需要编写和维护 JS 代码。
- 在极老浏览器（如 IE）中需要 Polyfill（但在已基本无需考虑 IE）。
### 3. 关键细节与避坑指南（专家级建议）

无论用哪种方案，以下细节决定了懒加载的成败：

#### 1. 防止布局抖动 (CLS - Cumulative Layout Shift)

这是懒加载最容易犯的错误。

- **现象**：图片没加载时高度为 0，加载完后突然撑开，导致下方的文字突然跳动。这会严重影响用户体验，并被 Google 搜索降权。
- **解决**：
    - **硬编码尺寸**：始终给 `<img>` 标签写上 `width` 和 `height` 属性。浏览器会根据这两个属性预留空间，即使图片还没加载。
    - **Aspect Ratio**：使用 CSS `aspect-ratio: 16/9;` 保持宽高比。
    - **占位容器**：用一个固定高度的 `div` 包裹图片，背景色设为灰色。

#### 2. 首屏图片不要懒加载

- **LCP 是什么**：首屏最大内容（通常是图或标题）渲染完成的时间。
- **注意**：LCP **只关注首屏**（用户不用滚动就能看到的区域）。如果你有一张巨大的图片在页面最底部，它不会被算作 LCP。
- **原则**：**LCP (Largest Contentful Paint)** 元素（通常是首屏最大的那张图或 Hero Banner）**必须直接加载**。
- **原因**：懒加载会增加一个“检测 -> 触发 -> 请求”的过程，哪怕只有 100ms 的延迟，也会显著拖慢首屏渲染速度。如果 LCP 元素（首屏大图）也被懒加载了，浏览器会故意推迟它的加载，导致 LCP 时间大幅增加（可能从 1.5s 变成 3.0s）。
- **做法**：
    - 首屏图片：`<img src="..." loading="eager">` (或者直接不加 loading 属性)。
    - 第二屏及以后：`<img src="..." loading="lazy">`。

#### 3. 背景图的懒加载

CSS 背景图无法使用 `loading="lazy"`。必须使用 JS 方案：

- **HTML**：`<div class="lazy-bg" data-bg="url(image.jpg)"></div>`
- **JS**：当观察到该 div 进入视口时，执行 `element.style.backgroundImage = element.dataset.bg`。

#### 4. 响应式图片 (`srcset`) 的处理

如果使用 `<img srcset="...">` 进行响应式加载：

- **原生方案**：直接使用 `loading="lazy"`，浏览器会自动处理 `srcset` 的懒加载。
- **JS 方案**：需要将真实地址存在 `data-srcset` 中，加载时将 `data-srcset` 的值赋给 `srcset` 属性，并移除 `data-` 前缀。

#### 5. 无限列表中的内存管理

在“无限滚动”场景中，DOM 节点会不断增加。

- **必须**：在图片加载完成后调用 `observer.unobserve(img)`。
- **必须**：当旧节点被移除（虚拟列表回收）时，确保断开观察，否则会导致内存泄漏。

### 4. 进阶体验优化技巧

为了让懒加载看起来更“高级”，可以配合以下技术：

1. **低质量占位图 (LQIP - Low Quality Image Placeholders)**：
    
    - 先加载一张极小（如 1KB）的模糊缩略图作为 `src`。
    - 当真实大图加载完成后，通过 CSS `transition: filter 0.5s` 实现从模糊到清晰的平滑过渡。
    - _效果_：用户永远不会看到白块，而是先看到模糊的轮廓，体验极佳。
2. **骨架屏 (Skeleton Screen)**：
    
    - 在图片加载前，显示一个带有 shimmer（闪光）动画的灰色块。
    - 这比单纯的留白更能安抚用户的等待焦虑。
3. **并发控制**：
    
    - 如果用户快速滚动，瞬间触发了 50 张图片的加载，可能会瞬时占满带宽。
    - 可以在 JS 层做一个简单的队列，限制同时进行的图片请求数量（例如最多同时加载 6 张），其他的排队等待。

### 总结对比表

|特性|原生`loading="lazy"`|JS`IntersectionObserver`|
|---|---|---|
|**开发成本**|极低 (HTML 属性)|中 (需写 JS 逻辑)|
|**性能开销**|最低 (浏览器内核级)|低 (异步 API)|
|**自定义阈值**|❌ 不可控 (浏览器决定)|✅ 完全可控 (`rootMargin`)|
|**背景图支持**|❌ 不支持|✅ 支持|
|**加载动画**|❌ 难实现 (需 CSS Hack)|✅ 容易实现 (淡入/LQIP)|
|**首屏优化**|需注意区分 (设 eager)|可精确控制 (排除首屏)|
|**兼容性**|现代浏览器 (2020+)|现代浏览器 (需 Polyfill 兼容旧版)|
|**推荐指数**|⭐⭐⭐⭐ (常规场景首选)|⭐⭐⭐⭐⭐ (复杂/高要求场景)|

**最终建议：**  
**90% 的场景直接使用 `<img loading="lazy">`**，简单高效。  
只有在需要**背景图懒加载**、**复杂的加载动画**、**极度精细的预加载控制**或**兼容特殊业务逻辑**时，才动用 `IntersectionObserver` 手写方案。
