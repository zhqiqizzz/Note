判断页面卡顿是**脚本执行慢**、**渲染耗时**还是**回流重绘过多**，核心工具是 **Performance 面板**（精确量化占比）+ **Rendering 面板**（直观定位问题区域）。下面从**录制准备→宏观占比→微观细节→问题定位**四个步骤，手把手教你精准排查。

## 一、第一步：正确录制性能数据
### 1. 环境准备（必做，否则数据完全不准）

- 关闭所有其他标签页和浏览器扩展（扩展会注入大量 JS，干扰结果）
- 使用**无痕模式**打开页面（彻底隔离扩展和缓存）
- 开启**CPU 节流**：模拟普通用户设备（Performance 面板右上角 → CPU → 4x slowdown）
- 关闭 DevTools 的其他面板（只保留 Performance）

### 2. 录制步骤

1. 按 `F12` 打开 DevTools，切换到 **Performance** 面板
2. 勾选面板顶部的 **Screenshots**（截图）和 **Memory**（内存）选项
3. 点击左上角的**录制按钮（●）**，或按 `Ctrl+E`
4. **复现卡顿操作**（如滚动页面、点击按钮、加载列表）
5. 点击**停止按钮（■）**，等待 Chrome 分析生成报告

## 二、第二步：宏观判断：哪个环节占比最高

录制完成后，先看两个地方快速判断瓶颈类型：

### 1. 底部 Summary 面板（最直接的占比统计）

这是 Chrome 自动统计的**总耗时占比**，一眼就能看出哪个环节是瓶颈：

| 颜色    | 类别        | 包含内容                                         | 占比高说明什么                        |
| ----- | --------- | -------------------------------------------- | ------------------------------ |
| 🟡 黄色 | Scripting | JS 解析、执行、事件处理、定时器回调                          | **脚本执行太慢**，有长任务、复杂计算或频繁 DOM 操作 |
| 🟣 紫色 | Rendering | 样式计算（Recalculate Style）+ **回流（Layout / 重排）** | **渲染耗时**，尤其是回流过多               |
| 🟢 绿色 | Painting  | 重绘（填充像素、绘制文本 / 颜色 / 阴影）                      | **重绘过多**，频繁修改颜色、背景等外观属性        |
| 🔵 蓝色 | Loading   | 网络请求、HTML/CSS 解析                             | 资源加载慢，和前端代码关系不大                |
| ⚪ 白色  | Idle      | 浏览器空闲时间                                      | 性能良好                           |

**快速判断**：

- 黄色占比 > 30% → 脚本执行是瓶颈
- 紫色占比 > 20% → 回流 / 样式计算是瓶颈
- 绿色占比 > 20% → 重绘是瓶颈

### 2. 顶部 CPU 总览图（看时间分布）

CPU 总览图的颜色和 Summary 完全对应，横向是时间轴，纵向是 CPU 占用率。

- 如果**大片黄色** → 全程 JS 执行繁忙
- 如果**大片紫色** → 全程渲染繁忙
- 如果**有很多细碎的紫色 / 绿色条** → 频繁触发回流重绘

## 三、第三步：微观定位：具体是哪个操作导致的

宏观判断出瓶颈类型后，深入**Main 线程火焰图**（Performance 面板中间的 Main 轨道），定位具体的代码和操作。

### 1. 怎么看脚本执行（黄色块）

- 所有**黄色的长条块**都是 JS 执行任务
- **宽度超过 50ms 的黄色块**就是**长任务（Long Task）**，是导致页面卡顿的直接原因
- 点击黄色块，下方会显示**Call Stack（调用栈）**，直接定位到具体的函数和代码行号

**常见脚本瓶颈**：

- 大数组循环、递归计算
- 未节流 / 防抖的事件处理（如 scroll、resize）
- 频繁的 DOM 操作（每次操作都可能触发回流）
- 第三方脚本（统计、广告、SDK）执行缓慢

### 2. 怎么看回流（Layout / 重排，紫色块）

回流是最昂贵的渲染操作，也是最常见的性能杀手。在 Main 线程中：

- 看到 **Layout** 字样的紫色块 → 就是回流
- 看到 **Forced Synchronous Layout**（强制同步布局）→ 最严重的回流问题，性能极差
- 看到 **Recalculate Style** 紫色块 → 重新计算样式，通常会紧跟一个 Layout

**怎么看 "哪个 JS 导致了回流"**：

1. 点击紫色的 Layout 块
2. 查看右侧的**Call Stack**
3. 调用栈中最上面的 JS 函数，就是触发回流的元凶

**典型强制同步布局代码**：

```javascript
// ❌ 错误：在循环中交叉读写DOM属性，每次读取都会强制触发回流
for (let i = 0; i < 100; i++) {
  const height = element.offsetHeight; // 读取布局属性 → 强制回流
  element.style.top = `${height + 10}px`; // 修改样式
}
```

### 3. 怎么看重绘（Paint，绿色块）

- 看到 **Paint** 字样的绿色块 → 就是重绘
- 绿色块越宽、越多 → 重绘越频繁
- 点击 Paint 块，右侧会显示**绘制的区域和耗时**

## 四、第四步：直观验证：Rendering 面板

Performance 面板是量化分析，**Rendering 面板**是直观可视化，能让你直接看到页面上哪些区域在频繁回流重绘。

### 打开方式

DevTools 右上角三点 → **More tools** → **Rendering**，或按 `Ctrl+Shift+P` 输入 `Show Rendering`。

### 核心调试选项

|选项|作用|怎么看问题|
|---|---|---|
|**Paint flashing**|重绘区域会**绿色高亮闪烁**|如果整个页面都在闪，或某个区域频繁闪烁 → 重绘过多|
|**Layout Shift Regions**|回流区域会**蓝色高亮闪烁**|如果滚动或交互时出现大片蓝色 → 回流过多|
|**FPS Meter**|实时显示帧率和 GPU 内存|帧率持续低于 60fps → 有性能问题|
|**Scrolling Performance Issues**|高亮滚动性能差的区域|红色区域 → 滚动时会触发大量回流重绘|

**使用技巧**：

1. 勾选 **Paint flashing** 和 **Layout Shift Regions**
2. 操作页面（滚动、点击）
3. 观察页面上的闪烁区域，直接定位到有问题的元素

## 五、常见问题排查流程

### 1. 脚本执行占比高

1. 在 Main 线程中找到**宽度超过 50ms 的黄色长任务**
2. 点击长任务，查看 Call Stack 定位具体函数
3. 优化方案：
    
    - 拆分长任务为多个小任务（使用`setTimeout`或`requestIdleCallback`）
    - 将复杂计算移到 Web Worker
    - 给 scroll、resize 等事件添加节流 / 防抖
    - 延迟加载非关键脚本

### 2. 回流（Layout）占比高

1. 在 Main 线程中找到**紫色的 Layout 块**，特别是标有`Forced Synchronous Layout`的块
2. 查看 Call Stack 找到触发回流的 JS 代码
3. 优化方案：
    
    - 批量修改 DOM 样式，避免逐条修改
    - 读写分离：先读取所有布局属性，再统一修改
    - 使用`transform`和`opacity`做动画（不会触发回流重绘）
    - 给频繁变化的元素添加`will-change: transform`，提升到独立合成层

### 3. 重绘（Paint）占比高

1. 开启 Rendering 面板的**Paint flashing**，找到频繁闪烁的区域
2. 检查该区域的 CSS 属性
3. 优化方案：
    
    - 避免频繁修改`color`、`background`、`box-shadow`等属性
    - 将频繁重绘的元素提升到独立合成层
    - 减少复杂的 CSS 效果（如渐变、阴影、模糊）

## 六、关键误区澄清

1. ❌ **Rendering 就是回流**
    
    纠正：Rendering 包含 ** 样式计算（Recalculate Style）**和**回流（Layout）** 两部分，回流只是其中最耗时的部分。
    
2. ❌ **重绘一定比回流快**
    
    纠正：大面积重绘（如整个页面重绘）的耗时可能比小范围回流更长。
    
3. ❌ **没有长任务就不会卡顿**
    
    纠正：多个短任务连续执行，总阻塞时间（TBT）过长也会导致卡顿，需要看 Summary 的总 Scripting 时间。
    
4. ❌ **合成层越多越好**
    
    纠正：每个合成层都会占用 GPU 内存，过度使用会导致内存占用过高，反而影响性能。

## 总结：一分钟快速排查流程

1. 打开 Performance 面板，录制卡顿操作
2. 看底部 Summary：黄色多→脚本问题，紫色多→回流问题，绿色多→重绘问题
3. 深入 Main 线程：黄色长任务→找 JS 函数，紫色 Layout 块→找触发回流的代码
4. 打开 Rendering 面板：勾选 Paint flashing 和 Layout Shift Regions，直观看到问题区域
5. 根据问题类型应用对应的优化方案
