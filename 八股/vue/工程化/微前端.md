**一、什么是微前端**

先看传统单页应用的问题：

```
一个仓库
一个项目
一个构建
一个部署
一个团队维护
```

当项目变大后，会遇到这些痛点：

```
代码多，启动慢
多人协作冲突多
技术栈难以升级
模块耦合严重
一个小改动也要全量构建部署
新人上手成本高
```

微前端的核心思想是：

```
把一个大应用拆成多个小应用
每个小应用独立开发、独立部署
运行时动态组合成一个完整应用
```

类似后端的微服务。

举例：

```
公司后台系统包含：
- 用户管理
- 订单管理
- 商品管理
- 统计分析
- 系统设置

传统方案：一个大 Vue 项目

微前端方案：
- 主应用：负责路由、菜单、权限
- 用户子应用：Vue3
- 订单子应用：Vue2
- 商品子应用：React
- 统计子应用：独立技术栈
```

每个子应用可以独立开发、独立上线，主应用负责加载和组合。

---

**二、为什么需要微前端**

不是所有项目都需要微前端。

适合微前端的场景：

```
多团队协作，每个团队负责独立业务模块
历史遗留系统和新系统共存
不同模块技术栈不同
需要独立部署
业务边界清晰
```

不适合微前端的场景：

```
项目很小
单一团队
技术栈统一
业务耦合严重
```

微前端不是银弹，它用**复杂度换灵活性**。

---

**三、微前端要解决的核心问题**

微前端本质是解决这几个问题：

**1. 应用加载**

```
主应用怎么加载子应用？
子应用 JS/CSS 怎么引入？
加载失败怎么办？
```

**2. 应用隔离**

```
JS 全局变量冲突怎么办？
CSS 样式冲突怎么办？
多个 Vue 实例冲突怎么办？
```

**3. 路由管理**

```
主应用和子应用路由怎么配合？
切换子应用时路由怎么同步？
浏览器前进后退怎么处理？
```

**4. 应用通信**

```
主应用怎么传数据给子应用？
子应用怎么通知主应用？
子应用之间怎么通信？
```

**5. 公共依赖**

```
Vue、React 要每个子应用都打包吗？
公共组件库怎么共享？
工具函数怎么共享？
```

**6. 性能**

```
子应用加载是否阻塞？
重复加载怎么避免？
首屏性能怎么优化？
```

所有微前端方案都是围绕这些问题设计的。

**四、微前端实现方案**

常见方案有 5 种：

**1. iframe**

最简单，但也最受限。

```
<iframe src="https://sub-app.com" />
```

优点：

```
完全隔离
实现简单
```

缺点：

```
路由不同步
性能差
UI 不连贯
通信复杂
SEO 不友好
```

适合：

```
第三方应用嵌入
不需要深度集成
```

**2. 路由分发**

Nginx 根据路由转发：

```
location /user/ {
  proxy_pass http://user-app:3000;
}

location /order/ {
  proxy_pass http://order-app:4000;
}
```

优点：

```
完全独立
部署简单
```

缺点：

```
页面刷新
不是单页应用
体验割裂
```

适合：

```
独立站点集合
不追求 SPA 体验
```

**3. Web Components**

把子应用封装成自定义元素：

```
<user-app></user-app>
<order-app></order-app>
```

优点：

```
标准化
天然隔离
```

缺点：

```
兼容性
生态不成熟
框架支持有限
```

适合：

```
组件共享
标准化场景
```

**4. JavaScript 沙箱 + 动态加载**

主流方案，包括 qiankun、micro-app。

核心思路：

```
主应用动态加载子应用 JS
使用沙箱隔离 JS 环境
劫持路由和生命周期
```

优点：

```
SPA 体验
技术栈无关
灵活
```

缺点：

```
实现复杂
需要改造子应用
有一定侵入性
```

适合：

```
大部分微前端场景
```

**5. Module Federation (Webpack 5)**

构建时共享模块。

```
new ModuleFederationPlugin({
  name: 'app1',
  remotes: {
    app2: 'app2@http://localhost:3002/remoteEntry.js'
  }
})
```

优点：

```
依赖共享
构建优化
```

缺点：

```
强依赖 Webpack 5
配置复杂
版本管理难
```

适合：

```
同技术栈
强依赖共享需求
```

**五、主流方案：qiankun**

qiankun 是阿里开源的微前端框架，基于 single-spa 封装。

它是目前国内最流行的微前端方案。

核心理念：

```
主应用注册子应用
主应用控制子应用加载/卸载
子应用暴露生命周期
沙箱隔离 JS
样式隔离
```

---

**六、qiankun 实现原理**

我拆开讲核心原理。

**1. 子应用加载流程**

```
主应用注册子应用配置（name、entry、container）
路由匹配到子应用
fetch 请求子应用 HTML
解析 HTML，提取 JS/CSS 链接
加载 JS，执行
调用子应用 mount 生命周期
渲染子应用到指定容器
```

**2. JS 沙箱隔离**

qiankun 有 3 种沙箱：

**LegacySandbox（已废弃）**

```
快照模式
mount 前记录 window 快照
unmount 时恢复快照
```

问题：

```
多个子应用同时运行会冲突
```

**ProxySandbox（推荐）**

```
使用 Proxy 代理 window
子应用修改只影响自己的 fakeWindow
不影响全局 window
支持多实例
```

简化原理：

```js
class ProxySandbox {
  constructor() {
    this.fakeWindow = {}
    
    this.proxy = new Proxy(window, {
      get(target, prop) {
        // 优先从 fakeWindow 取
        if (prop in fakeWindow) {
          return fakeWindow[prop]
        }
        
        return target[prop]
      },
      
      set(target, prop, value) {
        // 写入 fakeWindow
        fakeWindow[prop] = value
        return true
      }
    })
  }
  
  active() {
    // 激活沙箱
  }
  
  inactive() {
    // 失活沙箱
  }
}
```

子应用 JS 在这个 Proxy 环境执行，就隔离了。

**SnapshotSandbox（兼容模式）**

```
不支持 Proxy 时的降级方案
记录和恢复 window 属性
```

**3. CSS 隔离**

两种方式：

**strictStyleIsolation**

```
子应用容器外包一层 shadow DOM
完全隔离
```

问题：

```
某些 UI 库弹窗会有问题
```

**experimentalStyleIsolation**

```
给子应用 CSS 选择器加前缀
```

例如：

```
.button {
  color: red;
}
```

变成：

```
div[data-qiankun="app1"] .button {
  color: red;
}
```

**4. 路由劫持**

qiankun 劫持 `pushState`、`replaceState` 和 `popstate`：

```
const rawPushState = window.history.pushState
window.history.pushState = function(...args) {
  // 触发子应用切换逻辑
  rawPushState.apply(this, args)
}
```

这样主应用能感知路由变化，动态加载/卸载子应用。

**5. 生命周期**

子应用必须暴露这些钩子：

```js
export async function bootstrap() {
  // 首次加载时调用，只调用一次
}

export async function mount(props) {
  // 每次进入时调用
  // 渲染子应用
}

export async function unmount(props) {
  // 每次离开时调用
  // 卸载子应用
}
```

qiankun 在合适时机调用这些钩子。

**七、实际代码示例**

我用 Vue3 主应用 + Vue3 子应用示例。

**主应用**

安装：

```
pnpm add qiankun
```

注册子应用：

```js
// main.ts
import { registerMicroApps, start } from 'qiankun'

registerMicroApps([
  {
    name: 'user-app',
    entry: '//localhost:3001',
    container: '#subapp-container',
    activeRule: '/user'
  },
  {
    name: 'order-app',
    entry: '//localhost:3002',
    container: '#subapp-container',
    activeRule: '/order'
  }
])

start()
```

主应用布局：

```vue
<template>
  <div class="app">
    <header>
      <nav>
        <router-link to="/user">用户管理</router-link>
        <router-link to="/order">订单管理</router-link>
      </nav>
    </header>
    
    <main>
      <router-view />
      <!-- 子应用渲染容器 -->
      <div id="subapp-container"></div>
    </main>
  </div>
</template>
```

**子应用改造**

子应用需要改造，暴露生命周期。

`main.ts`：

```js
import { createApp } from 'vue'
import { createRouter, createWebHistory } from 'vue-router'
import App from './App.vue'

let app: any = null

function render(props: any = {}) {
  const { container } = props
  
  const router = createRouter({
    history: createWebHistory(
      // 子应用路由基础路径
      window.__POWERED_BY_QIANKUN__ ? '/user' : '/'
    ),
    routes: [
      // 子应用路由
    ]
  })
  
  app = createApp(App)
  app.use(router)
  
  const containerEl = container
    ? container.querySelector('#app')
    : document.querySelector('#app')
  
  app.mount(containerEl)
}

// 独立运行时
if (!window.__POWERED_BY_QIANKUN__) {
  render()
}

// qiankun 生命周期
export async function bootstrap() {
  console.log('子应用 bootstrap')
}

export async function mount(props: any) {
  console.log('子应用 mount', props)
  render(props)
}

export async function unmount() {
  console.log('子应用 unmount')
  app?.unmount()
  app = null
}
```

子应用 `vite.config.ts`：

```js
import { defineConfig } from 'vite'
import vue from '@vitejs/plugin-vue'

export default defineConfig({
  plugins: [vue()],
  server: {
    port: 3001,
    cors: true,
    origin: 'http://localhost:3001'
  },
  base: window.__POWERED_BY_QIANKUN__ ? '/user' : '/'
})
```

注意：

```
子应用要支持跨域
路由要用 history 模式
打包要用 library 格式或配置好入口
```

**子应用打包配置**

Vite 打包要改：

```
build: {
  lib: {
    entry: './src/main.ts',
    formats: ['umd'],
    name: 'userApp',
    fileName: 'user-app'
  }
}
```

或者用 `vite-plugin-qiankun` 插件简化配置。

**八、主子应用通信**

这是 qiankun 内置的全局状态方案，本质是发布订阅。

**主应用初始化：**

```js
import { initGlobalState } from 'qiankun'

// 初始化全局状态
const actions = initGlobalState({
  token: '',
  userInfo: null,
  language: 'zh'
})

// 监听变化
actions.onGlobalStateChange((state, prevState) => {
  console.log('主应用监听变化', state, prevState)
})

// 修改状态
actions.setGlobalState({
  token: 'abc123',
  userInfo: {
    id: 1,
    name: 'Tom'
  }
})

// 暴露给外部使用
export { actions }
```

**子应用接收：**

```js
// 子应用存储 actions
let globalActions: any = null

export async function mount(props: any) {
  // qiankun 会把 onGlobalStateChange 和 setGlobalState 注入 props
  const { onGlobalStateChange, setGlobalState } = props

  globalActions = { onGlobalStateChange, setGlobalState }

  // 监听主应用状态变化
  onGlobalStateChange((state, prevState) => {
    console.log('子应用监听变化', state)

    // 更新子应用自己的 store
    const userStore = useUserStore()
    userStore.setToken(state.token)
    userStore.setUserInfo(state.userInfo)
  }, true) // true 表示立即执行一次

  render(props)
}
```

**子应用修改全局状态：**

```js
// 子应用通知主应用
globalActions.setGlobalState({
  subAppReady: true,
  activeModule: 'user'
})
```

局限性：

```
只能合并更新，不能精确监听某个字段
子应用卸载后要注意取消监听
状态越来越多会变复杂
```

**九、常见问题和坑**

**1. 子应用静态资源 404**

原因：

```
子应用打包时用了相对路径
```

解决：

```
// vite.config.ts
base: window.__POWERED_BY_QIANKUN__ ? '/user/' : '/'
```

或者用 CDN 绝对路径。

**2. CSS 样式冲突**

解决：

```
开启 qiankun 的 CSS 隔离
子应用使用 scoped 或 CSS Modules
使用 CSS-in-JS
```

**3. 子应用白屏**

原因：

```
生命周期没正确导出
CORS 问题
路由 base 不对
打包格式不对
```

检查：

```
浏览器 Network 看资源是否加载成功
Console 看报错
检查 window.__POWERED_BY_QIANKUN__ 是否正确
```

**4. 子应用跳转主应用路由**

子应用不要直接用 `router.push('/other')`，而是：

```js
import { getMainApp } from '@/utils/qiankun'

// 通知主应用跳转
getMainApp().router.push('/other')
```

或者通过 props 传入主应用路由实例。

**5. 子应用刷新后 404**

需要主应用和子应用都配置好 history 路由 fallback。

Nginx：

```
location / {
  try_files $uri $uri/ /index.html;
}
```

**6. Vue3 + Vite 子应用加载问题**

Vite 默认用 ES Module，qiankun 需要 UMD。

解决：

```
使用 vite-plugin-qiankun
或者用 library 模式打包
```

**十、micro-app 对比**

除了 qiankun，还有京东开源的 micro-app。

它用 Web Components 思路：

```
<micro-app name="user-app" url="http://localhost:3001"></micro-app>
```

优点：

```
更接近原生
配置更简单
子应用改造更少
```

缺点：

```
社区相对小
文档相对少
```

两者都可以用，qiankun 更成熟，micro-app 更轻量。

---

**十一、Module Federation 对比**

Webpack 5 的 Module Federation 不是传统意义的微前端，它是**构建时模块共享**。

适合：

```
同技术栈
强依赖共享
多个独立应用复用代码
```

qiankun 是运行时加载，MF 是构建时联邦。
