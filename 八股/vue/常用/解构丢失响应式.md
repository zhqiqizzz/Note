在 Vue 里说“**解构丢失响应式**”，核心意思是：

> 原本 Vue 能追踪的数据，被你用普通 JS 解构成了一个普通变量，Vue 就没法继续知道它和原对象之间的关系了。

**一句话总结**

普通解构：

```js
const { count } = state
```

拿到的是“当前值”，不是“响应式连接”。

正确做法是：

```js
const { count } = toRefs(state)
```

或者 Pinia 里：

```js
const { count } = storeToRefs(store)
```

记住这个判断就行：

> 只要你从 `reactive` / `props` / `store` 里直接解构出属性，就要警惕响应式丢失。直接用 `对象.属性`，或者用 `toRefs` / `storeToRefs`。

先看一个最典型的例子。

```js
import { reactive } from 'vue'

const state = reactive({
  count: 0,
  name: 'Tom',
})

const { count } = state

state.count++

console.log(count) // 还是 0
console.log(state.count) // 1
```

这里 `count` 丢失了响应式。

原因是：

```js
const { count } = state
```

等价于：

```js
const count = state.count
```

也就是说，你只是把当时的值 `0` 取出来，赋给了一个新的普通变量 `count`。后面 `state.count` 再变，普通变量 `count` 不会自动跟着变。

---

**Vue 响应式的原理**

Vue 3 的响应式主要基于 `Proxy`。

比如：

```js
const state = reactive({ count: 0 })
```

Vue 会把原对象包一层代理，大概可以理解成：

```js
const state = new Proxy(obj, {
  get(target, key) {
    // 收集依赖
    return target[key]
  },
  set(target, key, value) {
    // 触发更新
    target[key] = value
  },
})
```

当你访问：

```js
state.count
```

Vue 能拦截这次 `get`，知道某个页面或计算属性依赖了 `count`。

当你修改：

```js
state.count++
```

Vue 能拦截这次 `set`，通知相关页面更新。

但是当你这样写：

```js
const { count } = state
```

后面的 `count` 已经不是 `state.count` 的代理访问了，而只是一个普通数字。

之后模板或函数里再访问：

```js
count
```

不会触发 `state` 这个 Proxy 的 `get`，所以 Vue 追踪不到。

---

**哪些情况会丢响应式**

**1. 解构 `reactive` 对象**

```js
const state = reactive({
  count: 0,
  user: 'Alice',
})

const { count, user } = state
```

这里 `count`、`user` 都会变成普通变量。

错误用法：

```js
const { count } = state

const double = computed(() => count * 2)
```

如果 `state.count` 变化，`double` 不会更新。

正确写法：

```js
const double = computed(() => state.count * 2)
```

或者：

```js
import { toRefs } from 'vue'

const { count, user } = toRefs(state)
```

这时 `count` 是一个 `ref`。

使用时：

```js
count.value++
```

模板里可以直接用：

```
{{ count }}
```

---

**2. 解构 Pinia store**

Pinia store 本质上也是响应式对象，所以普通解构也容易出问题。

错误写法：

```js
const userStore = useUserStore()

const { user, token, isAuthenticated } = userStore
```

这样可能导致 `user`、`token`、`isAuthenticated` 响应式丢失。

正确写法：

```js
import { storeToRefs } from 'pinia'

const userStore = useUserStore()

const { user, token, isAuthenticated } = storeToRefs(userStore)
```

如果是 action，正常解构可以：

```js
const { login, logout } = userStore
```

因为 action 是函数，不需要响应式。

常见组合是：

```js
const userStore = useUserStore()

const { user, token } = storeToRefs(userStore)
const { login, logout } = userStore
```

---

**3. 解构 props**

在 Vue 组件里，`props` 也是响应式的。

比如：

```js
const props = defineProps<{
  title: string
}>()

const { title } = props
```

在很多情况下，这种普通解构会让 `title` 脱离 `props.title` 的响应式追踪。

更稳的写法是：

```js
const props = defineProps<{
  title: string
}>()

const title = computed(() => props.title)
```

或者：

```js
import { toRefs } from 'vue'

const props = defineProps<{
  title: string
}>()

const { title } = toRefs(props)
```

---

**为什么 `ref` 不一样**

`ref` 是另一种响应式形式。

```js
const count = ref(0)
```

它的真实结构类似：

```js
{
  value: 0
}
```

Vue 追踪的是 `.value`。

所以如果你直接这样用：

```js
const count = ref(0)

count.value++
```

响应式不会丢。

但是如果你解构一个包含 ref 的对象，也要看情况：

```js
const obj = {
  count: ref(0),
}

const { count } = obj
```

这里 `count` 仍然是 ref，所以没丢。

但是如果是：

```js
const state = reactive({
  count: 0,
})

const { count } = state
```

这里拿出来的是普通数字，所以丢了。

---

**toRefs 是干什么的**

`toRefs` 的作用是把 `reactive` 对象的每个属性转换成 `ref`，并且这个 `ref` 仍然和原对象保持连接。

```js
const state = reactive({
  count: 0,
  name: 'Tom',
})

const { count, name } = toRefs(state)

count.value++

console.log(state.count) // 1
```

可以理解为：

```
count.value <==> state.count
name.value <==> state.name
```

它们不是复制值，而是建立了一个响应式引用。

---

**storeToRefs 和 toRefs 的区别**

`toRefs` 是 Vue 提供的，适合普通 `reactive` 对象。

```js
const { count } = toRefs(state)
```

`storeToRefs` 是 Pinia 提供的，专门适合 Pinia store。

```js
const { user, token } = storeToRefs(userStore)
```

为什么 Pinia 要单独提供一个？

因为 Pinia store 里面除了状态，还有：

```
state
getter
action
方法
内部属性
```

`storeToRefs` 会帮你只把适合响应式解构的状态和 getter 转成 ref，跳过 action 这类函数。

所以在 Pinia 里优先用：

```js
storeToRefs(store)
```

---

**什么时候不会丢**

如果你不解构，直接用对象属性，不会丢：

```js
const state = reactive({ count: 0 })

computed(() => state.count)
```

如果你解构的是函数，也没所谓：

```js
const { login } = userStore
```

如果你解构前用了 `toRefs` 或 `storeToRefs`，不会丢：

```js
const { count } = toRefs(state)
const { user } = storeToRefs(userStore)
```
