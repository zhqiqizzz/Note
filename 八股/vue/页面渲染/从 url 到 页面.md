整个过程是**浏览器多进程协作 + 网络层通信 + 渲染引擎流水线执行**的完整链路，核心可拆分为 **7 个环环相扣的大阶段**，同时覆盖前端性能优化的核心原理，全程兼顾底层逻辑与前端开发的核心关注点。

## 前置基础：现代浏览器的多进程架构

Chrome 等现代浏览器采用**多进程架构**，整个流程的核心协作进程如下，是理解全流程的前提：

|进程类型|核心职责|
|---|---|
|浏览器主进程|控制浏览器 UI（地址栏、前进后退、书签）、管理子进程生命周期、文件存储|
|网络进程|全局唯一，负责所有 HTTP/HTTPS 网络请求、DNS 解析、缓存处理|
|渲染进程|每个标签页 / 站点一个，沙箱隔离，核心负责 HTML 解析、CSS 解析、JS 执行、页面渲染、事件处理|
|GPU 进程|全局唯一，负责 GPU 硬件加速渲染、图层合成|

整个流程的核心协作链路：**主进程接收 URL → 交给网络进程处理网络通信 → 网络进程拿到响应数据 → 交给渲染进程执行渲染 → GPU 进程完成最终屏幕绘制**。

## 阶段一：URL 输入与预处理

1. **输入内容判断**
    
    用户在地址栏输入内容后，浏览器会先判断输入是「搜索关键词」还是「合法 URL」：
    
    - 若为搜索词：自动拼接默认搜索引擎，生成搜索 URL；
    - 若为合法 URL（如`https://www.baidu.com`）：自动补全协议（默认补`http://`），生成完整 URL。
    
2. **URL 结构解析**
    
    浏览器拆解完整 URL 的核心组成部分，为后续请求做准备：
    
```plaintext
    https://www.example.com:443/user/list?id=1#name
    协议(https)  域名(www.example.com) 端口(443) 路径(/user/list) 查询参数(?id=1) 哈希(#name)
```
    
3. **安全与合规校验**
    
    - 浏览器安全机制检测 URL 是否为恶意域名、钓鱼地址，若是则弹出安全警告；
    - 检查该域名是否有 HSTS（HTTP 严格传输安全）记录，若有则强制使用 HTTPS 发起请求，拒绝 HTTP 明文请求。
    
4. **进程交接**
    
    主进程将完整 URL 交给**网络进程**，正式进入网络请求阶段。

## 阶段二：浏览器缓存检查（性能优化核心）

网络进程拿到 URL 后，**不会立即发起网络请求**，而是先按优先级检查各级缓存，判断是否可以直接使用本地缓存资源，这是前端首屏性能优化的核心环节。

缓存检查遵循「**强缓存优先，协商缓存兜底**」的规则：

1. **第一步：强缓存检查**
    
    强缓存的核心是：不发起任何 HTTP 请求，直接使用本地缓存，性能最高。
    
    - 浏览器先读取请求头的`Cache-Control`（HTTP/1.1），优先判断`max-age`是否过期；无`Cache-Control`则读取`Expires`（HTTP/1.0）的绝对过期时间；
    - 若缓存未过期：直接从本地缓存读取资源，返回给渲染进程，流程直接跳转到渲染阶段，全程无网络请求；
    - 若缓存已过期：进入协商缓存检查。
    
2. **第二步：协商缓存检查**
    
    协商缓存的核心是：发起轻量 HTTP 请求，和服务器确认缓存是否可用，可用则继续使用本地缓存，无需返回完整资源。
    
    - 浏览器携带缓存标识发起请求：
        
        - 上次响应头有`Last-Modified`：请求头携带`If-Modified-Since`（上次的修改时间）；
        - 上次响应头有`ETag`：请求头携带`If-None-Match`（上次的文件唯一标识）；
        
    - 服务器校验后，若资源未修改：返回`304 Not Modified`，浏览器继续使用本地缓存；
    - 若资源已修改：返回`200 OK`，携带全新的资源内容和缓存规则，浏览器更新本地缓存。
    
3. **缓存优先级**
    
    内存缓存（Memory Cache）> 磁盘缓存（Disk Cache）> Service Worker Cache > Push Cache。

> 若缓存完全失效 / 无缓存，才会进入后续的 DNS 解析、TCP 连接环节，发起完整的网络请求。

## 阶段三：DNS 域名解析

核心目标：把 URL 中的域名（如`www.example.com`）翻译成服务器的**IP 地址**，因为 TCP 连接只能通过 IP 地址建立，无法直接通过域名通信。

DNS 解析是**递归查询 + 迭代查询**结合的过程，按优先级逐级查询，全程有缓存机制减少重复解析：

1. **浏览器 DNS 缓存**：检查浏览器本地是否有该域名的解析记录，有则直接返回；
2. **系统 DNS 缓存**：查询操作系统的 DNS 缓存（Windows 的 hosts 文件、Linux 的 /etc/hosts），有则直接返回；
3. **本地 DNS 服务器**：向运营商 / 自定义的本地 DNS 服务器（LDNS）发起查询，LDNS 有缓存则直接返回；
4. **根 DNS 服务器**：LDNS 无缓存时，向全球 13 台根 DNS 服务器发起查询，根服务器返回顶级域名服务器（.com/.cn 等）的地址；
5. **顶级域名服务器**：LDNS 向顶级域名服务器查询，返回该域名对应的**权威 DNS 服务器**地址；
6. **权威 DNS 服务器**：LDNS 向权威 DNS 服务器查询，拿到域名对应的最终服务器 IP 地址，返回给浏览器；
7. **缓存更新**：浏览器和系统将解析结果缓存下来，TTL（生存时间）内无需重复解析。

> 前端优化点：通过`<link rel="dns-prefetch" href="//example.com">`提前解析后续会用到的域名，减少请求时的 DNS 耗时。

## 阶段四：TCP 连接与 TLS 加密握手

拿到服务器 IP 地址后，网络进程会和服务器建立网络连接，根据协议不同分为两种场景：

- HTTP 协议：仅需**TCP 三次握手**建立连接；
- HTTPS 协议：TCP 三次握手后，额外执行**TLS/SSL 四次握手**，完成加密通道建立。

### 1. TCP 三次握手（建立可靠连接）

TCP 是面向连接的、可靠的传输层协议，三次握手的核心是**确认双方的发送、接收能力正常，同步初始化序列号，避免历史报文干扰**：

1. 第一次握手：浏览器（客户端）向服务器发送`SYN`报文，携带客户端初始化序列号`ISN(c)`，客户端进入`SYN_SENT`状态；
2. 第二次握手：服务器收到`SYN`报文后，返回`SYN+ACK`报文，确认客户端的序列号（`ACK=ISN(c)+1`），同时携带服务器初始化序列号`ISN(s)`，服务器进入`SYN_RCVD`状态；
3. 第三次握手：客户端收到`SYN+ACK`报文后，返回`ACK`报文，确认服务器的序列号（`ACK=ISN(s)+1`），客户端和服务器均进入`ESTABLISHED`状态，连接正式建立。

> 补充：HTTP/1.1 默认开启`Connection: keep-alive`长连接，同一个域名的多个请求可复用同一个 TCP 连接，避免重复三次握手的开销；HTTP/2 支持多路复用，同一个 TCP 连接可并行传输多个请求 / 响应，彻底解决队头阻塞问题。

### 2. TLS/SSL 四次握手（HTTPS 专属，加密通道建立）

HTTPS = HTTP + TLS/SSL，在 TCP 连接建立后，通过 TLS 握手完成身份认证和加密密钥协商，全程保证数据传输的机密性、完整性、身份真实性：

1. 第一次握手：客户端向服务器发送`Client Hello`报文，携带客户端支持的 TLS 版本、加密套件列表、随机数`Random1`；
2. 第二次握手：服务器返回`Server Hello`报文，确认使用的 TLS 版本和加密套件，携带随机数`Random2`；同时发送服务器的 SSL 数字证书、服务器公钥；
3. 第三次握手：客户端验证证书的合法性（由 CA 机构签发，防止伪造），验证通过后生成预主密钥`Pre-Master Secret`，用服务器公钥加密后发送给服务器；
4. 第四次握手：服务器用自己的私钥解密，拿到预主密钥；双方通过`Random1 + Random2 + Pre-Master Secret`生成相同的**会话对称密钥**；双方互相发送`Finished`报文，验证加密通道正常，握手完成。

> 后续的 HTTP 请求 / 响应，全部用会话对称密钥加密传输，兼顾加密安全性和传输性能。

## 阶段五：HTTP 请求发送与响应处理

连接建立完成后，网络进程正式发起 HTTP 请求，接收并处理服务器响应：

1. **构建 HTTP 请求报文**
    
    浏览器构建标准的 HTTP 请求报文，包含三部分：
    
    - 请求行：请求方法（GET/POST 等）、请求路径、HTTP 版本；
    - 请求头：携带`Host`、`User-Agent`、`Accept`、`Cookie`、`Authorization`等请求上下文；
    - 请求体：POST/PUT 等请求携带的业务数据（JSON、FormData 等）。
    
2. **发送请求与接收响应**
    
    请求报文通过 TCP 连接发送到服务器，服务器处理请求后，返回 HTTP 响应报文，同样包含三部分：
    
    - 状态行：HTTP 版本、状态码（200/301/304/404/500 等）、状态描述；
    - 响应头：携带`Content-Type`、`Content-Length`、`Cache-Control`、`Set-Cookie`等响应规则；
    - 响应体：核心返回内容，HTML 页面的响应体就是 HTML 源码，图片 / JS/CSS 则是对应资源的二进制 / 文本内容。
    
3. **响应结果处理**
    
    - 若状态码为**重定向（301/302/307/308）**：读取响应头的`Location`字段，拿到新的 URL，回到「阶段二缓存检查」重新执行全流程；
    - 若状态码为**错误码（4xx/5xx）**：浏览器根据错误类型展示错误页面，流程终止；
    - 若状态码为**200 成功**：根据`Content-Type`判断资源类型，HTML 文档则交给渲染进程，其他资源（JS/CSS/ 图片）则交给对应模块处理，同时更新缓存。

## 阶段六：渲染进程页面渲染（前端核心环节）

渲染进程拿到 HTML 文档后，开启**渲染流水线**，将 HTML/CSS/JS 代码转换成屏幕上的像素页面，核心分为 6 个核心步骤，也是前端渲染性能优化的核心依据。

### 前置准备：创建渲染上下文

渲染进程会为当前页面创建**渲染主线程**、**合成线程**、**多个工作线程**，同时创建 JS 引擎（V8）、渲染引擎（Blink）的执行上下文，所有核心解析、渲染、JS 执行都在渲染主线程完成。

#### 步骤 1：解析 HTML，构建 DOM 树

渲染主线程读取 HTML 字节流，按顺序解析生成 DOM 树（文档对象模型）：

1. 字节解码：将 HTML 二进制字节流，按编码格式（UTF-8）解码为字符；
2. 词法分析：将字符拆解为令牌（Token），区分标签、属性、文本内容；
3. 语法分析：将令牌转换成对应的 DOM 节点；
4. 树结构生成：根据节点的嵌套关系，生成完整的 DOM 树。

**关键阻塞规则**：

- 遇到`<script>`标签（无 defer/async）时，会**暂停 DOM 解析**，先下载并执行 JS 代码，因为 JS 可以修改 DOM 结构，必须保证执行顺序；
- 遇到`<img>`、`<video>`等资源，会发起异步下载，不阻塞 DOM 解析。

#### 步骤 2：解析 CSS，构建 CSSOM 树

渲染主线程解析 CSS 代码，生成 CSSOM 树（CSS 对象模型），流程和 DOM 树构建一致：

1. 解析所有 CSS 来源：行内样式、`<style>`内联样式、`<link>`外部样式表、浏览器默认样式；
2. 处理样式的层叠、继承规则，计算每个节点的最终样式；
3. 生成完整的 CSSOM 树，和 DOM 树一一对应，包含每个节点的最终样式信息。

**关键阻塞规则**：

- CSS 不会阻塞 DOM 解析，但会**阻塞页面渲染**，浏览器不会渲染没有 CSSOM 的页面，避免无样式内容闪烁（FOUC）；
- CSS 会阻塞后续`<script>`标签的执行，因为 JS 可能会读取、修改元素的样式，必须等待 CSSOM 构建完成。

#### 步骤 3：合并生成渲染树（Render Tree）

DOM 树和 CSSOM 树构建完成后，渲染主线程将二者合并，生成**渲染树（Render Tree）**：

- 渲染树只包含**页面可见的节点**；
- `display: none`的节点不会进入渲染树；`visibility: hidden`、`opacity: 0`的节点会进入渲染树（占据页面空间，只是不可见）；
- 排除`<head>`、`<script>`等无视觉输出的标签。

#### 步骤 4：布局（Layout / 重排）

渲染主线程根据渲染树，计算每个节点的**精确几何位置和大小**，这个过程就是布局（也叫重排）：

- 浏览器采用流式布局模型，从上到下、从左到右计算节点的坐标、宽高、边距；
- 父子节点互相影响，比如父节点宽度变化，会触发子节点的位置重新计算；
- 布局完成后，会生成每个节点的布局信息，保存在布局树中。

#### 步骤 5：绘制（Paint / 重绘）

渲染主线程将布局后的节点，拆分成多个绘制指令，生成**绘制记录**，这个过程就是绘制（也叫重绘）：

- 绘制的内容包括：颜色、背景、边框、阴影、文字、图片等视觉元素；
- 浏览器会按图层拆分绘制任务，比如普通文档流一个图层，定位元素、transform 动画、will-change 提升的元素会单独生成图层，避免局部修改影响整个页面；
- 绘制完成后，每个图层都会生成对应的绘制列表，交给合成线程。

#### 步骤 6：合成（Composite）与屏幕显示

合成线程接收绘制列表，将每个图层的绘制指令转换成位图（栅格化），再将所有图层合并成一个完整的页面，交给 GPU 进程渲染到屏幕上：

1. 栅格化：合成线程将图层分块，交给 GPU 进程完成栅格化，转换成像素位图；
2. 图层合并：将所有图层的位图合并，生成最终的页面画面；
3. 上屏显示：将最终画面提交给 GPU，显示到屏幕上，用户看到完整的页面。

**关键性能知识点**：

- `transform`、`opacity`的修改，只会触发合成阶段，不会触发重排和重绘，性能最高；
- 样式修改只影响视觉效果（如颜色、背景色），会触发重绘，不会触发重排；
- 布局相关的样式修改（如宽高、位置、display），会触发重排 → 重绘 → 合成全流程，性能损耗最大。

## 阶段七：页面加载完成与后续交互

### 1. 关键加载事件触发

渲染过程中会按顺序触发核心生命周期事件，前端开发可通过这些事件执行对应逻辑：

- `DOMContentLoaded`：DOM 树构建完成触发，无需等待图片、字体等非阻塞资源加载完成；
- `load`：页面所有资源（HTML、JS、CSS、图片、视频等）全部加载完成触发；
- `beforeunload`：页面即将关闭 / 跳转前触发，可用于提示用户保存未提交内容；
- `unload`：页面卸载时触发，可用于清理数据、上报日志。

### 2. 后续资源加载

- 页面渲染完成后，继续加载懒加载的图片、视频、异步 JS 等资源；
- 异步 JS（`async`/`defer`）下载完成后执行，`defer`会在`DOMContentLoaded`之前按顺序执行，`async`下载完成后立即执行，阻塞渲染。

### 3. 用户交互响应

页面渲染完成后，渲染主线程进入**事件循环机制**，监听用户的点击、输入、滚动等交互事件，触发对应的 JS 事件回调，执行对应的逻辑，如需修改页面样式 / 结构，会重新触发重排 / 重绘 / 合成流程，更新页面。

## 常见误区纠正

1. **误区**：CSS 会阻塞 DOM 解析
    
    纠正：CSS 不会阻塞 DOM 树的解析，只会阻塞页面渲染和后续 JS 执行，DOM 解析和 CSS 解析是并行执行的。
2. **误区**：三次握手只是为了确认双方收发能力
    
    纠正：核心除了确认收发能力，还要同步初始化序列号，避免网络中历史失效的报文干扰当前连接。
3. **误区**：重绘一定会触发重排
    
    纠正：重排一定会触发重绘，重绘不一定触发重排，仅修改颜色、阴影等不影响布局的样式，只会触发重绘。
4. **误区**：HTTPS 的 TLS 握手和 TCP 三次握手同时执行
    
    纠正：TLS 握手必须在 TCP 三次握手完成、连接建立后才能执行。
5. **误区**：`async`和`defer`的执行时机一致
    
    纠正：`defer`会在 DOM 解析完成后、`DOMContentLoaded`之前按顺序执行；`async`下载完成后立即执行，会阻塞渲染，不保证执行顺序。
