**SSR (Server-Side Rendering，服务端渲染)** 和 **SSG (Static Site Generation，静态站点生成)** 是现代前端框架（如 Next.js, Nuxt）中两种核心的渲染模式。

它们的核心区别在于：**HTML 是在什么时候生成的？由谁生成的？**

这直接决定了网站的 **SEO（搜索引擎优化）**、**首屏加载速度 (FCP)** 以及 **服务器负载**。

### 一、核心概念通俗解释

#### 1. SSR (服务端渲染)

> **“现点现做”** —— 像餐厅的**炒菜**。

- **工作流程**：
    1. 用户请求页面（例如 `/product/123`）。
    2. 服务器收到请求，**立刻**去数据库查数据。
    3. 服务器把数据和模板拼成完整的 **HTML**。
    4. 服务器把 HTML 发给浏览器。
    5. 浏览器显示内容（此时页面已可见），随后下载 JS 进行“注水”（Hydration），让页面可交互。
- **特点**：每次请求都是**动态**的，数据是最新的。
- **适用场景**：用户个人中心、实时新闻、股票行情、电商商品详情页（价格库存实时变）。

#### 2. SSG (静态站点生成)

> **“提前预制”** —— 像超市的**罐头**。

- **工作流程**：
    1. **构建时 (Build Time)**：开发者运行 `npm run build`。
    2. 框架遍历所有路由，去接口拉取数据。
    3. 生成一堆纯静态的 **HTML 文件** (`index.html`, `about.html`)。
    4. 把这些文件部署到 CDN 或静态服务器（如 Nginx, Vercel, Cloudflare）。
    5. **用户请求时**：服务器直接把现成的 HTML 文件扔给用户，**不需要**经过 Node.js 服务器计算。
- **特点**：速度极快（因为是静态文件），服务器零压力，但数据不是实时的（除非重新构建）。
- **适用场景**：博客、文档站、营销官网、关于我们、产品介绍页。

### 二、深度对比表

|特性|**SSR (服务端渲染)**|**SSG (静态站点生成)**|**CSR (客户端渲染 - 传统 SPA)**|
|---|---|---|---|
|**HTML 生成时机**|**请求时** (Request Time)|**构建时** (Build Time)|**浏览器端** (用户加载后)|
|**数据来源**|每次请求实时查询数据库/API|构建时抓取一次数据|浏览器加载 JS 后请求 API|
|**首屏速度 (FCP)**|🟡 快 (取决于服务器响应速度)|🟢 **极速** (直接从 CDN 读取)|🔴 慢 (需等待 JS 下载执行)|
|**SEO 友好度**|🟢 **完美** (爬虫直接拿到完整 HTML)|🟢 **完美** (爬虫直接拿到完整 HTML)|🟠 一般 (依赖爬虫执行 JS，有时效问题)|
|**服务器压力**|🔴 **高** (每个请求都要计算)|🟢 **极低** (只需托管静态文件)|🟡 中 (仅托管静态资源，但并发高)|
|**数据实时性**|🟢 **实时** (总是最新数据)|🟠 **滞后** (直到下次构建才更新)|🟢 实时 (加载后请求最新数据)|
|**典型应用**|淘宝商品页、Twitter 时间线|Hexo/Hugo 博客、公司官网|后台管理系统、SaaS 仪表盘|

### 三、技术原理图解

#### 1. SSR 流程

```text
[用户浏览器] 
      ⬇️ (请求 /news/1)
[Node.js 服务器] 
      ⬇️ (1. 查数据库获取最新新闻)
      ⬇️ (2. 渲染 Vue/React 组件为 HTML 字符串)
      ⬇️ (3. 返回完整 HTML)
[用户浏览器] -> 立即显示内容 -> 下载 JS -> 绑定事件 (Hydration)
```

_痛点：如果数据库慢，用户就得等；如果并发高，服务器 CPU 会爆。_

#### 2. SSG 流程

```text
[开发者电脑] (运行 npm run build)
      ⬇️ (1. 遍历所有路由)
      ⬇️ (2. 请求数据)
      ⬇️ (3. 生成 .html 文件)
      ⬇️ (4. 部署到 CDN/Nginx)
      
[用户浏览器] 
      ⬇️ (请求 /news/1)
[CDN / 静态服务器] 
      ⬇️ (直接返回预先生成的 .html 文件)
[用户浏览器] -> 瞬间显示内容 -> 下载 JS -> 绑定事件
```

_痛点：如果新闻更新了，必须重新运行 `build` 才能让用户看到。_

### 四、现代框架的“混合模式” (ISR / On-Demand)

在 **Next.js** 和 **Nuxt** 中，SSR 和 SSG 不再是二选一，而是可以**混合使用**，甚至出现了更高级的模式：

#### 1. ISR (Incremental Static Regeneration，增量静态再生)

- **Next.js 特产，Nuxt 也支持**。
- **原理**：页面默认是 SSG（静态的）。但当用户访问一个**过时**的页面时：
    1. 服务器先返回旧的静态页面（保证速度）。
    2. 后台触发重新生成该页面的 HTML。
    3. 下一个用户访问时，就看到新页面了。
- **优势**：既有 SSG 的速度，又有接近 SSR 的实时性，还不用全量重建网站。

#### 2. On-Demand SSG (按需静态生成)

- **Nuxt 3/4 强项**。
- **原理**：构建时不生成任何页面。当第一个用户访问 `/product/999` 时，服务器临时生成这个 HTML 并缓存起来。后续用户直接拿缓存。
- **优势**：适合有海量页面（如百万级商品），但不可能全部预先构建的场景。

### 五、代码实战对比 (以 Nuxt 为例)

#### 场景 A：开启 SSR (动态)

在 `nuxt.config.ts` 中设置 `ssr: true` (默认)，或者在特定页面强制 SSR。

```typescript
// pages/profile.vue (用户个人资料，必须实时)
export default defineNuxtPageMeta({
  ssr: true 
});
// 每次访问都会去数据库查最新的用户信息
```

#### 场景 B：开启 SSG (静态)

在 `nuxt.config.ts` 中设置 `ssr: false` 且开启 `nitro.static`。

```typescript
// nuxt.config.ts
export default defineNuxtConfig({
  ssr: false, // 关闭服务端渲染
  nitro: {
    preset: 'static' // 输出静态文件
  }
});
// 运行 `nuxt generate` 后，生成纯 HTML 文件
```

#### 场景 C：混合模式 (Nuxt 4 / Next.js)

```typescript
// pages/blog/[slug].vue
export default defineNuxtPageMeta({
  // 静态生成，但每 60 秒重新验证一次 (ISR)
  cache: {
    maxAge: 60, 
    swr: true 
  }
});
```

### 六、选型指南：我该用哪个？

|你的需求|推荐模式|理由|
|---|---|---|
|**公司官网 / 博客 / 文档**|**SSG**|内容不常变，追求极致速度和低成本托管 (GitHub Pages, Cloudflare Pages)。|
|**电商首页 / 活动页**|**SSG + ISR**|大部分时间不变，偶尔更新，需要兼顾速度和内容新鲜度。|
|**用户 Dashboard / 后台**|**CSR** (或 SSR)|不需要 SEO，登录后数据完全个性化，SSG 没意义。|
|**新闻门户 / 社交媒体**|**SSR**|内容每秒都在变，必须保证用户看到的是最新状态。|
|**大型电商 (百万商品)**|**SSG (按需) / ISR**|无法预先构建百万页面，采用“访问时生成并缓存”策略。|

### 七、总结

- **SSR** = **动态**。慢一点点（相对 SSG），费服务器，但数据永远最新。适合**千人千面**或**实时数据**。
- **SSG** = **静态**。快得飞起，省服务器，但数据有延迟。适合**内容固定**或**更新频率低**的场景。
- **现代趋势**：不再死磕某一种，而是利用 **Next.js/Nuxt** 的能力，**按页面粒度**灵活选择。首页用 SSG，个人中心用 SSR，商品页用 ISR，实现性能与体验的完美平衡。
