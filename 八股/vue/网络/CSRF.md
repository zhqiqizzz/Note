## CSRF（Cross-Site Request Forgery，跨站请求伪造）
### 1. 核心定义与原理

CSRF 全称跨站请求伪造，也被称为 “会话劫持”、“一键攻击”，**核心原理是：利用网站对用户浏览器的信任，诱导用户在已登录目标网站的情况下，访问攻击者的恶意页面，恶意页面自动发起对目标网站的请求，浏览器会自动携带目标网站的有效 Cookie，网站后端会误以为是用户本人发起的合法请求，从而执行攻击者预设的操作**。

CSRF 攻击的核心本质：**攻击者全程无法获取用户的 Cookie 或任何敏感信息，只是借用户的登录状态，冒充用户执行操作**。

#### CSRF 攻击成功的两个必要前提

1. 用户已经在浏览器中登录了目标网站，且浏览器保存了目标网站的有效登录 Cookie（会话未过期）；
2. 用户在同一个浏览器中，访问了攻击者构造的恶意页面，浏览器的同源策略不会阻止跨站请求的发送，且会自动携带对应域名的 Cookie。

### 2. CSRF 的攻击流程（典型示例）

我们以银行转账场景为例，完整还原 CSRF 攻击过程：

1. **用户登录目标网站**：用户在浏览器中登录了网上银行 `https://my-bank.com`，登录成功后，浏览器保存了该网站的有效登录 Cookie，会话有效期内持续有效。
2. **攻击者构造恶意页面**：攻击者提前摸清了银行的转账接口规则，构造了恶意页面 `https://evil.com/csrf.html`，页面代码如下：
    
```html
    <!-- 隐藏的转账表单，自动提交 -->
    <form action="https://my-bank.com/api/transfer" method="POST" id="csrfForm" style="display:none">
      <input type="hidden" name="toAccount" value="hacker-account-123">
      <input type="hidden" name="amount" value="10000">
    </form>
    <script>
      // 页面加载后自动提交表单，发起转账请求
      document.getElementById('csrfForm').submit();
    </script>
```
    
3. **诱导用户访问恶意页面**：攻击者通过邮件、社交平台、广告等渠道，诱导用户在登录了银行的同一个浏览器中，访问 `https://evil.com/csrf.html`。
4. **自动发起伪造请求**：用户访问恶意页面时，页面自动提交表单，向银行的转账接口发起 POST 请求，浏览器会**自动携带 `https://my-bank.com` 的有效登录 Cookie**。
5. **后端执行非法操作**：银行后端收到请求，校验 Cookie 有效，误以为是用户本人发起的转账操作，执行了转账逻辑，攻击完成。

### 3. CSRF 的主流攻击类型

#### （1）GET 型 CSRF

最简单、最常见的 CSRF 类型，利用 `img`、`script`、`link` 等 HTML 标签会自动发起 GET 请求的特性，构造恶意请求，用户只要访问页面，就会自动触发攻击。
##### 示例

```html
<!-- 页面加载时，自动发起GET转账请求 -->
<img src="https://my-bank.com/api/transfer?toAccount=hacker123&amount=10000" style="display:none">
```
##### 核心前提

目标网站的高危操作接口使用了 GET 方法，这是严重的开发规范问题 ——GET 请求应仅用于只读操作，绝对不能用于修改数据、转账等写操作。

#### （2）POST 型 CSRF

比 GET 型更通用的攻击方式，针对使用 POST 方法的接口，攻击者在恶意页面中构造隐藏表单，通过 JS 自动提交，模拟用户正常的表单提交操作，和用户本人提交的请求几乎无差异。

就是上文示例中的攻击方式，也是目前最主流的 CSRF 攻击类型。

#### （3）JSON 型 CSRF

针对接收 JSON 格式数据的接口，攻击者通过 `fetch`、`XMLHttpRequest` 发起跨域 POST 请求，携带 JSON 格式的攻击数据。

该攻击有严格的限制：只有目标网站的 CORS（跨域资源共享）配置存在严重漏洞（比如允许所有外域的跨域请求、允许携带凭证），才能攻击成功，否则会被浏览器的同源策略拦截。

### 4. CSRF 的核心危害

CSRF 的核心危害是**冒充用户执行其权限内的任意操作**，无法直接窃取用户信息，但可通过操作间接接管账号、造成财产损失：

1. **财产损失**：诱导用户执行转账、支付、充值、购买商品等操作，直接造成用户资金损失；
2. **账号接管**：修改用户的登录密码、绑定邮箱 / 手机号，再通过找回密码功能接管用户账号；
3. **数据篡改与删除**：修改用户的个人信息、收货地址，删除用户的帖子、文件、订单等数据；
4. **权限操作**：如果受害者是网站管理员，可执行添加管理员账号、修改网站配置、删除网站数据等高危操作，导致整个网站被接管；
5. **社交传播**：在社交网站中，自动给用户的好友发私信、发帖子，扩散恶意链接，实现蠕虫式传播。

### 5. CSRF 的完整防御方案

CSRF 防御的核心原则是：**验证请求的发起者是否为用户本人，而不是仅校验 Cookie 是否有效**。

#### （1）核心防御：CSRF Token（业界主流方案）

CSRF Token 是目前最有效、最通用的防御方案，核心原理是：在请求中加入一个攻击者无法获取、不可预测的随机 Token，后端验证 Token 有效性，只有 Token 正确的请求才会被处理。

##### 完整执行流程

1. 用户登录后，后端生成一个**随机、不可预测、唯一**的 CSRF Token，存储到用户的 Session 中，同时返回给前端；
2. 前端页面渲染时，将 Token 放入表单的隐藏域，或存入 `localStorage` 中；
3. 前端发起 POST/PUT/DELETE 等写操作请求时，必须将 Token 放入请求参数或请求头中（如 `X-CSRF-Token` 头）；
4. 后端收到请求后，对比请求中的 Token 和 Session 中存储的 Token 是否一致，不一致直接拒绝请求。

##### 防御有效性

攻击者的恶意页面处于外域，受浏览器同源策略限制，无法读取目标网站的页面内容、Session 或 `localStorage` 中的 Token，因此无法在伪造的请求中携带正确的 Token，后端会直接拦截攻击。

#### （2）浏览器层面防御：Cookie SameSite 属性

SameSite 是 Cookie 的核心属性，用来限制 Cookie 在跨站请求中是否被自动携带，是目前浏览器原生支持的、最有效的 CSRF 防御手段，Chrome、Firefox 等现代浏览器均已支持。

SameSite 有 3 个可选值：

|值|规则|安全性|适用场景|
|:--|:--|:--|:--|
|`Strict`（严格模式）|完全禁止跨站请求携带 Cookie，只有当前页面的域名和请求目标完全一致时，才会携带 Cookie|最高|银行、支付等对安全性要求极高的系统|
|`Lax`（宽松模式）|仅允许安全的 HTTP 方法（GET/HEAD/OPTIONS）在顶级导航时携带 Cookie，POST 请求、img/script 等标签的跨站请求，一律不携带 Cookie|高（浏览器默认值）|绝大多数业务系统，兼顾安全与用户体验|
|`None`（无限制）|跨站请求会自动携带 Cookie，必须同时设置 `Secure` 属性，仅在 HTTPS 下生效|极低|必须跨站使用 Cookie 的场景（如第三方登录、嵌入组件）|
##### 配置示例

```http
Set-Cookie: sessionId=abc123; SameSite=Lax; HttpOnly; Secure
```

设置 `SameSite=Lax` 后，攻击者恶意页面中的 POST 表单、img 标签的 GET 请求，都不会携带目标网站的 Cookie，CSRF 攻击直接失效。

#### （3）同源检测：验证请求来源

后端通过校验请求头中的 `Origin` 和 `Referer` 字段，判断请求是否来自本站的合法域名，外域发起的请求直接拦截。

- `Origin` 头：记录请求的源域名，POST 请求一定会携带，无法被前端篡改，优先使用；
- `Referer` 头：记录请求的来源页面完整 URL，兼容性更好，但可能被浏览器禁用或篡改，作为辅助校验。

该方案实现简单，适合作为辅助防御手段，不建议作为唯一防御方案。

#### （4）双重 Cookie 验证

适合分布式系统、无 Session 架构的场景，核心原理：

1. 用户登录后，后端生成一个随机 Token，设置到 Cookie 中；
2. 前端发起请求时，从 Cookie 中取出该 Token，放入请求头或请求参数中；
3. 后端收到请求后，对比 Cookie 中的 Token 和请求中的 Token 是否一致，不一致直接拒绝。

防御有效性：攻击者只能利用浏览器自动携带 Cookie，但受同源策略限制，无法读取 Cookie 中的 Token，因此无法在请求中携带正确的 Token。注意：该方案依赖网站无 XSS 漏洞，XSS 可读取 Cookie 中的 Token 绕过防御。

#### （5）高危操作增加二次验证

对于转账、修改密码、修改绑定信息、删除数据等高危操作，必须增加二次验证环节，例如：短信验证码、邮箱验证码、原密码校验、人脸识别等。

即使存在 CSRF 漏洞，攻击者也无法获取用户的验证码，无法完成攻击，是 CSRF 攻击的终极兜底方案。

#### （6）基础开发规范

- 严格区分 HTTP 方法，GET 请求仅用于只读操作，绝对不允许用 GET 处理任何修改数据的写操作；
- 禁止接口支持跨域 Credentials 模式，除非业务必须，且严格限制跨域白名单。
