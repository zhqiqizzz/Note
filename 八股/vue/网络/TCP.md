**TCP**（全称 **Transmission Control Protocol，传输控制协议**）是 TCP/IP 协议簇的核心传输层协议，是面向连接、可靠有序、基于字节流、支持全双工通信的传输标准。它为应用层提供了无差错的端到端数据传输服务，是 HTTP/HTTPS、FTP、SSH、数据库连接等绝大多数互联网应用的底层传输载体，与 IP 协议共同构成了互联网的通信基石。

## 一、核心基础特性

TCP 的所有设计都围绕「**在不可靠的 IP 网络之上，实现可靠的端到端数据传输**」展开，核心特性如下：

### 1. 面向连接

TCP 通信必须遵循「**先建立连接，再传输数据，最后断开连接**」的完整生命周期，不存在无连接的自发传输。

- 连接是**点对点、端到端**的，仅支持两个端点之间的一对一通信，不支持广播、多播。
- 连接的建立和断开都有严格的标准化流程（三次握手建连、四次挥手断连），保证双方的收发能力都正常。

### 2. 可靠有序、无差错传输

这是 TCP 最核心的价值，它能保证：

- 数据完整送达：发送的数据最终能完整到达接收方，不会丢失。
- 数据顺序一致：接收方收到的数据顺序，和发送方发出的顺序完全一致。
- 数据无重复、无篡改：自动丢弃重复报文，通过校验和检测数据篡改，异常则触发重传。

### 3. 基于字节流

TCP 将应用层交付的数据，视为一串无结构、无边界的连续字节流，不区分消息边界。

- 发送方可以分多次发送 1000 字节的数据，接收方也可以分多次接收，TCP 只保证字节流的完整有序，不保证应用层的消息边界。
- 这一特性带来了极高的灵活性，但也导致了**粘包 / 拆包**问题，需要应用层自行定义消息格式来处理边界。

### 4. 全双工通信

TCP 连接建立后，双方可以同时、独立地发送和接收数据，互不干扰。

- 连接两端都有独立的**发送缓冲区**和**接收缓冲区**，发送方的数据先进入发送缓冲区，由内核负责发送；接收方收到的数据先进入接收缓冲区，由应用层读取。
- 全双工特性决定了 TCP 的断开需要双方单独关闭各自的发送通道，也就是四次挥手的核心原因。

### 5. 内置流量控制与拥塞控制

- 流量控制：解决「发送方发得太快，接收方处理不过来，导致缓冲区溢出丢包」的问题。
- 拥塞控制：解决「发送方发得太快，中间网络路由器过载，导致网络拥塞、大面积丢包」的问题。

## 二、核心连接管理：三次握手与四次挥手

TCP 连接的管理分为 ** 连接建立（三次握手）**和**连接断开（四次挥手）** 两个核心流程，所有状态转换都有严格的标准定义。

### 1. 三次握手：建立可靠连接

三次握手的核心目标：**同步双方的初始序列号、确认双方的收发能力正常、协商核心参数，最终建立可靠的全双工连接**。

#### 核心术语提前说明

- `SYN`：同步标志位，用于建连时同步序列号。
- `ACK`：确认标志位，代表确认号有效，建连完成后所有报文必须开启 ACK。
- `seq`：序列号，本报文段数据第一个字节的编号，每个连接的初始序列号是随机生成的。
- `ack`：确认号，代表期望收到的下一个字节的序列号，ack = 对方的 seq + 本次收到的数据长度。

#### 完整流程

|步骤|动作|报文内容|客户端状态|服务端状态|
|---|---|---|---|---|
|1|客户端主动发起建连请求|客户端向服务端发送`SYN`报文，`seq=x`（随机初始序列号），无数据|CLOSED → SYN_SENT|LISTEN|
|2|服务端确认客户端的建连请求，同时发起自己的建连同步|服务端收到`SYN`后，回复`SYN+ACK`报文，`ack=x+1`（确认收到客户端的 seq），`seq=y`（自身随机初始序列号）|SYN_SENT|LISTEN → SYN_RCVD|
|3|客户端确认服务端的同步，完成建连|客户端收到`SYN+ACK`后，回复`ACK`报文，`ack=y+1`，`seq=x+1`；服务端收到该`ACK`后，建连完成|SYN_SENT → ESTABLISHED|SYN_RCVD → ESTABLISHED|

#### 关键问题：为什么是三次握手，不是两次？

1. **防止失效的连接请求浪费服务端资源**：如果客户端发送的`SYN`报文因网络延迟滞留，客户端超时重发了新的`SYN`并完成通信、断开连接后，滞留的旧`SYN`才到达服务端。两次握手的情况下，服务端会直接建立连接并一直等待客户端发数据，造成大量无效资源占用；三次握手时，客户端不会为无效`SYN`回复`ACK`，服务端收不到最终确认，就不会建立无效连接。
2. **双向同步初始序列号**：TCP 是全双工通信，需要双方都确认对方的序列号是可接收的，两次握手只能完成单向同步，三次才能完成双向的序列号确认，保证后续可靠传输的有序性。

### 2. 四次挥手：优雅断开全双工连接

TCP 是全双工通信，双方的发送通道是独立的，需要各自单独关闭，因此需要四次交互完成断开。主动关闭方可以是客户端，也可以是服务端。

#### 核心术语

- `FIN`：结束标志位，代表本方没有数据要发送了，请求关闭本方的发送通道。

#### 完整流程

|步骤|动作|报文内容|主动关闭方状态|被动关闭方状态|
|---|---|---|---|---|
|1|主动关闭方发起关闭请求|发送`FIN`报文，`seq=m`，告知对方：我没有数据要发了，要关闭我的发送通道|ESTABLISHED → FIN_WAIT_1|ESTABLISHED|
|2|被动关闭方确认关闭请求|收到`FIN`后，回复`ACK`报文，`ack=m+1`，告知对方：我收到了你的关闭请求，但我还有数据要发，暂时不能关闭|FIN_WAIT_1 → FIN_WAIT_2|ESTABLISHED → CLOSE_WAIT|
|3|被动关闭方数据发完，发起自己的关闭请求|发送`FIN`报文，`seq=n`，`ack=m+1`，告知对方：我也没有数据要发了，要关闭我的发送通道|FIN_WAIT_2|CLOSE_WAIT → LAST_ACK|
|4|主动关闭方确认最终关闭，等待 2MSL 后彻底关闭|收到`FIN`后，回复`ACK`报文，`ack=n+1`，`seq=m+1`；被动关闭方收到该`ACK`后，直接关闭连接|FIN_WAIT_2 → TIME_WAIT（等待 2MSL 后→CLOSED）|LAST_ACK → CLOSED|

#### 关键问题

- **为什么是四次挥手，不是三次？**
    
    因为 TCP 是全双工的，`FIN`仅代表关闭本方的发送通道，接收通道依然可以正常接收数据。被动关闭方收到`FIN`后，可能还有未发送完的数据，不能把确认`ACK`和自己的`FIN`合并发送，必须先回`ACK`确认，等数据全部发送完成后，再发`FIN`关闭自己的发送通道，因此需要四次交互。
    
- **为什么主动关闭方要等待 2MSL？**
    
    MSL（Maximum Segment Lifetime，报文最大生存时间）是 IP 报文在网络中的最长存活时间，2MSL 就是一个来回的最长时间。
    
    1. 保证最后一个`ACK`能被对方收到：如果被动关闭方没收到最终的`ACK`，会重发`FIN`报文，2MSL 的等待时间内，主动关闭方可以收到重发的`FIN`并再次回复`ACK`，避免对方无法正常关闭。
    2. 清除网络中的残留报文：等待 2MSL 可以让本次连接产生的所有报文都从网络中消失，避免后续新建的同名连接收到旧连接的残留报文，造成数据错乱。

## 三、TCP 可靠传输的核心实现机制

TCP 在不可靠的 IP 网络之上实现可靠传输，核心依赖以下 6 大机制：

1. **序列号与累计确认机制**
    
    TCP 为传输的每一个字节都分配了唯一的序列号，接收方收到数据后，会通过`ACK`报文回复确认号，确认号代表「前面所有字节都已收到，期望下次收到从该编号开始的数据」。
    
    例：发送方发送`seq=1`、长度 100 字节的报文，接收方回复`ack=101`，代表 1-100 字节全部收到，下次请从 101 字节开始发送。
    
2. **超时重传机制**
    
    发送方发送报文后，会启动**超时重传计时器（RTO）**，如果超过 RTO 时间仍未收到对应的`ACK`确认，就会重传该报文段。
    
    RTO 是动态计算的，基于网络往返时间 RTT 实时调整，适配网络延迟的变化，避免过早重传或等待过久。
    
3. **校验和机制**
    
    发送方会基于整个 TCP 报文（首部 + 数据）计算校验和，接收方收到后会重新计算校验和，若结果不一致，说明数据被篡改或传输出错，直接丢弃该报文，不回复`ACK`，触发发送方超时重传。
    
4. **失序重排与重复丢弃**
    
    IP 报文可能会乱序到达，TCP 会根据序列号将乱序的报文重新排序，再按顺序交给应用层；若收到重复的报文（如`ACK`丢失导致发送方重传），TCP 会自动丢弃重复数据，保证应用层只收到一次完整数据。
    
5. **选择性确认（SACK）**
    
    传统的累计确认只能确认到连续的最后一个字节，一旦中间有报文丢失，只能重传丢失位置之后的所有数据。SACK 是 TCP 的优化扩展，接收方可以通过 SACK 选项告知发送方「哪些不连续的字节段已经收到」，发送方只需重传真正丢失的字节段，大幅提升丢包场景下的传输效率。

## 四、流量控制与拥塞控制

### 1. 流量控制：端到端的发送速率匹配

流量控制的核心是**保证发送方的发送速率，不超过接收方的处理能力**，避免接收方缓冲区溢出导致丢包。

TCP 通过**滑动窗口协议**实现流量控制：

- 接收方在每一个`ACK`报文中，都会携带自己的**接收窗口大小 rwnd**，代表自己当前还能接收多少字节的数据。
- 发送方的发送窗口，始终不能超过接收方通告的 rwnd 大小，实现精准的速率控制。
- 若接收方缓冲区满了，rwnd=0，发送方会停止发送数据，同时启动持续计时器，定时发送窗口探测报文，直到接收方更新 rwnd>0，再恢复发送。

### 2. 拥塞控制：网络全局的发送速率控制

拥塞控制的核心是**避免发送方发送过多数据，导致中间网络路由器过载、网络拥塞**，是 TCP 最复杂的核心模块，由 4 个核心算法组成：

#### （1）慢启动

连接刚建立时，不了解网络状况，不会一次性发送大量数据。拥塞窗口 cwnd（发送方根据网络拥塞情况设置的发送上限）从 1 个 MSS（最大报文段长度）开始，每收到一个有效`ACK`，cwnd 就翻倍，呈指数增长，直到达到慢启动阈值 ssthresh，进入拥塞避免阶段。

核心目的：避免一开始就把网络堵死，逐步探测网络的承载能力。

#### （2）拥塞避免

当 cwnd 超过 ssthresh 后，停止指数增长，改为**每经过一个 RTT，cwnd 只增加 1 个 MSS**，呈线性缓慢增长。

核心目的：在避免网络拥塞的前提下，缓慢提升发送速率，最大化利用网络带宽。

#### （3）快重传

当发送方**连续收到 3 个重复的 ACK**，就可以判断对应的报文段已经丢失，无需等待超时计时器到期，直接重传丢失的报文段。

核心目的：大幅减少丢包后的等待时间，提升传输效率。

#### （4）快恢复

快重传触发后，不会直接回到慢启动阶段，而是执行：

1. 将慢启动阈值 ssthresh 减半；
2. 将 cwnd 设置为新的 ssthresh；
3. 直接进入拥塞避免阶段，线性增长。
    
    核心目的：避免网络刚出现丢包就把发送速率降到最低，保证传输的稳定性和带宽利用率。

## 五、TCP 报文段结构

TCP 报文段分为**固定首部**和**数据部分**，固定首部长度为 20 字节，可选扩展首部最多 40 字节，总首部最大 60 字节。

|字段|长度|核心作用|
|---|---|---|
|源端口 / 目的端口|各 16 位|标识通信两端的应用进程，实现传输层的复用与分用，端口范围 0-65535|
|序列号 seq|32 位|本报文段数据第一个字节的编号，保证数据有序|
|确认号 ack|32 位|期望收到的下一个字节的序列号，ACK 标志位为 1 时有效|
|数据偏移|4 位|标识 TCP 首部的长度，单位为 4 字节，因此首部最大 60 字节|
|标志位（控制位）|6 位|核心控制位：<br><br>• `URG`：紧急指针有效<br><br>• `ACK`：确认号有效，建连后所有报文必须为 1<br><br>• `PSH`：推送，要求接收方尽快将数据交给应用层<br><br>• `RST`：复位，强制断开异常连接<br><br>• `SYN`：同步，用于建连时同步序列号<br><br>• `FIN`：结束，用于断连时关闭发送通道|
|窗口大小|16 位|接收方通告的接收窗口 rwnd，用于流量控制，最大 65535 字节|
|校验和|16 位|校验整个 TCP 报文的完整性，防篡改、防传输错误|
|紧急指针|16 位|URG 标志位为 1 时有效，标识紧急数据的末尾位置|
|选项|可变长度|扩展功能，常见：MSS 最大报文段长度、SACK 选择性确认、窗口扩大、时间戳等|

## 六、TCP vs UDP 核心对比

|特性|TCP|UDP（用户数据报协议）|
|---|---|---|
|连接特性|面向连接，必须先建连再传输|无连接，无需建连，直接发送数据|
|可靠性|可靠有序，保证数据无丢失、无重复、无乱序|不可靠，不保证数据送达，不保证顺序|
|传输模式|基于字节流，无消息边界|基于数据报，有明确的消息边界，一次发送对应一次接收|
|双工特性|全双工|支持全双工|
|流量 / 拥塞控制|内置完整的流量控制、拥塞控制|无，发送速率完全由应用层控制|
|首部开销|固定 20 字节，最大 60 字节|固定 8 字节，开销极小|
|适用场景|对数据可靠性要求高的场景：HTTP/HTTPS、FTP、SSH、数据库连接、邮件传输|对实时性要求高、可容忍少量丢包的场景：直播、游戏、视频通话、DNS 查询|

## 七、常见应用场景与开发痛点

### 主流应用场景

所有对数据完整性、可靠性要求高的互联网应用，底层均基于 TCP 传输：

- Web 服务：HTTP/1.1、HTTP/2、HTTPS
- 文件传输：FTP、SFTP
- 远程登录：SSH、Telnet
- 邮件服务：SMTP、POP3、IMAP
- 数据库访问：MySQL、PostgreSQL、Redis
- 实时可靠通信：WebSocket

### 开发中常见的核心痛点

1. **粘包 / 拆包问题**
    
    因 TCP 是字节流无消息边界，多个小报文可能被合并成一个 TCP 报文发送（粘包），一个大报文可能被拆分成多个 TCP 报文发送（拆包）。
    
    解决方案：应用层定义消息格式，如固定长度、特殊分隔符、「消息头 + 消息体长度」的结构化格式。
    
2. **TIME_WAIT 过多**
    
    高并发短连接场景下，主动关闭方会产生大量 TIME_WAIT 状态的连接，占用系统端口和内存资源。
    
    解决方案：调整内核参数、优化连接复用、让服务端成为被动关闭方。
    
3. **传输延迟优化**
    
    TCP 内置的**Nagle 算法**（攒小报文一起发，减少网络包数量）和**延迟 ACK**（延迟回复 ACK，尝试和数据合并），在实时性要求高的场景下会导致延迟飙升。
    
    解决方案：实时性场景可通过`TCP_NODELAY`选项关闭 Nagle 算法。

## 八、TCP 的局限性与演进

TCP 的核心设计也带来了固有局限性：

1. **队头阻塞**：TCP 要求数据按顺序到达，一旦某个报文丢失，整个连接的所有数据都要停下来等待重传，哪怕后续报文已经到达接收缓冲区。这也是 HTTP/2 多路复用依然存在性能瓶颈的核心原因。
2. **握手延迟高**：HTTPS 场景下，TCP 三次握手 + TLS 握手需要 3-4 个 RTT，才能开始传输业务数据，弱网场景下延迟极高。
3. **连接迁移能力差**：移动端切换 Wi-Fi / 移动网络时，IP 地址变化，TCP 连接会直接断开，必须重新建连。

为了解决这些问题，**QUIC 协议**应运而生：它基于 UDP 封装，实现了 TCP 的可靠传输、拥塞控制、流量控制，同时彻底解决了队头阻塞，支持 0-RTT/1-RTT 握手、连接迁移，成为了 HTTP/3 的底层传输协议。
