**HTTPS**（全称 **HyperText Transfer Protocol Secure，超文本传输安全协议**），是 HTTP 的安全增强版本。它的核心本质是**HTTP + TLS/SSL**：在 HTTP 应用层和 TCP 传输层之间，新增了一层**TLS/SSL 加密安全层**，彻底解决了 HTTP 明文传输的三大致命安全缺陷，是目前互联网 Web 通信的绝对主流标准。

- HTTP 默认端口：**80**，明文传输，无任何安全保障；
- HTTPS 默认端口：**443**，全程加密传输，具备防窃听、防篡改、防伪造三大核心安全能力。

## 一、为什么需要 HTTPS？HTTP 的三大致命安全缺陷

HTTP 协议全程明文传输，数据在网络中像 “裸奔” 一样，经过路由器、运营商、WiFi 热点等任意中间节点，都可以被读取、修改、伪造，核心有三大安全问题：

|缺陷|具体风险|典型攻击场景|
|---|---|---|
|**明文传输，数据可窃听**|传输的所有内容（账号密码、Cookie、银行卡号、业务数据）都能被中间节点直接读取|公共 WiFi 下，攻击者抓取用户的登录请求，直接窃取账号密码|
|**无完整性校验，数据可篡改**|中间节点可以随意修改传输的内容，收发双方无法感知|运营商在网页中植入广告、恶意脚本；中间人篡改支付金额、收款账户|
|**无身份认证，身份可伪造**|无法确认通信的对方是官方服务器，还是攻击者伪造的钓鱼服务器|攻击者搭建钓鱼网站，冒充银行官网，即使用户输入了正确的域名，也无法验证对方身份，导致敏感信息泄露|

HTTPS 的诞生，就是为了彻底解决这三个问题，对应实现了三大核心能力：

1. **数据加密**：全程加密传输，中间节点无法读取明文内容，防窃听；
2. **完整性校验**：通过密码学校验机制，确保数据在传输过程中没有被篡改，防篡改；
3. **身份认证**：通过可信数字证书，确认服务器的真实身份，防伪造 / 钓鱼。

## 二、HTTPS 的核心安全基石：TLS/SSL 协议

HTTPS 的安全能力，完全由**TLS 协议**提供。

### 1. SSL 与 TLS 的演进

- **SSL**（Secure Sockets Layer，安全套接层）：是早期的加密协议，由网景公司开发，经历了 SSL 1.0、2.0、3.0 三个版本，目前已全部被废弃，存在严重安全漏洞。
- **TLS**（Transport Layer Security，传输层安全协议）：是 SSL 的标准化继任者，由 IETF 发布，是目前 HTTPS 的核心标准。
    
    - 已淘汰版本：TLS 1.0、TLS 1.1（主流浏览器已停止支持）；
    - 主流稳定版本：**TLS 1.2**（兼容性最好）；
    - 最新高性能版本：**TLS 1.3**（大幅优化握手延迟，安全性更强，目前已大规模普及）。
    

> 日常口语中常说的 “SSL 证书”，本质上都是 TLS 证书，只是沿用了旧的叫法。

### 2. TLS 的核心加密体系：对称加密 + 非对称加密的完美结合

TLS 不是单一的加密算法，而是一套密码学套件，核心结合了两种加密方式的优势，兼顾**安全性**和**传输性能**。
#### （1）两种加密方式的核心对比

| 加密类型      | 核心原理                                                                       | 优点                        | 缺点                          | 代表算法            |
| --------- | -------------------------------------------------------------------------- | ------------------------- | --------------------------- | --------------- |
| **对称加密**  | 加密和解密使用**同一个密钥**                                                           | 加密解密速度极快，性能损耗小，适合大数据量传输   | 密钥分发难题：如何安全地把密钥传给对方，不被中间人窃取 | AES、ChaCha20    |
| **非对称加密** | 生成一对密钥：**公钥（可公开）**和**私钥（仅服务器持有，绝对保密）**；公钥加密的内容，只有对应的私钥能解密；私钥签名的内容，公钥可验证合法性 | 解决了密钥分发问题，公钥可以全网公开，无需担心泄露 | 加密解密速度极慢，性能损耗大，不适合大数据量传输    | RSA、ECC（椭圆曲线加密） |
#### （2）TLS 的结合方案：各司其职，兼顾安全与性能

TLS 不会用非对称加密传输业务数据，而是采用 “非对称加密协商密钥，对称加密传输数据” 的模式：

1. **密钥协商阶段**：用非对称加密，安全地协商生成一个**仅本次通信有效的临时对称密钥（会话密钥）**；
2. **数据传输阶段**：用协商好的对称密钥，对 HTTP 报文进行加密解密，全程高速传输。

这个方案完美解决了两个核心问题：

- 非对称加密只用来协商密钥，数据量极小，规避了性能差的缺点；
- 对称密钥是临时生成的，仅本次通信有效，且通过非对称加密安全传输，解决了密钥分发的安全问题。

### 3. 身份认证的核心：数字证书与 CA 机构

这里有一个关键问题：哪怕用了非对称加密，中间人依然可以伪造公钥，实施 “中间人攻击”。比如：

1. 客户端向服务器要公钥，中间人拦截请求，把自己的公钥发给客户端；
2. 客户端用中间人的公钥加密数据，中间人可以用自己的私钥解密，窃取数据；
3. 中间人再用服务器的公钥加密数据，转发给服务器，全程双方都无法感知。

要解决这个问题，必须有一个**全网信任的第三方机构**，来证明 “这个公钥确实属于目标服务器”，这个机构就是**CA（Certificate Authority，证书颁发机构）**。

#### 数字证书的核心逻辑

1. 服务器生成自己的非对称密钥对，把公钥、域名、企业信息等提交给可信 CA；
2. CA 审核通过后，用 CA 自己的**私钥**，对服务器的公钥和信息进行数字签名，生成**数字证书**，颁发给服务器；
3. 客户端（浏览器）内置了所有全球可信 CA 的**公钥**，可以直接验证证书的合法性：
    
    - 用 CA 的公钥，验证证书上的数字签名是否有效；
    - 校验证书中的域名，是否和当前访问的域名一致；
    - 校验证书是否在有效期内、是否被吊销。
    
4. 只有证书验证通过，客户端才会使用证书里的服务器公钥，进行后续的密钥协商。

简单类比：CA 就是互联网世界的 “公安局”，数字证书就是服务器的 “身份证”，浏览器可以通过公安局的公章（CA 公钥），验证身份证的真伪，彻底杜绝伪造的钓鱼服务器。

#### 证书的常见类型

|证书类型|全称|认证级别|适用场景|
|---|---|---|---|
|DV 证书|域名验证型|基础级|个人网站、博客，仅验证域名所有权|
|OV 证书|组织验证型|企业级|企业官网、中小型业务，验证企业主体真实性|
|EV 证书|扩展验证型|最高级|银行、金融、支付平台，最严格的企业审核，浏览器地址栏会显示企业名称|
## 三、HTTPS 完整通信流程

HTTPS 的通信分为 4 个核心阶段，全程基于 TCP 协议，我们结合最常用的 TLS 1.2 和优化后的 TLS 1.3 分别讲解。

### 前置说明

HTTPS 的 TLS 握手，**必须在 TCP 三次握手完成之后**进行，只有 TCP 连接建立成功，才能开始 TLS 加密协商。

### 完整流程（以 TLS 1.2 标准握手为例）

|步骤|动作|核心内容|
|---|---|---|
|1|建立 TCP 连接|客户端和服务器完成 TCP 三次握手，建立可靠的 TCP 连接|
|2|客户端发送「Client Hello」|客户端向服务器发送：<br><br>・支持的 TLS 版本、加密套件列表<br><br>・客户端生成的随机数（Client Random）<br><br>・会话 ID（用于会话复用）|
|3|服务器返回「Server Hello」|服务器向客户端回应：<br><br>・选定的 TLS 版本、加密套件<br><br>・服务器生成的随机数（Server Random）<br><br>・会话 ID|
|4|服务器发送「数字证书」|服务器把自己的数字证书完整发给客户端|
|5|服务器发送「Server Hello Done」|告知客户端：服务器端的握手信息已发送完毕|
|6|客户端验证证书|客户端用内置的 CA 公钥，验证证书的合法性、域名匹配性、有效期；验证不通过直接中断连接，提示 “证书不安全”|
|7|客户端生成预主密钥，加密发送|客户端生成第三个随机数：**预主密钥（Pre-Master Secret）**，用证书里的服务器公钥加密，发送给服务器|
|8|双方协商生成会话密钥|客户端和服务器，用之前的 Client Random、Server Random、Pre-Master Secret，通过相同的算法，生成**完全一致的对称会话密钥**|
|9|客户端发送「Change Cipher Spec」|告知服务器：后续所有通信都用协商好的会话密钥加密|
|10|客户端发送「Finished」|把之前所有握手内容生成摘要，用会话密钥加密发送，供服务器校验握手完整性|
|11|服务器发送「Change Cipher Spec」|告知客户端：后续所有通信都用协商好的会话密钥加密|
|12|服务器发送「Finished」|把之前所有握手内容生成摘要，用会话密钥加密发送，供客户端校验|
|13|握手完成，加密传输 HTTP 数据|双方握手校验通过，TLS 连接建立成功；后续所有 HTTP 请求 / 响应，都用会话密钥加密后传输|
|14|连接断开|数据传输完成后，先断开 TLS 连接，再断开 TCP 连接|
### TLS 1.3 核心优化

TLS 1.3 对握手流程做了革命性优化，大幅降低延迟：

1. **1-RTT 握手**：把 TLS 1.2 的 2-RTT 握手，压缩到 1-RTT，客户端第一次发送「Client Hello」时，就带上密钥协商的所有参数，服务器直接返回证书和密钥信息，一次往返就完成握手；
2. **0-RTT 会话复用**：如果客户端之前和服务器建立过连接，可直接发送加密的 HTTP 请求，无需等待握手完成，延迟降到最低；
3. **更强的安全性**：禁用了所有不安全的加密算法和过时特性，只保留 5 个核心加密套件，大幅减少安全漏洞面。

## 四、HTTPS 的核心优势

1. **极致的通信安全**
    
    全程加密传输，彻底解决了明文窃听、数据篡改、身份伪造三大问题，是敏感业务（支付、金融、登录、企业管理）的强制要求。
2. **浏览器与搜索引擎的强制要求**
    
    - Chrome、Edge 等主流浏览器，对 HTTP 网站会明确标记 “不安全”，阻止用户访问；
    - 百度、谷歌等搜索引擎，优先收录 HTTPS 网站，HTTP 网站的搜索排名会大幅降低。
    
3. **新一代 Web 协议的基础**
    
    HTTP/2、HTTP/3 协议，主流浏览器都强制要求必须基于 HTTPS 运行，只有 HTTPS 才能使用多路复用、0-RTT 握手等高性能特性。
4. **规避运营商劫持与广告注入**
    
    HTTP 网站极易被运营商、中间节点植入广告、恶意脚本，HTTPS 加密后，中间节点无法修改内容，彻底杜绝劫持。
5. **提升 Web 应用的安全边界**
    
    配合 Cookie 的`Secure`属性（仅 HTTPS 下传输）、`HttpOnly`、`SameSite`，可以大幅降低 XSS、CSRF 攻击的风险，防止敏感 Cookie 被窃听。

## 五、常见误区与安全边界

1. **误区 1：HTTPS 是绝对安全的，不会被攻击**
    
    错。HTTPS 只是保障传输链路的安全，不是万能的：
    
    - 如果用户导入了不可信的根证书（比如公司内网代理、恶意软件），依然可以实施中间人攻击；
    - 服务器私钥泄露、证书配置错误、使用过时的 TLS 版本，都会导致安全漏洞；
    - HTTPS 无法防御 XSS、CSRF、SQL 注入等 Web 应用层漏洞，只能保障传输过程的安全。
    
2. **误区 2：HTTPS 性能损耗极大，会拖慢网站速度**
    
    错。早期 HTTPS 确实有明显的性能损耗，但现在硬件加速、TLS 1.3、会话复用、OCSP 装订等技术，已经把性能损耗降到了几乎可以忽略的程度，甚至配合 HTTP/2，速度比 HTTP 更快。
3. **误区 3：只要地址栏有小锁，就是安全的 HTTPS**
    
    错。小锁只代表证书是可信 CA 颁发的、传输是加密的，但不代表这个网站是正规网站。钓鱼网站也可以申请 DV 证书，地址栏也会显示小锁，依然需要核对域名是否正确。
4. **误区 4：内网系统、测试环境不需要 HTTPS**
    
    错。内网环境同样存在窃听、中间人攻击的风险，Chrome 等浏览器对 HTTP 网站的限制，会导致很多 Web 功能无法使用（比如摄像头、麦克风权限、Service Worker），测试环境也应和生产环境保持一致，使用 HTTPS。

## 六、生产环境最佳实践

1. **强制全站 HTTPS**
    
    配置 301/302 重定向，把所有 HTTP 请求跳转到 HTTPS；同时配置`HSTS`响应头，强制浏览器只能用 HTTPS 访问网站，防止降级劫持。
    
```http
    Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
```
    
2. **使用安全的 TLS 版本与加密套件**
    
    禁用 TLS 1.0、TLS 1.1，优先使用 TLS 1.3，兼容 TLS 1.2；禁用所有不安全的加密套件，只保留现代安全套件。
3. **规范的证书配置**
    
    - 选择全球可信的 CA 机构颁发的证书，不使用自签名证书、免费不可信证书；
    - 证书域名覆盖所有业务域名，避免证书名称不匹配错误；
    - 提前续期证书，避免证书过期导致网站无法访问；
    - 妥善保管服务器私钥，绝对不能泄露、不能提交到代码仓库。
    
4. **性能优化配置**
    
    - 开启会话复用（Session Ticket/ Session ID），减少重复握手的开销；
    - 开启 OCSP 装订，优化证书验证速度；
    - 配合 HTTP/2 或 HTTP/3，最大化传输性能。
    
5. **安全配套配置**
    - 给核心 Cookie 设置`Secure`属性，确保仅在 HTTPS 下传输；
    - 配置 CSP 内容安全策略、X-Content-Type-Options 等安全响应头，提升 Web 应用的整体安全性；
    - 定期用 SSL Labs 等工具，扫描 HTTPS 配置的安全等级，修复漏洞。
