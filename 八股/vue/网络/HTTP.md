**HTTP**（全称 **HyperText Transfer Protocol，超文本传输协议**）是互联网的核心应用层协议，基于 TCP/IP 协议簇，采用**请求 - 响应**的通信模型，负责客户端（最常见为浏览器）与 Web 服务器之间的超文本、数据、媒体等各类网络资源的传输，是万维网（WWW）的底层基石，也是前后端数据交互的核心载体。

它诞生之初仅用于传输 HTML 超文本，如今已成为通用的应用层协议，支持 JSON、图片、视频、音频、文件等几乎所有类型的资源传输，广泛用于 API 接口、网页加载、小程序 / App 网络请求等几乎所有 Web 相关场景。

## 一、核心基础特性

### 1. 经典请求 - 响应模型

HTTP 通信永远由**客户端主动发起请求**，服务器接收后处理并返回响应，不存在服务器主动向客户端推送数据的能力（HTTP/2 的服务器推送是特殊优化，并非原生双向通信，双向通信需依赖 WebSocket），一次请求对应一次响应，通信闭环清晰。

### 2. 无状态特性

HTTP 协议本身**不保留两次请求之间的任何上下文状态**，每一次请求都是完全独立的。服务器不会默认记录客户端的身份、历史请求信息，哪怕两次请求来自同一个客户端，服务器也会视为两个完全无关的请求。

这个特性带来了极强的扩展性和服务器容错能力，但也导致无法实现 “登录状态保持” 等业务需求，因此衍生出了**Cookie、Session、Token**等状态保持方案。

### 3. 可靠传输依赖

HTTP 默认基于**TCP 协议**实现传输，依靠 TCP 的三次握手、丢包重传、拥塞控制等机制，保证数据完整、有序、无差错地到达目的地，默认端口为**80**；其安全版本 HTTPS 基于 TLS/SSL 加密，默认端口为**443**。

### 4. 灵活可扩展

- 支持自定义请求方法、自定义 HTTP 头，可根据业务需求扩展能力（如 CORS 跨域、缓存控制、认证授权均通过 HTTP 头实现）
- 支持任意格式的资源传输，通过`Content-Type`头声明资源类型，无格式限制
- 向下兼容，新版本协议均保留对旧版本的兼容能力

## 二、HTTP 版本演进历程

HTTP 的版本迭代，核心始终围绕**解决性能瓶颈、提升安全性、增强传输能力**三大方向，目前主流使用的是 HTTP/1.1、HTTP/2，HTTP/3 正在快速普及。

| 版本       | 发布时间                  | 核心特性与优化                                                                                                                                                                                                                                 | 解决的核心问题                                                |
| -------- | --------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------ |
| HTTP/0.9 | 1991 年                | 原型版本，仅支持`GET`方法，只能传输纯 HTML 文本，无请求头 / 响应头、无状态码，请求完成立即断开 TCP 连接                                                                                                                                                                           | 实现最基础的超文本传输                                            |
| HTTP/1.0 | 1996 年                | 正式标准化，新增`POST`、`HEAD`方法；支持请求头 / 响应头、状态码；支持图片、视频等非 HTML 资源；默认短连接（每次请求新建一个 TCP 连接）                                                                                                                                                        | 完善协议能力，支持多媒体资源传输，奠定 HTTP 的基础框架                         |
| HTTP/1.1 | 1997 年                | 目前最广泛兼容的版本，核心优化：<br><br>1. 默认开启**长连接**（`Connection: keep-alive`），一个 TCP 连接可复用传输多次请求<br><br>2. 新增`PUT`、`DELETE`、`PATCH`、`OPTIONS`等方法（`OPTIONS`用于 CORS 预检请求）<br><br>3. 新增`Host`头，支持一台服务器部署多个网站<br><br>4. 完善缓存控制机制（`Cache-Control`）、支持断点续传 | 解决 HTTP/1.0 频繁新建 TCP 连接的性能损耗，完善 RESTful 语义，适配复杂 Web 业务 |
| HTTP/2   | 2015 年                | 基于谷歌 SPDY 协议，核心优化：<br><br>1. 二进制分帧传输，替代文本格式<br><br>2. **多路复用**：同一个 TCP 连接中可并行传输多个请求 / 响应，彻底解决 HTTP/1.1 的队头阻塞问题<br><br>3. HPACK 头部压缩，减少冗余传输<br><br>4. 支持服务器主动推送资源、流优先级控制                                                                 | 大幅提升高并发场景下的传输性能，降低延迟，默认基于 HTTPS 运行                     |
| HTTP/3   | 2022 年正式标准化（RFC 9114） | 底层抛弃 TCP，基于**QUIC 协议（UDP 封装）**，核心优化：<br><br>1. 彻底解决 TCP 层的队头阻塞，每个流独立传输，互不影响<br><br>2. 0-RTT/1-RTT 握手，大幅降低连接建立延迟<br><br>3. 连接迁移，移动端切换 Wi-Fi / 移动网络时连接不中断<br><br>4. 原生内置 TLS 加密，安全性更强                                                     | 解决 HTTP/2 在弱网、高丢包场景下的性能瓶颈，进一步降低传输延迟，适配移动互联网场景          |

> 补充：HTTP/1.1 的**队头阻塞**：同一个 TCP 连接中，请求必须按顺序处理，前一个请求未完成响应，后续所有请求只能排队等待，导致性能瓶颈。

## 三、核心报文结构

HTTP 的通信内容称为**报文**，分为**请求报文（客户端→服务器）**和**响应报文（服务器→客户端）**，两者结构高度一致，均由 4 部分组成。

### 1. 请求报文结构

```http
# 1. 请求行：方法 + 请求路径 + HTTP版本
GET /api/user?name=zhangsan HTTP/1.1
# 2. 请求头：键值对，描述请求的元信息
Host: api.example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) Chrome/120.0.0.0
Accept: application/json
Authorization: Bearer token123456
Content-Type: application/json
Content-Length: 40
# 3. 空行：固定分隔请求头和请求体，不可省略
# 4. 请求体：请求携带的业务数据，GET/HEAD请求通常无请求体
{"age": 20, "gender": "male"}
```

#### 常用请求头说明

|请求头|核心作用|
|---|---|
|`Host`|【HTTP/1.1 必填】请求的目标域名 + 端口，用于服务器区分虚拟主机|
|`User-Agent`|客户端标识（浏览器、设备、系统信息）|
|`Accept`|客户端能接收的资源类型（如`application/json`、`text/html`）|
|`Content-Type`|请求体的数据格式，常见：`application/json`、`multipart/form-data`、`application/x-www-form-urlencoded`|
|`Authorization`|认证信息，用于 Token、JWT 等身份校验|
|`Cookie`|客户端自动携带的 Cookie，用于状态保持|
|`Origin`|标记请求的来源域名，用于 CORS 跨域校验|

### 2. 响应报文结构

```http
# 1. 状态行：HTTP版本 + 状态码 + 状态描述
HTTP/1.1 200 OK
# 2. 响应头：键值对，描述响应的元信息
Content-Type: application/json; charset=utf-8
Content-Length: 65
Cache-Control: max-age=3600
Access-Control-Allow-Origin: https://example.com
Set-Cookie: sessionId=abc123; Path=/; HttpOnly
# 3. 空行：固定分隔响应头和响应体，不可省略
# 4. 响应体：服务器返回的业务资源/数据
{"code": 0, "message": "success", "data": {"userId": 1001}}
```

#### 常用响应头说明

|响应头|核心作用|
|---|---|
|`Content-Type`|响应体的数据格式与编码|
|`Content-Length`|响应体的字节长度|
|`Cache-Control`|缓存控制策略，控制浏览器缓存行为|
|`Set-Cookie`|向客户端写入 Cookie，用于状态保持|
|`Access-Control-Allow-*`|CORS 跨域相关头，控制跨域请求权限|
|`Location`|重定向目标地址，用于 3xx 状态码|
|`ETag` / `Last-Modified`|协商缓存标识，用于资源更新校验|
## 四、常用请求方法与语义

HTTP 方法定义了对目标资源的操作类型，具备明确的语义规范，是 RESTful API 设计的核心基础。

|方法|核心语义|特性|典型场景|
|---|---|---|---|
|`GET`|获取、查询资源|幂等、无请求体、可缓存、参数拼接在 URL|网页加载、数据查询接口|
|`POST`|提交数据、创建资源|非幂等、有请求体、默认不可缓存|用户注册、表单提交、文件上传|
|`PUT`|全量更新资源|幂等、有请求体|完整更新用户信息、覆盖资源|
|`PATCH`|部分更新资源|幂等、有请求体|仅更新用户昵称，不修改其他信息|
|`DELETE`|删除资源|幂等、通常无请求体|删除用户、删除文件|
|`HEAD`|与 GET 一致，仅返回响应头，不返回响应体|幂等、无响应体|检查资源是否存在、校验资源大小、预校验缓存|
|`OPTIONS`|获取服务器对目标资源支持的请求方法、跨域权限|幂等|CORS 预检请求、服务器能力探测|

> 幂等性：多次执行相同的请求，对服务器资源产生的影响与执行一次完全一致，是接口设计的重要规范。

## 五、标准状态码分类

HTTP 状态码是服务器对请求处理结果的标识，位于响应行，分为 5 大类，每一类都有明确的语义范围，客户端可通过状态码快速判断请求结果。

|分类|语义|常用核心状态码|
|---|---|---|
|1xx 信息性|请求已被服务器接收，正在处理，临时响应|100 Continue：继续发送请求体<br><br>101 Switching Protocols：协议切换（如 WebSocket 升级）|
|2xx 成功|请求已被服务器成功接收、处理并完成|200 OK：请求成功<br><br>201 Created：资源创建成功（POST）<br><br>204 No Content：请求成功，无响应体（DELETE/PUT）|
|3xx 重定向|需要客户端进一步操作才能完成请求|301 Moved Permanently：永久重定向<br><br>302 Found：临时重定向<br><br>304 Not Modified：协商缓存命中，资源无更新<br><br>307/308：严格临时 / 永久重定向，不允许修改请求方法|
|4xx 客户端错误|请求存在错误，服务器无法或拒绝处理|400 Bad Request：请求参数 / 格式错误<br><br>401 Unauthorized：未认证，需登录<br><br>403 Forbidden：已认证，无权限访问<br><br>404 Not Found：资源不存在<br><br>405 Method Not Allowed：请求方法不被允许<br><br>429 Too Many Requests：请求频率超限，限流|
|5xx 服务器错误|服务器处理请求时发生内部错误|500 Internal Server Error：服务器内部未知错误<br><br>502 Bad Gateway：网关 / 反向代理无法连接上游服务<br><br>503 Service Unavailable：服务暂时不可用（过载 / 维护）<br><br>504 Gateway Timeout：网关 / 代理请求上游服务超时|

## 六、核心配套机制

### 1. 缓存机制

HTTP 缓存是前端性能优化的核心手段，分为两大类，目的是减少重复网络请求，提升页面加载速度，降低服务器压力。

- **强缓存**：浏览器直接读取本地缓存，不向服务器发送任何请求，优先级最高。通过`Cache-Control: max-age=xxx`（相对过期时间）、`Expires`（绝对过期时间）控制。
- **协商缓存**：强缓存失效后，浏览器发送请求到服务器，服务器校验资源是否更新。无更新则返回`304 Not Modified`，浏览器复用本地缓存；有更新则返回`200 OK`和新资源。通过`ETag/If-None-Match`、`Last-Modified/If-Modified-Since`头控制。

### 2. 状态保持机制

针对 HTTP 无状态的特性，主流的状态保持方案：

- **Cookie**：服务器通过`Set-Cookie`头写入客户端的小型数据，后续请求会自动通过`Cookie`头携带，支持过期时间、域名、路径、`HttpOnly`（防 XSS 读取）、`Secure`（仅 HTTPS 传输）等安全配置。
- **Session**：基于 Cookie 实现，服务端存储用户状态，仅将 SessionId 写入客户端 Cookie，安全性更高。
- **Token/JWT**：无状态认证方案，客户端登录后获取 Token，后续请求通过`Authorization`头携带，服务端自行校验，无需存储会话信息，适配分布式、前后端分离架构。

### 3. 安全增强：HTTPS

HTTPS（HTTP Secure）是 HTTP 的安全版本，在 HTTP 与 TCP 之间新增了**TLS/SSL 加密层**，解决了 HTTP 明文传输的三大核心安全问题：

- 防窃听：数据全程加密，无法被第三方窃取
- 防篡改：数据完整性校验，无法被第三方修改
- 防伪造：基于数字证书验证服务器身份，防止钓鱼网站

目前主流浏览器已对 HTTP 网站标记 “不安全”，生产环境必须使用 HTTPS，HTTP/2、HTTP/3 也默认基于 HTTPS 运行。

### 4. 跨域控制：CORS

CORS（跨域资源共享）是基于 HTTP 头的浏览器安全机制，用于放宽同源策略的限制，允许受控地访问非同源资源，是目前解决前端跨域请求的标准方案，核心通过`Access-Control-Allow-*`系列响应头实现。

## 七、常见安全风险

1. **明文传输风险**：HTTP 明文传输数据，账号密码、业务数据极易被窃听、篡改、劫持，生产环境必须禁用 HTTP，强制使用 HTTPS。
2. **XSS 跨站脚本攻击**：恶意注入 JavaScript 脚本，窃取 Cookie、伪造用户操作，可通过`HttpOnly`Cookie、输入输出过滤、CSP 内容安全策略防护。
3. **CSRF 跨站请求伪造**：利用 Cookie 自动携带的特性，在恶意网站伪造用户请求，可通过 CSRF Token、同源校验、`SameSite`Cookie 防护。
4. **点击劫持**：通过 iframe 嵌套目标 HTTP 页面，诱导用户点击，可通过`X-Frame-Options`响应头禁止页面被 iframe 嵌套防护。
5. **HTTP 头注入攻击**：恶意构造请求头 / 响应头，实现 Cookie 篡改、跨站脚本、缓存投毒等攻击。
