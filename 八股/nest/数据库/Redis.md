## 常见用途

**缓存热点数据**

热点数据就是经常被访问的数据。

比如：

```text
商品详情
用户资料
文章内容
配置项
```

如果每次都查 MySQL，数据库压力会很大。可以先查 Redis：

```text
请求商品详情
  -> 先查 Redis
  -> 有：直接返回
  -> 没有：查 MySQL
  -> 写入 Redis
  -> 返回
```

**登录 session**

用户登录后，服务端可以把登录状态存在 Redis。

比如：

```
sessionId -> userId
```

前端 cookie 里保存 sessionId，后端每次请求用它去 Redis 查用户身份。

```text
Cookie: sessionId=abc123
Redis:
abc123 -> userId: 1001
```

**验证码**

短信验证码、邮箱验证码通常有有效期，比如 5 分钟。

Redis 很适合存这种临时数据：

```
login:code:13800138000 -> 839201
过期时间：5 分钟
```

验证成功或过期后就没了。

**限流计数**

限制用户或 IP 在一段时间内请求次数。

比如：

```text
同一个 IP 1 分钟最多请求 60 次
同一个手机号 1 分钟最多发 1 次验证码
```

Redis 可以做计数：

```
rate:ip:1.2.3.4 -> 42
过期时间：60 秒
```

超过阈值就拒绝请求。

**分布式锁**

当系统有多台服务器时，可能多个服务同时操作同一份数据。

比如：

```
多个 worker 同时处理同一个订单
多个请求同时抢同一张优惠券
```

Redis 可以用一个 key 当锁：

```
lock:order:123 -> locked
```

谁先设置成功，谁拿到锁；处理完再释放。

**排行榜**

比如：

```
游戏积分榜
文章热度榜
直播间贡献榜
学习打卡排名
```

Redis 有 Sorted Set，天然适合排行榜：

```
user:1001 score 980
user:1002 score 1200
user:1003 score 870
```

可以快速查：

```
前 10 名
某个用户排名
某个分数区间
```

**任务状态**

异步任务执行时间长，不能一直让 HTTP 请求等着。

比如：

```
生成报告
AI Agent 执行任务
视频转码
导入 Excel
```

可以把任务状态放 Redis：

```
task:abc123 -> running
task:abc123:progress -> 67%
```

前端轮询接口查状态。

**消息队列**

Redis 也可以做简单消息队列。

比如：

```
用户上传文件
  -> 后端把任务丢进队列
  -> worker 从队列取任务
  -> 后台慢慢处理
```

常见用途：

```
发邮件
发短信
生成报告
处理图片
AI 后台任务
```

不过复杂系统更常用专门消息队列，比如 RabbitMQ、Kafka、RocketMQ、BullMQ。

一句话总结：

```
Redis 适合存快、临时、高频、需要过期或计数的数据。
```

而 MySQL/PostgreSQL 更适合存长期、可靠、结构化的核心数据。

## 缓存问题

这是 Redis 缓存里的几个经典问题。你可以先把它们理解成：**缓存虽然快，但缓存和数据库之间多了一层，就会带来失效、一致性和高并发风险。**

**缓存过期**

Redis 里的缓存通常会设置过期时间：

```
user:1 -> 用户信息，10 分钟后过期
```

过期后，下次请求发现 Redis 没有，就要去查数据库，再写回 Redis。

问题是：

```
过期时间太短：频繁查数据库
过期时间太长：数据可能不新
```

比如商品价格改了，但 Redis 里还缓存着旧价格。

常见处理：

```
设置合理 TTL
更新数据库后删除缓存
热点数据续期
重要数据不用长时间缓存
```

**缓存和数据库不一致**

典型场景：

```
数据库里的用户昵称已经改了
Redis 里还是旧昵称
```

因为数据有两份：

```
数据库：真实数据
Redis：缓存副本
```

更新时如果只改数据库，不处理缓存，就会读到旧数据。

常见做法：

```
先更新数据库，再删除缓存
```

而不是直接更新缓存。

流程：

```
修改用户昵称
  -> 更新 MySQL
  -> 删除 Redis 里的 user:1
  -> 下次查询时重新从 MySQL 加载
```

这叫 **Cache Aside Pattern**，最常用。

**缓存击穿**

缓存击穿指：**某一个热点 key 突然失效，大量请求同时打到数据库。**

比如：

```
首页爆款商品 product:1001 很多人访问
这个 key 刚好过期
瞬间 1 万个请求都查 Redis 失败
于是一起查 MySQL
数据库压力暴增
```

特点：

```
一个热点 key 失效
大量并发请求打数据库
```

常见解决：

```
热点 key 不过期
加互斥锁，只允许一个请求查数据库
逻辑过期，后台刷新缓存
提前异步刷新热点缓存
```

简单锁思路：

```
请求发现缓存没了
  -> 抢锁
  -> 抢到锁的查数据库并写缓存
  -> 没抢到的等一下再读缓存
```

**缓存穿透**

缓存穿透指：**请求的数据在缓存和数据库里都不存在。**

比如攻击者一直请求：

```
/product/-1
/product/999999999999
/user/not-exist
```

Redis 查不到，MySQL 也查不到。每次都绕过缓存打到数据库。

特点：

```
查不存在的数据
缓存永远没命中
数据库一直被打
```

常见解决：

```
缓存空值
布隆过滤器
参数校验
接口限流
```

缓存空值例子：

```
查 user:999999
数据库也没有
Redis 存 user:999999 -> null，过期时间 1 分钟
```

这样短时间内重复请求不会再查数据库。

**缓存雪崩**

缓存雪崩指：**大量缓存 key 在同一时间过期，导致大量请求一起打到数据库。**

比如你给很多 key 都设置：

```
TTL = 10 分钟
```

服务启动时批量写入缓存，10 分钟后它们一起过期。

结果：

```
Redis 大量未命中
请求全部打到 MySQL
数据库被压垮
```

特点：

```
不是一个 key
是大量 key 同时失效
```

常见解决：

```
TTL 加随机值
热点数据分批过期
多级缓存
提前刷新缓存
限流和降级
Redis 高可用
```

TTL 加随机值：

```
基础过期时间 10 分钟
随机加 0-5 分钟
```

这样不会同一时间全部过期。

**三者区别**

```
缓存击穿：
一个热点 key 过期，大量请求打数据库。

缓存穿透：
请求不存在的数据，缓存和数据库都没有。

缓存雪崩：
大量 key 同时过期，大量请求打数据库。
```

记忆方式：

```
击穿：打穿一个热点
穿透：查不存在，穿过缓存直达数据库
雪崩：一大片缓存同时倒下
```

**最常用的一套缓存方案**

实际项目里常用：

```
读数据：
  先查 Redis
  有就返回
  没有就查数据库
  数据库有：写 Redis，设置随机 TTL
  数据库没有：缓存空值，短 TTL

写数据：
  先更新数据库
  再删除缓存
```

再加上：

```
热点 key 加锁或逻辑过期
接口限流
监控 Redis 命中率
监控数据库慢查询
```

一句话总结：

```
缓存提升性能，但必须处理过期、一致性和高并发下的失效问题。
```

## 限流

Redis 是目前最主流的分布式限流实现方案，凭借其**高性能原子操作**、**丰富数据结构**和**天然分布式支持**，能高效解决高并发场景下的流量控制问题，保护系统不被突发流量击垮。

**什么是限流**

限流是通过控制请求的**速率**或**并发数**，确保系统在承载能力范围内稳定运行的技术手段。当请求超过预设阈值时，会触发拒绝服务、排队等待或服务降级等策略。

**为什么用 Redis 做限流**

- **高性能**：纯内存操作，单节点 QPS 可达 10 万 +
- **原子性**：提供 INCR、EXPIRE、ZADD 等原子命令，避免并发竞争问题
- **分布式友好**：集中式存储，天然支持多实例集群的统一限流
- **灵活度高**：支持多种限流算法和粒度（IP、用户、接口、服务）

**典型应用场景**

- API 接口防刷（如短信验证码、登录接口）
- 秒杀、抢购等突发流量场景
- 微服务间的调用保护
- 爬虫、恶意攻击防护
- 消息队列消费速率控制

### 1. 固定窗口计数器算法（最简单）

**原理**

将时间划分为固定大小的窗口（如 1 分钟），每个窗口内维护一个计数器，每来一个请求计数器加 1。当计数器达到阈值时，拒绝该窗口内的后续请求。窗口过期后计数器自动重置。

**Redis 实现**

使用`INCR`+`EXPIRE`原子组合，key 为 "limit:{资源标识}:{时间窗口}"，过期时间等于窗口大小。

```javascript
// JavaScript实现（ioredis）
async function fixedWindowLimit(resource, windowSize, maxRequests) {
  const key = `limit:fixed:${resource}:${Math.floor(Date.now() / windowSize)}`;
  const count = await redis.incr(key);
  
  if (count === 1) {
    // 第一次请求时设置过期时间
    await redis.expire(key, windowSize / 1000);
  }
  
  return count <= maxRequests;
}

// 使用示例：限制用户123每分钟最多100次请求
const allowed = await fixedWindowLimit("user:123", 60 * 1000, 100);
if (!allowed) {
  throw new Error("请求过于频繁，请稍后再试");
}
```

**优缺点**

- ✅ 优点：实现最简单，性能最高，内存占用小
- ❌ 缺点：存在**临界突刺问题**—— 两个相邻窗口的交界处可能出现双倍流量
    
    - 例如：1 分钟窗口，59 秒 500ms 来了 100 个请求，0 秒 0ms 又来 100 个请求，1 秒内实际通过 200 个请求

### 2. 滑动窗口计数器算法（解决临界问题）

**原理**

将固定窗口拆分为多个更小的时间格子（如 1 分钟拆分为 6 个 10 秒格子），每次请求记录当前格子的计数。统计时，只计算**最近一个完整窗口**内所有格子的总和。随着时间推移，旧的格子会被自动丢弃。

**Redis 实现**

使用**有序集合 (ZSet)** 实现，member 为请求唯一标识（或时间戳），score 为请求时间戳。每次请求时：

1. 删除窗口外的所有历史记录
2. 添加当前请求记录
3. 统计集合内元素数量是否超过阈值

```javascript
async function slidingWindowLimit(resource, windowSize, maxRequests) {
  const key = `limit:sliding:${resource}`;
  const now = Date.now();
  const windowStart = now - windowSize;
  
  // 使用Lua脚本保证原子性（关键！避免并发问题）
  const luaScript = `
    redis.call('ZREMRANGEBYSCORE', KEYS[1], 0, ARGV[1])
    local count = redis.call('ZCARD', KEYS[1])
    if count < tonumber(ARGV[2]) then
      redis.call('ZADD', KEYS[1], ARGV[3], ARGV[3])
      redis.call('EXPIRE', KEYS[1], ARGV[4])
      return 1
    end
    return 0
  `;
  
  const result = await redis.eval(luaScript, 1, key, 
    windowStart, maxRequests, now, windowSize / 1000);
  
  return result === 1;
}
```

**优缺点**

- ✅ 优点：解决了固定窗口的临界突刺问题，限流精度更高
- ❌ 缺点：内存占用比固定窗口大，ZSet 操作性能略低；当窗口很大或请求很多时，会占用较多 Redis 资源

### 3. 漏桶算法（平滑流量）

**原理**

将请求比作水，先流入漏桶中，漏桶以**固定速率**流出水（处理请求）。当桶满时，新流入的水会溢出（拒绝请求）。

核心特点：**强制控制流出速率**，无论输入流量多么突发，输出流量始终保持平稳。

**Redis 实现**

可以使用 List 模拟队列（桶），或者使用 Redis 4.0 + 官方推荐的**Redis Cell 模块**（基于令牌桶算法，但提供了漏桶的语义）。

```javascript
// Redis Cell模块实现（推荐）
// 安装：redis-cli MODULE LOAD /path/to/redis-cell.so
async function leakyBucketLimit(resource, capacity, leakRate) {
  // CL.THROTTLE key max_bucket_count refill_time refill_amount take_count
  // 参数解释：
  // max_bucket_count: 桶的最大容量
  // refill_time: 令牌完全填满的时间（秒）
  // refill_amount: 每次填充的令牌数
  // take_count: 本次请求消耗的令牌数
  const result = await redis.call('CL.THROTTLE', 
    `limit:leaky:${resource}`, capacity, capacity, leakRate, 1);
  
  // result[0]: 0=允许，1=拒绝
  // result[1]: 桶的最大容量
  // result[2]: 桶中剩余令牌数
  // result[3]: 下次可以重试的时间（秒）
  // result[4]: 桶完全填满需要的时间（秒）
  return result[0] === 0;
}

// 使用示例：桶容量100，每秒流出10个请求
const allowed = await leakyBucketLimit("api:pay", 100, 10);
```

**优缺点**

- ✅ 优点：流量绝对平滑，不会出现突刺；能有效保护下游系统
- ❌ 缺点：无法应对突发流量；当流量超过平均速率时，会有大量请求被拒绝

### 4. 令牌桶算法（支持突发流量）

**原理**

系统以**固定速率**向桶中放入令牌，请求到来时需要先从桶中获取一个令牌才能执行。如果桶中没有令牌，则拒绝请求。桶的容量决定了最大突发流量。

核心特点：**允许一定程度的突发流量**，同时也能控制平均速率。

**Redis 实现**

Redis Cell 模块本质上就是令牌桶算法的实现，上面的漏桶示例其实也可以作为令牌桶使用。区别在于参数配置：

- 漏桶：容量小，填充速率慢
- 令牌桶：容量大，填充速率适中

```javascript
// 令牌桶示例：允许每秒10个请求，最大突发20个
const allowed = await leakyBucketLimit("api:query", 20, 10);
```

**优缺点**

- ✅ 优点：既能控制平均速率，又能应对突发流量；灵活性高
- ❌ 缺点：实现相对复杂；如果突发流量过大，可能会导致下游系统压力骤增

### 四种算法对比

|算法|实现难度|性能|临界问题|突发流量支持|适用场景|
|---|---|---|---|---|---|
|固定窗口|简单|极高|有|不支持|要求不高的简单限流|
|滑动窗口|中等|高|无|不支持|精度要求较高的限流|
|漏桶|中等（用 Redis Cell 简单）|高|无|不支持|需要严格平滑流量的场景|
|令牌桶|中等（用 Redis Cell 简单）|高|无|支持|大多数互联网应用场景|


**1. 原子性保证**

所有限流操作必须是原子的，否则在高并发下会出现计数不准的问题。Redis 提供了两种解决方案：

- 使用 Lua 脚本：将多个 Redis 命令打包成一个原子操作
- 使用 Redis 原生原子命令：如 INCR、ZADD 等

**2. 时间同步问题**

分布式系统中，各个服务器的时间可能存在偏差。解决方案：

- 使用 Redis 服务器的时间（通过`TIME`命令获取），而不是客户端时间
- 允许一定的时间误差（通常在几秒内）

**3. Redis 集群问题**

在 Redis 集群环境下，限流 key 可能分布在不同的节点上，导致无法统一计数。解决方案：

- 使用 HashTag 将限流 key 固定到同一个节点上：`limit:{user:123}`
- 使用 Redis Cluster 的哈希槽特性

**4. Redis 故障降级**

当 Redis 不可用时，限流功能会失效。需要设计降级策略：

- 本地限流兜底：使用 Guava RateLimiter 等本地限流组件
- 熔断机制：当 Redis 故障次数达到阈值时，暂时关闭限流或降级为本地限流

## Cache Aside

**Cache Aside** 也叫 **旁路缓存模式**，是业务系统最常用的缓存模式。

核心思想是：

```
应用自己负责读写缓存
数据库仍然是最终可信数据源
```

也就是：

```
应用 -> 先查 Redis
应用 -> Redis 没有再查数据库
应用 -> 查到数据库后写回 Redis
```

**读流程**

比如查询用户 user:1：

```
1. 先查 Redis：GET user:1
2. 如果 Redis 有，直接返回
3. 如果 Redis 没有，查数据库
4. 数据库查到后，写入 Redis
5. 返回数据
```

伪代码：

```ts
async function getUser(id: number) {
  const key = `user:${id}`;

  const cached = await redis.get(key);
  if (cached) {
    return JSON.parse(cached);
  }

  const user = await db.user.findUnique({ where: { id } });

  if (user) {
    await redis.set(key, JSON.stringify(user), 'EX', 600);
  }

  return user;
}
```

这里 Redis 只是加速层，数据库才是真正的数据源。

**写流程**

常见做法是：

```
先更新数据库
再删除缓存
```

例如：

```ts
async function updateUser(id: number, data: UpdateUserDto) {
  const user = await db.user.update({
    where: { id },
    data,
  });

  await redis.del(`user:${id}`);

  return user;
}
```

为什么是删除缓存，而不是更新缓存？

因为删除更简单、更稳。

如果直接更新缓存，可能遇到：

```
缓存里是复杂聚合数据
多个接口都会影响同一个缓存
并发写入顺序错乱
缓存字段和数据库字段不完全一致
```

删除缓存后，下次读取会重新查数据库并重建缓存。

**完整流程图**

```
读数据：

请求
  -> 查 Redis
      -> 命中：返回缓存
      -> 未命中：查数据库
             -> 写入 Redis
             -> 返回数据
```

```
写数据：

请求
  -> 更新数据库
  -> 删除 Redis 缓存
  -> 返回结果
```

**为什么不先删缓存再更新数据库**

假设你这样做：

```
1. 删除缓存
2. 更新数据库
```

可能发生并发问题：

```
线程 A 删除缓存
线程 B 查询缓存，发现没有
线程 B 查数据库，读到旧数据
线程 B 把旧数据写回缓存
线程 A 更新数据库为新数据
```

结果：

```
数据库是新数据
缓存是旧数据
```

所以更常见的是：

```
先更新数据库，再删除缓存
```

这样整体风险更低。

**但是先更新数据库再删缓存也不是绝对安全**

极端情况下也可能有问题：

```
线程 A 查询缓存未命中
线程 A 查数据库，读到旧值
线程 B 更新数据库
线程 B 删除缓存
线程 A 把旧值写回缓存
```

结果缓存又变旧了。

这个情况概率较低，但高并发场景要处理。

常见解决：

```
缓存设置较短 TTL
延迟双删
加分布式锁
用消息队列异步删缓存
更新时带版本号
强一致场景不用缓存或走事务/锁
```

**延迟双删**

流程：

```
1. 更新数据库
2. 删除缓存
3. 等一小段时间
4. 再删除一次缓存
```

伪代码：

```ts
await db.user.update(...);
await redis.del(`user:${id}`);

setTimeout(async () => {
  await redis.del(`user:${id}`);
}, 500);
```

目的是清理掉并发读可能写回来的旧缓存。

注意：真实项目里不建议直接依赖进程内 setTimeout 做关键任务，更稳的是用消息队列或延迟任务。

**Cache Aside 的优点**

```
简单
最常用
业务可控
缓存挂了还能查数据库
适合大多数读多写少场景
```

**Cache Aside 的缺点**

```
代码里要自己处理缓存逻辑
可能出现短暂不一致
缓存未命中时数据库压力会上来
需要处理穿透、击穿、雪崩
```

**适合的场景**

适合：

```
用户资料缓存
商品详情缓存
文章详情缓存
权限配置缓存
系统配置缓存
热点查询结果
```

不适合：

```
强一致金融余额
库存强扣减核心逻辑
不能容忍旧数据的业务
复杂多表频繁变更数据
```

**常见 key 设计**

```
user:1
product:1001
article:88
config:payment
permission:user:1
```

推荐加业务前缀：

```
cache:user:1
cache:product:1001
```

**TTL 要加随机值**

不要所有缓存都设置一样的过期时间。

比如不要全是：

```
600 秒
```

可以：

```
600 + random(0, 120)
```

避免大量 key 同一时间过期，造成缓存雪崩。

伪代码：

```ts
const ttl = 600 + Math.floor(Math.random() * 120);
await redis.set(key, JSON.stringify(user), 'EX', ttl);
```

**缓存空值防穿透**

如果数据库查不到，也可以短时间缓存空值：

```ts
if (!user) {
  await redis.set(key, 'null', 'EX', 60);
  return null;
}
```

下次再查同一个不存在的用户，就不会每次都打数据库。

读取时：

```ts
const cached = await redis.get(key);

if (cached === 'null') {
  return null;
}

if (cached) {
  return JSON.parse(cached);
}
```

**一句话总结**

Cache Aside 就是：

```
读：先缓存，缓存没有查数据库，再写回缓存
写：先更新数据库，再删除缓存
```

它牺牲一点点强一致性，换来高性能和简单可控，是实际项目里最常用的 Redis 缓存模式。
