**数据库事务并发问题**，也就是多个事务同时读写同一批数据时产生的数据一致性问题。

它们通常和 **事务隔离级别** 一起讲。

**常见并发读写问题有 4 个：**

```
脏读
不可重复读
幻读
丢失更新
```

**1. 脏读**

脏读是：**一个事务读到了另一个事务还没提交的数据**。

例子：

```
事务 A：把用户余额从 100 改成 50，但还没提交
事务 B：查询余额，看到 50
事务 A：回滚，余额又变回 100
```

这时事务 B 读到的 50 就是脏数据。

因为它读到了一个最终不存在的数据。

脏读通常发生在最低隔离级别：

```
Read Uncommitted，读未提交
```

实际项目一般很少用这个级别。

**2. 不可重复读**

不可重复读是：**同一个事务内，两次读取同一行数据，结果不一样**。

例子：

```
事务 A：第一次查询余额 = 100
事务 B：把余额改成 80，并提交
事务 A：第二次查询余额 = 80
```

事务 A 在同一个事务里，两次读到的结果不同。

这叫不可重复读。

它关注的是：

```
同一行数据被别人修改了
```

**3. 幻读**

幻读是：**同一个事务内，两次按条件查询，查到的记录数量不一样**。

例子：

```
事务 A：查询 age > 18 的用户，有 10 条
事务 B：新增一个 age = 20 的用户，并提交
事务 A：再次查询 age > 18 的用户，变成 11 条
```

事务 A 感觉像出现了一条“幻影记录”，所以叫幻读。

它关注的是：

```
符合条件的记录新增或删除了
```

不可重复读关注“某一行被改了”。

幻读关注“某个范围内多了或少了行”。

**4. 丢失更新**

丢失更新是：**多个事务同时更新同一条数据，后提交的覆盖了先提交的结果**。

例子：

```
库存 count = 10

事务 A 读到 count = 10
事务 B 读到 count = 10

事务 A 更新 count = 9
事务 B 也更新 count = 9
```

明明扣了两次库存，最后库存只少了 1。

正确结果应该是：

```
count = 8
```

这就是丢失更新。

解决方式：

```
1. 数据库行锁
2. 乐观锁 version
3. update count = count - 1 where count > 0
4. 分布式锁
```

**事务隔离级别**

数据库通过隔离级别来控制这些问题。

常见 4 种：

```
Read Uncommitted：读未提交
Read Committed：读已提交
Repeatable Read：可重复读
Serializable：串行化
```

从弱到强：

```
读未提交 < 读已提交 < 可重复读 < 串行化
```

隔离越强，一致性越好，但并发性能通常越差。

**1. Read Uncommitted：读未提交**

可以读到别人还没提交的数据。

可能出现：

```
脏读
不可重复读
幻读
丢失更新
```

很少使用。

**2. Read Committed：读已提交**

只能读到别人已经提交的数据。

可以避免：

```
脏读
```

但可能出现：

```
不可重复读
幻读
```

很多数据库默认是这个，比如 Oracle、PostgreSQL。

**3. Repeatable Read：可重复读**

同一个事务内，多次读取同一行结果一致。

可以避免：

```
脏读 不可重复读
```

但理论上可能出现：

```
幻读
```

不过 MySQL InnoDB 的 Repeatable Read 通过 MVCC + next-key lock，在很多场景下可以避免幻读。

MySQL 默认隔离级别就是：

```
Repeatable
Read
```

**4. Serializable：串行化**

最强隔离级别。

事务像排队一样一个个执行。

可以避免：

```
脏读
不可重复读
幻读
```

缺点是并发性能最低，容易阻塞。
