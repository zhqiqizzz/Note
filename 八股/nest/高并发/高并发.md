这些都是在“**不能只靠 Redis 锁**”时常用的并发控制和一致性手段。可以按一句话理解：

```
Redis 锁偏“挡并发入口”
数据库事务/唯一索引/乐观锁偏“保证数据最终写对”
ZooKeeper/etcd 偏“强一致协调”
状态机偏“控制业务状态流转”
MQ 串行化偏“把并发改成排队”
```

**1. 数据库事务**

数据库事务用来保证一组 SQL 要么全部成功，要么全部失败。

比如下单扣库存：

```
1. 创建订单
2. 扣减库存
3. 写支付流水
```

这几步必须放在一个事务里：

```sql
BEGIN;

INSERT INTO orders ...;

UPDATE stock
SET count = count - 1
WHERE product_id = 1001
  AND count > 0;

COMMIT;
```

如果中间失败，就回滚：

```sql
ROLLBACK;
```

事务主要保证：

```
原子性：要么都成功，要么都失败
一致性：数据从一个正确状态到另一个正确状态
隔离性：并发事务之间互不乱套
持久性：提交后数据落库
```

适合场景：

```
订单创建
余额扣减
库存扣减
积分变更
支付流水
```

注意：

```
事务只能保证数据库内部的一致性
跨 Redis、MQ、第三方接口时，事务管不了全部
事务不要包太大，不要在事务里调用远程接口
```

**2. 唯一索引**

唯一索引是数据库层面的“最后一道防线”。

它能保证某个字段或字段组合不能重复。

比如防止用户重复下单：

```sql
CREATE UNIQUE INDEX uk_user_product
ON seckill_order(user_id, product_id);
```

这样同一个用户对同一个商品只能有一条秒杀订单。

即使应用层并发没挡住，两个请求同时进来：

```
请求 A 插入成功
请求 B 插入失败，数据库报唯一索引冲突
```

适合场景：

```
防止重复注册
防止重复下单
防止重复支付回调
防止幂等请求重复落库
```

常见设计：

```
订单号唯一
支付流水号唯一
业务请求号 request_id 唯一
用户 + 活动 ID 唯一
用户 + 商品 ID 唯一
```

唯一索引的优点是非常可靠，因为它在数据库层面生效。

缺点是：

```
只能防重复，不能处理复杂业务流程
高并发冲突时会有数据库压力
需要合理设计唯一键
```

**3. 乐观锁版本号**

乐观锁的思想是：**我先不加锁，更新时检查数据有没有被别人改过**。

通常用 version 字段。

表结构：

```sql
stock_id | count | version
1001     | 10    | 1
```

查询时拿到：

```
count = 10
version = 1
```

更新时带上版本号：

```sql
UPDATE stock
SET count = count - 1,
    version = version + 1
WHERE stock_id = 1001
  AND version = 1
  AND count > 0;
```

如果更新成功，说明没人改过。

如果返回影响行数是 0，说明：

```
库存不足
或者 version 已经被别人改了
```

适合场景：

```
库存扣减
余额变更
配置更新
订单状态更新
```

还有一种更简单的库存扣减：

```sql
UPDATE stock
SET count = count - 1
WHERE product_id = 1001
  AND count > 0;
```

这也是一种常见的数据库层面防超卖方式。

乐观锁适合：

```
读多写少
冲突不太激烈
不想长时间持锁
```

不适合：

```
冲突非常高的秒杀热点
大量请求反复重试
```

因为冲突越高，失败重试越多。

**4. ZooKeeper**

ZooKeeper 是一个分布式协调服务，常用来做：

```
分布式锁
服务注册发现
配置管理
Leader 选举
分布式协调
```

ZooKeeper 的特点是强一致协调能力比较好。

它做分布式锁通常依赖：

```
临时节点
顺序节点
watch 机制
```

简单理解：

```
多个客户端去 ZooKeeper 创建临时顺序节点
谁的节点序号最小，谁拿到锁
没拿到锁的客户端监听前一个节点
前一个节点删除后，后一个客户端获得锁
```

例如：

```
/lock/order/seq-000001  A
/lock/order/seq-000002  B
/lock/order/seq-000003  C
```

A 拿锁，B 监听 A，C 监听 B。

A 执行完删除节点，B 获得锁。

ZooKeeper 锁的优势：

```
顺序性更好
强一致协调能力强
客户端断开后临时节点会自动删除
适合 Leader 选举和分布式协调
```

缺点：

```
性能不如 Redis
使用和运维复杂一些
不适合极高频热点锁
```

**5. etcd**

etcd 和 ZooKeeper 类似，也是分布式强一致 KV 存储，基于 Raft 协议。

常见用途：

```
服务发现
配置中心
分布式锁
Leader 选举
Kubernetes 元数据存储
```

etcd 做锁通常依赖：

```
lease 租约
revision 版本
watch 监听
事务 CAS
```

简单理解：

```
客户端创建一个带租约的 key
租约会自动续期
客户端断开，租约过期，key 删除
其他客户端通过 revision 顺序竞争锁
```

适合场景：

```
云原生系统
Kubernetes 相关基础设施
需要强一致协调能力的系统
Leader 选举
配置协调
```

和 Redis 相比：

```
Redis 更快，更适合高频缓存和轻量锁
etcd 更强调一致性和协调语义
```

如果是金融核心状态、集群主节点选举、控制面元数据，etcd/ZooKeeper 通常比 Redis 更合适。

**6. 状态机**

状态机不是一个具体中间件，而是一种业务设计方法。

它用来控制业务状态只能按规定路径流转。

比如订单状态：

```
待支付 -> 已支付 -> 已发货 -> 已完成
待支付 -> 已取消
```

不能出现：

```
已取消 -> 已发货
已完成 -> 已支付
已发货 -> 待支付
```

数据库更新时可以这样写：

```sql
UPDATE orders
SET status = 'PAID'
WHERE order_id = 1001
  AND status = 'WAIT_PAY';
```

只有当前状态是 WAIT_PAY，才能变成 PAID。

如果影响行数是 0，说明状态已经变了，不能重复处理。

适合场景：

```
订单状态流转
支付状态流转
退款状态流转
审批流程
任务执行状态
物流状态
```

状态机能解决很多重复请求、乱序消息、并发状态修改问题。

比如支付回调重复来了两次：

```
第一次：WAIT_PAY -> PAID，成功
第二次：WHERE status = WAIT_PAY 不满足，更新失败
```

自然就幂等了。

核心思想：

```
不要只判断“我要改成什么”
还要判断“当前能不能从这个状态改过去”
```

**7. MQ 串行化**

MQ 串行化的思想是：**把并发请求变成排队处理**。

比如同一个商品的库存扣减，如果很多请求同时处理，很容易竞争。

可以把消息发到 MQ：

```
扣库存消息 1
扣库存消息 2
扣库存消息 3
扣库存消息 4
```

然后让同一个商品的消息进入同一个队列或同一个分区，由一个消费者顺序处理。

这样：

```
并发扣库存
变成
排队扣库存
```

适合场景：

```
秒杀扣库存
订单异步创建
账户流水处理
用户积分变更
需要按用户或商品维度顺序处理的任务
```

关键点是分区顺序。

比如 Kafka：

```
message key = productId
```

同一个 productId 的消息会进入同一个 partition，保证分区内有序。

RocketMQ 也可以使用顺序消息，把同一业务 key 发到同一个队列。

优点：

```
削峰
异步
顺序处理
降低数据库瞬时压力
```

缺点：

```
有延迟
系统复杂度提高
需要处理消息丢失、重复消费、消费失败
需要做幂等
```

**怎么选择？**

```
防重复：唯一索引
防超卖：数据库条件更新 / 乐观锁 / MQ 串行化
保证一组数据库操作同时成功失败：事务
控制状态不能乱跳：状态机
跨服务高一致协调：ZooKeeper / etcd
高并发削峰排队：MQ 串行化
短时间互斥执行：Redis / Redisson 锁
```

举个秒杀下单组合方案：

```
Redis 预扣库存，挡住大部分流量
MQ 异步创建订单，削峰
数据库唯一索引，防止一人多单
数据库库存条件更新，防止超卖
订单状态机，防止重复支付/取消乱序
消费者幂等，防止 MQ 重复消费
监控告警，发现积压和异常
```

真正的高并发系统一般不是只靠一种机制，而是多层兜底。
