## 常见用途

**缓存热点数据**

热点数据就是经常被访问的数据。

比如：

```text
商品详情
用户资料
文章内容
配置项
```

如果每次都查 MySQL，数据库压力会很大。可以先查 Redis：

```text
请求商品详情
  -> 先查 Redis
  -> 有：直接返回
  -> 没有：查 MySQL
  -> 写入 Redis
  -> 返回
```

**登录 session**

用户登录后，服务端可以把登录状态存在 Redis。

比如：

```
sessionId -> userId
```

前端 cookie 里保存 sessionId，后端每次请求用它去 Redis 查用户身份。

```text
Cookie: sessionId=abc123
Redis:
abc123 -> userId: 1001
```

**验证码**

短信验证码、邮箱验证码通常有有效期，比如 5 分钟。

Redis 很适合存这种临时数据：

```
login:code:13800138000 -> 839201
过期时间：5 分钟
```

验证成功或过期后就没了。

**限流计数**

限制用户或 IP 在一段时间内请求次数。

比如：

```text
同一个 IP 1 分钟最多请求 60 次
同一个手机号 1 分钟最多发 1 次验证码
```

Redis 可以做计数：

```
rate:ip:1.2.3.4 -> 42
过期时间：60 秒
```

超过阈值就拒绝请求。

**分布式锁**

当系统有多台服务器时，可能多个服务同时操作同一份数据。

比如：

```
多个 worker 同时处理同一个订单
多个请求同时抢同一张优惠券
```

Redis 可以用一个 key 当锁：

```
lock:order:123 -> locked
```

谁先设置成功，谁拿到锁；处理完再释放。

**排行榜**

比如：

```
游戏积分榜
文章热度榜
直播间贡献榜
学习打卡排名
```

Redis 有 Sorted Set，天然适合排行榜：

```
user:1001 score 980
user:1002 score 1200
user:1003 score 870
```

可以快速查：

```
前 10 名
某个用户排名
某个分数区间
```

**任务状态**

异步任务执行时间长，不能一直让 HTTP 请求等着。

比如：

```
生成报告
AI Agent 执行任务
视频转码
导入 Excel
```

可以把任务状态放 Redis：

```
task:abc123 -> running
task:abc123:progress -> 67%
```

前端轮询接口查状态。

**消息队列**

Redis 也可以做简单消息队列。

比如：

```
用户上传文件
  -> 后端把任务丢进队列
  -> worker 从队列取任务
  -> 后台慢慢处理
```

常见用途：

```
发邮件
发短信
生成报告
处理图片
AI 后台任务
```

不过复杂系统更常用专门消息队列，比如 RabbitMQ、Kafka、RocketMQ、BullMQ。

一句话总结：

```
Redis 适合存快、临时、高频、需要过期或计数的数据。
```

而 MySQL/PostgreSQL 更适合存长期、可靠、结构化的核心数据。

## 缓存问题

这是 Redis 缓存里的几个经典问题。你可以先把它们理解成：**缓存虽然快，但缓存和数据库之间多了一层，就会带来失效、一致性和高并发风险。**

**缓存过期**

Redis 里的缓存通常会设置过期时间：

```
user:1 -> 用户信息，10 分钟后过期
```

过期后，下次请求发现 Redis 没有，就要去查数据库，再写回 Redis。

问题是：

```
过期时间太短：频繁查数据库
过期时间太长：数据可能不新
```

比如商品价格改了，但 Redis 里还缓存着旧价格。

常见处理：

```
设置合理 TTL
更新数据库后删除缓存
热点数据续期
重要数据不用长时间缓存
```

**缓存和数据库不一致**

典型场景：

```
数据库里的用户昵称已经改了
Redis 里还是旧昵称
```

因为数据有两份：

```
数据库：真实数据
Redis：缓存副本
```

更新时如果只改数据库，不处理缓存，就会读到旧数据。

常见做法：

```
先更新数据库，再删除缓存
```

而不是直接更新缓存。

流程：

```
修改用户昵称
  -> 更新 MySQL
  -> 删除 Redis 里的 user:1
  -> 下次查询时重新从 MySQL 加载
```

这叫 **Cache Aside Pattern**，最常用。

**缓存击穿**

缓存击穿指：**某一个热点 key 突然失效，大量请求同时打到数据库。**

比如：

```
首页爆款商品 product:1001 很多人访问
这个 key 刚好过期
瞬间 1 万个请求都查 Redis 失败
于是一起查 MySQL
数据库压力暴增
```

特点：

```
一个热点 key 失效
大量并发请求打数据库
```

常见解决：

```
热点 key 不过期
加互斥锁，只允许一个请求查数据库
逻辑过期，后台刷新缓存
提前异步刷新热点缓存
```

简单锁思路：

```
请求发现缓存没了
  -> 抢锁
  -> 抢到锁的查数据库并写缓存
  -> 没抢到的等一下再读缓存
```

**缓存穿透**

缓存穿透指：**请求的数据在缓存和数据库里都不存在。**

比如攻击者一直请求：

```
/product/-1
/product/999999999999
/user/not-exist
```

Redis 查不到，MySQL 也查不到。每次都绕过缓存打到数据库。

特点：

```
查不存在的数据
缓存永远没命中
数据库一直被打
```

常见解决：

```
缓存空值
布隆过滤器
参数校验
接口限流
```

缓存空值例子：

```
查 user:999999
数据库也没有
Redis 存 user:999999 -> null，过期时间 1 分钟
```

这样短时间内重复请求不会再查数据库。

**缓存雪崩**

缓存雪崩指：**大量缓存 key 在同一时间过期，导致大量请求一起打到数据库。**

比如你给很多 key 都设置：

```
TTL = 10 分钟
```

服务启动时批量写入缓存，10 分钟后它们一起过期。

结果：

```
Redis 大量未命中
请求全部打到 MySQL
数据库被压垮
```

特点：

```
不是一个 key
是大量 key 同时失效
```

常见解决：

```
TTL 加随机值
热点数据分批过期
多级缓存
提前刷新缓存
限流和降级
Redis 高可用
```

TTL 加随机值：

```
基础过期时间 10 分钟
随机加 0-5 分钟
```

这样不会同一时间全部过期。

**三者区别**

```
缓存击穿：
一个热点 key 过期，大量请求打数据库。

缓存穿透：
请求不存在的数据，缓存和数据库都没有。

缓存雪崩：
大量 key 同时过期，大量请求打数据库。
```

记忆方式：

```
击穿：打穿一个热点
穿透：查不存在，穿过缓存直达数据库
雪崩：一大片缓存同时倒下
```

**最常用的一套缓存方案**

实际项目里常用：

```
读数据：
  先查 Redis
  有就返回
  没有就查数据库
  数据库有：写 Redis，设置随机 TTL
  数据库没有：缓存空值，短 TTL

写数据：
  先更新数据库
  再删除缓存
```

再加上：

```
热点 key 加锁或逻辑过期
接口限流
监控 Redis 命中率
监控数据库慢查询
```

一句话总结：

```
缓存提升性能，但必须处理过期、一致性和高并发下的失效问题。
```
