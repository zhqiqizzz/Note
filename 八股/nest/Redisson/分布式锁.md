**1. 普通可重入锁 RLock**

这是 Redisson 最常用的锁。

它类似 Java 里的 ReentrantLock，作用是：**同一时刻只允许一个线程/服务实例执行某段代码**。

适合场景：

```
防止重复下单
防止库存扣减并发冲突
防止缓存击穿时多个线程同时重建缓存
防止多个服务实例重复执行同一个定时任务
```

示例：

```java
RLock lock = redissonClient.getLock("lock:order:" + orderId);

try {
    boolean locked = lock.tryLock(3, 10, TimeUnit.SECONDS);
    if (!locked) {
        throw new RuntimeException("系统繁忙，请稍后再试");
    }

    doBusiness();
} finally {
    if (lock.isHeldByCurrentThread()) {
        lock.unlock();
    }
}
```

这里：

```
tryLock(3, 10, TimeUnit.SECONDS)
```

表示：

```
最多等 3 秒获取锁
拿到锁后 10 秒自动释放
```

**可重入是什么意思？**

同一个线程已经拿到锁后，可以再次拿同一把锁。

比如：

```java
public void createOrder() {
    lock.lock();
    try {
        deductStock();
    } finally {
        lock.unlock();
    }
}

public void deductStock() {
    lock.lock();
    try {
        // 扣库存
    } finally {
        lock.unlock();
    }
}
```

如果不是可重入锁，createOrder() 已经拿到锁，里面调用 deductStock() 再次加锁，就会把自己卡死。

Redisson 会记录当前线程的加锁次数：

```
第一次 lock，计数 = 1
第二次 lock，计数 = 2
unlock 一次，计数 = 1
再 unlock 一次，锁才真正释放
```

注意点：

```
加锁几次，就要释放几次
unlock 要放在 finally 里
unlock 前最好判断 isHeldByCurrentThread()
锁 key 的粒度要合理
```

锁粒度例子：

```
加锁几次，就要释放几次
unlock 要放在 finally 里
unlock 前最好判断 isHeldByCurrentThread()
锁 key 的粒度要合理
```

**2. Watchdog 自动续期**

严格说 watchdog 不是一种锁，而是 Redisson 普通锁的重要机制。

它解决的问题是：**业务还没执行完，锁先过期了**。

普通 Redis 锁通常这样：

```
SET lock:order:1 uuid NX EX 10
```

如果业务 15 秒才执行完，但锁 10 秒就过期了：

```
第 0 秒：线程 A 拿到锁
第 10 秒：锁过期
第 11 秒：线程 B 拿到锁
第 15 秒：线程 A 还在执行业务
```

这时 A 和 B 就并发执行了，锁失效。

Redisson 的 watchdog 会自动续期：

```java
lock.lock();
```

如果你没有指定锁释放时间，Redisson 会默认给锁设置一个过期时间，比如 30 秒，然后后台定时续期。

大致逻辑：

```
第 0 秒：A 拿到锁，锁过期时间 30 秒
第 10 秒：watchdog 检查锁还属于 A，续期到 30 秒
第 20 秒：继续续期
第 30 秒：继续续期
直到 A unlock
```

适合场景：

```
业务执行时间不稳定
外部接口耗时不确定
批处理任务耗时较长
不希望锁因为超时提前释放
```

注意：

```java
lock.lock(10, TimeUnit.SECONDS);
```

如果你手动指定了 leaseTime，Redisson 一般不会启动 watchdog。10 秒到了锁会自动释放。

所以：

```
业务时间确定：可以指定 leaseTime
业务时间不确定：用 lock() 或 tryLock(waitTime, unit)，让 watchdog 续期
```

**3. 公平锁 FairLock**

公平锁解决的问题是：**多个线程抢锁时，尽量按先来后到的顺序获取锁**。

普通锁不保证顺序。

比如：

```
A 先等待
B 后等待
C 最后等待
```

锁释放后，可能 C 先抢到。

公平锁会维护一个等待队列：

```
A -> B -> C
```

释放锁后，尽量让 A 先拿锁，再 B，再 C。

用法：

```java
RLock fairLock = redissonClient.getFairLock("lock:pay:" + orderId);

try {
    boolean locked = fairLock.tryLock(5, 10, TimeUnit.SECONDS);
    if (!locked) {
        throw new RuntimeException("排队超时");
    }

    doBusiness();
} finally {
    if (fairLock.isHeldByCurrentThread()) {
        fairLock.unlock();
    }
}
```

适合场景：

```
需要按请求顺序处理
支付、排队、抢占资源时希望更公平
用户体验要求“先来先处理”
```

不适合场景：

```
极高并发秒杀
对性能要求特别高
不关心顺序，只关心吞吐量
```

原因是公平锁要维护队列，性能比普通锁低。

另外，公平锁只能做到“尽量公平”，分布式环境里会受网络延迟、客户端超时、节点状态影响，不要把它当成绝对严格的业务排队系统。如果业务强依赖顺序，应该考虑 MQ 排队。

**4. 读写锁 ReadWriteLock**

读写锁解决的问题是：**读多写少时，提高并发能力**。

普通互斥锁是：

```
读和读互斥
读和写互斥
写和写互斥
```

但很多业务里，多个线程同时读是安全的，只有写的时候才需要互斥。

读写锁规则：

```
读 + 读：可以并发
读 + 写：互斥
写 + 写：互斥
```

用法：

```java
RReadWriteLock rwLock = redissonClient.getReadWriteLock("lock:config");

RLock readLock = rwLock.readLock();
RLock writeLock = rwLock.writeLock();
```

读操作：

```java
readLock.lock();
try {
    queryConfig();
} finally {
    readLock.unlock();
}
```

写操作：

```java
writeLock.lock();
try {
    updateConfig();
} finally {
    writeLock.unlock();
}
```

适合场景：

```
系统配置读取多、修改少
商品基础信息读取多、更新少
黑名单、白名单、权限配置
缓存刷新期间保护数据一致性
```

举例：

```
100 个请求同时读取系统配置，可以一起读
1 个请求正在更新系统配置，其他读写都要等
```

注意点：

```
读写锁适合读多写少
如果写很多，读写锁优势不明显
写锁期间会阻塞读请求
不要在持锁期间执行很慢的远程调用
```

**5. 联锁 MultiLock**

联锁解决的问题是：**一个业务需要同时锁住多个资源**。

比如转账：

```
账户 A 扣钱
账户 B 加钱
```

为了防止并发修改，需要同时锁住两个账户：

```
lock:account:A
lock:account:B
```

Redisson MultiLock 的逻辑是：

```
多把锁全部获取成功，才算整体加锁成功
只要其中一把失败，就释放已经拿到的锁
```

用法：

```java
RLock lock1 = redissonClient.getLock("lock:account:" + fromUserId);
RLock lock2 = redissonClient.getLock("lock:account:" + toUserId);

RLock multiLock = redissonClient.getMultiLock(lock1, lock2);

try {
    boolean locked = multiLock.tryLock(3, 10, TimeUnit.SECONDS);
    if (!locked) {
        throw new RuntimeException("系统繁忙");
    }

    transfer();
} finally {
    if (multiLock.isHeldByCurrentThread()) {
        multiLock.unlock();
    }
}
```

适合场景：

```
转账同时锁两个账户
订单操作同时锁订单和库存
多个资源必须作为一个整体修改
```

风险点：

```
锁越多，失败概率越高
容易造成等待时间变长
业务复杂后更难排查
```

一个重要建议：

```
多个资源加锁时，尽量固定加锁顺序
```

比如账户转账，永远按账户 ID 从小到大加锁：

```
先锁 account:1001
再锁 account:2002
```

避免 A 线程锁 1001 等 2002，B 线程锁 2002 等 1001，造成互相等待。

虽然 MultiLock 会帮你处理部分失败释放，但业务设计上仍然应该尽量降低多锁复杂度。

**6. 红锁 RedLock**

红锁解决的问题是：**单个 Redis 节点故障时，锁可能不可靠**。

普通 Redis 锁依赖一个 Redis 实例或一个 Redis 主节点。

问题场景：

```
线程 A 在 Redis 主节点加锁成功
Redis 主节点还没同步给从节点就宕机
从节点升级成新主节点
线程 B 在新主节点也加锁成功
```

这样 A 和 B 都以为自己拿到了锁。

RedLock 的思路是用多个相互独立的 Redis 节点投票。

比如 5 个 Redis 节点：

```
Redis1
Redis2
Redis3
Redis4
Redis5
```

客户端尝试分别加锁。

如果满足：

```
超过半数节点加锁成功，比如 3/5 总耗时小于锁有效期
```

才认为加锁成功。

用法示意：

```java
RLock lock1 = redissonClient1.getLock("lock:order:1");
RLock lock2 = redissonClient2.getLock("lock:order:1");
RLock lock3 = redissonClient3.getLock("lock:order:1");

RLock redLock = redissonClient.getRedLock(lock1, lock2, lock3);
```

适合场景：

```
对 Redis 单点故障更敏感
希望多个独立 Redis 节点共同决定锁状态
```

但要注意：RedLock 在业界有争议，不是所有高一致性场景都推荐用它。

如果业务对一致性要求非常高，比如金融核心交易，通常更应该考虑：

```
数据库事务
唯一索引
乐观锁版本号
ZooKeeper
etcd
状态机
MQ 串行化
```

而不是只依赖 Redis 锁。

**7. 信号量 Semaphore**

Redisson 还支持分布式信号量。

它不是“只允许一个人进”，而是：**允许固定数量的人同时进入**。

比如最多允许 10 个任务同时执行：

```java
RSemaphore semaphore = redissonClient.getSemaphore("semaphore:task");
semaphore.trySetPermits(10);

boolean acquired = semaphore.tryAcquire();

try {
    if (!acquired) {
        throw new RuntimeException("当前任务太多");
    }

    doTask();
} finally {
    if (acquired) {
        semaphore.release();
    }
}
```

适合场景：

```
限制同时导出文件的任务数
限制同时调用第三方接口的并发数
限制某类后台任务最多并行 N 个
```

它和限流不完全一样。

```
限流：限制一段时间内多少请求
信号量：限制同一时刻多少并发
```

**8. 闭锁 CountDownLatch**

Redisson 也有分布式闭锁。

它的作用是：**等待多个分布式任务都完成后，再继续执行后续逻辑**。

例如：

`主任务拆成 5 个子任务 5 个子任务分别在不同机器执行 等 5 个都完成后，主任务继续汇总结果`

用法示意：

`RCountDownLatch latch = redissonClient.getCountDownLatch("latch:import:task1"); latch.trySetCount(5); // 主任务等待 latch.await(); // 子任务完成时 latch.countDown();`

适合场景：

`批量导入分片处理 多个异步任务全部完成后汇总 分布式任务编排`

不过现在很多项目会用 MQ、任务调度框架、工作流引擎来做更复杂的任务编排。

---

**重点对比**

`RLock： 只允许一个线程执行，最常用。 Watchdog： 自动续期，防止业务没结束锁过期。 FairLock： 按等待顺序获取锁，公平但性能低。 ReadWriteLock： 读读共享，读写互斥，适合读多写少。 MultiLock： 同时获取多把锁，适合多个资源一起操作。 RedLock： 多个独立 Redis 节点多数成功才算成功，增强容错但有争议。 Semaphore： 同一时刻允许 N 个并发，不是互斥。 CountDownLatch： 等待多个任务都完成，再继续。`
