Redisson 的**可重入锁**就是 Redis 版的 ReentrantLock：同一个线程/同一个锁持有者已经拿到锁后，可以再次获取同一把锁，而不会把自己阻塞住。

**1. 为什么需要可重入**

比如业务方法 A 拿了锁，又调用方法 B，方法 B 也需要同一把锁：

```java
public void methodA() {
    lock.lock();
    try {
        methodB();
    } finally {
        lock.unlock();
    }
}

public void methodB() {
    lock.lock();
    try {
        // do something
    } finally {
        lock.unlock();
    }
}
```

如果锁不可重入，methodB() 会被自己卡死。  
可重入锁允许：

```
同一个线程重复 lock 多次
也必须 unlock 相同次数
锁才真正释放
```

**2. Redisson 基本用法**

```java
RLock lock = redissonClient.getLock("lock:order:123");

lock.lock();

try {
    // 临界区业务
} finally {
    lock.unlock();
}
```

带等待时间和租约时间：

```java
boolean locked = lock.tryLock(3, 10, TimeUnit.SECONDS);

if (!locked) {
    throw new RuntimeException("获取锁失败");
}

try {
    // 业务逻辑
} finally {
    lock.unlock();
}
```

含义：

```
最多等待 3 秒获取锁
拿到锁后 10 秒自动释放
```

**3. Redisson 可重入锁底层怎么存**

Redisson 不是简单用：

```bash
SET lock_key value NX EX 30
```

它用的是 Redis Hash。

大概结构：

```bash
key: lock:order:123

value:
{
  "clientId:threadId": 2
}
```

其中：

```
clientId:threadId 表示锁持有者
2 表示重入次数
```

第一次加锁：

```
lock:order:123
  clientA:thread1 = 1
```

同一线程再次加锁：

```
lock:order:123
  clientA:thread1 = 2
```

释放一次：

```
clientA:thread1 = 1
```

再释放一次：

```
计数归零，删除 lock key
```

所以它能判断：

```
是不是同一个线程
重入了几次
什么时候真正释放
```

**4. 加锁逻辑简化版**

Redisson 加锁大概是这个思路：

```
如果锁不存在：
  创建锁
  field = 当前客户端线程
  value = 1
  设置过期时间
  加锁成功

如果锁存在，并且持有者就是当前线程：
  重入次数 +1
  刷新过期时间
  加锁成功

如果锁存在，但不是当前线程：
  加锁失败，等待或重试
```

伪 Lua 逻辑：

```lua
if redis.call('exists', KEYS[1]) == 0 then
  redis.call('hincrby', KEYS[1], ARGV[2], 1)
  redis.call('pexpire', KEYS[1], ARGV[1])
  return nil
end

if redis.call('hexists', KEYS[1], ARGV[2]) == 1 then
  redis.call('hincrby', KEYS[1], ARGV[2], 1)
  redis.call('pexpire', KEYS[1], ARGV[1])
  return nil
end

return redis.call('pttl', KEYS[1])
```

返回 nil 表示加锁成功；返回剩余 TTL 表示别人持有锁。

**5. 解锁逻辑简化版**

解锁时：

```
如果当前线程不是锁持有者：
  抛 IllegalMonitorStateException

如果当前线程是持有者：
  重入次数 -1

如果次数仍 > 0：
  只刷新过期时间，不释放锁

如果次数 == 0：
  删除锁
  发布解锁消息，通知等待者
```

这就是为什么：

```
lock 几次，就必须 unlock 几次
```

否则锁不会真正释放。

**6. Watchdog 看门狗机制**

这是 Redisson 很重要的能力。

如果你这样加锁：

```java
lock.lock();
```

没有指定 leaseTime，Redisson 默认会启用 watchdog。

默认锁过期时间通常是：

```
30 秒
```

Redisson 会定期续期，大概每 10 秒续一次：

```
只要持锁线程还活着
就自动延长锁 TTL
```

这样可以避免：

```
业务执行超过 30 秒
锁自动过期
其他线程拿到锁
导致并发问题
```

如果应用宕机，watchdog 停止续期，锁最终会过期释放。

**7. 指定 leaseTime 时没有 watchdog**

如果你这样写：

```java
lock.lock(10, TimeUnit.SECONDS);
```

或：

```java
lock.tryLock(3, 10, TimeUnit.SECONDS);
```

表示你明确告诉 Redisson：

```
锁 10 秒后自动释放
```

这时一般不会启用 watchdog 自动续期。

所以如果业务可能超过 10 秒，就有风险：

```
锁先过期
业务还没执行完
别人又拿到锁
```

初学时记住：

```
不确定业务耗时：lock()，让 watchdog 续期
明确知道最长耗时：lock(leaseTime)
```

**8. 等待者怎么被唤醒**

Redisson 不只是傻傻轮询 Redis。

当锁释放时，会发布一条消息到 Redis Pub/Sub channel。

等待锁的线程订阅这个 channel：

```
锁释放
  -> publish unlock message
  -> 等待者收到通知
  -> 尝试重新抢锁
```

这样比一直 sleep 轮询更高效。

**9. tryLock 推荐用法**

生产里更推荐 tryLock，避免无限等待：

```java
RLock lock = redissonClient.getLock("lock:order:" + orderId);

boolean locked = lock.tryLock(3, TimeUnit.SECONDS);

if (!locked) {
    return;
}

try {
    // do business
} finally {
    if (lock.isHeldByCurrentThread()) {
        lock.unlock();
    }
}
```

这里：

```java
lock.isHeldByCurrentThread()
```

是为了避免当前线程没持有锁却 unlock，导致异常。

**10. 常见坑**

```
1. lock 后忘记 unlock
2. lock 多次，unlock 次数不够
3. 指定 leaseTime 太短，业务没做完锁就过期
4. finally 里不判断就 unlock，可能抛异常
5. 锁粒度太大，导致并发能力下降
6. 用锁解决本该用幂等/事务/队列解决的问题
```

**一句话总结**

Redisson 可重入锁的核心是：

```
Redis Hash 记录“当前持锁线程 + 重入次数”
Lua 保证加锁/解锁原子性
watchdog 自动续期防止业务未完成锁过期
Pub/Sub 通知等待者锁已释放
```

它比手写 SET NX EX 更完整，尤其适合 Java/Spring Boot 项目里的分布式锁场景。
