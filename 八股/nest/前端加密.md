RSA 公钥加密登录，本质上是：

```
前端拿到一个“只能加密的钥匙”
用它把密码加密成密文
后端拿着“唯一能解密的钥匙”把密文解开
然后后端再验证密码
```

它解决的问题是：**让密码不要以原文形式出现在请求体里。**

**1. 什么是加密**  
假设你的密码是：

```
123456
```

如果直接提交，网络请求里可能是：

```json
{
  "password": "123456"
}
```

加密之后，可能变成：

```json
{
  "password": "aB8xK92...一大串看不懂的字符..."
}
```

这个“看不懂的一大串”就叫**密文**。

原始的 `123456` 叫**明文**。

```
明文：123456
加密：把明文变成密文
密文：aB8xK92...
解密：把密文还原成明文
```

---

**2. 什么是 RSA**  
RSA 是一种**非对称加密**算法。

“非对称”的意思是：加密和解密用的不是同一把钥匙。

它有两把钥匙：

```
公钥 public key
私钥 private key
```

它们是一对，互相配套。

公钥可以公开，给任何人看都可以。

私钥必须保密，只能后端服务器自己保存。

---

**3. 公钥和私钥分别干什么**  
在“前端加密登录”这个场景里，一般这样用：

```
公钥：给前端，用来加密密码
私钥：放后端，用来解密密码
```

也就是说：

```
前端：我只有公钥，所以我只能把密码锁起来
后端：我有私钥，所以我能把密码解开
```

公钥不能用来解密自己加密出来的内容。

这就是 RSA 的关键点。

---

**4. 用一个例子理解**  
你可以想象后端有一个保险箱系统。

后端把很多“开口的锁盒子”发给前端。

每个盒子都可以被任何人关上，但一旦关上，只有后端手里的钥匙能打开。

于是：

```
用户把密码写在纸条上
前端把纸条放进盒子
用公钥把盒子锁上
把锁好的盒子寄给后端
后端用私钥打开盒子
拿出纸条，看见原始密码
```

别人就算拦截到了盒子，也打不开。

---

**5. 登录时完整流程**  
一次登录通常是这样：

```
1. 后端提前生成一对 RSA 密钥
   公钥：可以公开
   私钥：后端保管

2. 前端加载登录页

3. 前端从后端拿公钥
   比如请求：GET /public-key

4. 用户输入账号和密码

5. 前端用公钥加密密码

6. 前端发送登录请求
   账号还是明文
   密码是密文

7. 后端收到密文密码

8. 后端用私钥解密，得到原始密码

9. 后端拿原始密码去验证用户身份

10. 验证成功后，后端返回登录凭证
```

发送前：

```
用户输入密码：123456
```

发送时：

```
密码密文：u3xTqZx93K...很长一串...
```

后端解密后：

```
还原密码：123456
```


逻辑是：前端拿公钥，用公钥加密密码；后端用私钥解密，再继续做登录验证。

**5.1. 生成 RSA 密钥**  
可以在后端项目里生成一对密钥：

```bash
openssl genrsa -out private.pem 2048
openssl rsa -in private.pem -pubout -out public.pem
```

会得到：

```
private.pem  私钥，后端保存，不能给前端
public.pem   公钥，可以给前端
```

**5.2. 后端提供公钥接口**  
后端把 `public.pem` 返回给前端。

```ts
import fs from 'node:fs'
import path from 'node:path'
import express from 'express'

const app = express()

const publicKey = fs.readFileSync(
  path.resolve('./keys/public.pem'),
  'utf8',
)

app.get('/public-key', (req, res) => {
  res.json({
    publicKey,
  })
})
```

前端拿到的公钥大概长这样：

```
-----BEGIN PUBLIC KEY-----
MIIBIjANBgkqhkiG9w0BAQEFAAOCAQ8A...
-----END PUBLIC KEY-----
```

**5.3. 前端安装加密库**  
浏览器端可以用 `jsencrypt`，简单好理解：

```bash
pnpm add jsencrypt
```

**5.4. 前端用公钥加密密码**  
前端登录前先请求公钥，然后加密密码。

```ts
import JSEncrypt from 'jsencrypt'

async function getPublicKey() {
  const res = await fetch('/public-key')
  const data = await res.json()
  return data.publicKey
}

function encryptPassword(password: string, publicKey: string) {
  const encryptor = new JSEncrypt()
  encryptor.setPublicKey(publicKey)

  const encrypted = encryptor.encrypt(password)

  if (!encrypted) {
    throw new Error('密码加密失败')
  }

  return encrypted
}

async function login(username: string, password: string) {
  const publicKey = await getPublicKey()

  const encryptedPassword = encryptPassword(password, publicKey)

  const res = await fetch('/login', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
    },
    body: JSON.stringify({
      username,
      password: encryptedPassword,
    }),
  })

  return res.json()
}
```

这时前端发送的不是：

```json
{
  "username": "alice",
  "password": "123456"
}
```

而是：

```json
{
  "username": "alice",
  "password": "一长串RSA密文"
}
```

**5.5. 后端用私钥解密**  
后端收到密文后，用私钥解密。

```ts
import fs from 'node:fs'
import path from 'node:path'
import { privateDecrypt } from 'node:crypto'

const privateKey = fs.readFileSync(
  path.resolve('./keys/private.pem'),
  'utf8',
)

function decryptPassword(encryptedPassword: string) {
  const buffer = Buffer.from(encryptedPassword, 'base64')

  const decrypted = privateDecrypt(
    {
      key: privateKey,
      padding: crypto.constants.RSA_PKCS1_PADDING,
    },
    buffer,
  )

  return decrypted.toString('utf8')
}
```

注意这里需要引入 `crypto`：

```bash
import crypto from 'node:crypto'
```

或者这样写更完整：

```ts
import fs from 'node:fs'
import path from 'node:path'
import crypto from 'node:crypto'

const privateKey = fs.readFileSync(
  path.resolve('./keys/private.pem'),
  'utf8',
)

function decryptPassword(encryptedPassword: string) {
  return crypto.privateDecrypt(
    {
      key: privateKey,
      padding: crypto.constants.RSA_PKCS1_PADDING,
    },
    Buffer.from(encryptedPassword, 'base64'),
  ).toString('utf8')
}
```

**5.6. 后端登录时使用解密后的密码**  
后端不要直接拿密文去比对，要先解密：

```ts
app.post('/login', express.json(), async (req, res) => {
  const { username, password } = req.body

  const rawPassword = decryptPassword(password)

  const user = await findUserByUsername(username)

  if (!user) {
    return res.status(401).json({
      message: '用户名或密码错误',
    })
  }

  const ok = await bcrypt.compare(rawPassword, user.passwordHash)

  if (!ok) {
    return res.status(401).json({
      message: '用户名或密码错误',
    })
  }

  return res.json({
    message: '登录成功',
    token: 'jwt-token',
  })
})
```

**6. 注册时也是一样**  
注册也可以用同样方式。

```
用户输入密码
前端用公钥加密
后端用私钥解密
后端拿到原始密码
后端再做密码哈希
数据库保存哈希值
```

这里有一个非常重要的点：

**数据库不应该保存 RSA 加密后的密文。**

数据库应该保存密码哈希，比如 bcrypt 哈希。

因为 RSA 是为了“传输过程加密”，不是为了“存储密码”。

```ts
app.post('/register', express.json(), async (req, res) => {
  const { username, password } = req.body

  const rawPassword = decryptPassword(password)

  const passwordHash = await bcrypt.hash(rawPassword, 10)

  await createUser({
    username,
    passwordHash,
  })

  return res.json({
    message: '注册成功',
  })
})
```

完整流程就是：

```
前端 password 明文
        ↓
RSA 公钥加密
        ↓
发送 password 密文
        ↓
后端 RSA 私钥解密
        ↓
得到 password 明文
        ↓
bcrypt.compare 或 bcrypt.hash
```

再稍微严谨一点，可以加时间戳和 nonce，不只加密密码：

```ts
const payload = JSON.stringify({
  password,
  timestamp: Date.now(),
  nonce: crypto.randomUUID(),
})

const encryptedPayload = encryptPassword(payload, publicKey)
```

后端解密后：

```ts
const raw = decryptPassword(encryptedPayload)

const payload = JSON.parse(raw)

if (Date.now() - payload.timestamp > 2 * 60 * 1000) {
  throw new Error('请求已过期')
}

const password = payload.password
```

这样可以降低别人复制旧密文重复登录的风险。

**7. RSA 和 bcrypt 的区别**  
这两个东西很多小白容易混。

RSA 是加密：

```
明文 -> 密文
密文 -> 明文
```

它是可以解密回来的。

bcrypt 是哈希：

```
明文 -> 哈希值
哈希值 -> 不能还原明文
```

bcrypt 不是加密，它是单向的。

比如：

```
密码：123456
bcrypt 后：$2a$10$xxxxxx...
```

后端不会从 bcrypt 结果里还原出 `123456`。

登录时是这样比对：

```
用户输入 123456
后端把 123456 用同样规则算一遍
看它是否能匹配数据库里的 bcrypt 哈希
```

所以两者职责不同：

```
RSA：保护前端到后端的提交过程
bcrypt：保护数据库里的密码
```

它们通常一起用。

---

**8. 为什么不能只用前端 RSA，然后数据库存密文**  
因为如果数据库存的是 RSA 密文，后端理论上可以用私钥解密出所有用户密码。

这很危险。

正确做法是：

```
前端 RSA 加密
后端 RSA 解密
后端 bcrypt 哈希
数据库保存 bcrypt 哈希
```

数据库里永远不应该有能被还原成明文密码的东西。

---

**9. 为什么还需要 HTTPS**  
RSA 前端加密看起来已经很安全了，但它不能代替 HTTPS。

原因是：如果没有 HTTPS，攻击者可能篡改网页内容。

比如正常情况：

```
前端拿到后端公钥 A
用公钥 A 加密密码
后端用私钥 A 解密
```

如果有人中间篡改：

```
攻击者把公钥 A 换成自己的公钥 B
前端用公钥 B 加密密码
攻击者用自己的私钥 B 解密
拿到你的密码
```

所以如果没有 HTTPS，前端拿到的公钥本身都不一定是真的。

HTTPS 的作用是保证：

```
你访问的真的是服务器
中间人不能随便改页面
中间人不能随便改公钥
中间人不能偷看完整通信内容
```

所以正确关系是：

```
HTTPS 是基础
RSA 前端加密是额外一层
```

**10. 什么是重放攻击**  
再讲一个重要问题：重放攻击。

假设用户登录时发出去的是：

```
{
  "username": "alice",
  "password": "加密后的密文"
}
```

攻击者虽然看不懂密文，也解不开密码，但他可能直接复制这次请求，再发一遍。

如果后端只检查“密文解出来是不是正确密码”，那这次请求可能仍然成功。

这叫**重放攻击**。

因为攻击者不需要知道密码，只需要重复使用这段合法请求。

---

**11. 怎么防重放**  
常见做法是加：

```
timestamp 时间戳
nonce 随机数
```

前端加密的内容不只是密码，而是：

```
{
  "password": "123456",
  "timestamp": 1780000000000,
  "nonce": "一串随机字符"
}
```

然后前端把整个对象加密后发给后端。

后端解密后检查：

```
1. timestamp 是否过期
   比如只允许 2 分钟内有效

2. nonce 是否已经用过
   如果用过，拒绝
```

这样即使别人复制了旧请求，也不能一直使用。

因为：

```
时间过期了，不接受
nonce 用过了，不接受
```

**12. RSA 适合加密大内容吗**  
RSA 不适合加密很大的内容。

它通常只适合加密较短的数据，比如：

```
密码
随机密钥
登录凭证片段
```

如果要加密很长的数据，通常会用“混合加密”：

```
1. 随机生成一个 AES 密钥
2. 用 AES 加密大量数据
3. 用 RSA 加密 AES 密钥
4. 后端用 RSA 解出 AES 密钥
5. 再用 AES 解出大量数据
```

不过登录密码很短，直接 RSA 加密就够理解了。

**13. 一次完整的安全登录可以这样理解**  
比较完整的版本：

```
1. 后端生成 RSA 公钥和私钥

2. 前端请求公钥

3. 用户输入密码

4. 前端准备一个对象：
   {
     password: "123456",
     timestamp: 当前时间,
     nonce: 随机字符串
   }

5. 前端把这个对象转成字符串

6. 前端用 RSA 公钥加密这个字符串

7. 前端发送：
   {
     username: "alice",
     encryptedData: "密文..."
   }

8. 后端用私钥解密 encryptedData

9. 后端得到：
   {
     password: "123456",
     timestamp: 当前时间,
     nonce: 随机字符串
   }

10. 后端检查 timestamp 是否过期

11. 后端检查 nonce 是否用过

12. 后端用 bcrypt 验证 password

13. 验证成功，登录成功
```

**14. 小白最容易混的几个点**  
第一，公钥不是密码。

```
公钥可以公开
公钥泄露通常没事
私钥泄露才严重
```

第二，RSA 加密后的密文不是最终密码。

```
它只是运输途中用的包装
后端还是要解开，拿到原始密码
```

第三，bcrypt 不是加密。

```
bcrypt 不能解密
bcrypt 是用来存密码的
```

第四，前端加密不能代替 HTTPS。

```
没有 HTTPS，别人可以换掉你的公钥或前端代码
```

第五，只做 `sha256(password)` 不等于安全。

```
sha256 后的值会变成新的“密码”
别人拿到这个值，也可以直接拿它登录
```
