**一、MQ 是什么**

MQ，全称 Message Queue，消息队列。

它的核心作用是：

```
把“直接调用”变成“发送消息”
```

原来是：

```
用户下单
  ↓
扣库存
  ↓
生成订单
  ↓
发优惠券
  ↓
发短信
  ↓
写日志
  ↓
返回结果
```

如果所有事情都同步做，请求会很慢，而且某个步骤失败可能拖垮主流程。

用了 MQ 后：

```
用户下单
  ↓
核心流程：扣库存、生成订单
  ↓
发送 MQ 消息
  ↓
立即返回

消费者异步处理：
发优惠券
发短信
写日志
同步数据
```

一句话：**MQ 用来解耦、异步、削峰。**

**二、MQ 削峰**

削峰是高并发系统里非常重要的能力。

比如秒杀开始的一瞬间：

```
10 万个请求同时进来
```

但数据库每秒可能只能处理：

```
3000 个写请求
```

如果请求直接打到数据库，数据库很容易崩。

使用 MQ 后：

```
10 万请求
  ↓
先进入 MQ
  ↓
消费者按数据库能承受的速度慢慢消费
```

就像：

```
前端洪峰很大
后端匀速处理
```

典型流程：

```
用户请求
  ↓
校验资格、库存预扣
  ↓
发送下单消息到 MQ
  ↓
快速返回“排队中”
  ↓
消费者慢慢创建订单
```

这种模式常见于：

```
秒杀
抢购
大促下单
批量任务
积分发放
优惠券发放
```

削峰的关键点：

```
1. MQ 能暂存大量消息
2. 消费者按固定能力消费
3. 数据库不会被瞬时流量打爆
4. 用户不一定立刻拿到最终结果，而是先拿到处理中状态
```

**三、MQ 异步任务**

异步任务是 MQ 最常用的场景。

很多操作不需要在用户请求里立即完成。

比如用户注册：

```
注册账号
  ↓
发送欢迎短信
  ↓
发送欢迎邮件
  ↓
赠送新人优惠券
  ↓
写用户行为日志
```

用户真正关心的是：

```
账号注册成功
```

短信、邮件、优惠券可以异步处理。

改成 MQ：

```
注册成功
  ↓
发送 user.registered 消息
  ↓
立即返回注册成功

消费者：
1. 发送短信
2. 发送邮件
3. 发优惠券
4. 写日志
```

好处：

```
1. 接口响应更快
2. 主流程更稳定
3. 各业务模块解耦
4. 某个异步任务失败，不影响注册成功
```

再比如订单支付成功：

```
支付成功消息
pay.success
```

消费者可以分别处理：

```
订单服务：修改订单状态
库存服务：确认扣减库存
积分服务：增加积分
优惠券服务：标记优惠券已使用
通知服务：发送通知
物流服务：准备发货
```

这就是事件驱动。

**四、MQ 幂等消费**

这是 MQ 面试里非常高频的问题。

幂等的意思是：

```
同一条消息被消费一次，和被消费多次，结果应该一样。
```

为什么 MQ 会重复消费？

因为 MQ 通常是“至少投递一次”。

也就是说：

```
MQ 尽量保证消息能送到
但不能保证绝对只送一次
```

常见重复场景：

```
消费者处理完业务，还没来得及 ack，宕机了
MQ 认为消息没消费成功
于是重新投递
```

例如：

```
消费者扣了库存
  ↓
还没 ack
  ↓
消费者宕机
  ↓
MQ 重新投递
  ↓
又扣了一次库存
```

这就出问题了。

所以消费者必须做幂等。

常见幂等方案：

```
1. 唯一业务 ID
2. 数据库唯一索引
3. Redis 去重
4. 状态机判断
5. 消息消费记录表
```

**1. 唯一业务 ID**

每条消息都带一个唯一 ID：

```
messageId = "pay_20260608_1001"
orderId = "order_1001"
```

消费者处理前先判断这个消息是否处理过。

**2. 数据库唯一索引**

比如创建订单时，给 order_id 或 user_id + activity_id 加唯一索引。

即使消息重复消费：

```sql
insert into order(user_id, activity_id, ...)
```

第二次插入会因为唯一索引失败。

这是一种非常可靠的幂等方式。

秒杀里常用：

```
user_id + sku_id 唯一索引
```

防止一个用户重复下单。

**3. Redis 去重**

消费前：

```
setnx mq:consume:messageId 1
```

如果成功，说明第一次消费。

如果失败，说明已经消费过，直接丢弃。

伪代码：

```java
String key = "mq:consume:" + messageId;

boolean first = redis.setIfAbsent(key, "1", 24, TimeUnit.HOURS);

if (!first) {
    return;
}

doBusiness();
ack();
```

注意：Redis 去重适合辅助，但核心业务最好还是有数据库唯一约束兜底。

**4. 状态机判断**

比如订单状态：

```
待支付 -> 已支付 -> 已发货 -> 已完成
```

如果重复收到支付成功消息：

```
第一次：待支付 -> 已支付
第二次：发现已经是已支付，直接忽略
```

伪代码：

```sql
update orders
set status = 'PAID'
where order_id = ?
and status = 'WAIT_PAY'
```

如果影响行数是 1，说明处理成功。

如果影响行数是 0，说明可能已经处理过。

这种方式很优雅，因为它把幂等和业务状态结合起来。

**5. 消息消费记录表**

建一张表：

```
mq_consume_log
```

字段：

```
message_id
consumer_group
status
create_time
update_time
```

消费前先插入记录：

```sql
insert into mq_consume_log(message_id, consumer_group, status)
values (?, ?, 'PROCESSING')
```

如果插入失败，说明这条消息已经处理过。

处理成功后：

```sql
update mq_consume_log set status = 'SUCCESS'
```

适合比较重要的业务消息。

**五、消息可靠性**

除了幂等，MQ 还经常问可靠性。

一条消息完整链路是：

```
生产者 -> MQ Broker -> 消费者
```

每一段都可能丢消息。

**A. 生产者到 MQ**

这一段的问题是：**生产者发送消息时，消息可能没有真正到达 MQ。**

比如：

```
网络抖动
Broker 临时不可用
生产者发送超时
生产者进程宕机
发送成功但 ACK 丢失
```

所以生产者不能“发完就不管”。

常见保证方式有：

```
发送确认机制
失败重试
本地消息表
事务消息
发送结果日志记录
```

**1. 发送确认**

生产者发送消息后，要等 MQ 返回确认。

```
生产者 send message
  ↓
Broker 接收并落盘
  ↓
Broker 返回 send success
```

只有收到成功响应，生产者才认为消息发送成功。

如果返回失败或超时：

```
生产者重试发送
```

比如 RocketMQ：

```java
SendResult result = producer.send(message);

if (result.getSendStatus() == SendStatus.SEND_OK) {
    // 发送成功
}
```

RabbitMQ 里也有：

```
publisher
confirm
```

Kafka 里有：

```
acks
```

比如 Kafka：

```
acks=0：发出去就算成功，不可靠
acks=1：leader 写入成功就算成功
acks=all：leader 和 ISR 副本都确认后才算成功，更可靠
```

**2. 失败重试**

如果发送失败，可以重试。

但是重试会带来一个问题：

```
可能重复发送
```

比如：

```
Broker 实际已经收到消息
但是返回 ACK 时网络断了
生产者以为失败，又发了一次
```

结果 MQ 里可能有两条相同业务消息。

所以：

```
可靠发送通常会导致重复消息
消费者必须做幂等
```

**3. 本地消息表**

本地消息表常用于“数据库操作 + 发 MQ”要保持一致的场景。

比如下单：

```
1. 创建订单
2. 发送订单创建消息
```

问题是：

```
订单创建成功了，但 MQ 发送失败怎么办？
```

可以用本地消息表：

```
开启数据库事务
  ↓
写订单表
  ↓
写本地消息表，状态 = 未发送
  ↓
提交事务
```

然后后台任务扫描本地消息表：

```
找到未发送消息
  ↓
发送到 MQ
  ↓
发送成功后，标记为已发送
```

这样可以保证：

```
只要订单入库成功，消息记录也一定入库
消息即使发送失败，也能后续补偿发送
```

**4. 事务消息**

事务消息也是解决“本地事务和 MQ 消息一致性”的。

典型流程：

```
生产者先发送半消息
Broker 暂时不投递给消费者
生产者执行本地事务
本地事务成功：提交消息
本地事务失败：回滚消息
```

RocketMQ 支持事务消息。

例子：

```
发送半消息：订单创建事件
执行本地事务：创建订单
如果订单创建成功：commit 消息
如果订单创建失败：rollback 消息
```

如果生产者执行完本地事务后宕机，Broker 还会回查生产者：

```
这个半消息到底应该 commit 还是 rollback？
```

**B、MQ Broker 自身**

这一段的问题是：**消息已经到 MQ 了，Broker 自己会不会把消息弄丢。**

可能丢失的情况：

```
Broker 宕机
消息只在内存中，还没落盘
磁盘损坏
主节点宕机，从节点还没同步
集群副本不足
```

常见保证方式：

```
消息持久化
同步刷盘
主从复制
多副本机制
高可用集群
死信队列
消息堆积监控
```

**1. 消息持久化**

Broker 收到消息后，不能只放内存，要写入磁盘。

否则 Broker 一宕机，内存消息就没了。

RabbitMQ 里通常要：

```
队列持久化
消息持久化
交换机持久化
```

Kafka 和 RocketMQ 默认就是基于日志文件持久化消息。

**2. 刷盘策略**

写入磁盘也分两种：

```
异步刷盘
同步刷盘
```

异步刷盘：

```
消息先写入操作系统 page cache
然后异步刷到磁盘
```

优点：

```
性能高
```

缺点：

```
机器突然宕机时，page cache 里的消息可能丢
```

同步刷盘：

```
消息写入磁盘成功后，才返回发送成功
```

优点：

```
可靠性更高
```

缺点：

```
性能更低
```

高可靠场景选同步刷盘，高吞吐场景常用异步刷盘。

**3. 主从复制 / 多副本**

单个 Broker 宕机也可能导致消息不可用或丢失。

所以 MQ 通常会有副本机制。

比如 Kafka：

```
一个 partition 有 leader 和 follower
生产者写 leader
follower 从 leader 同步数据
```

如果设置：

```
acks=all
min.insync.replicas=2
```

表示至少多个副本确认后，才认为消息写入成功。

RocketMQ 也有主从同步，常见配置有：

```
异步复制
同步双写
```

同步双写可靠性更高，但性能会下降。

**4. 死信队列**

死信队列不是防止 Broker 丢消息，而是防止消息处理失败后“无处可去”。

比如消息消费一直失败，超过最大重试次数后，可以进入死信队列：

```
正常队列 -> 消费失败多次 -> 死信队列
```

后续可以人工排查或补偿处理。

**三、MQ 到消费者**

这一段的问题是：**消息已经在 MQ 里了，消费者有没有真正处理成功。**

可能出问题的情况：

```
消费者收到消息后宕机
业务处理失败
数据库写入失败
消费者处理成功但 ACK 失败
消费者重复消费
消费者处理太慢导致消息积压
```

常见保证方式：

```
手动 ACK
消费失败重试
死信队列
消费者幂等
业务状态校验
消费日志表
监控告警
```

**1. 手动 ACK**

消费者不能一收到消息就自动确认。

因为可能出现：

```
消费者刚收到消息
还没处理业务
自动 ACK 已经发给 MQ
消费者宕机
消息丢失
```

所以关键业务一般用手动 ACK。

正确流程：

```
消费者收到消息
  ↓
执行业务逻辑
  ↓
业务成功
  ↓
手动 ACK
```

如果业务失败：

```
不 ACK / 返回失败
MQ 后续重新投递
```

RabbitMQ 里是：

```
basicAck
basicNack
basicReject
```

RocketMQ 里消费者返回：

```
CONSUME_SUCCESS
RECONSUME_LATER
```

Kafka 里消费者提交 offset：

```
业务处理成功后，再提交 offset
```

**2. 消费失败重试**

如果消费者处理失败，MQ 可以重新投递。

比如：

```
数据库临时异常
远程服务超时
库存服务短暂不可用
```

重试可以提高最终成功率。

但是重试也有风险：

```
重复消费
重试风暴
消息一直失败造成阻塞
```

所以一般要配：

```
最大重试次数
重试间隔
死信队列
告警
```

**3. 幂等消费**

MQ 的一个重要事实是：

```
消息通常只能保证至少一次投递，很难保证绝对只投递一次。
```

也就是：

```
消息可能重复
```

重复原因包括：

```
生产者重试导致重复发送
Broker 重试投递
消费者处理成功但 ACK 丢失
消费者超时后 MQ 重新投递
```

所以消费者必须幂等。

幂等意思是：

```
同一条消息处理一次和处理多次，最终结果一样。
```

常见幂等方案：

```
唯一索引
消费记录表
业务状态机
Redis setnx
消息 ID 去重
乐观锁
```

比如支付回调消息：

```sql
UPDATE orders
SET status = 'PAID'
WHERE order_id = 1001
  AND status = 'WAIT_PAY';
```

第一次消费：

```
WAIT_PAY -> PAID，成功
```

第二次重复消费：

```
status 已经是 PAID
更新影响行数 = 0
直接认为已处理
```

这就是状态机幂等。

再比如消费记录表：

```sql
CREATE UNIQUE INDEX uk_msg_id ON consume_log(message_id);
```

消费前先插入：

```
message_id = xxx
```

如果插入成功，说明第一次消费。

如果唯一索引冲突，说明已经消费过，直接跳过。

**四、整体可靠性链路**

一条可靠消息链路大概是这样：

```
生产者
  ↓ 发送确认 + 重试 + 本地消息表/事务消息
Broker
  ↓ 持久化 + 刷盘 + 副本复制 + 高可用
消费者
  ↓ 手动 ACK + 失败重试 + 死信队列 + 幂等消费
业务数据库
  ↓ 唯一索引 + 状态机 + 事务
```

注意一个现实点：

```
MQ 可靠性不是靠 MQ 单独完成的，而是生产者、Broker、消费者、数据库一起保证。
```
