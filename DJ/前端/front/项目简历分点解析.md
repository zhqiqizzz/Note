
# 瀑布流 + 长列表渲染优化

### 1. **业务列表组件**

> src/views/main/components/list/index.vue

- 请求图片列表数据
- 维护分页参数 `page / size`
- 判断是否加载完成
- 监听分类和搜索词变化
- 把数据传给瀑布流组件
- 外层包一层无限滚动组件 `m-infinite-list`

==作用==：源码里可以看到，业务列表通过 `getPexelsList(query)` 请求数据，初始页覆盖列表，后续页通过 `push(...res.list)` 追加数据；同时用 `isFinished` 判断是否已经加载完。首页里移动端使用 2 列，PC 端使用 5 列，并且传入了 `picturePreReading=false`

### 2. **瀑布流布局组件**

> src/libs/waterfall/index.vue

- 计算容器宽度
- 根据列数计算每列宽度
- 维护每一列的高度
- 按“最短列优先”分配每个 item 的位置
- 给每个 item 动态生成 `left / top`
- 设置容器总高度

==作用==：你的瀑布流组件外层是 `relative`，每个 item 是 `absolute`，通过动态样式控制 `width / left / top`，这就是典型的 JS 计算型瀑布流布局

### 3. **单个图片卡片组件**

> src/views/main/components/list/item.vue

- 渲染图片
- 根据接口返回的图片原始宽高计算展示高度
- 使用 `v-lazy` 懒加载图片
- 渲染标题、作者、头像
- 提供下载、分享、全屏、点击详情等交互
```css 
:style="{
	height: (width / data.photoWidth) * data.photoHeight + 'px'
}"
```

==作用==：这说明你的瀑布流不是等图片真实加载完成后才知道高度，而是根据图片原始尺寸比例提前计算高度，这也是为什么首页里把 `picturePreReading` 设置成了 `false`

## 瀑布流的核心原理

>**JS 计算列宽 + 维护列高数组 + 最短列优先插入 + absolute 定位渲染**


```JavaScript
{  0: 0,  1: 0,  2: 0,  3: 0,  4: 0}
```
假设 PC 端是 5 列，初始每列高度如上，每来一个图片卡片，组件会：
1. 找到当前高度最小的列
2. 把这个 item 放到这列
3. 它的 `left` 等于：列索引 × `(列宽 + 列间距)`
4. 它的 `top` 等于：当前最短列的高度
5. 该列高度增加：`itemHeight + rowSpacing`

## 面试时你应该怎么讲

>我这个项目首页是图片内容流，图片宽高比例不固定，所以普通网格布局会造成空间浪费。

>我封装了一个通用瀑布流组件，外层容器使用 `relative`，每个 item 使用 `absolute`，通过 JS 动态计算每个 item 的 `left` 和 `top`。核心算法是最短列优先：维护一个列高对象，每次把新卡片放到当前高度最小的列里，再更新该列高度，最后用最高列高度作为容器高度。

>在业务层，我把请求分页和布局渲染解耦。列表组件只负责请求数据、分页追加、分类和搜索条件变化时重置数据；瀑布流组件只负责位置计算。

>另外，图片卡片会根据接口返回的图片原始宽高提前计算展示高度，避免等图片真实加载后再回流。图片资源本身通过自定义 `v-lazy` 指令懒加载，基于 `IntersectionObserver` 判断图片是否进入可视区，进入后再加载真实图片。

>这个方案主要优化了图片内容流场景下的空间利用率、首屏加载压力和滚动浏览体验。不过它目前还不是虚拟列表，如果数据量进一步扩大，我会把它升级成虚拟瀑布流，只渲染可视区附近的 DOM 节点。

### list/index.vue：业务列表层

文件：`src/views/main/components/list/index.vue`它是整个首页内容流的业务入口，负责请求数据、维护分页、监听分类/搜索变化，并把数据交给瀑布流组件渲染

#### 1. template 结构
```vue
<m-infinite-list
  v-model="isLoading"
  :isFinished="isFinished"
  @onLoad="getPexelsData"
>
  <m-waterfall
    class="px-1 w-full"
    :data="pexelsList"
    :nodeKey="'id'"
    :column="isMobileTerminal ? 2 : 5"
    :picturePreReading="false"
  >
    <template v-slot="{ item, width }">
      <itemVue :data="item" :width="width" @click="onToPins" />
    </template>
  </m-waterfall>
</m-infinite-list>
```

==外层==：`m-infinite-list`  它负责 **“触底加载”**
- `isLoading`：当前是否正在请求数据
- `isFinished`：是否所有数据都已经加载完
- `onLoad`：触底时调用 `getPexelsData`
所以业务组件不用自己监听滚动到底部

 ==中层==：`m-waterfall` 它负责 **“怎么排版”**
- `data`：图片列表数据
- `nodeKey`：每个 item 的唯一 key，用 `id`
- `column`：移动端 2 列，PC 端 5 列
- `picturePreReading=false`：不等待图片真实加载完成后再计算高度
这个 `picturePreReading=false` 非常关键，说明你当前布局依赖的是**接口返回的图片宽高比例**提前计算高度，而不是等图片加载完再测量

==内层== ：slot 渲染 `itemVue`
`m-waterfall` 会把两个东西通过插槽传出来：
- `item`：当前图片数据
- `width`：瀑布流算出来的列宽
然后 `item.vue` 根据这个 `width` 计算图片高度

#### 2. 核心变量
```javascript
let query = {
  page: 1,
  size: 20
}
```
它最开始表示：
- 第 1 页
- 每页 20 条
后续切换分类、搜索时，会往里面追加：categoryId、searchText，所以最终可能变成：
```javascript
{
  page: 1,
  size: 20,
  categoryId: xxx,
  searchText: 'xxx'
}
```

```javascript
const isLoading = ref(false)
```
表示当前是否正在加载数据。
它和 `m-infinite-list` 的 `v-model` 绑定。
当触底加载时，`infinite-list` 会把它改成 `true`；请求结束后，`list/index.vue` 再把它改回 `false`
这就是父子组件之间的 loading 状态同步


```javascript
const isFinished = ref(false)
```
表示是否已经没有更多数据。
如果它是 `true`，`getPexelsData` 会直接 return，不再请求

```javascript
const pexelsList = ref([])
```

这是当前页面真正要渲染的数据源，它会传给：
```vue
<m-waterfall :data="pexelsList" />
```
也就是说，只要 `pexelsList` 变化，瀑布流组件就会重新计算布局

#### 3. 核心函数

 ##### getPexelsData
 这是业务列表最重要的函数，源码逻辑可以拆成 6 步：

1. 如果已经加载完，直接 return，避免重复请求
```javascript
if (isFinished.value) {
  return
}
```

2. 如果列表已有数据，页码自增
```javascript
if (pexelsList.value.length) {
  query.page += 1
}
```
这里逻辑是：
	- 第一次请求时，`pexelsList` 为空，还是第 1 页
	- 后续请求时，`pexelsList` 有数据，`page += 1`
   所以触底加载下一页，本质就是修改 `query.page`

3. 请求接口
```javascript
const res = await getPexelsList(query)
```
接口返回的数据应该包含：
```javascript
{
  list: [],
  total: number
}
```

4. 第一页覆盖，后续页追加
```javascript
if (query.page === 1) {
  pexelsList.value = res.list
} else {
  pexelsList.value.push(...res.list)
}
```
这段很重要：
- 如果是第一页，说明是初始化、分类切换、搜索切换，需要覆盖旧数据
- 如果不是第一页，说明是触底加载更多，需要追加数据

5. 判断是否加载完
```javascript
if (pexelsList.value.length === res.total) {
  isFinished.value = true
}
```
如果当前已加载数量等于接口返回的总数，就标记没有更多数据

6. 关闭 loading
```javascript
if (pexelsList.value.length === res.total) {
  isFinished.value = true
}
```
告诉 `m-infinite-list`： 当前这一页已经加载完成，可以继续监听下一次触底

##### resetQuery
这个函数负责“重置查询”：
```javascript
const resetQuery = (newQuery) => {
  query = { ...query, ...newQuery }
  isFinished.value = false
  pexelsList.value = []
}
```
比如分类变化时：
```javascript
resetQuery({
  page: 1,
  categoryId: currentCategory.id
})
```
搜索词变化时：
```javascript
resetQuery({
  page: 1,
  searchText: val
})
```
它做了三件事：
1. 合并新的查询条件
2. 把 `isFinished` 改回 `false`
3. 清空旧列表
清空 `pexelsList` 后，瀑布流组件会收到空数据；下一次 `m-infinite-list` 发现底部 loading 区域可见，就会重新触发请求。

##### 两个 watch：分类和搜索联动

**监听当前分类**
```javascript
watch(
  () => store.getters.currentCategory,
  (currentCategory) => {
    resetQuery({
      page: 1,
      categoryId: currentCategory.id
    })
  }
)
```
>用户切换分类后，列表回到第一页，并使用新的 `categoryId` 重新加载

**监听搜索词**
```javascript
watch(
  () => store.getters.searchText,
  (val) => {
    resetQuery({
      page: 1,
      searchText: val
    })
  }
)
```
>用户搜索后，列表回到第一页，并使用新的 `searchText` 重新加载

##### 数据流
```plaintext
分类变化 / 搜索变化
        ↓
watch 监听 store getter
        ↓
resetQuery()
        ↓
清空 pexelsList + page 回到 1
        ↓
m-infinite-list 触底触发 onLoad
        ↓
getPexelsData()
        ↓
getPexelsList(query)
        ↓
pexelsList 更新
        ↓
m-waterfall 重新布局
        ↓
item.vue 渲染图片卡片
```

### infinite-list/index.vue：触底加载组件

文件：`src/libs/infinite-list/index.vue`，它负责无限滚动，也就是“什么时候该加载下一页”
#### 1. template 结构
```vue
<div>
  <slot />
  <div ref="laodingTarget" class="h-6 py-4">
    <m-svg-icon
      v-show="loading"
      class="w-4 h-4 mx-auto animate-spin"
      name="infinite-load"
    />
    <p v-if="isFinished">
      已经没有更多数据了!
    </p>
  </div>
</div>
```
它做了两件事：
1. `<slot />` 渲染列表内容，也就是瀑布流
2. 底部放一个 `laodingTarget` 节点，用来观察是否进入可视区
这个 `laodingTarget` 就是“触底检测器”
#### 2. 核心变量
```javascript
const props = defineProps({
  modelValue: {
    type: Boolean,
    required: true
  },
  isFinished: {
    type: Boolean,
    default: false
  }
})
```

`modelValue`就是父组件传进来的 `isLoading`
`isFinished`表示是否加载完成，如果已经完成，就不再触发加载

**`useVModel`**
```javascript
const loading = useVModel(props, 'modelValue', emits)
```
这是 VueUse 提供的工具，它的作用是把：props.modelValue 包装成一个==可读可写的响应式变量==

**`laodingTarget`**
```javascript
const loading = useVModel(props, 'modelValue', emits)
```
这个变量指向底部 loading 节点

**`targetIsIntersecting`**
```javascript
const targetIsIntersecting = ref(false)
```
表示底部节点当前是否进入可视区域

#### 3. 核心函数

##### useIntersectionObserver
```javascript
useIntersectionObserver(
  laodingTarget,
  ([{ isIntersecting }], observerElement) => {
    targetIsIntersecting.value = isIntersecting
    emitLoad()
  }
)
```
这段逻辑是：
1. 监听底部节点
2. 如果它进入可视区，`isIntersecting=true`
3. 保存当前状态
4. 调用 `emitLoad`
##### emitLoad
```javascript
const emitLoad = () => {
  setTimeout(() => {
    if (targetIsIntersecting.value && !loading.value && !props.isFinished) {
      loading.value = true
      emits('onLoad')
    }
  }, 100)
}
```
它会判断三个条件：
1. targetIsIntersecting.value 底部节点是否可见
2. !loading.value 当前是否没有正在请求
3. !props.isFinished 是否还有更多数据
三个条件都满足，才会：loading.value = true  emits('onLoad')

也就是说：
> 只有触底、没在加载、没加载完，才请求下一页

这个逻辑避免了重复触发接口请求

##### 监听 loading 变化

```javascript
watch(loading, (val) => {
  setTimeout(() => {
    emitLoad()
  }, 100)
})
```
这段的作用是解决一个常见问题：

> 如果第一页数据太少，列表高度还没撑满一屏，那么底部 loading 区域一直在可视区内。  
> 第一次请求结束后，`loading` 变回 `false`，这时应该继续请求下一页，直到页面被撑满或者数据加载完。

所以这里监听 `loading` 变化，数据加载完成后再尝试触发一次 `emitLoad`。

这个设计是合理的，属于无限滚动组件里比较容易漏掉的细节。

### waterfall/index.vue：瀑布流核心组件

文件：`src/libs/waterfall/index.vue`，这是整个模块的布局核心

#### 1. template 结构
```vue
<div
  class="relative"
  ref="containerTarget"
  :style="{ height: containerHeight + 'px' }"
>
  <template v-if="columnWidth && data.length">
    <div
      class="m-waterfall-item absolute duration-300"
      :style="{
        width: columnWidth + 'px',
        left: item._style?.left + 'px',
        top: item._style?.top + 'px'
      }"
      v-for="(item, index) in data"
      :key="nodeKey ? item[nodeKey] : index"
    >
      <slot :item="item" :width="columnWidth" :index="index" />
    </div>
  </template>
</div>
```

 内外层容器是==子绝父相== ，子元素的 `left/top` 需要相对于这个父容器定位

 **为什么要设置 containerHeight**：因为子元素全部是 `absolute`，脱离了普通文档流，如果父容器不主动设置高度，父容器高度会塌陷成 0，所以瀑布流计算完所有列之后，要把最高列高度作为容器高度

 **v-if="columnWidth && data.length"**
 
> 只有列宽计算完成，并且有数据时，才渲染瀑布流 item。

为什么要等 `columnWidth`？因为 item 的宽度和图片高度都依赖列宽
如果列宽还没算出来就渲染，item 高度可能是错的，后续瀑布流布局也会错

**每个 item 的定位**
```javascript
:style="{
  width: columnWidth + 'px',
  left: item._style?.left + 'px',
  top: item._style?.top + 'px'
}"
```

每个瀑布流 item 的位置不是 CSS 自己排的，而是 JS 算出来的：
- `width`：列宽
- `left`：横向位置
- `top`：纵向位置
这些值来自：==item._style== （这就是瀑布流算法的最终输出）

#### 2. 核心变量
```javascript
const props = defineProps({
  data: Array,
  nodeKey: String,
  column: Number,
  columnSpacing: Number,
  rowSpacing: Number,
  picturePreReading: Boolean
})
```
 data 瀑布流的数据源，也就是 pexelsList
 nodeKey:用于 `v-for` 的 key，所以每个 item 用 `item.id` 作为 key。
 column：列数
 columnSpacing：列间距，也就是横向间隔
 rowSpacing：行间距，也就是纵向间隔
 picturePreReading：是否需要图片预读取，它决定高度计算方式：
```javascript
picturePreReading = true
    等图片真实加载完成后，测量 DOM 高度

picturePreReading = false
    直接根据 DOM 当前高度计算
```
**containerHeight**
容器总高度，因为 item 是 absolute，所以容器高度必须手动设置，最终它会等于所有列里的最大高度：
```javascript
containerHeight.value = getMaxHeight(columnHeightObj.value)
```
**columnHeightObj**
记录每一列当前高度，例如 PC 端 5 列时：
```javascript
{
  0: 620,
  1: 740,
  2: 590,
  3: 810,
  4: 700
}
```
瀑布流每放一个 item，就会更新这个对象
**containerTarget**
指向外层容器 DOM，用于计算容器宽度
**containerWidth**
容器内容宽度，不包含 padding
**containerLeft**
容器左侧 padding，计算 item 的 `left` 时要加上它
**columnWidth**
每一列的宽度：计算公式是：(containerWidth - columnSpacingTotal) / column
**columnSpacingTotal**
```javascript
const columnSpacingTotal = computed(() => {
  return (props.column - 1) * props.columnSpacing
})
```
**itemHeights**
记录每个 item 的高度，后续布局时，通过：itemHeights[index]，知道当前 item 应该增加多少列高

#### 3. 核心函数
##### useColumnHeightObj
```javascript
const useColumnHeightObj = () => {
  columnHeightObj.value = {}
  for (let i = 0; i < props.column; i++) {
    columnHeightObj.value[i] = 0
  }
}
```
作用：初始化每一列高度为 0，如果当前是 5 列，就生成：
```javascript
{
  0: 0,
  1: 0,
  2: 0,
  3: 0,
  4: 0
}
```
##### useContainerWidth
```javascript
const useContainerWidth = () => {
  const { paddingLeft, paddingRight } = getComputedStyle(
    containerTarget.value,
    null
  )

  containerLeft.value = parseFloat(paddingLeft)

  containerWidth.value =
    containerTarget.value.offsetWidth -
    parseFloat(paddingLeft) -
    parseFloat(paddingRight)
}
```
作用：计算容器真实可用宽度

为什么要减 padding？因为瀑布流 item 的布局区域不应该包含容器 padding

`getComputedStyle` 是浏览器内置 API，用于**读取元素最终实际渲染的 CSS 值**。你的代码用它来获取容器的左右内边距，以便计算瀑布流内容区的净宽度和 item 的排列起点。

##### useColumnWidth
```javascript
const useColumnWidth = () => {
  useContainerWidth()

  columnWidth.value =
    (containerWidth.value - columnSpacingTotal.value) / props.column
}
```
> 根据容器宽度、列数、列间距计算每列宽度
```javascript
containerWidth = 1180
column = 5
columnSpacing = 20
columnSpacingTotal = 80
columnWidth = (1180 - 80) / 5 = 220
```

```javascript
  })
}
```
这个函数用于：
> 等图片全部加载完成后，再测量 item 高度

```plaintext
拿到所有瀑布流 item DOM
        ↓
从 item 中找出所有 img
        ↓
拿到所有 img.src
        ↓
创建 Image 对象等待图片加载完成
        ↓
图片完成后读取每个 item 的 offsetHeight
        ↓
调用 useItemLocation 计算位置
```
这个适用于：
> 接口没有提供图片宽高，必须等图片真实加载后才能知道高度
##### useItemHeight
```javascript
const useItemHeight = () => {
  itemHeights = []

  let itemElements = [...document.getElementsByClassName('m-waterfall-item')]

  itemElements.forEach((el) => {
    itemHeights.push(el.offsetHeight)
  })

  useItemLocation()
}
```
这是你首页当前真正走的高度计算方式，它不等待图片加载完成，直接读取每个 item 的 DOM 高度。为什么这样也能工作？因为 `item.vue` 里提前写了图片高度
##### useItemLocation
```javascript
const useItemLocation = () => {
  useColumnHeightObj()
  props.data.forEach((item, index) => {
    item._style = item._style || {}
    item._style.left = getItemLeft()
    item._style.top = getItemTop()
    increasingHeight(index)
  })
  containerHeight.value = getMaxHeight(columnHeightObj.value)
}
```
这是瀑布流布局最核心的函数，它做了 4 件事：
第一步：重置列高 useColumnHeightObj()，每次重新计算位置前，先把每列高度清零
第二步：遍历所有数据 props.data.forEach((item, index) => {})，这里是全量遍历，每个 item 都会重新计算位置
第三步：给 item 设置位置 item.style.left = getItemLeft()，item.style.top = getItemTop()，这里会把布局结果写到 item 上。最终 template 里读取：item.style.left，item.style.top
第四步：更新列高 increasingHeight(index)，当前 item 放进去之后，这一列高度要增加
第五步：设置容器高度 containerHeight.value = getMaxHeight(columnHeightObj.value)，所有 item 排完后，取最高列作为容器高度
##### getItemLeft
```javascript
const getItemLeft = () => {
  const column = getMinHeightColumn(columnHeightObj.value)
  return (
    column * (columnWidth.value + props.columnSpacing) + containerLeft.value
  )
}
```
> 计算当前 item 应该放在哪一列的横向位置
1. 找到当前最短列
2. 用列索引计算 left
```javascript
column = 3
columnWidth = 220
columnSpacing = 20
containerLeft = 10
left = 3 * (220 + 20) + 10 = 730
```
##### getItemTop
```javascript
const getItemTop = () => {
  return getMinHeight(columnHeightObj.value)
}
```
> 当前 item 的 top 就是当前最短列的高度
##### increasingHeight
```javascript
const increasingHeight = (index) => {
  const minHeightColumn = getMinHeightColumn(columnHeightObj.value)

  columnHeightObj.value[minHeightColumn] +=
    itemHeights[index] + props.rowSpacing
}
```
当前 item 放入最短列后，更新这一列的高度
##### watch：数据变化时重新布局
```javascript
watch(
  () => props.data,
  (newVal) => {
    const resetColumnHeight = newVal.every((item) => !item._style)
    if (resetColumnHeight) {
      useColumnHeightObj()
    }
    nextTick(() => {
      if (props.picturePreReading) {
        waitImgComplate()
      } else {
        useItemHeight()
      }
    })
  },
  {
    immediate: true,
    deep: true
  }
)
```
为什么要 `nextTick`？因为数据变了以后，DOM 还没立即更新，必须等 Vue 把新的 item 渲染到页面上，再去读取
##### watch：列数变化时重新布局
```javascript
watch(
  () => props.column,
  () => {
    if (props.picturePreReading) {
      columnWidth.value = 0
      nextTick(reset)
    } else {
      reset()
    }
  }
)
```
当列数从 2 变 5，或者从 5 变 2 时，必须重新计算布局。
比如移动端切 PC 端，列宽、left、top、容器高度全部都要变
##### reset
```javascript
const reset = () => {
  setTimeout(() => {
    useColumnWidth()
    props.data.forEach((item) => {
      item._style = null
    })
  }, 100)
}
```
它做了两件事：
1. 重新计算列宽
2. 把所有 item 的 `_style` 清空
因为 `props.data` 被 deep watch 监听，所以：item.style = null，会触发数据 watch，进而重新计算瀑布流布局。

### lazy.js：图片懒加载指令
文件：`src/directives/modules/lazy.js`

#### 1. 指令 mounted
```javascript
mounted(el) {
  const imgSrc = el.src
  el.src = ''
  const { stop } = useIntersectionObserver(el, ([{ isIntersecting }]) => {
    if (isIntersecting) {
      el.src = imgSrc
      stop()
    }
  })
}
```
#### 2. 第一步：保存真实图片地址
const imgSrc = el.src，此时 `el.src` 就是：data.photo，也就是图片真实 URL

#### 3. 第二步：清空 src
el.src = ''，清空后，浏览器就不会立即请求真实图片资源，这一步是图片懒加载的关键

#### 4. 第三步：监听是否进入可视区
useIntersectionObserver(el, ([{ isIntersecting }]) => {})
它底层用的是浏览器的 `IntersectionObserver`，当图片进入可视区时：isIntersecting = true

#### 5. 第四步：恢复真实 src
el.src = imgSrc，这时浏览器才开始加载图片

#### 6. 第五步：停止监听
stop()
图片已经加载过了，就不需要继续观察，这样可以减少观察器开销
#### 7. 优化
**当图片快进入视口时，提前用 `new Image()` 加载真实图片；加载并解码完成后，再替换页面上的图片地址，并添加 loaded 类显示动画；加载失败则显示兜底图，最后停止监听。**
```javascript
useIntersectionObserver(
  el,
  ([{ isIntersecting }]) => {
    if (!isIntersecting) return

    const image = new Image()
    image.src = imgSrc

    image.onload = async () => {
      if (image.decode) {
        await image.decode().catch(() => {})
      }
      el.src = imgSrc
      el.classList.add('loaded')
      stop()
    }

    image.onerror = () => {
      el.src = fallbackUrl
      stop()
    }
  },
  {
    rootMargin: '200px'
  }
)
```
 >为什么要 `new Image()`？先创建一个临时图片对象，用它来**预加载图片**
 区别是：直接设置el.src = imgSrc，浏览器会立刻把页面上的图片换成真实图片。如果网络慢，用户可能会看到图片一点点加载、闪烁，或者布局变化。先用 `new Image()` 加载，先在内存里偷偷加载。等加载成功后，再把真实图片地址赋给页面上的 `img`，这样页面上的图片切换更平滑。
 
 >`image.onload` 是什么？表示图片资源已经加载成功，但注意：`onload` 代表图片文件下载完成，不一定代表浏览器已经完全解码并准备好绘制，所以后面又加了 `image.decode()`，`decode()` 会让浏览器提前把图片解码好，图片从网络下载下来之后，还需要经过解码才能真正显示。对于大图来说，如果直接显示，可能会有一瞬间卡顿。
 
 > 加载成功后做了什么？设置真实图片，添加样式类 ‘loaded’通常配合 CSS 做淡入动画，再停止监听。
 > `rootMargin: '200px'` 是什么？默认情况下，元素真正进入视口才会触发。但加了 `rootMargin: '200px'` 之后，浏览器会把视口范围向外扩展 200px。这样用户继续往下滚动时，图片可能已经加载好了，体验更流畅。

 

### item.vue：单个图片卡片
文件：`src/views/main/components/list/item.vue`，它是瀑布流中的单个卡片组件

####  图片高度计算
```vue
<img
  v-lazy
  ref="imgTarget"
  class="w-full rounded bg-transparent"
  :src="data.photo"
  :style="{
    height: (width / data.photoWidth) * data.photoHeight + 'px'
  }"
/>
```
这段是整个瀑布流稳定性的关键: **展示高度 = 展示宽度 / 原图宽度 * 原图高度**
这样即使图片还没加载，页面也已经知道它应该占多高
好处：
1. 减少图片加载完成后的布局抖动
2. 瀑布流可以提前计算 item 高度
3. 不必等图片预加载完成再排版
4. 配合 `v-lazy` 时也不会因为图片还没 src 而高度塌陷
这就是你项目里 `picturePreReading=false` 能成立的原因
#### 随机背景色
```javascript
:style="{
  backgroundColor: randomRGB()
}"
```
在图片加载之前，卡片有一个随机背景色作为视觉占位
这个点可以面试里轻轻带一句：
> 图片加载前通过背景色弱化白屏感

#### 下载逻辑 onDownload
```javascript
const onDownload = async () => {
  message('success', '图片开始下载')
  const url = props.data.photo
  const filename = (props.data.title || 'photo') + '.jpg'
  const res = await fetch(url, { mode: 'cors' })
  const blob = await res.blob()
  saveAs(blob, filename)
}
```
1. 提示用户开始下载
2. fetch 获取图片
3. 转成 blob
4. 用 `file-saver` 保存
这个功能不是瀑布流核心，但属于图片卡片交互

**Blob 是浏览器中用来表示“不可变的原始二进制数据”的对象，常被当作临时文件来使用。**
`Blob` 可以表示很多类型的数据，不只是图片：
图片：image/jpeg、image/png
PDF：application/pdf
Excel：application/vnd.openxmlformats-officedocument.spreadsheetml.sheet
视频：video/mp4
普通二进制：application/octet-stream

它常见属性有两个：
```typescript
blob.size // 文件大小，单位 byte
blob.type // MIME 类型，比如 image/jpeg
```
|类型|适合场景|
|---|---|
|`string`|文本、HTML、纯文本|
|`json`|接口返回的结构化数据|
|`ArrayBuffer`|更底层的二进制字节操作|
|`Blob`|下载、预览、上传文件|
|`File`|用户选择的本地文件，继承自 Blob|

用 `Blob` 保存，是因为：

URL 只是地址，不是文件内容
Response 是 HTTP 响应，不是文件对象
Blob 是浏览器里的文件数据容器
file-saver 正好可以把 Blob 触发下载成本地文件

### 最终你要能讲出来的版本

> 我这个首页内容流拆成了四层。最外层业务列表组件负责请求数据和维护分页状态；无限滚动组件负责监听底部 loading 节点是否进入可视区，进入后触发下一页加载；瀑布流组件负责布局计算；图片卡片组件负责具体展示和交互。
> 
> 瀑布流核心是最短列优先算法。组件挂载后先根据容器宽度、列数和列间距计算每列宽度，然后维护一个列高对象。每渲染一个 item，就找到当前高度最小的一列，把 item 的 `left` 设置为列索引乘以列宽加间距，`top` 设置为当前最短列高度，然后再把该 item 高度累加到这一列。所有 item 排完后，用最高列高度设置父容器高度，因为 item 都是 absolute 定位，父容器不会自然撑开。
> 
> 图片高度这块，我没有完全依赖图片真实加载完成再计算，而是利用接口返回的图片原始宽高，根据当前列宽提前算出展示高度。这样图片即使还没加载，也能先占位，减少布局抖动。图片资源本身通过全局 `v-lazy` 指令懒加载，进入可视区后才恢复真实 src，降低首屏图片请求压力。
> 
> 这个方案目前解决了图片比例不一致、长列表分页加载、首屏图片资源压力和瀑布流布局稳定性问题。不过它还不是虚拟列表，如果数据规模继续扩大，我会进一步做虚拟瀑布流和 DOM 回收。
> 
> （补充）：图片内容流里，图片加载是异步的，如果不提前设置高度，图片加载完成后会撑开 DOM，导致瀑布流重新计算和页面抖动。所以我利用接口返回的图片原始宽高，结合瀑布流计算出的列宽，提前按比例计算图片展示高度。这样图片还没加载时，页面已经预留了正确高度，瀑布流可以直接基于这个高度进行排布。再配合 `v-lazy`，图片资源只有进入可视区才会真正加载，但布局空间已经提前占好，所以既减少了首屏资源请求，也减少了布局抖动

### 虚拟瀑布流（优化）
#### 普通虚拟列表怎么做
先从最简单的“等高列表”理解，假设每个 item 高度固定为 `100px`，总共有 1000 条数据。
那真实列表总高度应该是：1000 × 100 = 100000px，但是页面只渲染当前屏幕附近的数据。
```javascript
scrollTop = 1200
itemHeight = 100
viewportHeight = 600
```
那么当前屏幕顶部对应的数据索引是：
```javascript
startIndex = Math.floor(scrollTop / itemHeight)
// 1200 / 100 = 12
```
当前屏幕最多能展示：
```javascript
visibleCount = Math.ceil(viewportHeight / itemHeight)
// 600 / 100 = 6
```
再加一点缓冲区，比如前后各 5 个：
```javascript
start = Math.max(0, startIndex - 5)
end = startIndex + visibleCount + 5
```
最终只渲染：
```
第 7 条 ~ 第 23 条
```
虽然总共有 1000 条，但 DOM 里只有十几二十个节点

#### 为什么“虚拟瀑布流”比普通虚拟列表难？
普通虚拟列表难度较低，因为 item 高度固定，但是瀑布流的问题是：
```
每个 item 高度不固定
每个 item 的 top 不固定
每个 item 分布在不同列
```
而且瀑布流是多列排列，不是简单的一维纵向列表，所以虚拟瀑布流必须提前维护每个 item 的布局信息：
```javascript
{
  id: 1,
  top: 320,
  left: 240,
  width: 220,
  height: 310,
  bottom: 630
}
```
然后根据当前滚动位置筛选：哪些 item 和当前视口有交集
#### 如何判断 item 是否在可视区？
```javascript
scrollTop = 1000
viewportHeight = 800
buffer = 300
```
那么真实要渲染的区域可以扩大一点：
```javascript
visibleTop = scrollTop - buffer
visibleBottom = scrollTop + viewportHeight + buffer
```
只要 item 满足：
```javascript
item.bottom >= visibleTop && item.top <= visibleBottom
```
就说明它在可视区附近，需要渲染
完整逻辑：
```javascript
const visibleItems = computed(() => {
  const visibleTop = scrollTop.value - buffer
  const visibleBottom = scrollTop.value + viewportHeight.value + buffer

  return layoutList.value.filter(item => {
    return item.bottom >= visibleTop && item.top <= visibleBottom
  })
})
```
#### 虚拟瀑布流整体流程
##### 第一步：拿到完整数据
```javascript
list = [
  { id: 1, photoWidth: 1000, photoHeight: 1500 },
  { id: 2, photoWidth: 800, photoHeight: 600 },
  ...
]
```
> 数据可以很多，虚拟化控制的是 DOM 渲染，不是数据本身
##### 第二步：计算每个 item 的高度
利用你现在已有的逻辑，再加上文本、作者、padding：
```javascript
imageHeight = columnWidth / photoWidth * photoHeight
itemHeight = imageHeight + titleHeight + authorHeight + padding
```
这里就引出一个优化点：
> 如果卡片的非图片区域高度固定，虚拟瀑布流就更容易做。所以前面我建议你固定标题区、作者区高度，不只是为了减少抖动，也是为了虚拟化更容易计算高度
##### 第三步：执行瀑布流布局算法
仍然是最短列优先
```javascript
const columnHeights = new Array(column).fill(0)
const layoutList = data.map((item, index) => {
  const minColumn = getMinColumn(columnHeights)
  const left = minColumn * (columnWidth + columnSpacing)
  const top = columnHeights[minColumn]
  const height = calcItemHeight(item, columnWidth)
  const bottom = top + height
  columnHeights[minColumn] += height + rowSpacing
  return {
    ...item,
    index,
    left,
    top,
    width: columnWidth,
    height,
    bottom
  }
})
```
##### 第四步：维护总高度
因为 DOM 只渲染一部分，但滚动条要看起来像完整列表，所以容器高度仍然是完整瀑布流高度：
```javascript
const containerHeight = Math.max(...columnHeights)
```
页面上需要一个撑高度的容器：
```javascript
<div
  class="waterfall-container"
  :style="{ height: containerHeight + 'px' }"
>
  ...
</div>
```
##### 第五步：根据滚动位置筛选可见 item
```javascript
const visibleItems = computed(() => {
  const start = scrollTop.value - buffer
  const end = scrollTop.value + viewportHeight.value + buffer
  return layoutList.value.filter(item => {
    return item.bottom >= start && item.top <= end
  })
})
```
然后 template 只渲染这些：
```javascript
<div
  v-for="item in visibleItems"
  :key="item.id"
  class="waterfall-item"
  :style="{
    width: item.width + 'px',
    height: item.height + 'px',
    transform: `translate3d(${item.left}px, ${item.top}px, 0)`
  }"
>
  <slot :item="item" />
</div>
```
##### 第六步：监听滚动更新 scrollTop
为了避免滚动时频繁计算，可以用 `requestAnimationFrame`：
```javascript
let ticking = false
const onScroll = () => {
  if (ticking) return
  ticking = true
  requestAnimationFrame(() => {
    scrollTop.value = window.scrollY
    ticking = false
  })
}
```
可以理解成：**滚动事件可以频繁触发，但真正更新 `scrollTop` 的操作，一帧最多只执行一次**
浏览器页面刷新通常是一帧一帧画出来的，比如常见屏幕大约是每秒 60 帧
也就是说，大概：
```
每 16.7ms 绘制一次页面
```
`requestAnimationFrame` 的特点是：**它会在浏览器下一次重绘前执行回调**

#### 你可以怎么把这个包装进面试？

>当前项目采用的是分页加载、瀑布流布局和图片懒加载，已经可以解决大多数图片流页面的性能问题。但它不是虚拟列表，DOM 会随着分页不断增长。

>如果面向更大规模数据，我会进一步升级为虚拟瀑布流。核心思路是提前维护每个 item 的布局信息，包括 top、bottom、left、height，然后根据 scrollTop 和 viewportHeight 计算当前可视区及 buffer 区域，只渲染这部分 item。外层容器仍然用完整的瀑布流总高度撑开滚动条，屏幕外节点则不渲染或复用，从而把 DOM 数量控制在稳定范围内。

#### 面试官追问：
##### 虚拟瀑布流怎么计算可见区域？

> 普通虚拟列表可以通过固定 itemHeight 直接算 startIndex 和 endIndex，但瀑布流 item 高度不固定，不能简单用 `scrollTop / itemHeight`。
> 
> 所以我会在布局阶段为每个 item 生成 layout 信息，例如 top、bottom、left、height。滚动时根据 `scrollTop`、`viewportHeight` 和一个 buffer 计算可见范围，然后判断 item 是否与这个范围相交：`item.bottom >= visibleTop && item.top <= visibleBottom`。满足条件的 item 才进入 visibleItems 渲染。
##### 虚拟瀑布流怎么保证滚动条高度正常？

> 虽然实际 DOM 只渲染可视区附近的 item，但组件内部会根据所有数据计算完整瀑布流布局，并取各列最大高度作为 containerHeight。外层容器用这个高度撑开滚动区域，所以用户看到的滚动条高度仍然对应完整列表。可见 item 再通过 absolute 或 transform 定位到它真实应该出现的位置。
##### 虚拟瀑布流的难点是什么？

> 难点主要有三个
> 第一是高度预测。瀑布流每个 item 高度不固定，如果图片或文本加载后高度变化，会导致后续 item 的 top 全部变化，所以要尽量根据图片原始宽高提前计算高度，并固定标题和作者区高度。
> 
> 第二是布局更新。分页追加时不能每次全量重排，最好只计算新增 item 的 layout；但分类切换、搜索切换、列数变化时又需要全量重算。
> 
> 第三是滚动性能。滚动时不能频繁全量 filter，可以用 requestAnimationFrame 合并滚动更新，数据更大时可以按 top 排序后二分查找，或者按列维护布局数据。
##### 虚拟列表和懒加载有什么区别？

> 图片懒加载解决的是资源加载问题，也就是图片什么时候发网络请求。
> 
> 虚拟列表解决的是 DOM 渲染问题，也就是页面上同时存在多少 DOM 节点。
> 
> 两者优化对象不同，可以同时使用。对于图片瀑布流场景，比较完整的方案是分页加载控制数据量，虚拟列表控制 DOM 数量，图片懒加载控制网络资源请求。
##### 如果接口不提供宽高，怎么提前占位？
###### 前端图片预加载后再排版
如果接口不提供宽高，前端可以先创建 `Image` 对象加载图片：
```javascript
const img = new Image()
img.src = imageUrl
img.onload = () => {
  const width = img.naturalWidth
  const height = img.naturalHeight
}
```
拿到：
```javascript
img.naturalWidth
img.naturalHeight
```
然后再计算比例。
优点：高度准确             
缺点：必须等图片加载    首屏会变慢    图片多时网络压力大    可能出现白屏等待
所以这个方案适合数据量少，或者图片尺寸必须准确的场景，不适合大规模图片流首屏体验
###### 默认比例占位，图片加载后修正
如果接口不提供宽高，也不想等图片加载，可以先给一个默认比例
优点：首屏快 不会完全高度塌陷
缺点：默认比例不一定准确  图片加载完成后仍然可能重排   瀑布流位置可能需要二次修正
这个方案适合：对首屏速度要求高，但可以接受轻微布局调整

```
src/libs/transition-router-view/index.vue
```
### 三、router/index.js：根据终端选择不同路由

`router/index.js` 做了一件很重要的事：

```javascript
routes: isMobileTerminal.value ? mobileTerminalRoutes : pcTerminalRoutes
```
也就是根据是否移动端，选择移动端路由或 PC 端路由，这就是你简历里“多端路由架构”的依据之一

#### 1. 移动端路由
`mobile-routes.js` 中，首页是一级路由：
```javascript
{
  path: '/',
  name: 'home',
  component: () => import('@/views/main/index.vue')
}
```
详情页也是一级路由：
```javascript
{
  path: '/pins/:id',
  name: 'pins',
  component: () => import('@/views/pins/index.vue')
}
```
移动端还有登录、注册、个人中心、会员、支付结果、404 等页面
移动端的特点是：
> 每个页面基本都是一级页面，更适合做类 App 的全屏切换动画
#### 2. PC 端路由
PC 端路由中，首页外面包了一层 layout：
```javascript
{
  path: '/',
  name: 'main',

    {
      path: '',
      name: 'home',

    }
  ]
}
```
而 `/pins/:id`、`/profile`、`/member` 等是独立路由，PC 端和移动端不一样：

> PC 首页有 layout，包括 header、main、floating；移动端更偏独立页面切换

`views/layout/index.vue` 也能看到它由 header、main、floating 三块组成

### 四、store/modules/app.js：路由动画类型存在哪里

`app.js` 里维护了一个状态：routerType: 'none'
并提供 mutation：
```javascript
changeRouterType(state, newType) {
  state.routerType = newType
}
```
源码中 `routerType` 和 `changeRouterType` 都在 `app` 模块里
这个状态的含义是：
```
none：不执行路由动画
push：前进动画
back：后退动画
```
它不是 Vue Router 自带的，而是项目自己维护的“路由跳转方向”

### 五、store/getters.js：PC 端禁用路由动画
`getters.js` 中的 `routerType` 很关键：
```javascript
routerType: (state) => {
  if (!isMobileTerminal.value) {
    return 'none'
  }
  return state.app.routerType
}
```
也就是说：**PC 端：永远返回 none移动端：返回 app.routerType**
这说明：
> 类 App 路由切换动画主要是为移动端服务的，PC 端不走这种左右滑动动画

这个点面试里很适合讲，因为它体现了你对多端体验差异的处理

### 六、transition-router-view/index.vue：全局路由动画核心
这个文件是“类 App 路由切换”的核心
#### 1. template 结构
```vue
<router-view v-slot="{ Component }">
  <transition :name="transitionName">
    <keep-alive :include="virtualTaskStack">
      <component
        :is="Component"
        :class="{ 'fixed top-0 left-0 w-screen z-50': isAnimation }"
        :key="$route.fullPath"
      />
    </keep-alive>
  </transition>
</router-view>
```
源码里确实是 `router-view + transition + keep-alive + dynamic component` 这套组合
这一层做了三件事：
```
router-view：拿到当前路由组件
transition：给路由组件加动画
keep-alive：缓存任务栈里的页面
component：真正渲染当前页面
```
#### 2. props
```javascript
// 路由跳转的类型，对应 ROUTER_TYPE_ENUM
  routerType: {
    type: String,
    default: NONE,
    validator(val) {
      const result = ROUTER_TYPE_ENUM.includes(val)
      if (!result) {
        throw new Error(
          `你的 routerType 必须是 ${ROUTER_TYPE_ENUM.join('、')} 中的一个`
        )
      }
      return result
    }
  },
  // 首页的组件名称，对应任务栈中的第一个组件
  mainComponentName: {
    type: String,
    required: true
  }
```
`routerType` 支持三个值：none、push、back
源码里定义了：

```javascript
const NONE = 'none'
const PUSH = 'push'
const BACK = 'back'
const ROUTER_TYPE_ENUM = [NONE, PUSH, BACK]
```
这些枚举都在文件顶部，它的作用是：
> 决定当前路由应该使用哪一种动画

比如：
```
push：新页面从右进入，老页面向左退出
back：老页面向右退出，上一页从左回来
none：没有动画
```

`App.vue` 里传的是：home，也就是首页组件名

它的作用是：
> 作为任务栈的初始页面，也是回到首页时清空任务栈的依据。

源码里初始化任务栈：
```javascript
const virtualTaskStack = ref([props.mainComponentName])
```
也就是一开始任务栈是：['home']，这一行在源码中明确存在
### 七、virtualTaskStack：虚拟任务栈是什么
`virtualTaskStack` 是这个组件最值得讲的变量，它的作用有两个：
#### 1. 维护页面栈
当路由前进时：
```javascript
virtualTaskStack.value.push(to.name)
```
当路由后退时：
```javascript
virtualTaskStack.value.pop())
```
源码在 `router.beforeEach` 里判断 `routerType`，如果是 `PUSH` 就入栈，如果是 `BACK` 就出栈
所以它模拟了一个 App 的页面任务栈：
```
['home']
['home', 'pins']
['home', 'pins', 'profile']
```
后退时：
```
['home', 'pins', 'profile']
↓
['home', 'pins']
↓
['home']
```
#### 2. 作为 KeepAlive include
模板里：
```vue
<keep-alive :include="virtualTaskStack">
```
这表示：
> 只有在 `virtualTaskStack` 里的组件会被缓存

源码中确实把 `virtualTaskStack` 传给了 `keep-alive` 的 `include`，也就是说，任务栈不仅控制动画，还控制页面缓存

这是一个很适合面试讲的点：

> 路由栈和 KeepAlive 缓存联动，栈内页面保留，出栈页面释放

不过这里有一个重要注意点，我后面会讲风险

### 八、router.beforeEach：路由切换时如何更新动画和任务栈
```javascript
router.beforeEach((to, from) => {
  transitionName.value = props.routerType
  if (props.routerType === PUSH) {
    virtualTaskStack.value.push(to.name)
  } else if (props.routerType === BACK) {
    virtualTaskStack.value.pop()
  }
  if (to.name === props.mainComponentName) {
    clearTask()
  }
})
```
#### 1. 设置动画名
transitionName.value = props.routerType, 如果 routerType = 'push'`，那transition name就是：push
Vue 会自动匹配：
```CSS
.push-enter-active
.push-leave-active
```
如果 `routerType = 'back'`，就匹配：
```CSS
.back-enter-active
.back-leave-active
```
如果是 `none`，则没有动画
#### 2. 维护任务栈
```javascript
if (props.routerType === PUSH) {
  virtualTaskStack.value.push(to.name)
} else if (props.routerType === BACK) {
  virtualTaskStack.value.pop()
}
```
这就是前进入栈、后退出栈
#### 3. 回到首页清空栈
```javascript
if (to.name === props.mainComponentName) {
  clearTask()
}
const clearTask = () => {
  virtualTaskStack.value = [props.mainComponentName]
}
```
也就是说，只要进入首页，任务栈重置为：['home']
### 九、isAnimation：动画期间为什么加 fixed
组件模板里：
```vue
:class="{ 'fixed top-0 left-0 w-screen z-50': isAnimation }"
```
也就是动画期间给页面加：
```css
fixed top-0 left-0 w-screen z-50
```

`isAnimation` 的控制逻辑是：
```JavaScript
const beforeEnter = () => {  isAnimation.value = true}
const afterLeave = () => {  isAnimation.value = false}
```
源码中 `beforeEnter` 和 `afterLeave` 就是这样设置动画状态
为什么动画期间要加 fixed？因为路由页面切换时，两个页面会同时存在，如果它们都在普通文档流里，可能互相挤占布局。
加上 fixed 后：
```
页面脱离文档流
覆盖在视口上
可以做横向滑动
不会撑开或挤压布局
```
这也是类 App 页面滑动常见处理
### 十、首页图片详情动画：item.vue 负责提供动画起点
`item.vue` 是单个瀑布流卡片组件，它不只是展示图片，还负责记录点击图片的位置
```vue
<img
  v-lazy
  ref="imgTarget"
  :src="data.photo"
  :style="{
    height: (width / data.photoWidth) * data.photoHeight + 'px'
  }"
/>
```
这里 `ref="imgTarget"` 指向图片 DOM
#### 1. 获取图片 DOM 位置
```JavaScript
const imgTarget = ref(null)
const {
  x: imgContainerX,
  y: imgContainerY,
  width: imgContainerWidth,
  height: imgContainerHeight
} = useElementBounding(imgTarget)
```
`useElementBounding` 是 **VueUse** 库提供的一个组合式函数，用于**响应式获取 DOM 元素的边界信息**
#### 2. 计算图片中心点
```JavaScript
const imgContainerCenter = computed(() => {
  return {
    translateX: parseInt(imgContainerX.value + imgContainerWidth.value / 2),
    translateY: parseInt(imgContainerY.value + imgContainerHeight.value / 2)
  }
})
```
这个中心点会作为详情展开动画的起始位置
#### 3. 点击时向父组件传递 id 和 location
```JavaScript
const onToPinsClick = () => {
  emits('click', {
    id: props.data.id,
    location: imgContainerCenter.value
  })
}
```
源码中点击图片后 emit 的就是 `id + location`
所以 `item.vue` 在动画链路里的职责是：
> 点击卡片时，把当前图片 id 和图片中心点坐标传给父组件

### 十一、list/index.vue：详情弹层、History API、GSAP 都在这里串起来
`list/index.vue` 里有两块模板：
第一块是瀑布流：
```vue
<itemVue :data="item" :width="width" @click="onToPins" />
```
父组件监听 `item.vue` 的 click
第二块是详情动画：
```vue
<transition
  :css="false"
  @before-enter="beforeEnter"
  @enter="enter"
  @leave="leave"
>
  <pins-vue v-if="isVisiblePins" :id="currentPins.id" />
</transition>
```
这里使用 Vue transition 的 JS hooks，并且 `css=false`，说明动画不走 CSS，而是由 JS/GSAP 接管
 `:css="false"` 的含义

|属性|作用|
|:--|:--|
|`:css="false"`|**禁用 CSS 过渡检测**，Vue 不会自动嗅探 `transitionend`/`animationend` 事件|

**默认行为**：Vue 会等 CSS 动画结束后自动清理 DOM  
**禁用后**：Vue 不知道动画何时结束，必须由你手动告诉它
`:css="false"` 把动画控制权完全交给 JS，配合 `beforeEnter/enter/leave` 钩子 + `done()` 回调，实现 CSS 做不到的高精度动画控制

|概念|一句话|
|:--|:--|
|CSS Animation|`@keyframes` 定义的多阶段动画|
|CSS Transition|两个状态之间的平滑渐变|
|`transitionend`|浏览器通知"CSS 过渡做完了"|
|`animationend`|浏览器通知"CSS 动画做完了"|
|Vue 默认|自动监听这两个事件，自动管理 DOM|
|`:css="false"`|禁用自动监听，你自己喊 `done()`|

#### 1. 状态变量
```JavaScript
const isVisiblePins = ref(false)
const currentPins = ref({})
```
`isVisiblePins` 控制详情弹层是否显示，`currentPins` 保存当前点击的图片 id 和 location
#### 2. 点击图片后执行 onToPins
```JavaScript
const onToPins = (item) => {
  history.pushState(null, null, `/pins/${item.id}`)
  currentPins.value = item
  console.log('onToPins', item)
  isVisiblePins.value = true
}
```
源码中 `onToPins` 做了三件事：
##### 第一，写入浏览器历史栈
history.pushState(null, null, `/pins/${item.id}`)，这会把地址栏改成：/pins/xxx，但它没有调用 `router.push`，所以它的特点是：

```
URL 变了
首页组件没有销毁
详情组件作为当前页面内的弹层显示
```
##### 第二，保存当前图片信息
currentPins.value = item，`item` 中包含：
```JavaScript
{  id,  
	location: {    
		translateX,    
		translateY  
	}
}
```
##### 第三，显示详情组件
isVisiblePins.value = true
这会让：
```vue
<pins-vue v-if="isVisiblePins" :id="currentPins.id" />
```
挂载，触发 transition 的 `beforeEnter` 和 `enter`
### 十二、GSAP 进入动画
#### 1. beforeEnter
```JavaScript
const beforeEnter = (el) => {
  gsap.set(el, {
    scaleX: 0,
    scaleY: 0,
    transformOrigin: '0 0',
    x: currentPins.value.location?.translateX || 0,
    y: currentPins.value.location?.translateY || 0,
    opacity: 0
  })
}
```
源码中进入前设置了初始状态，含义：
```
详情层一开始缩放为 0
位置在被点击图片的中心点
透明度为 0
```
这就是“从图片点击位置展开”的基础
#### 2. enter
```JavaScript
const enter = (el, done) => {
  gsap.to(el, {
    duration: 0.3,
    scaleX: 1,
    scaleY: 1,
    opacity: 1,
    x: 0,
    y: 0,
    onComplete: done
  })
}
```
源码中进入动画让详情层从缩放 0、透明，变成正常大小和正常位置
重点是：
```JavaScript
onComplete: done
```
因为 `transition` 设置了：
```vue
:css="false"
```
Vue 不知道 GSAP 动画何时结束，所以必须在动画完成后调用 `done`

### 十三、GSAP 离开动画
```JavaScript
const leave = (el, done) => {
  gsap.to(el, {
    duration: 0.3,
    scaleX: 0,
    scaleY: 0,
    x: currentPins.value.location?.translateX || 0,
    y: currentPins.value.location?.translateY || 0,
    opacity: 0,
    onComplete: done
  })
}
```
源码中离开动画就是反向过程
含义：
```
详情层从全屏状态缩小并移动回原图片中心点最后透明并卸载
```
这就形成了：
```
点击图片 → 从图片处展开返回关闭 → 收回到图片处
```
### 十四、浏览器返回：popstate
`list/index.vue` 里监听了：
```JavaScript
useEventListener(window, 'popstate', () => {  isVisiblePins.value = false})
```
源码中是这样处理浏览器后退的
完整流程是：

```plaintext
点击图片
    ↓
history.pushState('/pins/id')
    ↓
详情层打开

点击浏览器返回
    ↓
浏览器触发 popstate
    ↓
isVisiblePins = false
    ↓
触发 leave 动画
    ↓
详情层关闭
```

这就是“URL 和弹层状态同步”的关键

**修改浏览器历史记录，不触发页面刷新**

|方法|作用|
|:--|:--|
|`pushState(state, title, url)`|**添加**一条新历史记录|
|`replaceState(state, title, url)`|**替换**当前历史记录|
对比
```plaintext
初始:  /home
pushState(null, '', '/detail')
  → 历史栈: [/home, /detail]
  → 地址栏显示 /detail
  → 页面不刷新
replaceState(null, '', '/detail')
  → 历史栈: [/detail]  （home 被替换掉）
  → 地址栏显示 /detail
  → 页面不刷新
history.back()
  → 从 /detail 回到 /home
  → 触发 popstate 事件
```
 **导航方法**

|方法|作用|
|:--|:--|
|`history.back()`|后退一页（等价 `history.go(-1)`）|
|`history.forward()`|前进一页（等价 `history.go(1)`）|
|`history.go(-2)`|后退两页|
|`history.go(0)`|刷新当前页|
 **与 popstate 的关系**

```plaintext
用户操作:           历史栈变化:            是否触发 popstate:
─────────────────────────────────────────────────────────────
pushState()         添加新记录              ❌ 不触发
replaceState()      替换当前记录            ❌ 不触发
点击链接跳转        添加新记录（刷新页面）    ❌ 页面已刷新
history.back()      后退                    ✅ 触发
history.forward()   前进                    ✅ 触发
点击浏览器后退按钮    后退                    ✅ 触发
点击浏览器前进按钮    前进                    ✅ 触发
```

**为什么isVisiblePins的改变会触发过渡动画呢**

```plain
isVisiblePins.value = true
  ↓
v-if 条件变为真
  ↓
<pins-vue> 被**插入** DOM
  ↓
<transition> 检测到"新元素进入"
  ↓
触发 @before-enter → @enter（你执行动画，调 done()）
  ↓
元素显示完成

────────────────────────────

isVisiblePins.value = false
  ↓
v-if 条件变为假
  ↓
<pins-vue> 即将被**移除** DOM
  ↓
<transition> 检测到"元素离开"
  ↓
触发 @leave（你执行动画，调 done()）
  ↓
Vue 真正移除 DOM
```
 
 ==关键点==

|机制|说明|
|:--|:--|
|**不是监听变量**|Vue 不监听 `isVisiblePins`，监听的是 **DOM 的插入/移除**|
|`v-if` 控制挂载|`true` 插入节点，`false` 移除节点|
|`<transition>` 拦截|在真正插入/移除前，先执行你的钩子|
|`done()` 控制时机|你决定何时让 Vue 完成最终的 DOM 操作|
==对比其他指令==

|指令|是否触发过渡|
|:--|:--|
|`v-if`|✅ 触发（插入/移除）|
|`v-show`|❌ 不触发（只是 `display` 变化，DOM 一直在）|
|动态组件 `:is`|✅ 触发（组件切换 = 旧移除 + 新插入）|
|`v-for` 列表变化|✅ 触发（配合 `<transition-group>`）|
`isVisiblePins` → 控制 `v-if` → 控制 DOM 插入/移除 → `<transition>` 拦截这个 DOM 变化 → 触发你的钩子。  
**过渡的本质是 DOM 生命周期钩子，不是响应式监听**

### 十五、pins/components/pins.vue：详情组件内部如何关闭

 关闭逻辑 `onPop`
```JavaScript
const onPop = () => {  
	store.commit('app/changeRouterType', 'back')  
	router.back()
}
```
源码里关闭时先把 `routerType` 改成 `back`，再调用 `router.back()`
这点很关键：
```
详情页点击关闭
    ↓
设置 routerType = back
    ↓
router.back()
    ↓
浏览器历史后退
    ↓
list/index.vue 监听 popstate
    ↓
关闭详情层
```
所以关闭按钮和浏览器返回最终走的是同一套“历史后退”逻辑

###  十六、完整流程：从点击图片到关闭详情

#### 1. 用户点击瀑布流图片

`item.vue` 图片外层绑定：
```javascript
@click="onToPinsClick"
```
点击时执行：
```JavaScript
emits('click', {  id: props.data.id,  location: imgContainerCenter.value})
```
`location` 来自 `useElementBounding` 计算出的图片中心点
#### 2. 父组件接收点击
`list/index.vue` 中：
```vue
<itemVue :data="item" :width="width" @click="onToPins" />
```
然后执行：
```JavaScript
history.pushState(null, null, `/pins/${item.id}`)
currentPins.value = item
isVisiblePins.value = true
```
#### 3. 详情组件挂载
`isVisiblePins = true` 后：
```vue
<pins-vue v-if="isVisiblePins" :id="currentPins.id" />
```
开始挂载，并触发 `beforeEnter`
#### 4. GSAP 设置初始状态
`beforeEnter` 中：
```
scaleX = 0
scaleY = 0
x = 图片中心点 x
y = 图片中心点 y
opacity = 0
```
源码对应 `gsap.set`
#### 5. GSAP 展开详情
`enter` 中：
```
scaleX → 1
scaleY → 1
x → 0
y → 0
opacity → 1
```
源码对应 `gsap.to`
#### 6. 详情组件请求数据
`pins.vue` 根据 `props.id` 调用：
```JavaScript
getPexelsFromId(props.id)
```
然后把结果放到 `pexelData`
#### 7. 用户关闭详情
关闭方式有两种：
```
浏览器返回
详情页关闭按钮 / navbar 返回
```
如果是关闭按钮：
```JavaScript
store.commit('app/changeRouterType', 'back')
router.back()
```
如果是浏览器返回，则直接触发 `popstate`
#### 8. popstate 关闭弹层
`list/index.vue` 监听：
```JavaScript
useEventListener(window, 'popstate', () => {  isVisiblePins.value = false})
```
然后触发 `leave` 动画
#### 9. GSAP 收起详情
`leave` 中：
```
scaleX → 0
scaleY → 0
x → 原图片中心点
y → 原图片中心点
opacity → 0
```
源码中离开动画确实回到 `currentPins.location`
