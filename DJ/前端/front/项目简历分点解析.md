
# 瀑布流 + 长列表渲染优化

### 1. **业务列表组件**

> src/views/main/components/list/index.vue

- 请求图片列表数据
- 维护分页参数 `page / size`
- 判断是否加载完成
- 监听分类和搜索词变化
- 把数据传给瀑布流组件
- 外层包一层无限滚动组件 `m-infinite-list`

==作用==：源码里可以看到，业务列表通过 `getPexelsList(query)` 请求数据，初始页覆盖列表，后续页通过 `push(...res.list)` 追加数据；同时用 `isFinished` 判断是否已经加载完。首页里移动端使用 2 列，PC 端使用 5 列，并且传入了 `picturePreReading=false`

### 2. **瀑布流布局组件**

> src/libs/waterfall/index.vue

- 计算容器宽度
- 根据列数计算每列宽度
- 维护每一列的高度
- 按“最短列优先”分配每个 item 的位置
- 给每个 item 动态生成 `left / top`
- 设置容器总高度

==作用==：你的瀑布流组件外层是 `relative`，每个 item 是 `absolute`，通过动态样式控制 `width / left / top`，这就是典型的 JS 计算型瀑布流布局

### 3. **单个图片卡片组件**

> src/views/main/components/list/item.vue

- 渲染图片
- 根据接口返回的图片原始宽高计算展示高度
- 使用 `v-lazy` 懒加载图片
- 渲染标题、作者、头像
- 提供下载、分享、全屏、点击详情等交互
```css 
:style="{
	height: (width / data.photoWidth) * data.photoHeight + 'px'
}"
```

==作用==：这说明你的瀑布流不是等图片真实加载完成后才知道高度，而是根据图片原始尺寸比例提前计算高度，这也是为什么首页里把 `picturePreReading` 设置成了 `false`

## 瀑布流的核心原理

>**JS 计算列宽 + 维护列高数组 + 最短列优先插入 + absolute 定位渲染**


```JavaScript
{  0: 0,  1: 0,  2: 0,  3: 0,  4: 0}
```
假设 PC 端是 5 列，初始每列高度如上，每来一个图片卡片，组件会：
1. 找到当前高度最小的列
2. 把这个 item 放到这列
3. 它的 `left` 等于：列索引 × `(列宽 + 列间距)`
4. 它的 `top` 等于：当前最短列的高度
5. 该列高度增加：`itemHeight + rowSpacing`

## 面试时你应该怎么讲

>首页图片流采用了瀑布流布局，主要是为了解决图片宽高比例不一致时，普通网格布局会产生大量空白、内容密度低的问题。

>我把它拆成了业务列表层和底层布局层。业务列表层只负责分页请求、分类搜索条件变化和数据状态维护；瀑布流组件只负责布局计算，包括容器宽度、列宽、每列高度以及 item 的 `left/top` 定位。

>布局算法采用“最短列优先”：每次渲染新 item 时，先找到当前高度最小的一列，把 item 放进去，然后更新该列高度，最后用最高列高度作为容器高度。

>同时，图片卡片会根据接口返回的 `photoWidth` 和 `photoHeight` 提前计算展示高度，避免等待图片真实加载后再回流布局；再结合自定义 `v-lazy` 指令，在图片进入可视区域后再加载真实图片，从而降低首屏请求压力和长列表滚动时的渲染压力。
